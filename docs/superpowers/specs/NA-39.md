# Wire SEO meta, JSON-LD, and llms.txt — Technical Spec

**Story:** [NA-39](https://whimzylive.atlassian.net/browse/NA-39)
**Epic:** [NA-29 — Nightshift marketing site — design-handoff build](https://whimzylive.atlassian.net/browse/NA-29)
**Feature:** docs/features/2026-07-12-nightshift-marketing-site-design-handoff.md
**Date:** 2026-07-12

## Overview

Wire the search/AI-discovery layer into the existing `apps/marketing` Next.js app: per-page meta/OG/Twitter tags, `@graph` JSON-LD (SoftwareApplication + Organization + FAQPage + HowTo on Home; WebPage + BreadcrumbList + FAQPage on Why SDLC; FAQPage on the full FAQ page), and a root `llms.txt`. Every value is sourced from `docs/gtm/site-brief.md` — no new SEO copy is authored — and each FAQPage answer is derived from the same Payload CMS Lexical value the page renders, so schema text is text-identical to visible copy by construction.

## Scope note — non-visual story

This story ships no rendered UI (no components, no layout, no styling). Its entire surface is document `<head>` metadata, inert `<script type="application/ld+json">` tags, and two static text files. The only brand-token surface — the `og-image.png` / `og-why-sdlc.png` assets — is explicitly **out of scope** (placeholder paths per the brief; asset generation is a separate concern). The `nightshift-design` brand/adherence gate therefore does not apply to this story.

## Data Model

**No data-model changes.** No new Payload collections, globals, or fields. The `faq` collection and `whySdlc` global (NA-31) are the source of the dynamic FAQPage answers and are consumed read-only through the existing `lib/faq.ts` and `lib/why-sdlc.ts` data functions. No migration is generated.

## API Surface

**No new REST/API endpoints.** No new Payload endpoints, route handlers behind auth, or mutations. The story adds document-level metadata to existing rendered routes and two static, publicly served text assets.

### Request / Response shapes

No request bodies — these are GET-only document/asset responses. The load-bearing TypeScript is the SEO module's internal shapes and the JSON-LD node types. All fully typed; no `any`.

```typescript
// lib/seo/config.ts — single source for absolute URLs (canonical host = GitHub repo per brief §3/§4)
export interface SeoSite {
  /** Canonical site origin/base. Brief §4: https://github.com/whimzyLive/nightshift-ai */
  readonly siteUrl: string;
  readonly siteName: string; // "nightshift"
  readonly ogImageHome: string; // absolute raw.githubusercontent.com URL (brief §4)
  readonly ogImageWhySdlc: string; // absolute raw.githubusercontent.com URL (brief §4)
}

// lib/seo/jsonld.ts — narrow, fully-typed schema.org node interfaces (no schema-dts dependency required)
export interface JsonLdOffer {
  '@type': 'Offer';
  price: string;
  priceCurrency: string;
}

export interface SoftwareApplicationNode {
  '@type': 'SoftwareApplication';
  '@id': string;
  name: string;
  applicationCategory: string;
  applicationSubCategory: string;
  operatingSystem: string;
  description: string;
  url: string;
  downloadUrl: string;
  license: string;
  isAccessibleForFree: boolean;
  offers: JsonLdOffer;
  featureList: readonly string[];
  author: { '@id': string };
  publisher: { '@id': string };
}

export interface OrganizationNode {
  '@type': 'Organization';
  '@id': string;
  name: string;
  url: string;
  sameAs: readonly string[];
}

export interface FaqQuestionNode {
  '@type': 'Question';
  name: string;
  acceptedAnswer: { '@type': 'Answer'; text: string };
}

export interface FaqPageNode {
  '@type': 'FAQPage';
  '@id': string;
  mainEntity: FaqQuestionNode[];
}

export interface HowToStepNode {
  '@type': 'HowToStep';
  position: number;
  name: string;
  text: string;
}

export interface HowToNode {
  '@type': 'HowTo';
  '@id': string;
  name: string;
  description: string;
  step: readonly HowToStepNode[];
}

export interface WebPageNode {
  '@type': 'WebPage';
  '@id': string;
  name: string;
  description: string;
  isPartOf: { '@id': string };
  about: { '@id': string };
}

export interface BreadcrumbItemNode {
  '@type': 'ListItem';
  position: number;
  name: string;
  item: string;
}

export interface BreadcrumbListNode {
  '@type': 'BreadcrumbList';
  '@id': string;
  itemListElement: readonly BreadcrumbItemNode[];
}

export type JsonLdNode = SoftwareApplicationNode | OrganizationNode | FaqPageNode | HowToNode | WebPageNode | BreadcrumbListNode;

export interface JsonLdGraph {
  '@context': 'https://schema.org';
  '@graph': JsonLdNode[];
}

// The FAQ input the FAQPage builder consumes — a { question, answer } pair whose
// `answer` is the SAME Lexical value the page renders (identity guarantee).
export interface FaqSchemaInput {
  question: string;
  answer: SerializedEditorState; // from '@payloadcms/richtext-lexical/lexical' (already used across the app)
}

// Builds the FAQPage node from rendered FAQ data. Each answer.text is produced by
// convertLexicalToPlaintext on the same Lexical value, guaranteeing text-identity (AC4).
export function buildFaqPageNode(id: string, faqs: FaqSchemaInput[]): FaqPageNode;
```

## Backend Implementation

There is no separate backend tier — this is a Next.js App Router app. "Backend" here means the server-rendered SEO module and the static assets.

**SEO module (new): `apps/marketing/src/lib/seo/`**

| File               | Responsibility (WHAT)                                                                                                                                                                     |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `config.ts`        | Frozen `SeoSite` constants: canonical `siteUrl`, `siteName`, absolute OG image URLs. All values verbatim from site-brief §3/§4.                                                           |
| `metadata.ts`      | Per-page Next `Metadata` builders (Home, Why SDLC) returning the exact title/description/canonical/OG/Twitter values from brief §4.                                                       |
| `jsonld.ts`        | The node interfaces above + static node constants (SoftwareApplication, Organization, HowTo, Why-SDLC WebPage, BreadcrumbList) verbatim from brief §3, plus `buildFaqPageNode(id, faqs)`. |
| `faq-plaintext.ts` | Wraps `convertLexicalToPlaintext` (from `@payloadcms/richtext-lexical/plaintext`, verified present at v3.85.2) to turn one answer's Lexical value into the JSON-LD `answer.text`.         |
| `json-ld.tsx`      | Tiny presentational `<JsonLd graph={...} />` server component that renders one `<script type="application/ld+json">` with the serialized graph.                                           |

**Static assets (new):**

- `apps/marketing/public/llms.txt` — verbatim content from site-brief §5 (the fenced `# nightshift …` block). Served at `/llms.txt` as `text/plain` by Next's static handler.
- `apps/marketing/public/robots.txt` — allows `GPTBot`, `ClaudeBot`, `PerplexityBot`, `Google-Extended` (and a default `User-agent: *` Allow), per the site-brief §5 build note.

**Layout base (modify): `apps/marketing/src/app/(frontend)/layout.tsx`**

- Add `metadataBase: new URL(siteUrl)` and default `openGraph.siteName` / `twitter.card` to the existing `metadata` export so per-page relative canonical/OG values resolve to absolute URLs (AC5). Keep the existing default title/description as the fallback for routes that do not override (e.g. `/team`).

**Text-identity mechanism (AC4 — the load-bearing decision):** the FAQPage `answer.text` is produced by running `convertLexicalToPlaintext` on the **same** Lexical `answer` value the page passes to `<RichText>`. The page already resolves the correct per-surface field before rendering (`homeAnswer ?? answer` on Home via `getHomeFaqs`; `whySdlcAnswer ?? answer` on Why SDLC via `getWhySdlcFaqs`; `answer` on `/faq` via `getFaqPageGroups`). The JSON-LD builder consumes that identical resolved value, so visible copy and schema text can never drift. Do **not** re-fetch or re-source answer text from the site brief for the FAQPage nodes.

## Web UI

No visual components, pages, routes, hooks, or client state are added. The changes are confined to `<head>` metadata and inert JSON-LD scripts on three existing routes plus two static files.

**Route wiring (modify existing routes):**

| Route       | File                                   | Change                                                                                                                                                                                |
| ----------- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/`         | `src/app/(frontend)/page.tsx`          | Export `metadata` (Home builder). In the existing async body, build `@graph` (static nodes + `buildFaqPageNode` from the already-fetched `faqs`) and render `<JsonLd graph={...} />`. |
| `/why-sdlc` | `src/app/(frontend)/why-sdlc/page.tsx` | Export `metadata` (Why-SDLC builder). Build `@graph` (WebPage + BreadcrumbList + `buildFaqPageNode` from the already-fetched `faqs`) and render `<JsonLd .../>`.                      |
| `/faq`      | `src/app/(frontend)/faq/page.tsx`      | Keep existing NA-38 `metadata`. Build a single-node `@graph` (`buildFaqPageNode` from the flattened `getFaqPageGroups` items) and render `<JsonLd .../>`.                             |

- All three routes are already `export const dynamic = 'force-dynamic'` async Server Components that fetch their FAQ data at the top-level boundary — the JSON-LD is built from that already-resolved data, adding no extra Payload fetch and preserving the single-async-boundary rule (NA-16 memory).
- `metadata` is a static `const` export and coexists with an async default export and `dynamic = 'force-dynamic'` — no `generateMetadata` needed since none of the meta values depend on request/CMS data (all meta copy is static from the brief).
- Rendering an empty JSON-LD graph is avoided: when a route's `faqs` array is empty (data-fetch fallback), omit the FAQPage node rather than emit an empty `mainEntity` (see Error Handling).

**Route permissions:** all three routes and both static files are public (no route-permission config exists or is needed).

## Error Handling

| Scenario                                                | Behaviour                                                                                                                                                                                  | Status |
| ------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------ |
| Payload/DB fetch fails for a route's FAQs               | Existing `lib` functions already return `[]`/`null`. JSON-LD omits the FAQPage node (or its `mainEntity` is skipped); the static nodes still render; the page still renders. Non-blocking. | 200    |
| A route has zero surfaced FAQs                          | Do not emit a FAQPage node with an empty `mainEntity` array — drop the node from the `@graph`.                                                                                             | 200    |
| `convertLexicalToPlaintext` returns empty for an answer | Emit the question node with an empty `text` only if the visible answer is genuinely empty; otherwise the identity holds. Never throw — a converter failure must not break the page.        | 200    |
| `/llms.txt` or `/robots.txt` requested                  | Served verbatim as `text/plain` by Next static handling.                                                                                                                                   | 200    |
| Unknown path                                            | Existing Next 404 behaviour — unchanged.                                                                                                                                                   | 404    |

**Validation rules (AC5):** every `url`/`@id`/`item` in every emitted node MUST be absolute/fully-qualified. Static nodes carry absolute URLs verbatim from the brief; `metadataBase` makes page canonical/OG URLs absolute. The FAQPage `@id` values are the absolute anchors from brief §3 (`…#faq`). A Rich Results spot-check (FAQPage + SoftwareApplication) is a manual verification step, not automated in this story.

## Out of Scope

- **Authoring any new SEO copy** — every title/description/OG/JSON-LD/llms.txt value is verbatim from `docs/gtm/site-brief.md`.
- **Generating the `og-image.png` / `og-why-sdlc.png` assets** — placeholder `raw.githubusercontent.com` paths per the brief; asset generation (brand-token render) is a separate concern.
- **Team page (`/team`) meta/OG/JSON-LD** — the brief supplies meta copy only for Home and Why SDLC; `/team` keeps whatever metadata it already has.
- **Enhancing `/faq` meta/OG copy** — `/faq` only gains the FAQPage JSON-LD (AC4); its existing NA-38 `metadata` is left unchanged (no brief copy exists to author from).
- **`sitemap.xml`** — not in the acceptance criteria.
- **Adding a `WebSite` node or host-swapping to a dedicated domain** — the brief keeps the GitHub repo as canonical; the host-swap is a documented future step, not this story.
- **Any Payload schema / collection / migration change.**

## Open Questions

- [ ] **llms.txt / robots.txt delivery — static file vs route handler?** — Suggested default: static files in `apps/marketing/public/` (`llms.txt`, `robots.txt`). Content is fully static, `public/` serves at site root ahead of the `(payload)` catch-all admin route, and Next serves `.txt` as `text/plain`. Use a `route.ts` handler only if a future requirement makes the content dynamic.
- [ ] **JSON-LD typing — hand-rolled interfaces vs `schema-dts`?** — Suggested default: the narrow local interfaces defined above. They fully type every emitted field with no `any` and add no dependency; adopt `schema-dts` later only if node coverage grows.
- [ ] **`metadata` on Home via `const` export vs `generateMetadata`?** — Suggested default: static `const metadata`. All Home/Why-SDLC meta values are static brief copy; no request/CMS dependency justifies `generateMetadata`.
- [ ] **OG image absolute-URL host** — Suggested default: `https://raw.githubusercontent.com/whimzyLive/nightshift-ai/main/og-image.png` (and `og-why-sdlc.png`) exactly as the brief specifies (raw bytes, not the GitHub HTML wrapper); host-swap when the site gets its own domain.
