# Create Marketing Website — Technical Spec

**Story:** [NA-22](https://whimzylive.atlassian.net/browse/NA-22) — Create Marketing Website
**Feature / scope source:** `docs/gtm/site-brief.md` (the story AC references this brief as the page/IA/copy scope)
**Date:** 2026-07-10

## Overview

Build the nightshift marketing website in `apps/marketing` (Next.js App Router + embedded Payload CMS 3) so every page renders from Payload as the CMS-first content source, with zero hardcoded copy in page components. Extract the nightshift design tokens into a new `packages/design-system` and the shared UI primitives into `packages/ui`, and theme the Payload admin to match the design system.

This story is **web-scoped only** — the sole active agent is **web-engineer** (owns `apps/marketing/`, `apps/marketing-e2e/`, `packages/ui/`, `brand/`, and the new `packages/design-system/`). No `platform-engineer` (`tools/`), `ai-enablement-engineer` (`plugins/`, `skills/`, `.agents/`), mobile, or offline-sync work is in scope. The "Data Model" and "API Surface" sections below describe Payload content types and Payload's auto-generated API — there is no bespoke backend service.

## Data Model

The marketing app persists content through **Payload CMS 3** on **Neon Postgres** via `@payloadcms/db-postgres` (`apps/marketing/src/payload.config.ts`). There are **no hand-written ORM entities**; content types are Payload `GlobalConfig` / `CollectionConfig` objects, and the Postgres schema + `payload-types.ts` are generated (`payload generate:types`) with schema changes applied by Payload migrations (`payload migrate:create` → `payload migrate`). "Base class" per this project's convention is therefore **Payload's config type** — `GlobalConfig` for singleton, single-edit-form pages (chosen for the two bespoke landing pages, which have fixed, section-specific shapes rather than a repeatable list) and `CollectionConfig` for repeatable records (Media, Users — already present).

### New entities (Payload types)

All content lives in **three new Globals** plus the existing `Media` collection. Globals are used because the site is a fixed set of bespoke pages (Home long-scroll + `/why-sdlc`), each edited as one form — not a list of like records.

Shared chrome + the single install/repo CTA values, so nav/footer/CTA copy is CMS-sourced (not hardcoded).

### Modified entities

| Entity              | Change                                                                                                  |
| ------------------- | ------------------------------------------------------------------------------------------------------- |
| `Media` collection  | Add `read: () => true` already present — no change required; used for any hero/section imagery uploads. |
| `Users` collection  | No field change. `useAPIKey: true` already present — used for programmatic (agent) content writes.      |
| `payload.config.ts` | Register the three new globals in `globals: [...]`; keep `collections: [Media, Users]`.                 |

Image fields (`relationship`/`upload` to `media`) are added to a section group **only if** the built section renders imagery. Default for this pass: hero/sections are type-and-terminal driven (no photographic imagery per brand kit), so no `upload` fields are added unless a section design requires one — see Open Questions.

### Relationships

- `home` (1) → (1) `siteSettings` — logical reference at render time; the frontend reads both globals. No FK.
- `whySdlc` (1) → (1) `siteSettings` — same.
- `Media` (1) → (many) content globals via optional `upload` fields — only if imagery is introduced (none required this pass).
- Globals are singletons (cardinality 1 per site) by Payload definition — no per-row relationships.

## API Surface

No custom route handlers are authored. Payload auto-generates REST + GraphQL + Local API. The frontend (`(frontend)` route group, React Server Components) reads content via the **Payload Local API** (`getPayload({ config })` → `payload.findGlobal({ slug })`) — in-process, no network hop. The generated REST endpoints exist for the admin UI and programmatic agent writes.

### Payload-generated endpoints (relevant subset)

| Method | Path                           | Auth/Route Type                          | Description                                          |
| ------ | ------------------------------ | ---------------------------------------- | ---------------------------------------------------- |
| GET    | `/api/globals/site-settings`   | public read (access `read: () => true`)  | Read shared chrome/CTA content                       |
| GET    | `/api/globals/home`            | public read                              | Read Home page content                               |
| GET    | `/api/globals/why-sdlc`        | public read                              | Read `/why-sdlc` page content                        |
| POST   | `/api/globals/{slug}`          | authenticated (admin session or API key) | Update a global (admin UI / agent write)             |
| GET    | `/api/media`, `/api/media/:id` | public read                              | Read uploaded media                                  |
| POST   | `/api/media`                   | authenticated (admin session or API key) | Upload media                                         |
| ALL    | `/api/[...slug]`               | Payload-managed                          | Existing catch-all (`(payload)/api/...`) — unchanged |

Frontend rendering uses the Local API, not the HTTP endpoints above — but the access-control rules below govern both paths identically.

### Request / Response shapes

Payload generates these into `apps/marketing/src/payload-types.ts` (`payload generate:types`). The frontend consumes the generated types — the interfaces below are the intended shape (fully typed, no `any`):

```typescript
// Read model consumed by the frontend (subset — mirrors generated payload-types.ts)
interface NavLink {
  label: string;
  href: string;
}

interface SiteSettings {
  installCommand: string;
  githubUrl: string;
  navLinks: NavLink[];
  footerColumns: Array<{ heading: string; links: NavLink[] }>;
}

interface HomeGlobal {
  hero: { headline: string; subheadline: string; installCtaLabel: string; starCtaLabel: string };
  proofBar: { items: Array<{ value: string; label: string }> };
  problem: { eyebrow: string; body: SerializedLexicalNode; points: Array<{ lead: string; body: string }> };
  howItWorks: { eyebrow: string; steps: Array<{ number: number; title: string; body: string }>; autoRunCaption: string };
  workflow: { eyebrow: string; blocks: Array<{ label: string; body: SerializedLexicalNode }> };
  team: { eyebrow: string; intro: string; agents: Array<{ name: string; role: string }> };
  whyDifferent: { eyebrow: string; cards: Array<{ heading: string; body: string }> };
  control: { eyebrow: string; body: SerializedLexicalNode; linkLabel: string; linkHref: string };
  faq: { eyebrow: string; items: Array<{ question: string; answer: SerializedLexicalNode }> };
  finalCta: { heading: string; body: string };
}

interface WhySdlcGlobal {
  breadcrumb: string;
  hero: { eyebrow: string; h1: string; intro: SerializedLexicalNode };
  arguments: Array<{ heading: string; body: SerializedLexicalNode }>;
  faq: { items: Array<{ question: string; answer: SerializedLexicalNode }> };
  finalCta: { heading: string; body: string };
}

// SerializedLexicalNode is Payload's exported Lexical rich-text tree type
// (@payloadcms/richtext-lexical) — never `any`.
```

### Permissions

Payload has **no custom role model** in this app — the `Users` collection has no `roles` field. The effective principals are therefore: **anonymous** (public site visitors + the frontend read path), an **authenticated admin user** (email/password login to `/admin`), and an **API-key user** (agents writing content programmatically via `Users.useAPIKey`). Access is set per global/collection through Payload `access` functions.

| Role                     | Read (globals/media)   | Update globals | Create/Upload media | Access `/admin` |
| ------------------------ | ---------------------- | -------------- | ------------------- | --------------- |
| Anonymous (public)       | ✓ (`read: () => true`) | ✗              | ✗                   | ✗               |
| Authenticated admin user | ✓                      | ✓              | ✓                   | ✓               |
| API-key user (agents)    | ✓                      | ✓              | ✓                   | ✗ (API only)    |

Rule: every new global sets `access.read = () => true` (public content) and leaves `update` to Payload's default (authenticated). No global exposes write to anonymous.

## Backend Implementation

**Config layer (the single API/infra definition):** `apps/marketing/src/payload.config.ts` — register `globals: [siteSettings, home, whySdlc]`.

**Content-type layer:** one file per global, following the existing `src/collections/*.ts` convention:

- `apps/marketing/src/globals/SiteSettings.ts` — `GlobalConfig`, slug `site-settings`, `access.read: () => true`.
- `apps/marketing/src/globals/Home.ts` — `GlobalConfig`, slug `home`, section `group` fields per the Data Model.
- `apps/marketing/src/globals/WhySdlc.ts` — `GlobalConfig`, slug `why-sdlc`.

**Generated artifacts (do not hand-edit):** `apps/marketing/src/payload-types.ts` is regenerated by `payload generate:types`; the Postgres schema delta is produced by `payload migrate:create` and applied by `payload migrate`. The spec changes the config; the types/migrations are generated.

**Admin theming:** `apps/marketing/src/app/(payload)/custom.scss` (the CLAUDE.md-sanctioned exception to the no-SCSS rule) overrides Payload admin CSS variables to the nightshift palette (`--night-*`, `--moon-*`, `--terra-*`, fonts). Values must equal the design tokens (see Web UI → design-system).

## Web UI

### Pages

| Route       | File                                                  | Description                                                                                                |
| ----------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `/`         | `apps/marketing/src/app/(frontend)/page.tsx`          | Home long-scroll — replaces the current `Hello World` stub; renders all 10 sections from the `home` global |
| `/why-sdlc` | `apps/marketing/src/app/(frontend)/why-sdlc/page.tsx` | Standalone `/why-sdlc` page from the `whySdlc` global                                                      |

All header nav targets except `/why-sdlc` are on-page anchors (`/#how-it-works`, `/#workflow`, `/#agents`, `/#faq`) per brief §URL map. Future slots (`/docs`, `/agents`, `/changelog`, `/blog`) are **not** built — see Out of Scope.

### Components

Two tiers, per the AC:

**`packages/ui` (shared primitives — `@nightshift-ai/ui`, exported from `src/index.ts`):** presentational, brand-token-driven, reusable across surfaces. Minimum set:

- `Button` (primary terracotta / secondary outline variants)
- `Badge`
- `Card`
- `InstallSnippet` (copyable install command block on `--surface-terminal`)
- `Eyebrow` (mono `//` section label — the `.ns-eyebrow` treatment)

These mirror the primitives already prototyped in `.claude/skills/nightshift-design/components/core/` (`Button.jsx`, `Badge.jsx`, `Card.jsx`, `InstallSnippet.jsx`) — port to typed `.tsx` in `packages/ui`. All components are `.tsx` + Tailwind utility classes (CLAUDE.md); no CSS Modules/SCSS.

**`apps/marketing/src/app/(frontend)/_components/` (site-specific sections):** compose `packages/ui` primitives into page sections. One component per Home section + `/why-sdlc` blocks, plus `SiteHeader` and `SiteFooter`. These read no content directly — content is passed as typed props from the page (server) components.

### Data hooks

Payload has no client data-fetching hook layer here — content is read in **React Server Components** via the Local API:

- `apps/marketing/src/lib/payload.ts` — `getPayloadClient()` wrapper around `getPayload({ config })`.
- Server helpers (one per global): `getSiteSettings()`, `getHomeContent()`, `getWhySdlcContent()` calling `payload.findGlobal({ slug })`. Pages/layout call these and pass typed data down. No client-side fetching, no SWR/React Query.

### State

No client-state store. The site is content-driven and statically renderable. Interactive bits (copy-to-clipboard on `InstallSnippet`, mobile nav toggle, FAQ accordion) are local component state (`useState`) in the relevant client component only.

### Design tokens — `packages/design-system` (`@nightshift-ai/design-system`)

New non-buildable TS/CSS package (source-resolved via the workspace `@nightshift-ai/source` custom condition), owning the token layer per the `tailwind-design-system` guideline (CLAUDE.md: Tailwind v4, CSS-first `@theme`):

- `packages/design-system/src/tokens/{colors,fonts,typography,spacing,base}.css` — the CSS-variable tokens, value-identical to the canonical source at `.claude/skills/nightshift-design/tokens/*.css`.
- `packages/design-system/src/theme.css` — `@import "tailwindcss";` + an `@theme` block mapping the CSS variables to Tailwind v4 tokens (e.g. `--color-accent: var(--terra-500)`), so utilities like `bg-page`, `text-strong`, `text-accent`, `font-mono` resolve to brand tokens.
- `packages/design-system/src/index.ts` + package `exports` for the stylesheet(s).

`packages/ui` **imports its tokens from `packages/design-system`** (AC §6 — no duplicated token definitions): add `@nightshift-ai/design-system` as a dependency and consume its Tailwind theme; `packages/ui` defines zero raw token values. `apps/marketing` imports the design-system theme stylesheet in `(frontend)/layout.tsx` and adds Tailwind v4 (`@tailwindcss/postcss` in `next.config`/PostCSS) — this introduces Tailwind into the marketing app for the first time.

### Route permissions

All frontend routes are **public** (anonymous read) — a marketing site. `/admin/**` stays behind Payload authentication (unchanged). No per-route role gating is added.

## Error Handling

| Scenario                                               | Behaviour                                                                                  | HTTP Status |
| ------------------------------------------------------ | ------------------------------------------------------------------------------------------ | ----------- |
| Global has no content yet (empty fields)               | Section renders its empty/placeholder state — never throws                                 | 200         |
| Global slug not found via Local API                    | Payload returns the global with default/empty values (globals always exist) — page renders | 200         |
| `/why-sdlc` requested before content seeded            | Renders the page shell with empty sections                                                 | 200         |
| Unknown route (not `/`, `/why-sdlc`, `/admin`, `/api`) | Next.js `not-found`                                                                        | 404         |
| Anonymous attempts a content write                     | Payload access denies                                                                      | 403         |
| Media upload validation failure (missing `alt`)        | Payload field validation error                                                             | 400         |

Rendering is **defensive**: section components guard on nullable/empty arrays (`items ?? []`) so a partially-populated CMS never crashes a page. Because production content entry is out of scope, pages must render correctly against empty or seed content.

**Offline behaviour:** N/A — no offline sync in this project.

## Out of Scope

- **Authoring/populating final production content** in Payload — content entry is a separate workflow (story Out of Scope #1). Seed/placeholder values are acceptable.
- **SEO metadata, JSON-LD structured data, OG/meta tags, `llms.txt`, analytics, and tracking** (story Out of Scope #2) — brief §§3, 4, 5 are explicitly excluded this pass.
- **Shared design-system component adoption beyond `packages/ui` + the marketing site** — Tailwind v4 remains the workspace-wide styling default; no other surface is migrated (founder amendment, story comment).
- **Future page slots:** `/docs`, `/agents`, `/changelog`, `/blog` (brief §URL map "Future slot") — anchor links only for `/#how-it-works`, `/#workflow`, `/#agents`, `/#faq`.
- **Custom Payload role model / per-user permissions** — the existing anonymous / authenticated / API-key model is used as-is.
- **Mobile app, offline sync, `tools/`, `plugins/`, `skills/`** — not touched; not owned by the web-engineer.
- **A/B headline test, live GitHub star count, dedicated domain / canonical swap** (brief Open copy decisions #2, #3, #4).

## Open Questions

- [ ] **Design-system ↔ canonical-skill token sync.** The tokens exist canonically at `.claude/skills/nightshift-design/tokens/*.css` (used by the design skill's validators). — Suggested default: `packages/design-system` **vendors a value-identical copy** of those CSS files as the workspace runtime source (no build-time coupling to the skill dir); parity is asserted in review. A later story can automate the copy if drift becomes a risk.
- [ ] **Content shape: Globals vs a `Pages` collection.** — Suggested default: **Globals** (as specced) — two bespoke, singleton pages with fixed section shapes edit better as single forms; revisit only if repeatable marketing pages (blog/docs) land.
- [ ] **Rich text vs plain text per field.** Some section bodies could be plain `textarea`. — Suggested default: use **Lexical `richText`** for multi-paragraph/linked bodies (problem, workflow, control, FAQ answers, why-sdlc arguments/intro) and `text`/`textarea` for single-line labels, as tabled in the Data Model.
- [ ] **Section imagery.** Brand is type/terminal-driven, so no `upload` fields are specced. — Suggested default: **no media fields** this pass; add an `upload` field to a section group later if a design needs an image (Media collection already supports it).
- [ ] **Fonts loading.** Tokens `fonts.css` imports Inter + JetBrains Mono from Google Fonts. — Suggested default: keep the Google Fonts `@import` for this pass; self-hosting via `next/font` is a later optimization (not blocking, no SEO/perf ACs in scope).
