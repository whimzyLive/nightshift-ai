# Generic activation-gated reference-doc generation for /sdlc:docs — Technical Spec

**Story:** [NA-65](https://whimzylive.atlassian.net/browse/NA-65)
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Date:** 2026-07-21

## Overview

Make the `/sdlc:docs` reference-doc pipeline repo-agnostic and activation-gated: strip every
Nightshift-specific assumption out of the shipped `sdlc` plugin, gate artifact-reference doc-types
on declared config instead of assuming every consumer authors Claude Code artifacts, and add a
product-reference family (api/config/schema/cli/error) that generates from a repo's actual
contracts. Nightshift keeps reproducing its exact current docs scope — but now via its own
`docs-manifest.md`, not via strings baked into the plugin.

This is a **spec-change story**: the deliverable is edits to plugin-owned prose refs
(`plugins/sdlc/refs/*.md`) plus Nightshift's own `.claude/project/docs-manifest.md`. There is no
runtime code, no generator script, no data model, no API, no UI — the "algorithm" is prose executed
inline by the `knowledge-engineer` agent (see `docs-pipeline.md` §3's "instructions not code"
standing style). Sections of the standard spec template that assume DB / API / web / mobile / offline
surfaces are therefore omitted as non-applicable.

## Background — current state (what leaks and what is unconditional)

Verified against the current plugin files:

- **`refs/doc-types.md`** — 15 rows, **every row `applies-when = always`**. Six `reference`-quadrant
  `auto` rows exist: `command-reference`, `agent-reference`, `skill-reference`, `config-reference`,
  `hooks-contract`, `error-reference`, plus the `meta` `llms-txt` row. Every one is unconditional and
  every `source-of-truth` cell hardcodes `plugins/{sdlc,gtm}/…` globs. There is **no** `api-reference`,
  `schema-reference`, or `cli-reference` row.
- **`refs/docs-pipeline.md`** (168 KB) — §3 "Deterministic regen algorithm" is the core. Its
  **"Scope rule — plugin-only source-of-truth"** banner (lines 121–137) and the §3 affected-when
  table + steps 1–8 name `plugins/{sdlc,gtm}` **12 times**, name the `gtm` `marketing-context-template`
  3 times, and name the nx-generated mirror exclusions (`agents/`, `.opencode/**`, `.codex/**`,
  `.gemini/**`, `.github/**`) 3 times. `reference-roots`/`reference-excludes` appear **0 times**.
- **`refs/docs-manifest-template.md`** — the consumer manifest is a **3-column** row table
  (`type`, `enabled`, `target-path`) plus two optional free-form sections. No per-row `source:` or
  `contract:` field, and no repo-level `reference-roots`/`reference-excludes`.
- **`.claude/project/docs-manifest.md`** (Nightshift's own) — activates all six artifact/config
  reference rows + `llms-txt` + narrative rows. Carries no roots/excludes today because the plugin
  hardcodes them.

The defect: a plain product consumer inherits the four artifact-reference rows
(`command`/`agent`/`skill`/`hooks`) **and** the two product-reference rows (`error`/`config`) — all of
which scan `plugins/{sdlc,gtm}/**`, find nothing, and emit **empty artifact-reference and
product-reference pages**, while the product's real api/cli/config/schema/error surface goes
undocumented.

## Locked decisions (from the story — do not re-litigate)

1. **De-leak.** `docs-pipeline.md` and `doc-types.md` carry **zero** Nightshift-specific strings.
   All `plugins/{sdlc,gtm}`, gtm-template, and nx-mirror-exclusion specifics move **out of the
   shipped plugin** and live **only** in Nightshift's `.claude/project/docs-manifest.md`.
2. **No reference row stays `applies-when: always` except `llms-txt`.** Every reference row becomes
   activation-gated (narrative/release rows are out of scope and unchanged — they remain `always`).
3. **Two doc-type families.**
   - **artifact-reference** — `command`/`agent`/`skill`/`hooks` — activate on **presence of
     configured `reference-roots`** in the consumer manifest.
   - **product-reference** — `api`/`cli`/`config`/`schema`/`error` — activate on a **declared
     contract or a configured source**.
4. **One shared source resolver**, ordered:
   1. **declared contract** at a conventional-or-configured path (api ← OpenAPI/Swagger;
      config/schema ← JSON-Schema / env-schema / GraphQL schema);
   2. **configured source** (a manifest row's `source:` path-or-command);
   3. **convention scan** (artifact types only: `reference-roots` frontmatter walk + link transform,
      honoring the NA-64 transform-not-mirror rule);
   4. **skip** (no page, no empty stub).
5. **Product-reference detection split.** `api`/`config`/`schema` get **contract detection +
   configured fallback**; `cli`/`error` are **configured-source-only** (no invented contract
   detection — no route/CLI parsers).
6. **Single family-resolved `config-reference`.** Product repos resolve it from a JSON/env/GraphQL
   schema contract; artifact repos (Nightshift) resolve it from configured plugin config-contract
   templates. One registry row, family-resolved by the resolver — not two rows.
7. **New optional manifest fields** with convention defaults: repo-level `reference-roots` and
   `reference-excludes`, and per-row `source:` / `contract:`. A plain product repo configures
   nothing and still gets contract-detected product docs.
8. **Delivered in 3 dependency-ordered phases** (P1 → P2 → P3), each mapping to the story's ACs.

## Design

### 1. Registry schema change (`refs/doc-types.md`)

**`applies-when` grammar — add three activation predicates** to the existing grammar
(`always` / `framework:` / `language:` / `dep:`). These evaluate against the **consumer manifest**,
not just repo-detect + `package.json`, so the "evaluation inputs" prose in `doc-types.md` gains a
third declared input: **the consumer's `.claude/project/docs-manifest.md`** (its repo-level
`reference-roots` and per-row `source:`/`contract:` fields).

| New predicate               | Matches when…                                                                                                              |
| --------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `reference-roots:present`   | the consumer manifest declares a non-empty repo-level `reference-roots` list (gates the 4 artifact-reference rows)          |
| `contract-or-source:<kind>` | a declared contract of `<kind>` resolves at a conventional/configured path, **or** the row carries a manifest `source:`     |
| `source:present`            | the row carries a manifest `source:` (the `cli`/`error` gate — configured-source-only, no contract detection)              |

**Row changes** (the 8-column schema is unchanged; only `applies-when` values change, plus 3 new
rows):

| type                | family              | applies-when (new)                    | notes                                                                       |
| ------------------- | ------------------- | ------------------------------------- | --------------------------------------------------------------------------- |
| `command-reference` | artifact-reference  | `reference-roots:present`             | was `always`; convention-scan resolver over `reference-roots`               |
| `agent-reference`   | artifact-reference  | `reference-roots:present`             | was `always`                                                                |
| `skill-reference`   | artifact-reference  | `reference-roots:present`             | was `always`                                                                |
| `hooks-contract`    | artifact-reference  | `reference-roots:present`             | was `always`                                                                |
| `api-reference`     | product-reference   | `contract-or-source:openapi`          | **new row**; OpenAPI/Swagger contract detection + `source:` fallback        |
| `config-reference`  | product-reference   | `contract-or-source:config`           | was `always`; **family-resolved** (schema contract OR configured templates) |
| `schema-reference`  | product-reference   | `contract-or-source:schema`           | **new row**; JSON-Schema / GraphQL / env-schema contract + `source:`        |
| `cli-reference`     | product-reference   | `source:present`                      | **new row**; configured-source-only                                         |
| `error-reference`   | product-reference   | `source:present`                      | was `always`; configured-source-only, but a **special aggregating type** — `source:` is a cross-plugin scan directive (§3), not a lone file/command; guards preserved |
| `llms-txt`          | meta                | `always`                              | **unchanged** — the only `always` reference-family row (narrative/release rows are out of scope, still `always`) |

> The `family` concept is **descriptive prose in the registry**, not a new schema column — the 8-column
> row schema and its self-check are preserved. Add a short "Doc-type families" subsection to
> `doc-types.md` naming which types are artifact-reference vs product-reference and how each family's
> `applies-when` predicate resolves.

**Registry self-check updates** (`doc-types.md` "Registry self-check"): the current invariant
"Every row currently ships with `applies-when = always`" and "the 15 mandatory doc types below are
all stack-agnostic" must be **rewritten** — the new invariant is **"every reference row is
activation-gated; only `llms-txt` and the untouched narrative/release rows remain `always`."** The
new `api`/`schema`/`cli`-reference rows are `generation-mode = auto`, but they are **contract/source-driven**
(resolved by the source resolver of §2), **not** frontmatter-driven — add them to the registry's `auto`
rows without listing them under the "Frontmatter-driven rows are `auto`" set (that set stays the
`reference-roots`-scanned artifact rows only). No blank cells; new rows populate all 8 columns.

### 2. Source resolver (`refs/docs-pipeline.md` §3)

Replace §3's hardcoded plugin-only affected-when table with a **generic resolver**, applied per
enabled+activated row. The resolver is the ordered ladder from locked decision 4:

```
resolve(row):
  1. contract  — if row.family == product-reference and row.type ∈ {api, config, schema}:
                 if row.contract is explicitly configured in the manifest it MUST resolve and parse —
                 a missing OR malformed configured contract → FAIL LOUD at the confirm gate (surface
                 the bad path to the founder); never fall through, never emit an empty page.
                 Otherwise look for the conventional contract path for that kind (see Contract
                 conventions below). If a contract is found → generate from it, emit a "Source:" link
                 to the contract file. STOP. If NO `contract:` is configured and the conventional path
                 is merely absent → fall through to rung 2 (an absent conventional contract is not an
                 error).
  2. source    — if row carries a manifest `source:` (path, glob/scan directive, or `command:`):
                 read/execute it, applying reference-excludes to any glob/scan match set (see §4);
                 generate from its output, emit a "Source:" link. STOP.
                 (cli/error reach the resolver only through this rung; error-reference's source is an
                 aggregating scan directive — see §3.)
  3. scan      — if row.family == artifact-reference and repo-level reference-roots is non-empty:
                 each declared root MUST exist — a declared-but-missing root → FAIL LOUD at the confirm
                 gate (surface the bad root); never scan-as-empty (scanning a missing root as empty
                 would delete previously-generated artifact pages and break the always-present
                 guarantee). Walk each existing root for the type's artifact files (commands/**,
                 agents/**, skills/**/SKILL.md, hooks/hooks.json), minus reference-excludes; transform
                 each (frontmatter + Source link, NA-64 transform-not-mirror — never copy the body). STOP.
  4. skip      — no contract, no source, no roots → generate NOTHING. No page, no empty stub,
                 no llms.txt entry for this row. (This is the no-roots-configured / no-source case, an
                 intended inactive row — distinct from a declared-but-missing root/contract, which errors.)
```

**Idempotence (AC3 — byte-identical re-run):** each rung's output is a pure function of its resolved
source bytes + a fixed template + a fixed Source-path string. Re-running against an unchanged
contract/source/root set yields byte-identical pages. This is the existing §3 idempotence contract,
now applied to the generic resolver.

**Change-gate keying (drives `sync`'s regenerate-vs-skip decision):** the affected-when keying
(path vs content vs always) is preserved and resolves per row as follows:

- **path-keyed** — a row whose resolved source is a concrete file path (a contract file, a path-valued
  `source:`, or a file under `reference-roots`) is "affected" only when that path intersects
  `CHANGED_FILES`/`CHANGED_DIFF`.
- **content/always-keyed** — a row whose resolved source has **no path to intersect** is regenerated
  on **every** `sync`, so it can never silently go stale. This covers: any `command:`-prefixed
  `source:` (a command has no input file path to match), any row whose source otherwise can't be
  path-matched (e.g. the `error-reference` aggregating scan), and **`llms-txt`**, which keeps its
  `always` keying independent of the source-intersection rule — it has no contract/source/root file
  and is regenerated every run (AC4).

**De-leak requirement:** §3's "Scope rule — plugin-only source-of-truth" banner and every
`plugins/{sdlc,gtm}` / `marketing-context-template` / nx-mirror string in §3 (and anywhere else in
`docs-pipeline.md` and `doc-types.md`) is **deleted** and replaced by the generic resolver language.
The nx-mirror exclusions and the `plugins/{sdlc,gtm}` allowlist move into **Nightshift's manifest**
as `reference-roots` + `reference-excludes` (see §5 below). The transform-not-mirror standing rule
(NA-64) is referenced, not restated, at the scan rung.

### 3. Contract conventions (product-reference detection)

The resolver's rung-1 contract lookup uses a **conventional path per kind**, overridable by a row's
manifest `contract:` value. No framework-specific route parsing — contract files only.

| type               | contract kind          | conventional paths (first match wins)                                                 |
| ------------------ | ---------------------- | ------------------------------------------------------------------------------------- |
| `api-reference`    | OpenAPI / Swagger      | `openapi.json`, `openapi.yaml`, `openapi.yml`, `swagger.json`, `docs/openapi.*`       |
| `schema-reference` | JSON-Schema / GraphQL  | `schema.json`, `schema.graphql`, `*.schema.json`, `docs/schema.*`                     |
| `config-reference` | JSON-Schema / env      | `config.schema.json`, `.env.schema`, `.env.example` (env-schema); artifact repos: configured `source:`/templates |

- **`cli-reference`, `error-reference`** have **no** conventional contract — they resolve only via a
  manifest `source:` (rung 2). This is locked decision 5: no invented CLI/error parsers.
- When rung-1 finds a contract, the page is **generated from the contract's own structure** (paths +
  methods for OpenAPI; properties for JSON-Schema) with a prominent **Source:** link to the contract
  file. The generator never parses application source to synthesize a contract.
- A `contract:` **explicitly configured** in the manifest but missing or malformed is an **error**,
  not a fall-through: the resolver **FAILs LOUD** at the confirm gate (rung 1) rather than skipping.
  A merely-absent **conventional** contract path (none configured) is not an error and falls through.

**`error-reference` — special aggregating type (guards preserved).** `error-reference` reaches the
resolver via rung-2 `source:`, but it is **not** a plain single-source row — its `source:` is an
aggregating **scan directive**, and the generator retains all of its current guarantees:

- **Exhaustive cross-plugin scan** of the configured roots' `commands/**` + `agents/**` + `refs/**`
  (scoped by `reference-roots` / `reference-excludes`), not a lone file or command.
- **Case-insensitive section match** — both `## Error Handling` and `## Error handling` headings match.
- **Real-section-vs-stub exclusion** — a heading whose body is a stub, placeholder, or a
  deferral/"see X" pointer is excluded; only real Error Handling sections are aggregated.

These guards are part of the row's contract; a family-resolved plain single-source reading that drops
them is a defect.

### 4. Manifest schema additions (`refs/docs-manifest-template.md`)

Add to the consumer manifest, with **convention defaults so a plain product repo configures nothing**:

**Repo-level fields** (new optional section below the row table):

```markdown
## Reference roots

reference-roots: <comma-separated dirs the artifact-reference scan walks; omit for a product repo>
reference-excludes: <comma-separated globs excluded from the scan; omit if none>
```

- **Absent `reference-roots`** → the 4 artifact-reference rows never activate (product-repo default).
- **`reference-excludes` is a global exclusion guarantee** — it filters matching paths on **every**
  resolver rung that walks a glob or a scan directive, not only the artifact-scan rung. A glob/scan
  `source:` (e.g. `error`/`config-reference`'s broad plugin scan) has `reference-excludes` applied to
  its match set too, so a broad source glob can never sweep in mirror/settings content. It is a no-op
  only for rungs that resolve a single explicit file (a path-valued `contract:` or path-valued `source:`).

**Per-row optional fields** — extend the row table from 3 columns to allow **two optional trailing
columns**, `source:` and `contract:` (blank when unused, so existing 3-column manifests stay valid):

| Column        | Meaning                                                                                             |
| ------------- | --------------------------------------------------------------------------------------------------- |
| `type`        | registry key (unchanged)                                                                            |
| `enabled`     | `true`/`false` (unchanged)                                                                          |
| `target-path` | output path (unchanged)                                                                             |
| `source`      | **new, optional** — path or `command:`-prefixed command feeding the resolver's rung-2               |
| `contract`    | **new, optional** — explicit contract-file path overriding rung-1's conventional lookup             |

- Update the manifest **Fill rules** and self-check: the two new columns are optional and default to
  empty; `/sdlc:init` writes them empty; the founder fills them to configure product-reference rows.
- Update the manifest header-comment prose that currently says "the row table itself remains 3-column
  activation data only" to reflect the two new optional columns (still activation data, no generation
  logic).
- The existing "Voice & format" and "Additional Jira project keys" free-form sections are unchanged.

### 5. Nightshift's own manifest (`.claude/project/docs-manifest.md`)

Nightshift reproduces its **exact current reference-docs scope** under the new model — all
Nightshift-specific config that used to live in the plugin now lives here:

- Add repo-level **`reference-roots: plugins/sdlc, plugins/gtm`** — activates the 4 artifact-reference
  rows (`command`/`agent`/`skill`/`hooks`) exactly as today.
- Add **`reference-excludes`** covering the nx-generated mirrors and non-plugin skill dirs that §3's
  scope-rule banner used to hardcode: repo-root `agents/`, `.opencode/**`, `.codex/**`, `.gemini/**`,
  `.github/**`, repo-root `skills/`, `.agents/skills/`, `.claude/skills/`, and `.claude/settings*.json`.
- Set **`config-reference`** row `source:` (or `contract:`) to the plugin config-contract templates
  (`plugins/{sdlc,gtm}/refs/*-template.md`) so it resolves family-side to the same config surface it
  documents today.
- Set **`error-reference`** row `source:` to the aggregating scan directive it runs today — a
  cross-plugin walk of `plugins/{sdlc,gtm}/**` `commands/**` + `agents/**` + `refs/**` Error Handling
  sections. `error-reference` is now configured-source-only, but its `source:` is that **scan
  directive** (not a lone file/command) and it stays a **special aggregating type** carrying the §3
  guards (exhaustive scan, case-insensitive `## Error Handling`/`## Error handling` match, and
  real-section-vs-stub/deferral exclusion), scoped by the same `reference-roots`/`reference-excludes`.
- Leave `api`/`schema`/`cli`-reference rows **absent/disabled** — Nightshift ships no OpenAPI/CLI
  contract, so they correctly do not activate.

**Acceptance target (AC2):** after this change, `/sdlc:docs sync` on Nightshift must produce a
docs/reference tree **equivalent to today's** — same command/agent/skill/hooks/config/error pages,
same `llms.txt`.

## Phased delivery (each phase maps to the story's ACs)

**Phase 1 — de-leak + activation foundation (AC1, AC2).**
Remove every Nightshift-specific string from `doc-types.md` + `docs-pipeline.md`; make the 4
artifact-reference rows gate on `reference-roots:present`; add the repo-level `reference-roots` /
`reference-excludes` manifest fields + self-check; rewrite Nightshift's `docs-manifest.md` to
reproduce its current scope via those roots/excludes. Exit: `grep` finds zero leaks in the two plugin
refs; Nightshift `sync` reproduces today's artifact-reference tree.

**Phase 2 — product-reference framework + `api-reference` (AC3).**
Introduce the shared source resolver (contract → source → scan → skip) in §3; add the `api-reference`
row with OpenAPI/Swagger contract detection + `source:` fallback; add the per-row `source:`/`contract:`
manifest columns. Exit: fixture (A) product repo with `openapi.json` generates an `api-reference` page
+ Source link, byte-identical on re-run.

**Phase 3 — remaining product types (AC4).**
Add `schema-reference` (contract + source), `cli-reference` + `error-reference` (configured-source-only),
and make `config-reference` family-resolved. Exit: fixture (C) artifact repo generates artifact rows +
a family-resolved `config-reference`; `cli`/`error` activate only when a `source:` is configured.

Phases are strictly ordered: P2 depends on P1's activation model; P3 depends on P2's resolver.

**dispatch-and-snapshot** check (run the sync algorithm against the fixture, diff the produced
`docs/reference/**` + `llms.txt` against the committed snapshot), not a unit test of a script.

## Error handling / edge cases

| Scenario                                                    | Behaviour                                                                                             |
| ----------------------------------------------------------- | ---------------------------------------------------------------------------------------------------- |
| Row enabled but no contract, no `source:`, no roots         | Resolver rung-4 **skip** — generate nothing; no empty page, no `llms.txt` entry (fixture B invariant) |
| No `reference-roots` configured at all                      | The 4 artifact-reference rows are simply **inactive** (intended product-repo default) — not an error   |
| `reference-roots` declares a root dir that is missing       | **STOP** at the confirm gate and surface the bad root — never scan-as-empty (that would delete previously-generated artifact pages and break the always-present guarantee); other roots are not silently substituted |
| `contract:` explicitly configured but missing/malformed     | **FAIL LOUD** at the confirm gate (surface the bad path); do **not** fall through and do **not** emit an empty page |
| No `contract:` configured and the conventional path absent  | Not an error — **fall through** to the next resolver rung (source → scan → skip)                       |
| `source:` is a `command:` (or otherwise unmatchable to a path) | Keyed as **content/always** — regenerated every `sync`, never silently stale                        |
| `cli`/`error` row enabled with no `source:`                 | Does not activate (rung-1/3 don't apply, rung-2 absent → skip) — never invents a parser               |
| A leaked `plugins/{sdlc,gtm}` string survives in a plugin ref | AC1 fails — treat as a P1 defect; grep gate in Testing catches it                                    |
| Legacy 3-column consumer manifest (no new columns)          | Valid — new columns default empty; artifact rows simply never activate without `reference-roots`      |

## Out of scope

- Bespoke per-framework source parsing (Hono/Express/FastAPI route walkers, CLI arg parsers) —
  contract-based detection only, no invented parsers.
- Narrative doc types (`tutorial`/`how-to`/`concept`/`integration-guide`) — unchanged by this story.
- Release-mode types (`changelog`/`release-notes`/`migration-guide`) — unchanged.
- Publishing/deploying the updated plugin to consumers — tracked separately: cache-sync
  (`~/.claude/plugins/cache/nightshift/sdlc/…`) for **local** validation vs. a **real** sdlc plugin
  release for durable deploy. This story lands the source under `plugins/sdlc/`; the release is a
  separate step.
- Changing the 8-column registry schema or the manifest's core 3-column identity (both preserved;
  `family` is prose, the two new manifest columns are optional).

## Open questions (each with a suggested default so implementation proceeds)

- [ ] **Family as prose vs. a 9th registry column.** Suggested default: **prose** — keep the 8-column
      schema + self-check intact; describe families in a new `doc-types.md` subsection. Adding a column
      would ripple through every self-check assertion for no activation benefit.
- [ ] **Two new manifest columns vs. a per-row YAML block.** Suggested default: **two optional trailing
      columns** (`source`, `contract`) — preserves the existing table shape and keeps legacy 3-column
      manifests valid; a YAML block would be a larger template rewrite.
- [ ] **`source:` command execution safety.** A `source:` value may be a `command:`-prefixed shell
      command. Suggested default: allow it, run it at the same confirm gate the pipeline already gates
      generation behind (founder-confirmed), inside the session temp dir — no new sandbox in this story.
- [ ] **Fixture harness location.** Suggested default: `plugins/sdlc/scripts/__tests__/fixtures/<A|B|C>/`
      with committed expected-output snapshots, driven by a `*.test.sh` sibling to
      `auto-merge-pr.test.sh`. Confirm with Tech Lead if a different test convention is preferred.
