# writing-docs templates emit title/description + ADR-reference frontmatter — Technical Spec

**Story:** NA-61 — writing-docs templates emit no description:/ADR-reference frontmatter — forces per-mode skip workarounds, blocks audit AC4
**Epic / Feature:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer (`docs/features/2026-07-17-public-docs-generation-sdlc-docs.md`)
**Relates:** NA-55 (audit mode — this story is the deferred template fix its AC4 carry-over names)
**Date:** 2026-07-18

## Overview

The four `writing-docs` Diátaxis templates emit no `description:` frontmatter (only the How-to template emits any frontmatter at all, and none emits `description:`), so every `/sdlc:docs` mode's `llms.txt` regen — which derives each page's `title — description — link` from frontmatter — carries a duplicated "skip-and-surface frontmatter-less page" workaround, and NA-55 audit's AC4 ADR-drift flag has no frontmatter join key on template-authored pages. This story makes all four templates emit `title:` + `description:` + a standardised `related-adrs:` key at source, re-scopes (not removes) the skip path, and adds the **frontmatter arm** of the join key audit's §22 already looks for — **without weakening the existing inline-link arm**. The core invariant: an empty `related-adrs: []` means "the frontmatter arm is unpopulated → fall through to the inline-link check," **never** "this page has no ADR." So the claim is scoped honestly — the key is a deterministic join source on **template-authored pages an author fills**, not across every mode: pages that link ADRs only in body prose (e.g. release-notes' §12-generated links, which stay out of scope for the key) remain covered by audit's inline-link arm, which the empty default preserves.

## The frontmatter contract — what each template emits (AC1, AC2)

All four templates emit the same three keys by default. `title:` matches the page H1. `description:` is a one-line summary the author fills (never derived — deriving published index copy is the fabrication boundary the pipeline holds everywhere, §17). `related-adrs:` is the standardised ADR-reference key: a YAML list of repo-root-relative `docs/adr/NNNN-*.md` paths, **default `[]`**. An empty list is **not** a positive "no ADR" assertion — it means the frontmatter arm is unpopulated, and audit falls through to its inline-link check (OR-semantics; see [ADR-reference key + audit AC4](#adr-reference-key--audit-ac4--what-audit-does-with-the-key-crux-4)).

### Why `related-adrs:` (name + shape)

- **Name.** `related-adrs:` — distinct from the ADR-side `source-stories:` (ADR → story) and from the how-to `source:` (page → code globs). No collision, reads plainly as "the ADRs this page relates to."
- **Shape = full `docs/adr/NNNN-*.md` paths, not bare numbers.** Audit §22 already resolves the inline-link form (`docs/adr/NNNN-*`) by path; a path-valued list reuses that resolution **verbatim** — audit needs no new number→file mapping. This is the minimal-churn shape (crux 4).
- **Default empty (`[]`), flow-style.** Most pages populate no ADR path in frontmatter; an empty flow-style list adds no 2-space-indented list item, so it dodges the NA-62 landmine (below). Crucially, empty ≠ "no ADR reference": audit treats an empty/absent key as "frontmatter arm silent" and **still runs its inline-link check** (OR-semantics), so a page carrying an inline `docs/adr/…` link but a default-empty key **keeps the drift detection it had pre-change** — no regression.
- **Explanation quadrant is the primary consumer.** Its existing "Related decisions" prose section links ADRs for **readers**; `related-adrs:` is the machine-readable mirror **audit** reads. They coexist by design — one human section, one machine key. Audit's ADR-drift check is an **OR** over both arms (non-empty `related-adrs:` **or** an inline `docs/adr/…` link), so keeping the reader-facing links **and** the key never causes a page to lose detection; where both are present, either arm firing is enough.

### The four verbatim blocks

**Placeholder convention.** The scalar string fields (`title:`, `description:`) use a **bracket-free** `TODO — …` placeholder, **not** `[…]` or `<…>`. `title: [x]` is valid YAML **flow-sequence** syntax — it parses as a one-element list, not a string — so an unreplaced `[…]` placeholder reaching §8's `llms.txt` regen (which consumes `description` as a scalar) emits a garbled list or errors. A bare `TODO — …` is a plain scalar: valid, and obviously unfilled. `related-adrs: []` stays an **intentional** empty flow-sequence (a list is what the key is). The How-to `source:` line is the **shipped** template line, left verbatim (a pre-existing list item, out of this story's scope).

## ADR-reference key + audit AC4 — what audit does with the key (crux 4)

**NA-61 does not change audit's ADR-drift _direction_ logic** (ADR is source of truth; the reverse is never proposed). §22 was already written anticipating this key — it flags a page that declares "an inline link to `docs/adr/NNNN-*` **or an explicit ADR-reference frontmatter key**." NA-61's job at the audit layer is four small, precise edits:

1. **Pin the key name.** §22 currently says "an explicit ADR-reference frontmatter key" generically; change to name it `related-adrs:` (path-valued list, resolved identically to the inline-link form).
2. **Keep the reliability caveat honest — do NOT rewrite it to "reliable for template pages now."** The flag is an **OR** over two arms: it fires for a page that **opted in** — an inline `docs/adr/…` link **OR** a **non-empty** `related-adrs:`. A template-produced page left at the empty `[]` default is **not** flagged via the key (it opted out); its inline links, if any, still are. Rewrite the caveat to state the OR — "flagged if an inline ADR link is present OR `related-adrs:` is non-empty" — **not** "the templates now emit the key, so the flag is reliable," which would falsely tell a founder audit checked their `[]`-defaulted pages.
3. **Add an explicit dangling-`related-adrs` check.** A `related-adrs:` path that does **not** resolve to an existing `docs/adr/*.md` file at the base ref must be **flagged as dangling** — a **new** rule, distinct from the newer-commits drift check. A non-existent ADR has no commits, so the drift check (`git log` on the path → empty) **silently passes**; without a separate existence check, a typo'd or deleted `docs/adr/` path permanently advertises a phantom ADR audit never catches. §22 has a dangling **code-path** check today, but it is worded for code paths and does not cover `docs/adr/` frontmatter paths — this extends the same idea to the ADR arm.
4. **Retire §24's ADR half, but re-home its `title`/`description` no-flag decision.** The ADR-reference half of §24's carry-over is resolved (the key exists). §24 **also** carries the normative decision "audit does **not** add a narrative frontmatter-presence flag for `title`/`description`, because it would fire on nearly every legacy/hand-authored page (a template-source defect, not per-page drift)." That decision **must survive** — move it into §22 (the reference-integrity flag home), not delete it with §24. Dropping it invites a later audit edit to add a presence flag believing the templates now guarantee frontmatter → firing on every legacy page → an audit PR flagging nearly every doc as drift. The skip-and-surface half of §24 is handled by the re-scope in [Simplification safety](#simplification-safety--the-skip-stays-universal-crux-2-ac3).

Churn to shipped audit stays small: three of the four edits are prose (pin name, honest caveat, re-home a decision); the one behavioural addition is the dangling-ADR-path flag, which reuses the existing path-existence primitive. No reverse-direction proposal, no write to `docs/adr/` — every invariant §22 already states holds.

## Simplification safety — the skip stays universal (crux 2, AC3)

**Can the regen distinguish a template-produced page from a legacy page at regen time? No.** The `llms.txt` regen (§8) walks every enabled `public: yes` page in the tree and can observe only one fact per page: whether it **has** `title`/`description` frontmatter or not. There is no origin marker, no "produced by writing-docs vN" stamp, and adding one would not help retroactively (legacy pages already written could never carry it). A date/age heuristic would be exactly the ambient, non-deterministic input the pipeline forbids.

**Therefore the skip-and-surface code path stays universal** — it fires for **any** frontmatter-less page regardless of origin. AC3's "simplify the workaround" resolves to **"the skip now rarely fires (only for legacy/hand-authored pages)," not "remove the skip."** State this explicitly in the ref so no later edit deletes the skip wholesale and silently breaks legacy pages (Standing lesson: never collapse distinct cases — a template page and a legacy page stay distinct; the skip is what keeps the legacy case safe).

**What actually gets simplified is the duplicated _rationale_, not the rule.** The rule itself already lives in one normative home (§19) and is referenced — not recopied — by §21 ("§19's, verbatim"). The genuine duplication is the **explanatory claim** "the templates emit no frontmatter / three of four emit no block," restated as live justification in §17, §22's caveat, and §24. That claim is now **factually stale** and must be corrected in each live ref to: _template-produced pages carry frontmatter; the skip remains for legacy/hand-authored pages that predate it._ This removes the duplicated stale prose (the class the epic objects to) while preserving the one skip path all four modes share.

**Scope fence:** only the **live contract refs** (`docs-pipeline.md`, `SKILL.md`, `doc-types.md`) are edited. The shipped NA-53/54/55/56 **spec docs** are point-in-time records — not rewritten (see Out of Scope).

## Migration decision — no automated backfill (crux 3, AC4)

**Decision: do NOT run a backfill pass. The skip path stays for legacy pages; audit's surface list is the backfill worklist.** Rationale:

- A backfill pass mutating every already-written page is a big hammer that touches **published** docs, and `description:` cannot be machine-generated (it is authored, one-line human copy) — so a backfill would either fabricate index copy (forbidden) or demand bulk authoring, exactly the churn AC4 asks us to weigh against.
- The skip-and-surface path already handles legacy pages gracefully: they are **surfaced** (phase-2 output + PR body), never silently dropped. That surfaced list **is** a ready-made, opt-in worklist — a founder backfills the two lines on a legacy page the next time a run surfaces it, or when they next touch it.
- The docs pipeline is generic and runs per consumer repo; a mandated backfill would fan out across every consumer's published tree. Even less reason to force it centrally.

Flagged as an Open Question with the tradeoff, per the story's instruction, in case the founder wants a one-shot opt-in backfill later.

## NA-62 Prettier-safe frontmatter blocks

Every frontmatter block authored above lives inside a fenced code block in SKILL.md. The NA-62 landmine is a **2-space-indented fenced block with an embedded blank line under a list item**; the writing-docs templates are top-level `###`-section fences (not under list items), which lowers but does not eliminate risk. The implementer must, per block:

| Block                              | Landmine check                                                                                                                                                                                                                                          |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tutorial / Reference / Explanation | New blocks — no indented list item (`related-adrs: []` is flow-style). Confirm the block is flush-left inside its fence and carries **no embedded blank line**. Reference's outer fence is **four backticks** — keep it four.                             |
| How-to                             | Retains the existing 2-space-indented `source:` list. Do **not** introduce a blank line between `source:`'s list item and the new `related-adrs:` comment lines. Keep `related-adrs: []` flow-style (no new indented `-`).                                |
| All four                           | After editing, run the repo formatter and confirm it is **idempotent** on SKILL.md (a second run produces no diff) before committing — the NA-62 dedent/idempotency gate. Comment lines (`#`) inside the YAML must survive the formatter unchanged.       |

## Out of Scope

- **Any new `/sdlc:docs` mode or mode-contract change.** The mode set (`sync`/`release`/`seed`/`audit`/`distill`) is untouched; no mode's algorithm changes — the only behavioural addition is the §22 dangling-ADR-path flag, which is a new sibling check inside the existing audit mode, not a mode change.
- **Removing the skip-and-surface path.** Explicitly rejected — template vs legacy pages are indistinguishable at regen time, so the skip stays universal (it merely fires rarely now). See [Simplification safety](#simplification-safety--the-skip-stays-universal-crux-2-ac3).
- **An automated frontmatter backfill of already-written pages.** Rejected as the default; the skip path + audit's surface list cover legacy pages. See [Migration decision](#migration-decision--no-automated-backfill-crux-3-ac4) and the Open Question.
- **Rewriting the shipped NA-53/54/55/56 spec docs.** They are point-in-time records; the live contract is `docs-pipeline.md` + `SKILL.md`. Only live refs are edited.
- **Populating `related-adrs:` in release-mode (§12) pages.** Release-notes link their motivating ADRs in **body prose** (§12's ADR-link resolution), not the frontmatter key; this story does **not** wire §12 to populate `related-adrs:`. Detection is **not** lost — audit's inline-link arm (§22 OR-check) catches drift on those body links, and the empty-default key does not suppress it. Bringing §12 into scope (populate the key from its resolved ADRs) would touch NA-53's shipped release contract — deferred; see the Open Question. The Overview is scoped to "template-authored pages an author fills" precisely so it does not overclaim coverage of release-mode pages.
- **Adding a `related-adrs` presence check to any completion checklist.** The key is optional (empty default); a presence check would false-fire on every non-ADR page.
- **Adding a narrative `title`/`description` frontmatter-presence flag to audit.** Explicitly rejected and re-homed into §22 from §24 — it would fire on nearly every legacy page (a template-source defect, not per-page drift). The skip-and-surface path is how frontmatter-less pages are handled, not a drift flag.
- **The NA-62 Prettier sweep beyond the blocks this story touches.** Only the four blocks authored here get the dedent + idempotency check; the repo-wide sweep is NA-62's.
- **Auto-generated reference rows** (`command-reference`, `agent-reference`, etc.). They are machine-generated with their own frontmatter derivation (§3) and do **not** use the `writing-docs` Reference template — unaffected by this change.

## Open Questions

- [ ] **ADR-reference key name — `related-adrs:` vs `adr:` vs `source-adrs:`.** The story offered `adr:` / `related-adrs:` as candidates; NA-55's OQ floated `source-adrs:`. _Suggested default (adopt):_ **`related-adrs:`** — distinct from the ADR-side `source-stories:` and the how-to `source:`, reads plainly, and its path-valued list reuses audit §22's existing inline-link resolution with zero new resolution logic. Change requires only pinning one word in §22.
- [ ] **`related-adrs:` value shape — full `docs/adr/NNNN-*.md` paths vs bare ADR numbers.** _Suggested default (adopt):_ **full repo-root-relative paths** — audit §22 already resolves ADR references by path, so paths need no number→file mapping; bare numbers would add a resolution step and a new failure mode.
- [ ] **Scalar placeholder form — bracket-free `TODO —` vs `[…]`/`<…>`.** _Suggested default (adopt):_ **bracket-free `TODO — …`** for `title:`/`description:`. `[x]` is a YAML flow-sequence (parses as a list, not a scalar) and would garble §8's regen if it leaked unfilled; `<…>` is not YAML-reserved but is still a placeholder that should not reach output. `related-adrs: []` stays an intentional empty list. Additionally **add a cheap confirm-gate check** that rejects a `title:`/`description:` still holding a `TODO —` value (small, prevents an unfilled placeholder reaching published `llms.txt`).
- [ ] **Migration / backfill of legacy pages (AC4).** Ship a one-shot pass that backfills `title`/`description`/`related-adrs:` into already-written pages, or leave the skip path for legacy pages? _Suggested default (adopt):_ **no backfill** — `description:` is un-derivable authored copy (a backfill would fabricate or demand bulk authoring), the skip-and-surface path already handles legacy pages non-destructively, and audit's surfaced list is a ready opt-in worklist. Revisit only if a founder wants a guided one-shot backfill of a specific tree.
- [ ] **Populate `related-adrs:` for release-notes from §12's resolved ADRs?** Release-notes already link their motivating ADR in body prose (§12); should release-mode also write those ADR paths into the `related-adrs:` frontmatter key so the flag has a frontmatter arm there too? _Suggested default (adopt):_ **no** — the inline-link arm already detects drift on the body links (OR-semantics), so the key adds no detection, only churn to NA-53's shipped release contract. Revisit if key-based (rather than link-based) audit reporting is later wanted.
