# writing-docs templates emit title/description + ADR-reference frontmatter — Technical Spec

**Story:** NA-61 — writing-docs templates emit no description:/ADR-reference frontmatter — forces per-mode skip workarounds, blocks audit AC4
**Epic / Feature:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer (`docs/features/2026-07-17-public-docs-generation-sdlc-docs.md`)
**Relates:** NA-55 (audit mode — this story is the deferred template fix its AC4 carry-over names)
**Date:** 2026-07-18

## Overview

The four `writing-docs` Diátaxis templates emit no `description:` frontmatter (only the How-to template emits any frontmatter at all, and none emits `description:`), so every `/sdlc:docs` mode's `llms.txt` regen — which derives each page's `title — description — link` from frontmatter — carries a duplicated "skip-and-surface frontmatter-less page" workaround, and NA-55 audit's AC4 ADR-drift flag has no deterministic page→ADR join key. This story makes all four templates emit `title:` + `description:` + a standardised `related-adrs:` key at source, then re-scopes (not removes) the skip path and pins the ADR key audit already looks for.

## The frontmatter contract — what each template emits (AC1, AC2)

All four templates emit the same three keys by default. `title:` matches the page H1. `description:` is a one-line summary the author fills (never derived — deriving published index copy is the fabrication boundary the pipeline holds everywhere, §17). `related-adrs:` is the standardised ADR-reference key: a YAML list of repo-root-relative `docs/adr/NNNN-*.md` paths, **default `[]`** (empty = no ADR reference = never flagged by audit).

### Why `related-adrs:` (name + shape)

- **Name.** `related-adrs:` — distinct from the ADR-side `source-stories:` (ADR → story) and from the how-to `source:` (page → code globs). No collision, reads plainly as "the ADRs this page relates to."
- **Shape = full `docs/adr/NNNN-*.md` paths, not bare numbers.** Audit §22 already resolves the inline-link form (`docs/adr/NNNN-*`) by path; a path-valued list reuses that resolution **verbatim** — audit needs no new number→file mapping. This is the minimal-churn shape (crux 4).
- **Default empty (`[]`), flow-style.** Most pages reference no ADR; an empty flow-style list adds no 2-space-indented list item, so it dodges the NA-62 landmine (below). Audit "never attributes an ADR to a page with no declared reference" — an empty list is exactly "no declared reference."
- **Explanation quadrant is the primary consumer.** Its existing "Related decisions" prose section links ADRs for **readers**; `related-adrs:` is the machine-readable mirror **audit** reads. They coexist by design — one human section, one machine key — and the author keeps them in sync. This is not normative-prose duplication (the two serve different consumers); audit reads the frontmatter key and falls back to inline links for legacy pages (§22).

## ADR-reference key + audit AC4 — what audit does with the key (crux 4)

**NA-61 does not change audit's flag logic.** §22 was written anticipating this key — it already flags a page that declares "an inline link to `docs/adr/NNNN-*` **or an explicit ADR-reference frontmatter key**." NA-61's job at the audit layer is therefore documentation-only:

1. **Pin the key name.** §22 currently says "an explicit ADR-reference frontmatter key" generically; change to name it `related-adrs:` (path-valued list, resolved identically to the inline-link form).
2. **Update the Reliability caveat.** §22's caveat ("the current `writing-docs` templates emit no ADR-reference key, so this flag fires only for pages carrying an inline `docs/adr/NNNN-*` link by convention") becomes: template-produced pages now carry `related-adrs:`, so the flag is reliable for them; the inline-link path remains as the fallback for legacy/hand-authored pages that predate the key.
3. **Retire §24's OQ carry-over** for the ADR half: the follow-up story it names (NA-61) has landed; the key exists. The skip-and-surface half of §24 is handled by the re-scope below.

No new resolution code, no reverse-direction proposal, no write to `docs/adr/` — all the invariants §22 already states hold unchanged. Churn to shipped audit is minimal (prose in §22 + §24), which is the story's stated goal.

## Simplification safety — the skip stays universal (crux 2, AC3)

**Can the regen distinguish a template-produced page from a legacy page at regen time? No.** The `llms.txt` regen (§8) walks every enabled `public: yes` page in the tree and can observe only one fact per page: whether it **has** `title`/`description` frontmatter or not. There is no origin marker, no "produced by writing-docs vN" stamp, and adding one would not help retroactively (legacy pages already written could never carry it). A date/age heuristic would be exactly the ambient, non-deterministic input the pipeline forbids.

**Therefore the skip-and-surface code path stays universal** — it fires for **any** frontmatter-less page regardless of origin. AC3's "simplify the workaround" resolves to **"the skip now rarely fires (only for legacy/hand-authored pages)," not "remove the skip."** State this explicitly in the ref so no later edit deletes the skip wholesale and silently breaks legacy pages (Standing lesson: never collapse distinct cases — a template page and a legacy page stay distinct; the skip is what keeps the legacy case safe).

**What actually gets simplified is the duplicated _rationale_, not the rule.** The rule itself already lives in one normative home (§19) and is referenced — not recopied — by §21 ("§19's, verbatim"). The genuine duplication is the **explanatory claim** "the templates emit no frontmatter / three of four emit no block," restated as live justification in §17, §22's caveat, and §24. That claim is now **factually stale** and must be corrected in each live ref to: *template-produced pages carry frontmatter; the skip remains for legacy/hand-authored pages that predate it.* This removes the duplicated stale prose (the class the epic objects to) while preserving the one skip path all four modes share.

**Scope fence:** only the **live contract refs** (`docs-pipeline.md`, `SKILL.md`, `doc-types.md`) are edited. The shipped NA-53/54/55/56 **spec docs** are point-in-time records — not rewritten (see Out of Scope).

## Migration decision — no automated backfill (crux 3, AC4)

**Decision: do NOT run a backfill pass. The skip path stays for legacy pages; audit's surface list is the backfill worklist.** Rationale:

- A backfill pass mutating every already-written page is a big hammer that touches **published** docs, and `description:` cannot be machine-generated (it is authored, one-line human copy) — so a backfill would either fabricate index copy (forbidden) or demand bulk authoring, exactly the churn AC4 asks us to weigh against.
- The skip-and-surface path already handles legacy pages gracefully: they are **surfaced** (phase-2 output + PR body), never silently dropped. That surfaced list **is** a ready-made, opt-in worklist — a founder backfills the two lines on a legacy page the next time a run surfaces it, or when they next touch it.
- The docs pipeline is generic and runs per consumer repo; a mandated backfill would fan out across every consumer's published tree. Even less reason to force it centrally.

Flagged as an Open Question with the tradeoff, per the story's instruction, in case the founder wants a one-shot opt-in backfill later.

## NA-62 Prettier-safe frontmatter blocks

Every frontmatter block authored above lives inside a fenced code block in SKILL.md. The NA-62 landmine is a **2-space-indented fenced block with an embedded blank line under a list item**; the writing-docs templates are top-level `###`-section fences (not under list items), which lowers but does not eliminate risk. The implementer must, per block:

| Block                          | Landmine check                                                                                                                                                                                 |
| ------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Tutorial / Reference / Explanation | New blocks — no indented list item (`related-adrs: []` is flow-style). Confirm the block is flush-left inside its fence and carries **no embedded blank line**. Reference's outer fence is **four backticks** — keep it four. |
| How-to                         | Retains the existing 2-space-indented `source:` list. Do **not** introduce a blank line between `source:`'s list item and the new `related-adrs:` comment lines. Keep `related-adrs: []` flow-style (no new indented `-`). |
| All four                       | After editing, run the repo formatter and confirm it is **idempotent** on SKILL.md (a second run produces no diff) before committing — the NA-62 dedent/idempotency gate. Comment lines (`#`) inside the YAML must survive the formatter unchanged. |

## Error Handling

| Scenario                                                                        | Behaviour                                                                                                                                                            |
| ------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| A **template-produced** page reaches `llms.txt` regen                            | Carries `title` + `description` by construction → included in `llms.txt`. The skip does not fire. This is the whole point.                                            |
| A **legacy/hand-authored** page lacking `title`/`description` reaches the regen  | **Skip-and-surface, unchanged** (§19): omitted from `llms.txt`, path surfaced in phase-2 output + PR body. Never a STOP, never inferred, never fabricated.            |
| A page declares `related-adrs:` whose referenced ADR has newer commits           | **Audit flags** it (§22 AC4) — "the doc diverges from the ADR"; direction fixed, ADR is source of truth. `related-adrs:` resolved identically to an inline ADR link. |
| A page has `related-adrs: []` (or omits it)                                       | No ADR association — audit **never attributes** an ADR to it (§22). Not flagged. Not an error.                                                                       |
| Author leaves `description:` as the `[...]` placeholder                           | Out of this story's mechanical scope — same class as any unfilled template placeholder; the seed founder-confirm gate (§17) and PR review catch it. Not a new guard here. |
| `related-adrs:` lists a `docs/adr/NNNN-*.md` path that does not exist             | Audit's **dangling-reference** check (§22) already flags a non-existent referenced path — reused, no new rule.                                                       |
| The repo formatter corrupts a frontmatter block on write                         | Caught by the NA-62 idempotency check (above) **before** commit — a non-idempotent formatter pass blocks the commit.                                                 |

## Out of Scope

- **Any new `/sdlc:docs` mode or mode-contract change.** The mode set (`sync`/`release`/`seed`/`audit`/`distill`) is untouched; no mode's algorithm changes — only the prose that describes the now-closed template gap.
- **Removing the skip-and-surface path.** Explicitly rejected — template vs legacy pages are indistinguishable at regen time, so the skip stays universal (it merely fires rarely now). See [Simplification safety](#simplification-safety--the-skip-stays-universal-crux-2-ac3).
- **An automated frontmatter backfill of already-written pages.** Rejected as the default; the skip path + audit's surface list cover legacy pages. See [Migration decision](#migration-decision--no-automated-backfill-crux-3-ac4) and the Open Question.
- **Rewriting the shipped NA-53/54/55/56 spec docs.** They are point-in-time records; the live contract is `docs-pipeline.md` + `SKILL.md`. Only live refs are edited.
- **Changing §12's release-mode frontmatter block.** Release pages independently require `title`/`description` (§12); that requirement is now additionally template-backed but still correct — not re-specified here.
- **Adding a `related-adrs` presence check to any completion checklist.** The key is optional (empty default); a presence check would false-fire on every non-ADR page.
- **The NA-62 Prettier sweep beyond the blocks this story touches.** Only the four blocks authored here get the dedent + idempotency check; the repo-wide sweep is NA-62's.
- **Auto-generated reference rows** (`command-reference`, `agent-reference`, etc.). They are machine-generated with their own frontmatter derivation (§3) and do **not** use the `writing-docs` Reference template — unaffected by this change.

## Open Questions

- [ ] **ADR-reference key name — `related-adrs:` vs `adr:` vs `source-adrs:`.** The story offered `adr:` / `related-adrs:` as candidates; NA-55's OQ floated `source-adrs:`. _Suggested default (adopt):_ **`related-adrs:`** — distinct from the ADR-side `source-stories:` and the how-to `source:`, reads plainly, and its path-valued list reuses audit §22's existing inline-link resolution with zero new logic. Change requires only pinning one word in §22.
- [ ] **`related-adrs:` value shape — full `docs/adr/NNNN-*.md` paths vs bare ADR numbers.** _Suggested default (adopt):_ **full repo-root-relative paths** — audit §22 already resolves ADR references by path, so paths need no number→file mapping; bare numbers would add a resolution step and a new failure mode.
- [ ] **Migration / backfill of legacy pages (AC4).** Ship a one-shot pass that backfills `title`/`description`/`related-adrs:` into already-written pages, or leave the skip path for legacy pages? _Suggested default (adopt):_ **no backfill** — `description:` is un-derivable authored copy (a backfill would fabricate or demand bulk authoring), the skip-and-surface path already handles legacy pages non-destructively, and audit's surfaced list is a ready opt-in worklist. Revisit only if a founder wants a guided one-shot backfill of a specific tree.
- [ ] **Placeholder style — `[...]` vs `<...>`.** The four blocks use `[...]` to match the existing How-to template's convention inside SKILL.md; §17 of the pipeline ref uses `<...>`. _Suggested default (adopt):_ **`[...]`** for intra-SKILL.md consistency; the pipeline ref keeps its own `<...>` convention (different file, different audience). No cross-file normalisation in this story.
