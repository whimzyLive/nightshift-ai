# Prettier non-idempotency gate for plugin docs — Technical Spec

**Story:** NA-62 — Prettier 3.6.2 non-idempotency corrupts indented fenced command blocks in plugin docs — CI affected-only gate misses it
**Feature:** — (systemic-fix story, no parent Epic)
**Date:** 2026-07-18

## Overview

Prettier 3.6.2 silently **corrupts** a common markdown pattern in the plugin docs — it is **non-idempotent** on certain fenced code blocks, so the committed form is not a Prettier fixed point and the next `prettier --write` (a pre-commit hook, an editor-on-save, or any PR that happens to touch the file) shatters runnable command blocks into broken four-backtick fences with unterminated shell quotes. CI misses it because the format gate (`ci.yml` line 34) is **affected-only** (`--base`-scoped), so a latent-corrupt file that is not in the current diff passes untouched. This story makes each vulnerable block a genuine Prettier fixed point and adds an **all-files idempotency gate** over `plugins/**/*.md` so the class stops recurring.

## Scope of this story (authoring surface)

This is a **plugin-authoring + CI-infrastructure** story spanning **two owners**. It adds no product code, no database entity, no HTTP surface. It ships (1) a reformat sweep of the vulnerable blocks under `plugins/**`, (2) a new idempotency-check script alongside the existing `check-agent-skill-preloads.sh`, and (3) the `.github/workflows/ci.yml` wiring that invokes it — the last being CI infrastructure, not a `plugins/` artifact.

## The four crux decisions

### 1. Detection method — write-on-copy idempotency check (not `prettier --check`)

**The invariant the gate enforces:** every committed `plugins/**/*.md` file is already a **Prettier fixed point** — `format(file) == file`. When that holds, no later `prettier --write` (pre-commit hook, editor-on-save, a PR touching the file) can silently change anything, so no command block can be shattered.

**The detector:** for each target file, copy it to an **in-tree, same-extension** scratch path, run `prettier --write` on the copy, and `diff` the written copy against the original. **Any non-empty diff = the file is not a fixed point = FAIL**, naming the file and printing the diff. Zero diff on every file = clean.

**Why a single write-on-copy pass is sufficient:** `format` is a function, so if `format(file) == file` the file is stable under every future `--write`. One pass proves (or disproves) the fixed-point invariant; a second pass would only re-confirm it.

**Why `prettier --check` alone is not specified, even though it is the same predicate:** a single write-on-copy pass fails on exactly the same files `prettier --check` reports (`format(X) != X`). Write-on-copy is chosen over a bare `--check` for three concrete reasons, each tied to a real failure this session hit:

- **It prints the corrupting diff** — the reviewer/implementer sees *what* Prettier would do (a benign reflow vs. the quad-fence shatter), which is exactly the triage signal the reformat sweep needs. `--check` only emits pass/fail per file.
- **It does not trust Prettier's exit code / stdout.** The RTK CLI-proxy wrapper active in this environment has been observed printing "All files formatted correctly" while still exiting 1 (agent memory, NA-57). The script diffs the bytes itself and never keys its verdict on Prettier's stdout or exit code.
- **It is the finder for the reformat sweep** (decision 2) — the same run that gates future regressions enumerates today's backlog.

**Which files:** all `plugins/**/*.md` (both `plugins/sdlc/` and `plugins/gtm/`, all owned by ai-enablement-engineer). The story AC names `plugins/sdlc/**`; widening to `plugins/**` is safe (same owner) and strictly more durable — see Open Question 2. **Not** repo-wide (see CI gate design for why an all-files-all-types gate would trip on out-of-scope non-plugin files and block this story).

**False-negative guard (NA-51 landmine):** `prettier --write` **silently no-ops** on any path it considers ignored or has no parser for — a scratch copy outside the repo tree, under a gitignored dir, or without a `.md` extension reports `{ "ignored": true }` or `{ "inferredParser": null }` and yields a false "IDEMPOTENT" with zero signal. The script MUST confirm, via `prettier --file-info <scratch>`, that the scratch path reports **both** `ignored: false` **and** a non-null `inferredParser` before trusting any diff from it. If it cannot, that is a hard FAIL, not a pass.

### 2. The reformat sweep — the detector is the finder

There is no separate audit step. **Run the detector; every file it flags is a vulnerable block to fix.** For each flagged file, reformat its offending block to a Prettier-stable fixed point that preserves meaning, then re-run the detector until it reports clean. The three documented stable-form techniques (all in agent memory), one per known variant:

- **NA-56 variant** (2-space-indented ` ```bash ` fence with an embedded blank line, nested under a list item): **dedent the fence to column 0**. Per the NA-57 caveat, also dedent any **trailing list-continuation prose after the fence** to column 0 — a column-0 fence terminates the list item, so a still-indented following paragraph is itself non-idempotent.
- **NA-61 variant** (a freestanding paragraph directly above a list, zero blank lines between): **insert a blank line** between the paragraph and the list.
- Fallback where dedent changes meaning: **remove the embedded blank line** inside the quoted block, whichever preserves the runnable command intact.

The detector both **fixes the backlog** (flags every current landmine for the sweep) and **gates future regressions** (fails CI the moment a new non-fixed-point file is committed). A clean detector run over `plugins/**/*.md` is the story's done condition for the sweep.

### 3. The CI gate change — add the all-files idempotency gate, keep `format:check`

**The hole:** `ci.yml` line 34 runs `pnpm nx format:check --base="remotes/origin/main"` — affected-only. A latent-corrupt file not in the current diff is never checked, so it ships green.

**The change:** add a new step invoking the idempotency script, mirroring line 36:

```yaml
- run: pnpm nx format:check --base="remotes/origin/main"   # unchanged
- run: pnpm nx run-many -t lint test build typecheck e2e
- run: bash plugins/sdlc/scripts/check-agent-skill-preloads.sh
- run: bash plugins/sdlc/scripts/check-prettier-idempotency.sh   # NEW — all plugins/**/*.md, every PR
```

**Replace or supplement — and does the idempotency gate subsume `format:check`?** A fixed-point check and a `--check` (formatted) check are the **same predicate per file**; the only real axes are (a) file-set scope and (b) affected-only vs. all-files. The hole is axis (b). Over the file set it covers, the all-files idempotency gate therefore **fully subsumes** an affected-only `format:check`. But the new script is scoped to `plugins/**/*.md`, whereas `format:check` covers **all file types repo-wide** (`.ts`, `.json`, non-plugin `.md`). So:

- **For `plugins/**/*.md`:** the new gate subsumes `format:check` — it checks the same predicate over all files, not just affected ones.
- **For everything else:** `format:check` is the only remaining coverage. Removing it would *reduce* coverage for non-plugin files.

**Decision: supplement, do not replace.** Keep line 34 (its residual value is affected-only formatting coverage of non-plugin files) and add the plugins-scoped all-files idempotency gate. The gate **must fail loudly** — non-zero exit, naming each corrupt file — never a warning (Standing lesson: a green gate must not imply safety it does not provide; a warn-only gate reintroduces exactly the silent pass this story exists to close).

**Why the new gate is scoped to `plugins/**/*.md` and not the whole repo:** an all-files-all-types gate turned on today would fail on any latent non-fixed-point file under `apps/marketing/`, `tools/`, `brand/`, etc. — other domains' files, whose corruption is **out of scope** for this story. Scoping the gate to `plugins/**` lets this story guarantee its own surface is clean (the sweep), turn the gate green, and ship — without depending on or editing another domain's files. The remaining affected-only hole for non-plugin files is a separate, per-domain concern (Out of Scope).

### 4. Config vs. reformat — reformat + gate; do NOT rely on a Prettier config change

Investigated whether a config setting makes these blocks idempotent, which would be a smaller, more durable fix than reformatting every block:

| Option                              | Effect on this corruption class                                                                                                                                                                                                                            | Verdict                      |
| ----------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------- |
| `proseWrap` (`preserve`/`never`/…)  | Governs prose line-wrapping, not fenced-code re-fencing. The corruption is remark **mis-parsing an indented fence with an embedded blank line under a list item** and re-emitting a four-backtick fence — not a wrap decision. No effect expected.         | Does not fix                 |
| `embeddedLanguageFormatting: "off"` | Disables formatting of the code **inside** embedded blocks. The shatter is a **markdown-structure** (backtick-count / list-boundary) rewrite by the outer parser, not embedded-language reformatting. Unlikely to change remark's fence parsing.           | Speculative; not relied upon |
| Pin/downgrade Prettier version      | Already pinned `~3.6.2`; the bug is in this version. Non-idempotency bugs are longstanding and span versions — no version is *provably* idempotent, and a version change does **not** fix the already-latent files.                                          | Speculative; does not fix backlog |

**Recommendation: reformat-all-blocks + the all-files idempotency gate. Do not adopt a config change as the fix.** Rationale — the reformat makes each block a genuine fixed point that is stable **regardless of config**, and the gate is **root-cause-agnostic**: it catches this variant, the NA-61 variant, and any *future* non-idempotency, because it enforces the observable invariant (committed == `format(committed)`) rather than betting Prettier is globally idempotent. A config setting, even if it made *these* blocks stable, would (a) not fix the existing latent files, and (b) not prove the repo has no other non-idempotent inputs. The gate + sweep are strictly more durable. Impl **may** empirically test `embeddedLanguageFormatting: "off"` as defense-in-depth (adopt only if it demonstrably makes the known blocks idempotent *and* churns no other files), but it must **never** substitute for the gate — see Open Question 3.

## Permissions Detail

Not a multi-role RBAC surface. The actors are the **CI runner** (executes the gate on every PR to `develop`) and the **founder** (reviews the two PRs). No role/permission keys. The one authority statement: the gate is **blocking** — a non-zero exit fails the `main` CI job, so a non-fixed-point `plugins/**/*.md` file cannot merge.

## Error Handling

`check-prettier-idempotency.sh` failure modes — every failure is a **loud, non-zero exit that names the offending file** (never a warning, never a silent no-op):

| Scenario                                                                                     | Behaviour                                                                                                                            | Exit |
| -------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | ---- |
| A `plugins/**/*.md` file is not a Prettier fixed point (write-on-copy diff non-empty)         | Print the file path **and the diff**; mark as offender; continue scanning the rest (report all offenders, not just the first).      | 1    |
| Scratch copy reports `ignored: true` or `inferredParser: null` (NA-51 false-negative trap)    | **Hard FAIL** — a silent `--write` no-op would fake an "IDEMPOTENT" pass; treat an unusable scratch path as a checker failure, not a pass. | 1    |
| No `plugins/**/*.md` files found (misconfigured glob / wrong dir)                              | **Hard FAIL** — mirrors `check-agent-skill-preloads.sh`'s empty-file-set guard; a vacuously-green gate is worse than none.          | 1    |
| Prettier / `node_modules` unavailable (`ERR_PNPM_RECURSIVE_EXEC_FIRST_FAIL`)                  | **Hard FAIL** with an actionable message (run `pnpm install --frozen-lockfile` first). CI runs install at line 31, so this is a local-run guard. | 1    |
| Every `plugins/**/*.md` file is a fixed point                                                 | Print "OK"                                                                                                                          | 0    |

**Local-dev note (RTK wrapper):** the script diffs bytes itself and never keys its verdict on Prettier's exit code or stdout, so the observed RTK-proxy "prints success while exiting 1" artifact cannot produce a wrong verdict. If a maintainer runs Prettier by hand to reproduce, `rtk proxy pnpm exec prettier …` gives the un-summarized output.

## Out of Scope

- **Fixing corruption in non-plugin markdown.** `docs/superpowers/` and `docs/features/` are **prettier-ignored** (specs/features are exempt). `apps/marketing/`, `tools/`, `brand/`, `packages/ui/` markdown belongs to **other agents' domains** (web-engineer / platform-engineer) — not touched here.
- **Closing the affected-only hole for non-plugin file types repo-wide.** Line 34 stays affected-only; widening it repo-wide risks tripping on out-of-scope non-plugin latent files and would block this story. A per-domain follow-up can address it.
- **Non-markdown idempotency.** The script is scoped to `*.md`; `.ts` / `.json` idempotency (if any) stays under `format:check`.
- **Changing or pinning a different Prettier version.** Pin stays `~3.6.2`; a version change is neither necessary nor sufficient (decision 4).
- **Adopting a Prettier config change as the fix.** Investigated and not adopted (decision 4); only permitted as optional, empirically-verified defense-in-depth, never as a substitute for the gate.
- **A pre-commit hook / local git hook.** The gate lives in CI (the enforcement point the story names); a local hook is a separate ergonomics enhancement.
