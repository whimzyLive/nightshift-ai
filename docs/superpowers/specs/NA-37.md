# Team org-chart page — Technical Spec

**Story:** NA-37 — Build the team org-chart page
**Epic:** NA-29 — Nightshift marketing site — design-handoff build
**Design source:** `docs/design/marketing-site-handoff/designs/team.dc.html` (authoritative) · screenshots `docs/design/marketing-site-handoff/screenshots/0*-team.png`
**Date:** 2026-07-12

## Overview

Build the standalone `/team` marketing page as a pixel-faithful recreation of the committed design handoff: a hero, a 4-card philosophy strip, a department org chart of static agent profile cards, an enforced-handbook terminal, and an install CTA. Agent roster and model-tier facts are static in code (sourced from the real agent frontmatter under `plugins/sdlc/agents/`) — no CMS, no DB, no API for this page.

## Scope classification

This story is **web-only** (owner: `web-engineer`; workspaces `apps/marketing/`, `packages/ui/` per project-context). The following spec sections are **omitted as not applicable** and the reason is stated for each:

- **Data Model** — omitted: no persisted entity. Roster/tier/handbook content is static TypeScript in code per the Epic's CMS-vs-static split ("agent rosters, commands and install snippets stay static in code").
- **API Surface** — omitted: no new endpoints. The page performs zero server data fetches (contrast the `/why-sdlc` and `/faq` pages, which read Payload).
- **Backend Implementation** — omitted: no handler/infra change.
- **Mobile UI** — omitted: no mobile app in this workspace.
- **Offline Sync** — omitted: project has no offline-sync layer.

**Reused UI-kit primitives (`@nightshift-ai/ui`):** `Eyebrow` (section/hero eyebrows), `InstallSnippet` (two install lines), `CtaButton` (`variant="secondary" size="lg"` star button, `href` form → renders an anchor). Do not rebuild these.

**Model-tier badge — decision:** the design's badge is a 9.5px sharp mono pill whose border/text color is driven by tier (OPUS→terra, SONNET→indigo, HUMAN→green). The kit `Badge` is a fixed 12px pill with variant colors `accent`(terra)/`info`(indigo)/`success`(green) that map cleanly to the three tiers. **Suggested default:** render the tier badge as a small local element inside `AgentProfileCard` (matching the design's exact 9.5px size + `letter-spacing:.12em`), NOT the kit `Badge`, because the design size/weight differs; keep the tier→color mapping identical to `Badge`'s tokens so the palette stays consistent. This is a component-level choice left to implementation.

### Static data module

| File                                   | Exports                                                                                                                                                      |
| -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `components/team/roster-data.ts` (new) | `DEPARTMENTS: Department[]`, the philosophy cards array, and the handbook steps array — all transcribed verbatim from `team.dc.html`'s `renderVals().depts`. |

Keep this module **separate from** the existing `components/home/team-data.ts`: that file backs the Home terminal-tree widget (`ORG_LEVELS` phase model) and is explicitly out of scope here. The two data shapes differ (phase-tree vs department-roster) — do not overload one for both.

Fully-typed shapes (no `any`):

```typescript
export type ModelTier = 'human' | 'opus' | 'sonnet';

export interface AgentProfile {
  /** Two/three-letter mono initials, e.g. 'SA', 'YOU'. */
  ini: string;
  /** Agent file stem / roster name, e.g. 'solutions-architect'. */
  name: string;
  /** Human job title, e.g. 'Solutions Architect'. */
  title: string;
  /** Model tier — drives the badge label + color. */
  tier: ModelTier;
  /** Optional badge text override (unused today; keep for parity with design). */
  badgeOverride?: string;
  /** Charter summary shown as the card body. */
  owns: string;
  /** Sequence/handoff line, e.g. 'after: scrum-master · hands to: tech-lead'. */
  flow: string;
  /** Artifact produced, e.g. 'docs/superpowers/specs/<KEY>.md'. */
  artifact: string;
  /** One-liner personality fact. */
  fact: string;
  /**
   * Agent charter file name under plugins/sdlc/agents/, e.g.
   * 'solutions-architect.md'. `null` for the human (no charter file).
   */
  file: string | null;
}

export interface Department {
  /** Mono eyebrow, e.g. '// product'. */
  eyebrow: string;
  /** Display title, e.g. 'Product'. */
  title: string;
  /** Cluster description line. */
  blurb: string;
  members: AgentProfile[];
}
```

**Charter link derivation:** `charter ↗` href = `https://github.com/whimzyLive/nightshift-ai/blob/main/plugins/sdlc/agents/{file}` when `file` is non-null; the `YOU` card has `file: null` and renders no charter link.

**Roster content (verbatim from the design + verified against real frontmatter):** transcribe all `owns`/`flow`/`artifact`/`fact`/`title` strings exactly from `team.dc.html`. The department set and membership is:

| Department (`title`)    | `eyebrow`                    | Members (`name` · `tier`)                                                                                                                                                   |
| ----------------------- | ---------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Leadership _(YOU card)_ | `// leadership`              | you · `human`                                                                                                                                                               |
| Product                 | `// product`                 | product-manager · `opus`, scrum-master · `sonnet`                                                                                                                           |
| Architecture & Planning | `// architecture + planning` | solutions-architect · `opus`, tech-lead · `opus`                                                                                                                            |
| Engineering             | `// engineering`             | principal-engineer · `opus`, database-administrator · `sonnet`, platform-engineer · `sonnet`, sync-engineer · `sonnet`, web-engineer · `sonnet`, mobile-engineer · `sonnet` |
| Quality                 | `// quality`                 | qa-engineer · `opus`                                                                                                                                                        |
| On contract             | `// opt-in specialist`       | ai-enablement-engineer · `sonnet`                                                                                                                                           |

The five clusters rendered under the spine (per AC2) are all departments **except Leadership** — the `YOU` card renders as the standalone centered card above the clusters. **Tier facts are authoritative:** every `tier` above was verified against the real `model:` frontmatter in `plugins/sdlc/agents/*.md` (opus: product-manager, solutions-architect, tech-lead, principal-engineer, qa-engineer; sonnet: all others; human: you). The badge text maps `human→HUMAN`, `opus→OPUS`, `sonnet→SONNET`.

## Design-fidelity requirements (AC-driven)

- **AC1 — Hero + philosophy:** `TeamHero` renders headline + stats line; `PhilosophyStrip` renders all 4 cards with the approved copy, verbatim.
- **AC2 — Org chart structure:** the YOU card, the centered dashed spine (`1px dashed rgba(217,119,87,.4)`), and all five department clusters with their pill eyebrow labels render.
- **AC3 — Profile card completeness:** every card shows mono initials, name, human title, tier-colored model badge sourced from real frontmatter, charter summary, `→ artifact`, the one-liner fact, and a working `charter ↗` link to the agent's `.md` under `plugins/sdlc/agents/`.
- **AC4 — Handbook terminal:** numbered steps 1–6 render, including step 3's project-override path `.claude/project/agents/<you>.md`.
- **AC5 — CTA:** "Hire the whole team in 60 seconds" heading, both install snippets, and a working star action render.
- **AC6 — Visual parity:** the built page must match `0*-team.png` screenshots side-by-side; QA captures Playwright screenshots and raises any deviation as a review finding. (Verification concern — belongs to the QA phase, not the build.)

**Styling constraints (inherited from the Epic / CLAUDE.md):** Tailwind v4 CSS-first utilities + inline `style` with design-token CSS variables; sharp edges (`border-radius: 0`, except the `50%` twinkle/dot circles); no CSS Modules / SCSS. Reference the token names already used across `home/` and `why-sdlc/` components (`--surface-card`, `--surface-terminal`, `--border-default`, `--border-accent`, `--bg-void`, `--moon-100`, `--text-muted`, `--text-dim`, `--accent`, `--terra-400`, `--indigo-400`, `--green-400`, `--cyan-400`, `--red-400`, `--link`) — do not introduce new tokens.

## Error Handling

| Scenario                                | Behaviour                                                                                                                                                                                                   |
| --------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Any runtime error (none expected)       | Static content only; no fetch to fail. There is no error/empty state — the roster is compiled in.                                                                                                           |
| An agent `file` renamed/removed in repo | `charter ↗` would 404 on GitHub. Mitigation: `roster-data.ts` `file` values must match actual `plugins/sdlc/agents/*.md` filenames (12 present today) — implementers verify against the directory listing. |
| `prefers-reduced-motion`                | Hover lift/transition and `CursorGlow` suppressed via `motion-safe:` / the reduced-motion guard.                                                                                                            |

**Offline behaviour:** not applicable (no writes, no offline layer).
