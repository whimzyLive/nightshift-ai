# On-Demand ISR for CMS Pages — Technical Spec

**Story:** [NA-70](https://whimzylive.atlassian.net/browse/NA-70) — Add on-demand ISR for CMS pages
**Epic:** [NA-29](https://whimzylive.atlassian.net/browse/NA-29) — Nightshift marketing site — design-handoff build
**Feature:** docs/features/2026-07-12-nightshift-marketing-site-design-handoff.md
**Date:** 2026-07-24

## Overview

Introduce a general-purpose On-Demand Incremental Static Regeneration (ISR) mechanism in `apps/marketing`: a new Payload `Pages` collection that is pre-rendered to static HTML at build time via a generic `[slug]` App Router route, plus a Payload `afterChange` hook that calls `revalidatePath` so a single edited page regenerates instantly on save while every unchanged page keeps serving pre-rendered static HTML. This story delivers the mechanism only — it does not migrate the four existing hand-built pages (Home, Why SDLC, Team, FAQ) into the collection.

## Data Model

### New Entities

New Payload collection `Pages` — slug `pages`. Registered in `apps/marketing/src/collections/Pages.ts` and added to the `collections` array of `apps/marketing/src/payload.config.ts` (currently `[Faq, Media, Users]` → `[Faq, Media, Pages, Users]`). Payload generates the underlying Postgres table and TypeScript types (`payload-types.ts`) — a Payload migration is generated under `apps/marketing/src/migrations/` by the standard `payload migrate:create` step, never a hand-written column DDL.

| Field    | Type                       | Required | Unique | Default | Notes                                                                                                                             |
| -------- | -------------------------- | -------- | ------ | ------- | ------------------------------------------------------------------------------------------------------------------------------- |
| id       | serial/uuid (Payload-managed) | —     | Yes    | —       | PK, generated by Payload/postgres adapter — do not declare manually                                                              |
| slug     | text                       | Yes      | Yes    | —       | URL path segment. Indexed + unique. `admin.useAsTitle`-adjacent; used by `[slug]` route and `generateStaticParams`. `home` is reserved for the root route `/` (see Revalidation Hook). |
| title    | text                       | Yes      | No     | —       | Human page title, `admin.useAsTitle`. Feeds `<title>`/metadata on the rendered page.                                            |
| content  | richText (lexical)         | Yes      | No     | —       | Page body, edited with the config's `lexicalEditor()`. Rendered server-side via `@payloadcms/richtext-lexical/react` `<RichText>`. |
| _status  | select `draft`\|`published` | —       | No     | `draft` | Provided by Payload `versions: { drafts: true }` — not hand-declared. Read path serves `published` only (preview is out of scope). |

Field-level `access` mirrors the existing `Faq` collection convention: `read: () => true`, and `create`/`update`/`delete` gated on `Boolean(user)`.

### Modified Entities

| Entity                       | Change                                                                     |
| ---------------------------- | -------------------------------------------------------------------------- |
| `payload.config.ts` collections | Add `Pages` to the `collections` array (import from `./collections/Pages`) |

No changes to `Faq`, `Media`, `Users`, or the `WhySdlc` global. `Pages` is fully additive and independent of them.

### Relationships

- `Pages` has **no relationships** to other collections. It is a standalone content collection.
- Cardinality within the site: one `Pages` document (1) → (1) rendered route `/{slug}` (except the reserved `home` slug, which maps to `/`).

## Route Surface

New generic dynamic route: `apps/marketing/src/app/(frontend)/[slug]/page.tsx` (a server component). It lives inside the existing `(frontend)` route group alongside the hand-built `faq/`, `team/`, `why-sdlc/`, and root `page.tsx` routes. Next.js resolves an explicit static segment (e.g. `/faq`) ahead of the dynamic `[slug]` segment, so the four existing routes are unaffected by this addition.

### Behaviour contract (from ACs)

- **Direct Local API only.** Data is read inside the server component via Payload's Direct Local API — `import config from '@payload-config'; import { getPayload } from 'payload'; const payload = await getPayload({ config });` then `payload.find({ collection: 'pages', where: { slug: { equals: params.slug }, _status: { equals: 'published' } }, limit: 1, depth: 0 })`. Never an HTTP `fetch` to the CMS.
- **`generateStaticParams()`** queries the `pages` collection with the projection `select: { slug: true }` and maps results to the params matrix `[{ slug: 'example' }, …]`, so every published page is pre-rendered as static HTML at build. It uses the same Direct Local API client. It MUST be resilient to an unmigrated/unavailable DB at build time (the existing `(frontend)` routes deliberately use `dynamic = 'force-dynamic'` for exactly this reason — NA-16 memory): wrap the read in try/catch and return `[]` on failure, letting `dynamicParams` cover first-hit rendering.
- **`export const dynamicParams = true`** — a brand-new CMS page (not present in the build-time params matrix) compiles and serves statically on its first visitor hit instead of 404ing.
- **Static-path isolation guarantee.** On the default (non-preview) request path the route MUST NOT read `cookies()`, `headers()`, or `searchParams`, and MUST NOT set `dynamic = 'force-dynamic'`. Any such dynamic API opts the whole route out of the static cache and defeats ISR. (This is the deliberate difference from the four existing force-dynamic routes.)
- **Not-found handling.** When no published page matches `params.slug`, the route calls Next.js `notFound()` (404).

## Revalidation Hook

New standalone module `apps/marketing/src/hooks/revalidatePage.ts` exporting a `CollectionAfterChangeHook` (typed from `payload`). Behaviour:

- **Slug → path mapping.** A pure helper maps a slug to its route path: the reserved slug `home` → `/`; every other slug → `/${slug}`. This mapping is the single source of truth shared by the current-doc and previous-doc revalidation calls.
- **Current-doc revalidation.** Calls Next.js `revalidatePath(slugToPath(doc.slug))` so the saved page serves updated content on its next request without a rebuild.
- **Previous-doc slug-change delta.** When `previousDoc?.slug` exists and differs from `doc.slug` (an editor changed a page's URL), it also calls `revalidatePath(slugToPath(previousDoc.slug))` against the historical path so the old route stops serving the now-dead static snapshot.
- **Async / non-blocking.** The hook returns `doc` immediately; the `revalidatePath` work must not block or measurably delay the admin save/publish response (fire the revalidation without `await`-ing it in the critical save path, per the async-non-blocking AC). Any error inside revalidation is caught and logged (`console.error`) — a revalidation failure must never fail the editor's save.
- `revalidatePath` is imported from `next/cache`.

## Collection Wiring

`apps/marketing/src/collections/Pages.ts` registers the hook in `hooks.afterChange`:

```ts
hooks: {
  afterChange: [revalidatePage],
}
```

Saving, publishing, or updating a `Pages` document from the Payload admin triggers the `afterChange` hook, which runs the revalidation described above.

## Web UI

**Pages:**

| Route     | File                                       | Description                                           |
| --------- | ------------------------------------------ | ----------------------------------------------------- |
| `/[slug]` | `src/app/(frontend)/[slug]/page.tsx`       | Generic CMS page — statically rendered + on-demand ISR |

**Components:** No new brand components required. The page body renders the `content` lexical field by reusing the existing server-side renderer `<RichText data={page.content} disableContainer />` from `@payloadcms/richtext-lexical/react` — the same component already used by `components/faq/faq-groups.tsx`, `components/home/faq-preview.tsx`, and `components/why-sdlc/argument-rail.tsx`. Page chrome (nav/footer/`NightSky`) is inherited from the `(frontend)/layout.tsx` already in place.

**Data hook / data layer:** Add `src/lib/pages.ts` following the existing `lib/faq.ts` / `lib/why-sdlc.ts` convention (`getPayload({ config })` + graceful empty/`null` fallback on any Payload/DB error):

```typescript
import type { Page } from '../payload-types'; // generated by Payload typegen after Pages is added

export interface PageContent {
  title: Page['title'];
  content: Page['content'];
}

// Returns null when no published page matches the slug (route then calls notFound()).
export async function getPageBySlug(slug: string): Promise<PageContent | null>;

// Published slugs for generateStaticParams; returns [] on any DB/build failure.
export async function getPublishedPageSlugs(): Promise<Array<{ slug: string }>>;
```

Both helpers use `depth: 0` and query `_status: { equals: 'published' }`. `getPublishedPageSlugs` applies the `select: { slug: true }` projection.

**State:** No client-state changes. The route and its render are server-only; the reused `<RichText>` output is static server-rendered markup.

**Route permissions:** Public, no auth — the rendered `/[slug]` pages are public marketing pages (`read: () => true` on the collection). Editing happens only in the Payload admin, gated by `Boolean(user)`.

## Permissions

The only mutating surface is the Payload admin acting on the `pages` collection; the public route is read-only static HTML.

| Role                         | Read page (`/[slug]`) | Create Page | Update Page | Delete Page |
| ---------------------------- | --------------------- | ----------- | ----------- | ----------- |
| Anonymous visitor            | ✓ (published only)    | ✗           | ✗           | ✗           |
| Authenticated admin (Payload `users`) | ✓            | ✓           | ✓           | ✓           |

`Pages.access` uses the existing collection convention: `read: () => true` (published filtering enforced in the read path/`_status`), `create`/`update`/`delete` require `Boolean(req.user)`.

## Error Handling

| Scenario                                              | Behaviour                                                                                       | Result       |
| ---------------------------------------------------- | ---------------------------------------------------------------------------------------------- | ------------ |
| No published `Pages` doc matches `params.slug`        | Route calls `notFound()`                                                                        | 404          |
| Slug collides with an existing static route (`faq`, `team`, `why-sdlc`) | Next.js serves the explicit static route; the `[slug]` route never matches that segment | Static page (see Out of Scope) |
| Payload/DB read failure inside `getPageBySlug`        | Caught, logged, returns `null` → route `notFound()` (graceful, no 500 stack to visitor)         | 404          |
| Payload/DB failure inside `generateStaticParams`      | Caught, logged, returns `[]`; `dynamicParams = true` renders pages on first hit                 | Build succeeds |
| `revalidatePath` throws inside the hook               | Caught + logged; editor save still succeeds                                                     | Save OK      |
| Editor changes a slug                                 | Both new and old paths revalidated (old snapshot invalidated)                                   | Both routes correct |

**Offline behaviour:** N/A — this project has no offline-sync layer.

## Out of Scope

- **Migrating the four existing hand-built pages** (`/`, `/why-sdlc`, `/team`, `/faq`) into the `Pages` collection. Those routes remain their current hand-built, `force-dynamic` server components. This story ships the ISR mechanism and the generic `[slug]` route only; moving existing content into `Pages` is a separate migration story. Consequently, a `Pages` doc created with slug `faq`/`team`/`why-sdlc` is shadowed by the explicit static route today, and the `home` → `/` mapping is forward-looking (the root `/` remains the hand-built force-dynamic home).
- Draft/preview-mode rendering of unpublished content (the `_status = published` filter and the no-`cookies()`/`headers()` isolation guarantee explicitly exclude a preview path).
- Revalidation hooks for any other collection (`Faq`, `Media`) — `Pages` only.
- Any manual/webhook-triggered revalidation endpoint outside the Payload `afterChange` hook.
- A block/layout page builder — v1 `content` is a single lexical `richText` field (see Open Questions).
- Redesign or restyling of any existing page.

## Open Questions

- [ ] **`content` field shape** — a single lexical `richText` field vs a Payload `blocks`/layout builder. **Suggested default: single `richText` field** for v1 (matches the existing FAQ `answer`/`homeAnswer` pattern and the reused `<RichText>` renderer); a blocks-based layout builder is a later enhancement.
- [ ] **Status mechanism** — Payload `versions: { drafts: true }` (native `_status`) vs a plain `select` status field. **Suggested default: `versions: { drafts: true }`**, with the read path filtering `_status = published`. It is Payload-native, keeps a clean published/draft split, and leaves room for a future preview story without a schema change; preview rendering itself stays out of scope.
- [ ] **Reserved-slug guard** — should the collection reject slugs that collide with the four existing static routes (`home`, `faq`, `team`, `why-sdlc`) to avoid silently-shadowed pages? **Suggested default: add a field-level `validate` on `slug` rejecting the four reserved values**, so editors get an immediate error instead of an invisible no-op page. Low cost, prevents a confusing failure mode.
