# Ship doc-type registry + docs-manifest scaffold — Technical Spec

**Story:** NA-51 — Ship doc-type registry + docs-manifest scaffold
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md (PRD, breadcrumb only — not read as product context)
**Date:** 2026-07-17

## Overview

Ship the plugin's generic Diataxis-based doc-type registry (`refs/doc-types.md`) plus an opt-in
`/sdlc:init` step that scaffolds `.claude/project/docs-manifest.md` pre-filled with the registry rows
relevant to the detected stack. This is the foundation the whole NA-50 Epic reads from — this story
ships the registry and the opt-in scaffold **only**, with zero generation logic and zero runtime
manifest consumption.

## Scope & Applicable Layers

This is a **plugin-authoring** story. No product database, API, web, mobile, or offline-sync surface
is touched — those spec sections are omitted as not applicable.

| Active agent | Involved? | Why |
| ------------ | --------- | --- |
| `ai-enablement-engineer` | **Yes** | Owns `plugins/` and `.agents/` — all writes land in `plugins/sdlc/` |
| web-engineer / platform-engineer / database-administrator / sync/mobile | No | No product-surface change |

All paths below are relative to the SDLC plugin source tree in this repo: `plugins/sdlc/`. The
consumer-repo artifact (`.claude/project/docs-manifest.md`) is written by `/sdlc:init` at the
consumer's repo root — never committed into the plugin.

## Artifact 1 — Doc-type registry (`refs/doc-types.md`)

### Purpose

A generic, stack-agnostic taxonomy of documentation types keyed on the Diataxis framework
(tutorials, how-to guides, reference, explanation). Each row is one doc type. The registry is
plugin-owned and versioned; consumer repos never edit it — they activate rows via their own
`docs-manifest.md` (Artifact 2).

### Registry row schema

Every row MUST carry exactly these columns, in this order:

| Column | Meaning | Allowed values |
| ------ | ------- | -------------- |
| `type` | Stable doc-type identifier (kebab-case) | e.g. `command-reference`, `how-to`, `tutorial`, `concept`, `changelog`, `adr`, `llms-txt` |
| `quadrant` | Diataxis quadrant | `tutorial` \| `how-to` \| `reference` \| `explanation` \| `meta` (`meta` for cross-cutting artifacts such as `llms.txt` and the ADR index that do not sit in a single quadrant) |
| `trigger` | The verb/moment that regenerates it (maps to a future `/sdlc:docs` mode) | `sync` \| `release` \| `seed` \| `audit` \| `distill` |
| `source-of-truth` | Where the content is derived from | e.g. `command/agent/skill frontmatter`, `merged stories since last tag`, `founder-authored`, `learnings corpus`, `accepted ADRs` |
| `generation-mode` | How much automation is allowed | `auto` \| `draft-for-review` \| `manual-only` |
| `target-path` | Default output path token (a placeholder the consumer manifest overrides) | e.g. `docs/reference/`, `docs/adr/`, `<repo-docs-root>/…` |
| `public` | Whether the type is ever published to public docs | `yes` \| `no` |
| `applies-when` | Stack condition gating whether the row is pre-filled into a consumer manifest by `/init` | `always`, or a condition string over the repo-detect output contract (see below) |

### Rows that MUST ship (from Epic NA-50)

The registry MUST include at least the following rows. `generation-mode` follows the Epic's
hallucination-control rule: frontmatter-driven reference docs and `llms.txt` are `auto`
(deterministic); narrative types are `draft-for-review`; founder-authored types are `manual-only`.

| type | quadrant | trigger | generation-mode | public | applies-when |
| ---- | -------- | ------- | --------------- | ------ | ------------ |
| `command-reference` | reference | sync | auto | yes | `always` |
| `agent-reference` | reference | sync | auto | yes | `always` |
| `skill-reference` | reference | sync | auto | yes | `always` |
| `config-reference` | reference | sync | auto | yes | `always` |
| `hooks-contract` | reference | sync | auto | yes | `always` |
| `error-reference` | reference | sync | auto | yes | `always` |
| `llms-txt` | meta | sync | auto | yes | `always` |
| `changelog` | reference | release | draft-for-review | yes | `always` |
| `release-notes` | explanation | release | draft-for-review | yes | `always` |
| `migration-guide` | how-to | release | draft-for-review | yes | `always` |
| `tutorial` | tutorial | seed | manual-only | yes | `always` |
| `how-to` | how-to | seed / sync | draft-for-review | yes | `always` |
| `integration-guide` | how-to | seed | draft-for-review | yes | `always` |
| `concept` | explanation | seed | draft-for-review | yes | `always` |
| `adr` | meta | seed / distill | draft-for-review | **no** | `always` |

Constraints the row set MUST satisfy (self-checkable):

- The `adr` row MUST have `public = no` and `target-path = docs/adr/` — it is an internal doc type,
  never published to public docs, per the Epic's "ADR as a registry row" decision.
- Every reference-quadrant row driven by frontmatter MUST have `generation-mode = auto`.
- Every narrative row (tutorial/how-to/concept/integration-guide/release-notes/migration-guide)
  MUST be `draft-for-review` or `manual-only` — never `auto`.
- The exact final column values for `source-of-truth` and `target-path` are the author's to set
  from the Epic text; the table above fixes the load-bearing values (`quadrant`, `trigger`,
  `generation-mode`, `public`) that later Epic stories depend on.

### `applies-when` condition grammar

`applies-when` is evaluated by `/init` (Artifact 2) against the **repo-detect output contract**
(`refs/repo-detect.md` — the seven canonical fields: `language`, `framework`, `package_manager`,
`test_runner`, `typecheck`, `runtime`, `commit_scopes`). Grammar:

- `always` — row is always pre-filled (all SDLC-artifact doc types are stack-agnostic and use this).
- `framework:<value>` — matches when `framework` contains `<value>`.
- `language:<value>` — matches when `language` contains `<value>`.
- `dep:<pkg>` — matches when `<pkg>` is a dependency in the repo (same semantics as
  `skills-map.yml`'s `dep:` condition).

For this story every shipped row uses `applies-when = always` (the doc types above are all
stack-agnostic). The grammar is specified now so future stack-specific rows (e.g. an
API-reference row gated on `framework:Hono`) slot in without a schema change. This keeps AC3
("registry rows relevant to the detected stack") satisfiable while the v1 row set is universal.

### File placement & conventions

- Lives at `plugins/sdlc/refs/doc-types.md`, alongside `adr-pipeline.md` and `skills-map.yml`.
- Mirrors the existing ref-file shape: a top comment/heading explaining the file is a data registry
  read by `/sdlc:init` and (later) `/sdlc:docs`, then the schema, then the row table. Copy the
  documentary style of `skills-map.yml`'s header. Do **not** abstract a shared registry format —
  the Epic explicitly says "copy the shape; do not abstract yet."

## Artifact 2 — `/sdlc:init` docs-manifest scaffold

### Where it slots into the init flow

`/sdlc:init` already has: Step 0 (re-init guard), Step 2.5 (repo scan → detected stack), Step 3
(collect config, incl. the `ai-enablement-engineer` opt-in confirm), Step 3.5 (skills picker),
Step 4 (write files: 4a–4f), Step 5 (checklist). The docs-manifest scaffold is a **new opt-in
prompt presented after stack detection completes** (AC2 says "after stack detection completes"),
and a **new write sub-step in Step 4**.

Concretely:

1. **New prompt — Step 3, docs opt-in (after stack detection / alongside the other Step 3
   confirms).** Present an `AskUserQuestion` binary confirm modeled on the existing AI-context
   opt-in (Step 3, lines ~304–345 of `init.md`):

   ```
   AskUserQuestion(
     header: "Docs generation",
     question: "Scaffold a docs-manifest so /sdlc:docs can generate public documentation for this repo?",
     multiSelect: false,
     options: [
       { label: "Opt in", description: "Write .claude/project/docs-manifest.md pre-filled with doc-type rows relevant to the detected stack. Activates docs generation." },
       { label: "Skip",   description: "Write no manifest. All docs features stay a silent no-op. Re-run /init later to opt in." }
     ]
   )
   ```

2. **New write sub-step — Step 4g, `.claude/project/docs-manifest.md`.** Only when the docs opt-in
   was accepted. Fill the manifest template (Artifact 3) with the subset of `refs/doc-types.md` rows
   whose `applies-when` matches the detected stack (all `always` rows for v1). Declining writes
   nothing (AC4).

### Manifest content written on opt-in (AC3)

The written `.claude/project/docs-manifest.md` is a per-repo activation file. Each pre-filled row
carries, at minimum: `type`, `trigger`, `generation-mode`, `public`, and an **editable
`target-path`** the founder points at their real docs tree (for this repo, the Mintlify `docs/`
MDX tree — but `/init` writes the registry default token and leaves retargeting to the founder; see
Epic Open Question on Mintlify mapping, out of scope here). The manifest MUST:

- Include one row per registry row whose `applies-when` matches (v1: all `always` rows).
- Carry a header comment stating it was scaffolded by `/sdlc:init`, that it activates
  `/sdlc:docs`, and that editing/removing rows deactivates those doc types.
- NOT include any generation logic, schedule, or runtime directive — it is declarative activation
  data only.

### Re-run / merge behaviour (AC5)

Re-running `/sdlc:init` on a repo that **already has** `.claude/project/docs-manifest.md` MUST use
the standard merge/confirm flow and never silently overwrite existing entries. This reuses the
existing Step 0 "Merge new findings" mechanic and the Step 4d `skills.json` merge pattern:

- On the **Merge new findings** path, treat `refs/doc-types.md` as the canonical row set. For every
  registry row whose `applies-when` matches the detected stack and which is **absent** from the
  existing manifest, offer to append it (prompted, consistent with the merge flow). Rows already in
  the manifest are kept **verbatim** — their edited `target-path` and any removals are preserved.
  Never remove or rewrite an existing manifest row.
- If the manifest does not exist yet on a re-run and the founder did not opt in this pass, write
  nothing (same as AC4).
- The "Keep existing" and "Refresh skills" Step 0 paths do not touch the manifest.

### Step 5 checklist update

Add `.claude/project/docs-manifest.md` (when opt-in accepted) to the "Files written" summary printed
in Step 5, so the post-init checklist reflects the new artifact.

## Artifact 3 — docs-manifest template

A canonical template for the written `.claude/project/docs-manifest.md`, so `/init` produces a
consistent, no-placeholder-remaining file. It defines: the header comment, the row table columns
(`type`, `trigger`, `generation-mode`, `public`, `target-path`, plus an `enabled` flag defaulting to
`true` so a founder can deactivate a row without deleting it), and the fill rules. See Open
Questions Q1 for whether this template lives as a section inside `refs/doc-types.md` or as a sibling
ref file.

## Permissions Detail

No product roles are involved (no runtime auth surface). The only actor is the founder/solo-dev
running `/sdlc:init` locally. Ownership/write-scope rules that DO apply:

| Actor / scope | Can | Cannot |
| ------------- | --- | ------ |
| `ai-enablement-engineer` (implementer) | Create `plugins/sdlc/refs/doc-types.md`; edit `plugins/sdlc/commands/init.md`; add the manifest template | Touch `.claude/project/project-context.md`, other agents' overrides, or any product workspace |
| `/sdlc:init` at consumer runtime | Write `.claude/project/docs-manifest.md` **only on explicit opt-in**; merge-append on re-run | Overwrite an existing manifest row; write the manifest when the founder declines |

## Error Handling

| Scenario | Behaviour |
| -------- | -------- |
| Founder declines the docs opt-in | Write no manifest file; all downstream docs features remain a silent no-op (AC4). No error. |
| Re-run `/init`, manifest already present, new registry rows exist | Offer to append missing rows via merge/confirm; keep existing rows verbatim (AC5). Never overwrite. |
| Re-run `/init`, manifest already present, no new rows | No-op on the manifest; print it as unchanged. |
| `refs/doc-types.md` unreadable/malformed at scaffold time | Surface the failure and skip the manifest write (do not write a half-filled manifest); consistent with `/init`'s "no placeholder tokens remain" rule. |
| Repo has a manifest but repo declined ai-enablement-engineer opt-in | Independent — the docs opt-in is a separate confirm from the AI-context opt-in; either may be accepted without the other. |

Validation rules for the registry (author self-check before commit):

- Every row has all schema columns populated (no blank cells).
- `generation-mode ∈ {auto, draft-for-review, manual-only}`; `public ∈ {yes, no}`;
  `quadrant ∈ {tutorial, how-to, reference, explanation, meta}`; `trigger` values are among the
  five `/sdlc:docs` modes.
- The `adr` row is `public = no`.

## Open Questions

- [ ] **Q1 — Manifest template location.** Should the docs-manifest template live as a dedicated
      "Manifest Template" section inside `refs/doc-types.md`, or as a separate `refs/docs-manifest-template.md`
      sibling? — **Suggested default:** a section inside `refs/doc-types.md`, keeping the registry and
      its consumer-manifest shape in one file (the Epic says do not over-abstract yet). Split it out
      only if a second consumer of the template appears.
- [ ] **Q2 — `enabled` column on manifest rows.** Include a per-row `enabled: true|false` flag so a
      founder can deactivate a doc type without deleting the row? — **Suggested default:** yes; include
      `enabled` defaulting to `true`. A deleted row and a disabled row read the same to a no-op-safe
      consumer, but keeping the row documents the available type.
- [ ] **Q3 — Which rows are plugin-default vs opt-in-only** (Epic Open Question). — **Suggested
      default:** ship every row in the table above as `applies-when = always` (all pre-filled on
      opt-in); defer any stack-gated rows until a later story introduces a stack-specific doc type.
