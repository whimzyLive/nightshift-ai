# Enforce project-skill loading in domain-agent dispatches — Technical Spec

**Story:** NA-26
**Epic:** NA-24 — SDLC Improvements
**Date:** 2026-07-11

## Overview

The 6 SDLC domain agents load project-specific skills via model-discretionary prose, so skill loading is non-deterministic (NA-22 transcripts: 0/9 skills on the most design-critical dispatch). This spec adds a contract-level enforcement loop — the dispatcher names the applicable override skills in the prompt, the agent declares which skills it loaded in its return, and the orchestrator verifies the returned set against the named set — without reverting the frontmatter workaround for #76337 (NA-23).

## Nature of this change

This is a **plugin-prose / contract** change. It touches only Markdown playbooks, agent profiles, and refs under `plugins/sdlc/`. There is **no** data model, API surface, web/mobile UI, or offline sync involved — those sections are omitted per the writing-specs applicability rule. The one structured artifact introduced is the **`Skills loaded:` return line** (defined below), which is a contract string, not a code type.

## The return-contract artifact — `Skills loaded:` line

The shared domain-agent return format in `plugins/sdlc/refs/domain-agent-handoff.md` gains a **required `Skills loaded:` line**, appended last. Because `Note:` is emitted only on a blocked return, the exact line count depends on status — stated explicitly to remove the "three vs four lines" ambiguity:

**Complete return — exactly 3 lines** (`Note:` omitted):

```
Status: complete
Summary: <one line — files changed, key entities/handlers/screens touched>
Skills loaded: <comma-separated override skill names | none>
```

**Blocked return — exactly 4 lines** (`Note:` present):

```
Status: blocked
Note: <one line — why blocked>
Summary: <one line — what was attempted>
Skills loaded: <comma-separated override skill names | none>
```

Field rules:

- **Value** is a comma-separated list of the exact project-skill names the agent invoked via the Skill tool this dispatch (e.g. `tailwind-design-system, react-components`).
- **`none`** is the literal value emitted when no applicable override skill was loaded. Whether `none` is a pass or a failure is decided mechanically against the dispatch prompt — see "`none` semantics" below.
- The line is **required on every return** (`complete` or `blocked`, including early aborts). An absent line is a contract violation.
- Only runtime-loaded **override** (project) skills are listed — not the generic frontmatter skills (`executing-plans`, etc.), since those are already harness-enforced and were not the NA-22 failure.

### `none` semantics — mechanical, keyed off the dispatch prompt

The dispatch prompt is **required to state exactly one of two things**: (a) the explicit list of applicable override skills to invoke, or (b) an explicit "no project skills apply for this task" declaration. Verification is then mechanical (no agent judgement):

- **Prompt named skills** `S = {s1…sn}`: the return **passes** iff `Skills loaded` is present and **every** skill in `S` appears in it. A missing line, an empty value, `none`, or any named skill absent → **failure**. (This closes the "loaded 1 of N passes" gap — partial loading fails.)
- **Prompt declared no applicable skills**: the return **passes** iff `Skills loaded: none` (extra skills the agent chose to load are tolerated). A **missing/absent** line is still a contract violation (the line is mandatory) → **failure**.

So `Skills loaded: none` passes **iff the dispatch prompt declared no applicable skills**; if the prompt named skills, `none`/missing/empty/partial is a failure.

## Error handling

US spelling used throughout ("behavior").

| Scenario                                                              | Path             | Behavior                                                                                       |
| --------------------------------------------------------------------- | ---------------- | ---------------------------------------------------------------------------------------------- |
| Dispatch prompt omits the skill-naming item / no-skills declaration   | prose/contract   | Contract defect — caught in review; the contract requires the prompt to state one or the other |
| Return omits `Skills loaded` line (any status)                        | principal Step 5 | Verification failure → STOP-and-redispatch the phase once; persistent → STOP and report        |
| Return omits `Skills loaded` line (any status)                        | QA Step 3        | Verification failure → return `blocked` immediately (no redispatch)                            |
| Prompt named skills; return misses ≥1 named skill (incl. `none`)      | principal Step 5 | Failure → redispatch once → then STOP                                                          |
| Prompt named skills; return misses ≥1 named skill (incl. `none`)      | QA Step 3        | Failure → `blocked` immediately                                                                |
| Prompt declared no applicable skills; return is `Skills loaded: none` | both             | **Pass** — `none` is correct                                                                   |
| Prompt declared no applicable skills; line absent                     | both             | Contract violation (line mandatory) → redispatch-once (principal) / `blocked` (QA)             |
| Redispatch still returns no valid `Skills loaded`                     | principal Step 5 | STOP and report to the user (same shape as the existing silent-failure STOP)                   |
| `Status: blocked` (unrelated to skills)                               | both             | Existing blocked handling unchanged; the `Skills loaded` line is still required on the return  |

There is no offline/disconnected behavior — this change is entirely orchestration prose.

## Out of Scope

- Reverting project skills back into agent frontmatter (would reintroduce the #76337 re-injection bug tracked as NA-23).
- Changing the runtime skill-loading mechanism of the pipeline agents. (Note: `principal-engineer.md`'s **dispatch-prompt contract** is in scope per file #4 — only its own skill-loading is exempt.)
- Harness-level interception or blocking of Skill tool calls — this story changes only plugin prose/contracts, not the Claude Code harness.
- Any change to the `gtm` plugin or its agents — scoped to `plugins/sdlc/` only.
- Auto-detecting which override skills are "relevant to the task" programmatically — the dispatcher (a human-role orchestrator) selects them from the override; no tooling is built for this.
