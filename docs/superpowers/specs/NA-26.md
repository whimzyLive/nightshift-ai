# Enforce project-skill loading in domain-agent dispatches — Technical Spec

**Story:** NA-26
**Epic:** NA-24 — SDLC Improvements
**Date:** 2026-07-11

## Overview

The 6 SDLC domain agents load project-specific skills via model-discretionary prose, so skill loading is non-deterministic (NA-22 transcripts: 0/9 skills on the most design-critical dispatch). This spec adds a contract-level enforcement loop — the dispatcher must name the applicable override skills in the prompt, the agent must declare which skills it loaded in its return, and the orchestrator must treat a missing declaration as a verification failure — without reverting the frontmatter workaround for #76337 (NA-23).

## Nature of this change

This is a **plugin-prose / contract** change. It touches only Markdown playbooks, agent profiles, and refs under `plugins/sdlc/`. There is **no** data model, API surface, web/mobile UI, or offline sync involved — those sections are omitted per the writing-specs applicability rule. The one structured artifact introduced is the **`Skills loaded:` return line** (defined below), which is a contract string, not a code type.

## The return-contract artifact — `Skills loaded:` line

The shared domain-agent return format in `plugins/sdlc/refs/domain-agent-handoff.md` (currently 3 lines: `Status` / `Note` / `Summary`) gains a **fourth required line**. Contract shape (this is a return-string format, not a TypeScript type — no code interface applies):

```
Status: complete|blocked
Note: <one line if blocked, else omit>
Summary: <one line — files changed, key entities/handlers/screens touched>
Skills loaded: <comma-separated override skill names | none>
```

Field rules:

- **Value** is a comma-separated list of the exact project-skill names the agent invoked via the Skill tool this dispatch (e.g. `tailwind-design-system, react-components`).
- **`none`** is the literal value the agent MUST emit when its override lists **no** applicable project skills for the task (i.e. the override's `## Project skills` section is the "no skills configured" placeholder, or none apply). `none` is a valid, passing value — it is only a failure when the override _did_ list applicable skills.
- The line is **required on every return** (`complete` or `blocked`). An absent line is a contract violation.
- Generic frontmatter skills (`executing-plans`, etc.) are NOT listed — only the runtime-loaded **override** (project) skills, since those are the ones NA-22 showed were dropped.

## Contract changes per file — WHAT each file must say

### 1. `refs/domain-agent-handoff.md` — Return format (§ "Return format", ~lines 59-66)

- Extend the return block to 4 lines, adding `Skills loaded: <comma-separated override skill names | none>` after `Summary:`.
- Update the surrounding prose ("Return exactly three lines") to "exactly four lines".
- State that `Skills loaded` reports only the runtime override skills invoked via the Skill tool, and that `none` is required (not omission) when no override skill applied.

### 2. `refs/principal-engineer-playbook.md`

**Step 4 — Domain-agent prompt contract (§ "Domain-agent prompt contract", ~lines 303-327):**

- Add a new **mandatory numbered item** to the prompt contract: the dispatch prompt MUST name the specific project skills (drawn from the **target agent's override**, `.claude/project/agents/<agent>.md`, `## Project skills`) that are relevant to this task, and instruct: **"Invoke these via the Skill tool BEFORE starting Task 1."** When the agent's override lists no applicable skills, the prompt says so explicitly (so the agent knows `none` is the correct return).
- Amend the existing return-format item (item 9) to require the 4-line return including `Skills loaded:`.

**Step 5 — Phase completion verification (§ "Step 5", ~lines 330-351):**

- Add a verification rule: after extracting the agent return, if the dispatched agent's override lists applicable skills for that task **and** the return has a **missing or empty `Skills loaded` line** (or the value is `none` despite applicable skills existing), treat it as a **verification failure → STOP-and-redispatch** that phase once with the skill instruction re-emphasized. A persistent failure on redispatch → STOP and report (same shape as the existing silent-failure STOP).
- A `Skills loaded: none` return is a **pass** when the override genuinely lists no applicable skills — it is only a failure when applicable skills existed.

### 3. `refs/qa-engineer-playbook.md`

**Step 3 — Fix loop prompt (§ "Step 3", ~lines 209-233):**

- Add the same mandatory skill-naming item to the fix-dispatch prompt (item between current 2 and 6): name the target agent's applicable override skills and instruct "invoke via the Skill tool BEFORE fixing."
- Amend the return-format item (item 6) to require the 4-line return including `Skills loaded:`.
- Add to the post-dispatch check (the "After the agent returns" block): a missing/empty `Skills loaded` line, when the fixing agent's override lists applicable skills, is treated the same as the existing silent-failure condition → `blocked`/redispatch, consistent with Step 5 of the principal playbook.

### 4-9. The 6 domain-agent profiles

For each of `web-engineer`, `platform-engineer`, `database-administrator`, `sync-engineer`, `mobile-engineer`, `ai-enablement-engineer`:

- **First steps §2** (the "Read your override … invoke each project skill it lists" step): rewrite from descriptive to **imperative-with-checkpoint**, e.g. "Invoke every applicable project skill your override lists via the Skill tool. **Do not begin Task 1 until every applicable override skill is invoked.** List the skills you loaded in your return's `Skills loaded:` line (`none` if your override lists no applicable skills)."
- **`## Skills` section**: reinforce the same checkpoint — loading override skills at runtime is a gate on starting work, and the loaded set MUST be declared in the return.
- **Return** (via the shared handoff ref): each profile already delegates its return to `refs/domain-agent-handoff.md`, so the new `Skills loaded` line flows in automatically once file #1 changes — no per-profile return block to edit, but each profile's completion/return prose must not contradict the 4-line contract.

## Error handling

| Scenario                                                                     | Behaviour                                                                                              |
| ---------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| Dispatch prompt omits the skill-naming item                                  | Prose/contract defect — caught in review, not a runtime branch; the contract requires it               |
| Agent return omits `Skills loaded` line AND override lists applicable skills | Verification failure → STOP-and-redispatch the phase once (principal Step 5) / `blocked` (QA Step 3)   |
| Agent returns `Skills loaded: none` AND override lists applicable skills     | Treated as skills-not-loaded → same verification failure                                               |
| Agent returns `Skills loaded: none` AND override lists no applicable skills  | **Pass** — `none` is the correct value; no failure                                                     |
| Redispatch still returns no `Skills loaded` line                             | STOP and report to the user (same shape as the existing silent-failure STOP) — do not silently proceed |
| Agent returns `Status: blocked`                                              | Existing blocked handling is unchanged; the `Skills loaded` line is still required on a blocked return |

There is no offline/disconnected behavior — this change is entirely orchestration prose.

## Out of Scope

- Reverting project skills back into agent frontmatter (would reintroduce the #76337 re-injection bug tracked as NA-23).
- Changing the 6 pipeline agents (`principal-engineer`, `product-manager`, `qa-engineer`, `scrum-master`, `solutions-architect`, `tech-lead`) — their skills are already frontmatter-injected and harness-enforced.
- Harness-level interception or blocking of Skill tool calls — this story changes only plugin prose/contracts, not the Claude Code harness.
- Any change to the `gtm` plugin or its agents — this story is scoped to `plugins/sdlc/` only.
- Auto-detecting which override skills are "relevant to the task" programmatically — the dispatcher (a human-role orchestrator) selects them from the override; no tooling is built for this.
