# /sdlc:docs release mode — Technical Spec

**Story:** NA-53 — Add /sdlc:docs release mode
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-17

## Overview

Add the second `/sdlc:docs` mode — `release <version>` — which aggregates every story merged since
the last tag into a changelog entry, ADR-linked release notes, and a headings-only migration-guide
stub, holds the narrative content at the founder-confirm gate, then regenerates the doc index +
`llms.txt` deterministically and opens a PR. It reuses the command shell, two-phase dispatch, and
`refs/docs-pipeline.md` contract shipped by NA-52 (the `sync` core), and the `changelog` /
`release-notes` / `migration-guide` registry rows already present in `refs/doc-types.md` (NA-51).

extended, not created. No new registry rows are added: `changelog`, `release-notes`, and

## Command Contract — `/sdlc:docs release <version>`

### Frontmatter `description` update

`docs.md`'s frontmatter `description` currently ends: "`release`/`seed`/`audit`/`distill` are
not-yet-implemented stubs (see Epic NA-50)". This story **must** update it — it is not cosmetic.
`refs/doc-types.md`'s `command-reference` row has `source-of-truth = command frontmatter + body`
and `generation-mode = auto`, so the very next `/sdlc:docs sync` regenerates
`docs/reference/commands/` **from this string** and publishes it into the public docs surface and
`llms.txt`. Leaving it stale would ship "release is not yet implemented" as public reference copy
for a shipped mode, and `/help` would advertise the same falsehood.

Required change: drop `release` from the stub list and state the mode it now ships — the
`description` must name **both** live modes (`sync` and `release`) and list only
`seed`/`audit`/`distill` as not-yet-implemented stubs. Mirrors the equivalent `description` update
this story makes to the agent (see [Agent Extension](#agent-extension--knowledge-engineermd)).

### Argument validation

Parse `$ARGUMENTS` into `<mode>` (first token) and the mode's remaining args. Modify the existing
validation ladder in `docs.md` so that:

1. **Empty `$ARGUMENTS`** → STOP with the usage message (extended to advertise `release`):
   `usage: /sdlc:docs sync <STORY-KEY> | release <version>  (seed|audit|distill land in later stories)`.
2. **Unrecognised first token** → STOP with the same usage message.
3. **Recognised future-mode token** (`seed`/`audit`/`distill`) → print
   `mode "<mode>" is not yet implemented (see Epic NA-50)` and exit cleanly. `release` is **removed**
   from this stub set — it now routes to the release contract below.
4. **`release` with a missing or empty `<version>`** → STOP with the usage message. This is a
   **usage STOP** (a caller error), distinct from the manifest-absent silent no-op and the
   no-stories-merged clean no-op below.
5. **`release` `<version>` — hard-validate, then normalise.** `<version>` is **not** free text: it
   becomes a git branch name (`docs/release-<VERSION>`), page ids
   (`docs/release-notes/<VERSION>.md`, `docs/migration-guides/<VERSION>.md`), and the changelog
   `## <VERSION>` heading that the [upsert rule](#changelog-upsert-rule-cumulative-file) keys on.
   It is validated exactly as `sync`'s story key is (`docs.md` validation step 4 hard-validates
   `^[A-Z][A-Z0-9]*-[0-9]+$` **because** it forms a branch/ref) — same discipline, different regex:

   ```text
   raw token must match:  ^v?[0-9A-Za-z][0-9A-Za-z.]*(-[0-9A-Za-z.]+)*$
   normalise:             strip a single leading "v"  →  VERSION
   VERSION must be non-empty, must NOT contain "..", and must not end in "." or "-"
   ```

   - **Fails the regex, or is empty / `..`-bearing after normalisation** → **usage STOP** with
     `invalid version "<raw>" — expected a token like 1.4.0 or v1.4.0 (letters, digits, ".", "-"; no path separators)`.
     This runs **before** any branch, dispatch, draft, or founder gate — a caller passing
     `release ../../oops` is rejected at the ladder, never after the founder has confirmed drafts.
     The regex admits no `/`, no whitespace, and no `..`, so `<VERSION>` can never escape the docs
     tree or produce a git ref-format violation.
   - **Normalisation is identity-forming, not cosmetic.** `1.4.0` and `v1.4.0` normalise to the
     **same** `VERSION` (`1.4.0`) and therefore the same branch, the same PR, the same changelog
     section, and the same release-notes / migration-guide pages. Without it the two spellings
     produce a duplicate branch, a duplicate PR (the re-run dedupe in
     [Branch / PR / Control Flow](#branch--pr--control-flow) keys on branch name), a duplicate
     changelog section, and duplicate pages. This is what makes the
     [re-run / idempotence contract](#re-run--idempotence-contract) well-defined.
   - **`VERSION` (normalised), not the raw token, is used everywhere downstream** — branch name, PR
     title, commit string, page ids, changelog heading, and the no-op notices. The raw token is
     echoed only in the invalid-version STOP message above.
   - Beyond this token-safety regex, `<VERSION>` is **not** hard-validated against semver: a
     date-based or otherwise non-semver label (e.g. `2026.07.1`) passes. See
     [Open Questions](#open-questions) for the (declined) strict-semver option.

### `release <version>` — behavioural contract

The release run mirrors `sync`'s two-phase dispatch and command-layer founder-confirm gate — the
shared skeleton lives in `refs/docs-pipeline.md` §2 and is not re-derived. What is release-specific:

1. **Manifest gate (AC6).** Resolve `.claude/project/docs-manifest.md` **checkout-independently**
   (via `git show origin/<BASE-BRANCH>:.claude/project/docs-manifest.md`, the same way `sync`
   resolves it — never the working tree). If **absent** → **silent no-op**: no branch, no dispatch,
   no PR, no error, **no stdout**, exit 0. This is the zero-setup-cost guarantee for repos that
   declined the `/init` docs opt-in, identical in discipline to `sync`'s AC5.

2. **Enabled-row gate — resolve `ENABLED_ROWS` once, then thread it everywhere.** Resolve, from the
   manifest, which of the three `release`-triggered registry rows (`changelog`, `release-notes`,
   `migration-guide`) are **enabled**; call that resolved subset `ENABLED_ROWS`.

   - If `ENABLED_ROWS` is **empty** → **clean no-op**: print a one-line informational notice
     (`no release-triggered doc types enabled in docs-manifest.md — nothing to generate`) and exit
     without a PR. (Informational, not silent — the manifest exists, so the founder opted in; a
     fully-disabled release surface is worth surfacing, mirroring `sync`'s "found nothing to do" vs
     "opted out" distinction.)
   - **`ENABLED_ROWS` is a per-row filter, not a one-time collective gate.** It is an explicit input
     to phase 1, to the founder-confirm gate, and to phase 2, and each of those three **must**
     re-consult it per row: phase 1 drafts **only** the enabled rows' artifacts, the gate presents
     **only** those drafts, and phase 2 writes **only** those pages. This is the release-mode
     instance of the invariant `refs/docs-pipeline.md` §2 step 4 / §3 already state for `sync` —
     regeneration "never touches an unaffected row's pages". A founder who disables `release-notes`
     or `migration-guide` in their manifest must never have those pages drafted, shown, or created.
     A run whose `ENABLED_ROWS` is only `{changelog}` produces exactly one artifact, one gate item,
     and one written page.

3. **Resolve the last tag and the merged-story set (AC1).** Diff source is the **base branch**, not
   a story branch — release aggregates work already merged to base. Every git failure here is a
   **STOP**, never a silent fallthrough into the "no tags yet" branch — the two must be
   distinguished by inspecting `git describe`'s stderr, because git exits `128` for *both* "no tags
   exist" and unrelated fatals (shallow clone, unresolvable ref, corrupt refs), so the exit code
   alone cannot tell them apart:

   ```bash
   git fetch origin --tags --quiet || STOP "git fetch failed"

   # The base ref must resolve BEFORE any describe — an unresolvable base is a STOP,
   # never a "no tags yet" first-release run.
   git rev-parse --verify --quiet "origin/<BASE-BRANCH>^{commit}" >/dev/null \
     || STOP "cannot resolve origin/<BASE-BRANCH>"

   DESCRIBE_ERR="$(mktemp)"
   if LAST_TAG=$(git describe --tags --abbrev=0 "origin/<BASE-BRANCH>" 2>"$DESCRIBE_ERR"); then
     RANGE="$LAST_TAG..origin/<BASE-BRANCH>"        # tagged: exclusive of the tag, inclusive of head
   elif grep -qE 'No names found|No tags can describe' "$DESCRIBE_ERR"; then
     LAST_TAG=""                                    # genuine "no tags yet" — first-ever release
     RANGE="origin/<BASE-BRANCH>"                   # SINGLE-ENDED: full history to head, root inclusive
   else
     STOP "git describe failed: $(cat "$DESCRIBE_ERR")"
   fi
   ```

   - **When `LAST_TAG` is set** → the commit range is `"$LAST_TAG..origin/<BASE-BRANCH>"`.
   - **When the repo has NO tags** (first-ever release) → the range is the **single-ended**
     `origin/<BASE-BRANCH>` — every commit reachable from base head, the root commit included.
     Deliberately **not** an empty range start: `"..origin/<BASE-BRANCH>"` would let git default the
     omitted left side to `HEAD`, which on an up-to-date base checkout yields an **empty** range →
     zero keys → a "no stories merged" no-op → the first-ever release silently shipping nothing
     while reporting success. Deliberately **not** `<root-sha>..origin/<BASE-BRANCH>` either — that
     form **excludes** the root commit itself. The single-ended form is the only one that is both
     non-empty and root-inclusive.
   - **Enumerate merged stories** from `RANGE`: extract the **distinct** set of Jira story keys
     (regex `[A-Z][A-Z0-9]*-[0-9]+`) from commit subjects **and** bodies across the range
     (`git log $RANGE --format='%s%n%b'`), de-duplicated. Alongside each key, retain the **commit
     subject** of the commit that carried it — that subject is the offline source for the
     changelog's change type and summary (see
     [Changelog aggregation inputs](#changelog-aggregation-inputs-offline-first)). Optionally enrich
     each key's human title from the merged PR that carried it (`gh pr list --search "…"` /
     `gh pr view`), but the git-log extraction is the **authoritative** source — `gh` enrichment is
     best-effort and never gates the run. This key set (with subjects) is what the enabled artifacts
     are built from.
   - **No stories merged in `RANGE` (AC6)** → **clean no-op**: print a one-line notice and exit
     without a PR. The notice interpolates `LAST_TAG` **only when it is set** — never emit an empty
     interpolation:
     - `LAST_TAG` set → `no stories merged since <LAST_TAG> — nothing to release`
     - `LAST_TAG` empty (no tags yet) → `no stories merged since the start of history — nothing to release`

4. **Phase 1 dispatch — `knowledge-engineer` computes & drafts, writes nothing.** Pass it
   `VERSION`, `ENABLED_ROWS`, the resolved commit range, the merged-story key set (with commit
   subjects and any `gh`-enriched titles), and `origin/<BASE-BRANCH>`. Phase 1 drafts **only the
   artifacts whose row is in `ENABLED_ROWS`** — each bullet below is conditional on its row:
   - if `changelog ∈ ENABLED_ROWS` → draft the **changelog** entry for `VERSION` (see
     [Narrative artifacts](#narrative-artifacts-changelog--release-notes--migration-guide));
   - if `release-notes ∈ ENABLED_ROWS` → draft the **release notes** for `VERSION`, resolving each
     story's motivating ADR link (see [ADR-link resolution](#adr-link-resolution-ac2ac5));
   - if `migration-guide ∈ ENABLED_ROWS` → draft the **migration-guide stub** for `VERSION`
     (headings/scaffold only — no body prose);
   - returns the drafts **it actually produced** (plus the resolved merged-story set and ADR-link
     map) to the command layer, each labelled with its row so the gate and phase 2 can filter on the
     same subset. **Nothing is written to disk in phase 1.**

5. **Founder-confirm gate (command layer, AC3).** Present **every narrative draft phase 1 returned —
   and only those** (i.e. the `ENABLED_ROWS` subset, never a fixed set of three) — and wait for
   explicit founder confirmation. The founder may accept, edit, or reject each. AC3 names the
   release notes and migration stub explicitly; an enabled changelog draft rides the **same** gate
   because its registry `generation-mode` is `draft-for-review` (see
   [Founder-confirm gate scope](#founder-confirm-gate-scope-decision)). If `ENABLED_ROWS` resolved to
   a single row, the gate presents a single draft. A dispatched subagent cannot pause for
   interactive input — the gate lives at the command layer between the two dispatches, exactly as
   `sync` and `/sdlc:adr` do.

6. **Phase 2 dispatch — `knowledge-engineer` writes confirmed, fresh dispatch.** The command hands
   phase 2 `VERSION`, `ENABLED_ROWS`, and the founder-confirmed drafts **verbatim** (inline or via
   `scripts/tmp-dir.sh` temp files by path — never re-derived). Phase 2:
   - checks out the release branch per the [re-run / idempotence contract](#re-run--idempotence-contract)
     (a fresh branch cut from `origin/<BASE-BRANCH>` head on a first run; the **existing** branch,
     preserved, on a re-run);
   - writes the confirmed content **for the rows in `ENABLED_ROWS` only** under their
     manifest-resolved `target-path`s — a row absent from `ENABLED_ROWS` gets no page, and an
     existing page belonging to a now-disabled row is left untouched (never created, never deleted);
   - applies the [changelog upsert rule](#changelog-upsert-rule-cumulative-file) when writing the
     cumulative changelog;
   - **deterministically regenerates the doc index + `llms.txt` (AC4)** — see
     [Deterministic index + llms.txt regen](#deterministic-index--llmstxt-regen-ac4);
   - **only if `git status --porcelain` on the written paths is non-empty** — commits via
     `conventional-commit`, pushes, and opens or updates the release PR (AC4). If the founder
     rejected every draft and the deterministic regen produced byte-identical output → no commit,
     no PR, clean exit.

### Narrative artifacts (changelog / release-notes / migration-guide)

Each artifact maps onto an existing `writing-docs` quadrant template and its registry `target-path`;
the **normalised** `VERSION` is used as the identifier. A row is only in play when it is in
`ENABLED_ROWS`:

| Artifact          | Registry quadrant | `writing-docs` template | Default `target-path`     | File convention                                                                                             |
| ----------------- | ----------------- | ----------------------- | ------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `changelog`       | reference         | Reference               | `docs/changelog/`         | Single **cumulative** file (e.g. `docs/changelog/index.md`, Keep-a-Changelog style) — each release **upserts** its `## <VERSION>` section (see [upsert rule](#changelog-upsert-rule-cumulative-file)); the file self-indexes releases newest-first |
| `release-notes`   | explanation       | Explanation             | `docs/release-notes/`     | **One page per version** (e.g. `docs/release-notes/<VERSION>.md`); each note links its motivating ADR(s) |
| `migration-guide` | how-to            | How-to                  | `docs/migration-guides/`  | **One stub page per version** (e.g. `docs/migration-guides/<VERSION>.md`) — headings/scaffold only (AC2)  |

- **Changelog** entry: deterministically **aggregated** from the merged-story key set — grouped by
  conventional-commit change type (Added / Changed / Fixed / …), one line per merged story
  (`<KEY> — <summary>`). Its inputs are specified in
  [Changelog aggregation inputs](#changelog-aggregation-inputs-offline-first) — they must be
  available with **no** `gh` dependency. Presented at the founder-confirm gate as a draft (its
  registry mode is `draft-for-review`), but its content is a mechanical aggregation, not free prose.
- **Release notes**: explanation-quadrant prose per `writing-docs`, one entry per merged story,
  each linking its motivating ADR when one exists (AC2). The prose is narrative — fully gated.
- **Migration-guide stub**: headings/scaffold **only** (AC2) — e.g. a `## Breaking changes`,
  `## Upgrade steps`, `## Rollback` skeleton with no body content. Deep migration content is
  explicitly out of scope (Epic + story out-of-scope; deferred open question).

### Changelog aggregation inputs (offline-first)

The changelog needs a **change type** and a **summary** per merged story. `gh` PR-title enrichment
is best-effort and never gates the run, so neither may depend on it — the guaranteed input is
`git log` alone. Both are derived from the **conventional-commit prefix of the commit subject** that
carried the key, which `git log --format='%s'` already provides:

| Need        | Offline source (authoritative, always available)                                                                                                                                                                                                                                                                                                                                                                        |
| ----------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Change type | The subject's conventional-commit `type` (this repo commits via the `conventional-commit` skill, so it is present): `feat` → **Added**; `fix` → **Fixed**; any other recognised type (`refactor`/`perf`/`docs`/`chore`/…) → **Changed**; a `!` marker or a `BREAKING CHANGE:` footer → **Breaking**. A subject with **no** parseable prefix → **Changed** (the neutral bucket — never dropped, never guessed at).           |
| Summary     | The subject's description text — the remainder after `type(scope):` — used verbatim, trimmed.                                                                                                                                                                                                                                                                                                                            |

- **`gh` enrichment, when it succeeds**, may replace the summary with the merged PR title (a
  friendlier human label). When `gh` is unavailable, rate-limited, or returns no match, the commit
  subject's description stands. Change type is **always** derived from the commit subject —
  never from `gh`.
- **A story whose key appears in multiple commits** takes the highest-precedence change type among
  them (Breaking > Added > Fixed > Changed) and the subject of its most recent commit as the
  summary — deterministic, so a re-run yields byte-identical output.
- **Never fabricate.** The changelog emits exactly what the commit subjects support. No inferred
  prose, no invented summaries — this is the same no-fabrication discipline the ADR-link rule
  applies at [AC5](#adr-link-resolution-ac2ac5). A bare-key line is never emitted, because the
  fallbacks above always resolve both fields.

### Changelog upsert rule (cumulative file)

The cumulative changelog is maintained by **upsert**, not by unconditional prepend — a prepend-only
rule contradicts the [idempotence contract](#re-run--idempotence-contract) below, because `release`
creates no tags (see [Out of Scope](#out-of-scope)), so a re-run of a version whose PR already
merged would find its `## <VERSION>` section already present and prepend a **second** one:

1. **Scan** the cumulative file for an existing `## <VERSION>` heading (exact match on the
   normalised `VERSION` — which is precisely what
   [validation step 5](#argument-validation)'s normalisation guarantees is a single stable
   spelling; without it `## 1.4.0` and `## v1.4.0` would be distinct sections).
2. **Section exists** → **replace it in place**: swap the existing section's body for the freshly
   aggregated content, leaving its position in the file and every other section byte-identical. A
   re-run with no new merged stories therefore rewrites the section with identical bytes → empty
   `git status --porcelain` → no commit, no PR.
3. **Section absent** → **prepend** a new `## <VERSION>` section above the newest existing section,
   preserving the file's newest-first ordering.

The upsert boundary is the `## <VERSION>` heading and everything up to (not including) the next
`## ` heading or EOF. The file's preamble (title, Keep-a-Changelog blurb) is never touched.

### ADR-link resolution (AC2/AC5)

For each merged-story key, resolve its **motivating ADR** by scanning `docs/adr/**` frontmatter for
any ADR whose `source-stories:` list contains that key:

- **Match found** → the release note for that story links the ADR (relative link to the
  `docs/adr/NNNN-*.md` page, using the ADR title).
- **No match (AC5)** → the release note for that story **omits** the ADR link entirely — never
  fabricate, guess, or invent an ADR reference. A story with no ADR simply has an ADR-less note.
- Multiple ADRs may cite the same story; link each. The scan is read-only against `docs/adr/` at the
  release branch's base (`origin/<BASE-BRANCH>` head); if the repo has **no** `docs/adr/` directory
  at all, every note is ADR-less (not an error).

### Founder-confirm gate scope (decision)

AC3 names "release notes, migration stub" as the narrative content held at the gate. The
`refs/doc-types.md` registry marks **all three** release artifacts (`changelog`, `release-notes`,
`migration-guide`) as `generation-mode = draft-for-review`. This spec resolves the two by gating
**every enabled release artifact**: an enabled changelog draft is presented at the same
founder-confirm gate as the release notes and migration stub. This honours the registry's
`draft-for-review` mode for the changelog and satisfies AC3 (which names the prose-heavy artifacts
as a minimum, not an exhaustive list). The changelog's content is a mechanical aggregation, so in
practice the founder confirms it with little or no editing — but it is never written un-confirmed.
The gate's scope is always `ENABLED_ROWS`, never a hardcoded three — a disabled row has no draft to
gate.

### Deterministic index + llms.txt regen (AC4)

After the confirmed narrative writes, phase 2 deterministically regenerates the doc index and
`llms.txt` — the deterministic, un-gated half of the run (the same "auto rows are un-gated"
discipline `sync` applies):

- **`llms.txt`** — regenerated by reusing `sync`'s `llms-txt` algorithm
  (`refs/docs-pipeline.md` §8): an index-only file, grouping the generated pages of every enabled
  `public: yes` manifest row by Diátaxis quadrant, each entry a `title — one-line description —
  relative link` derived from page frontmatter. The newly written release pages now appear in it.
  Idempotent; committed only if changed.
- **The doc index** — for the `changelog` target-path, the cumulative changelog file **is** the
  index of releases (newest-first), maintained in place by the
  [upsert rule](#changelog-upsert-rule-cumulative-file). For `release-notes` / `migration-guides`
  target-paths, if the row is enabled **and** the consumer's docs tree carries a section index page
  at that path, regenerate it deterministically (**upsert** the `<VERSION>` entry — add it if
  absent, rewrite it in place if present; never append a duplicate); if none exists, `llms.txt` is
  the sole index and no separate section index is created. See
  [Open Questions](#open-questions) for the resolved default on "the doc index" identity.

### Re-run / idempotence contract

Both regenerations, plus the changelog upsert, are idempotent by construction: re-running
`release <VERSION>` with no new merged stories and no founder edits recomputes **byte-identical**
content for every enabled row, leaves `git status --porcelain` empty, and therefore commits nothing
and opens no PR (AC6). Three things are load-bearing for that claim, and none may be dropped:

1. **`VERSION` normalisation** ([validation step 5](#argument-validation)) — one spelling per
   release, so a re-run targets the same branch, the same PR, the same pages, and the same changelog
   section rather than creating a parallel set.
2. **Changelog upsert** ([upsert rule](#changelog-upsert-rule-cumulative-file)) — a re-run rewrites
   its section in place instead of prepending a duplicate.
3. **Non-destructive branch re-use** ([Branch / PR / Control Flow](#branch--pr--control-flow)) — a
   re-run writes **on top of** the existing branch and never resets or force-pushes it, so a re-run
   that genuinely changes nothing produces no commit, and one that does changes only what changed.

## Branch / PR / Control Flow

| Item              | Value                                                                                                                                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Branch            | `docs/release-<VERSION>` (the **normalised** version — see [validation step 5](#argument-validation); `1.4.0` and `v1.4.0` resolve to the same branch, never two), cut from the **base branch head** (`origin/<BASE-BRANCH>`) — **not** a story branch. Release aggregates work already merged to base, so the base tree is the correct source. |
| Commit            | `docs(docs): release <VERSION>` (via `conventional-commit`)                                                                                                                                                               |
| PR title          | `docs(docs): release <VERSION>`                                                                                                                                                                                           |
| PR base           | `<BASE-BRANCH>` from project-context (never assume `main`)                                                                                                                                                                |
| First-run PR      | Branch absent on `origin` → create it from `origin/<BASE-BRANCH>` head, write, commit, push, open the PR (`gh pr create`) against `<BASE-BRANCH>`.                                                                        |
| Re-run behaviour  | **Open or update, non-destructively** (AC4/AC6). If `docs/release-<VERSION>` already exists on `origin` (`git rev-parse --verify origin/docs/release-<VERSION>` / `gh pr list --head docs/release-<VERSION>`) → fetch, check out, and `git reset --hard origin/docs/release-<VERSION>` **to align with the remote branch as it stands** — then write the regenerated content **on top of it** as a new commit and `git push` (a plain fast-forward). **Never `git reset --hard` onto a freshly regenerated state, and never `git push --force`/`--force-with-lease`.** Rationale: the control-flow tail below drives `/loop /sdlc:loop` against this same branch, so it carries **review-fix commits that are not derived from the drafts** — a hard reset to regenerated state would silently destroy them (`--force-with-lease` would *pass*, since the local ref was just fetched) and revert the PR to unreviewed content. Convergence comes from idempotence, not from resetting: per the [re-run / idempotence contract](#re-run--idempotence-contract), a re-run with no new stories and no founder edits recomputes byte-identical content → empty `git status --porcelain` → **no commit, no push, no duplicate PR**. |
| Re-run guard      | Before writing on a re-run, if the existing branch's history relative to `origin/<BASE-BRANCH>` contains commits this pipeline did not author (anything other than `docs(docs): release <VERSION>` commits) — i.e. review-fix commits — that is **expected and fine**: proceed with the write-on-top above. The guard exists only to forbid the destructive path: **no re-run may reset or force-push this branch**, regardless of what it contains. |
| Control-flow tail | Mirror `sync` / `commands/adr.md`: after the PR is raised, drive the review loop to convergence via `/loop /sdlc:loop <PR_URL>` (falling back to `ScheduleWakeup` if the harness cannot nest `/loop`). If the run ended **before** any PR (manifest-absent silent no-op, no-row-enabled no-op, no-stories-merged no-op, an invalid-version STOP, or another usage STOP), release directly via `scripts/session-complete.sh` — only the manifest-absent path is silent. |

## Permissions Detail

Not a multi-role RBAC surface. The single actor is the **founder** running `/sdlc:docs release`. The
one authority boundary is the **founder-confirm gate**: the deterministic doc index + `llms.txt`
regen is written without confirmation; the enabled narrative artifacts (changelog, release notes,
migration stub) are written only after explicit founder confirmation at the command layer. A
dispatched subagent never runs the gate itself — the command layer owns it, exactly as
`commands/docs.md` (`sync`) and `commands/adr.md` do.

A second, quieter authority boundary: the founder's manifest is what enables a row. `release` never
creates a page for a row the founder disabled (`ENABLED_ROWS`, step 2), and never rewrites history
on a branch under review (non-destructive re-run, above).

## Error Handling

| Scenario                                                                                                        | Behaviour                                                                                                                                                                                                                 |
| --------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `.claude/project/docs-manifest.md` absent                                                                       | **Silent** no-op — no branch, no dispatch, no PR, no error, **no stdout** (AC6). Distinct from a usage STOP, which prints.                                                                                                  |
| Empty `$ARGUMENTS` / unrecognised first token                                                                   | Usage STOP (prints the extended usage message).                                                                                                                                                                            |
| `release` with missing/empty `<version>`                                                                        | Usage STOP (prints the usage message).                                                                                                                                                                                     |
| `release` with a `<version>` failing the token regex, or containing `..` / a path separator (e.g. `../../oops`) | Usage STOP with `invalid version "<raw>" — expected a token like 1.4.0 or v1.4.0 …`, raised **at the validation ladder** — before any branch, dispatch, draft, or founder gate.                                            |
| Manifest present but no `release`-triggered row (`changelog`/`release-notes`/`migration-guide`) enabled          | Clean no-op — print `no release-triggered doc types enabled…`, no PR.                                                                                                                                                       |
| Manifest enables only **some** release rows                                                                     | Not an error — `ENABLED_ROWS` is that subset; only those rows are drafted, gated, and written. Disabled rows' pages are never created or touched.                                                                          |
| No stories merged in the range, `LAST_TAG` set (AC6)                                                            | Clean no-op — print `no stories merged since <LAST_TAG> — nothing to release`, no PR.                                                                                                                                       |
| No stories merged in the range, no tags exist (AC6)                                                             | Clean no-op — print `no stories merged since the start of history — nothing to release`, no PR (never an empty interpolation).                                                                                             |
| Repo has no tags yet                                                                                            | Not an error — range is the **single-ended** `origin/<BASE-BRANCH>` (full history, root inclusive); proceed if any stories merged.                                                                                          |
| `git fetch` fails, or `origin/<BASE-BRANCH>` will not resolve                                                    | Surface the git failure and STOP — resolved **before** `git describe`, so it can never be mistaken for "no tags yet".                                                                                                       |
| `git describe --tags` fails for any reason other than "no names found"                                          | Surface the git failure and STOP — the no-tags branch is entered **only** on stderr matching `No names found` / `No tags can describe`; a shallow clone or corrupt refs must never fall through to a whole-history release. |
| `git log` failure                                                                                               | Surface the git failure and STOP — do not silently release an empty or partial range.                                                                                                                                      |
| `gh` unavailable / rate-limited during title enrichment                                                          | Not an error — enrichment is best-effort. Change type and summary come from the commit subject's conventional-commit prefix (never fabricated, never a bare key).                                                          |
| A merged story has no motivating ADR (AC5)                                                                      | Its release note omits the ADR link — never fabricated.                                                                                                                                                                    |
| Repo has no `docs/adr/` directory                                                                               | Every release note is ADR-less; not an error.                                                                                                                                                                              |
| `refs/doc-types.md` unreadable/malformed                                                                        | Surface the failure and STOP — never generate from a partial registry.                                                                                                                                                     |
| Re-run: `docs/release-<VERSION>` exists on `origin` and carries review-fix commits                              | Not an error, and **never** reset/force-pushed — check out the remote branch as-is, write regenerated content on top, commit + fast-forward push only if it changed.                                                       |
| Re-run: `## <VERSION>` section already present in the cumulative changelog                                      | Not an error — the section is **replaced in place** (upsert), never prepended a second time.                                                                                                                                |
| Founder rejects every enabled draft and deterministic regen is byte-identical                                   | No commit, no PR (AC4/AC6) — write phase detected an empty `git status --porcelain`.                                                                                                                                       |
| Founder rejects the narrative drafts but the deterministic `llms.txt`/index changed                             | Still commit + PR the deterministic regen (content changed).                                                                                                                                                               |
| `gh` / `raise-pr.sh` failure                                                                                    | STOP and surface — the write is on a branch and reviewable; never leave a raised-but-broken state silently.                                                                                                                |

## Out of Scope

- **seed / audit / distill modes** — own stories (NA-54 / NA-55 / NA-57). `docs.md` keeps them as
  clean "not yet implemented" stubs; only `release` graduates from stub in this story.
- **Deep migration-guide content generation** — the migration guide is a headings/scaffold **stub
  only** (AC2). Generating actual upgrade prose/steps is a deferred open question (Epic + story).
- **Auto-triggering `release` on tag push / CI events** — later enhancement (Epic out-of-scope).
- **Creating the git tag itself** — `release <version>` documents a release; it does not cut or push
  the version tag. The tag is assumed to be created by the existing release/versioning process.
  (This is exactly why the changelog needs the
  [upsert rule](#changelog-upsert-rule-cumulative-file): a version can legitimately be re-run after
  its PR merged but before its tag is cut.)
- **`llms-full.txt`** (full-content variant) — v1 ships index-only `llms.txt` (inherited from NA-52).
- **Editing `refs/doc-types.md` registry rows** — the `changelog` / `release-notes` /
  `migration-guide` rows already exist (NA-51); `release` reads them, never edits them.
- **Deleting or migrating pages for a newly-disabled row** — if a founder disables `release-notes`
  after earlier releases created pages, `release` simply stops writing them; it never deletes the
  existing pages.
- **Building or hosting a docs site** — Mintlify is this repo's choice; the plugin stays
  stack-agnostic via manifest `target-path`s (Epic out-of-scope).
- **Marketing/SEO release copy** — `gtm:docs` owns that; `sdlc release` is technical accuracy only.
