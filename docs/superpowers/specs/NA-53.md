# /sdlc:docs release mode — Technical Spec

**Story:** NA-53 — Add /sdlc:docs release mode
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-17

## Overview

Add the second `/sdlc:docs` mode — `release <version>` — which aggregates every story merged since
the last tag into a changelog entry, ADR-linked release notes, and a headings-only migration-guide
stub, holds the narrative content at the founder-confirm gate, then regenerates the doc index +
`llms.txt` deterministically and opens a PR. It reuses the command shell, two-phase dispatch, and
`refs/docs-pipeline.md` contract shipped by NA-52 (the `sync` core), and the `changelog` /
`release-notes` / `migration-guide` registry rows already present in `refs/doc-types.md` (NA-51).

## Command Contract — `/sdlc:docs release <version>`

### Mode table update

The `docs.md` mode table changes the `release` row's v1 status from a stub to in-scope; the other
three future modes remain stubs:

| Invocation                       | Mode        | Status after this story                                                     |
| -------------------------------- | ----------- | -------------------------------------------------------------------------- |
| `/sdlc:docs sync <STORY-KEY>`    | sync        | Shipped (NA-52)                                                            |
| `/sdlc:docs release <version>`   | **release** | **In scope — this story**                                                  |
| `/sdlc:docs seed <type> [topic]` | seed        | Not yet implemented (NA-54) — clean stub, not an error                     |
| `/sdlc:docs audit [--dry-run]`   | audit       | Not yet implemented (NA-55) — clean stub, not an error                     |
| `/sdlc:docs distill`             | distill     | Not yet implemented (NA-57) — clean stub, not an error                     |

### Argument validation

Parse `$ARGUMENTS` into `<mode>` (first token) and the mode's remaining args. Modify the existing
validation ladder in `docs.md` so that:

1. **Empty `$ARGUMENTS`** → STOP with the usage message (extended to advertise `release`):
   `usage: /sdlc:docs sync <STORY-KEY> | release <version>  (seed|audit|distill land in later stories)`.
2. **Unrecognised first token** → STOP with the same usage message.
3. **Recognised future-mode token** (`seed`/`audit`/`distill`) → print
   `mode "<mode>" is not yet implemented (see Epic NA-50)` and exit cleanly. `release` is **removed**
   from this stub set — it now routes to the release contract below.
4. **`release` with a missing or empty `<version>`** → STOP with the usage message. This is a
   **usage STOP** (a caller error), distinct from the manifest-absent silent no-op and the
   no-stories-merged clean no-op below.
5. **`release` `<version>` format** — `<version>` is accepted as any non-empty single token; it is
   **not** hard-validated against semver (a leading `v` is allowed, e.g. `1.4.0` or `v1.4.0`). It is
   used verbatim as the changelog heading label and the release-notes / migration-guide page
   identifiers. See [Open Questions](#open-questions) for the (declined) strict-semver option.

### `release <version>` — behavioural contract

The release run mirrors `sync`'s two-phase dispatch and command-layer founder-confirm gate — the
shared skeleton lives in `refs/docs-pipeline.md` §2 and is not re-derived. What is release-specific:

1. **Manifest gate (AC6).** Resolve `.claude/project/docs-manifest.md` **checkout-independently**
   (via `git show origin/<BASE-BRANCH>:.claude/project/docs-manifest.md`, the same way `sync`
   resolves it — never the working tree). If **absent** → **silent no-op**: no branch, no dispatch,
   no PR, no error, **no stdout**, exit 0. This is the zero-setup-cost guarantee for repos that
   declined the `/init` docs opt-in, identical in discipline to `sync`'s AC5.

2. **Enabled-row gate.** Resolve, from the manifest, which of the three `release`-triggered registry
   rows (`changelog`, `release-notes`, `migration-guide`) are **enabled**. If **none** is enabled →
   **clean no-op**: print a one-line informational notice
   (`no release-triggered doc types enabled in docs-manifest.md — nothing to generate`) and exit
   without a PR. (Informational, not silent — the manifest exists, so the founder opted in; a
   fully-disabled release surface is worth surfacing, mirroring `sync`'s "found nothing to do" vs
   "opted out" distinction.)

3. **Resolve the last tag and the merged-story set (AC1).** Diff source is the **base branch**, not
   a story branch — release aggregates work already merged to base:

   ```bash
   git fetch origin --tags --quiet
   LAST_TAG=$(git describe --tags --abbrev=0 "origin/<BASE-BRANCH>" 2>/dev/null || echo "")
   RANGE_START=${LAST_TAG:+$LAST_TAG}          # empty when the repo has no tags yet
   ```

   - **When `LAST_TAG` is set** → the commit range is `"$LAST_TAG..origin/<BASE-BRANCH>"`.
   - **When the repo has NO tags** (first-ever release) → the range is the full history to
     `origin/<BASE-BRANCH>` (root..head). Aggregate every merged story from the beginning.
   - **Enumerate merged stories** from the range: extract the **distinct** set of Jira story keys
     (regex `[A-Z][A-Z0-9]*-[0-9]+`) from commit subjects **and** bodies across the range
     (`git log <RANGE> --format='%s%n%b'`), de-duplicated. Optionally enrich each key's human title
     from the merged PR that carried it (`gh pr list --search "…"` / `gh pr view`), but the git-log
     key extraction is the **authoritative** source — `gh` enrichment is best-effort and never
     gates the run. This key set is what the three artifacts are built from.
   - **No stories merged since `LAST_TAG` (AC6)** → **clean no-op**: print a one-line notice
     (`no stories merged since <LAST_TAG> — nothing to release`) and exit without a PR.

4. **Phase 1 dispatch — `knowledge-engineer` computes & drafts, writes nothing.** Pass it
   `<version>`, the resolved commit range, the merged-story key set (with any `gh`-enriched titles),
   and `origin/<BASE-BRANCH>`. Phase 1:
   - drafts the **changelog** entry for `<version>` (see [Narrative artifacts](#narrative-artifacts-changelog--release-notes--migration-guide));
   - drafts the **release notes** for `<version>`, resolving each story's motivating ADR link (see
     [ADR-link resolution](#adr-link-resolution-ac2ac5));
   - drafts the **migration-guide stub** for `<version>` (headings/scaffold only — no body prose);
   - returns all three drafts (plus the resolved merged-story set and ADR-link map) to the command
     layer. **Nothing is written to disk in phase 1.**

5. **Founder-confirm gate (command layer, AC3).** Present all three narrative drafts — changelog,
   release notes, migration stub — and wait for explicit founder confirmation. The founder may
   accept, edit, or reject each. AC3 names the release notes and migration stub explicitly; the
   changelog draft rides the **same** gate because its registry `generation-mode` is
   `draft-for-review` (see [Founder-confirm gate scope](#founder-confirm-gate-scope-decision)). A
   dispatched subagent cannot pause for interactive input — the gate lives at the command layer
   between the two dispatches, exactly as `sync` and `/sdlc:adr` do.

6. **Phase 2 dispatch — `knowledge-engineer` writes confirmed, fresh dispatch.** The command hands
   phase 2 the founder-confirmed drafts **verbatim** (inline or via `scripts/tmp-dir.sh` temp files
   by path — never re-derived). Phase 2:
   - checks out the release branch cut from `origin/<BASE-BRANCH>` head (see [Branch / PR / Control Flow](#branch--pr--control-flow));
   - writes the confirmed changelog / release-notes / migration-guide content under their
     manifest-resolved `target-path`s;
   - **deterministically regenerates the doc index + `llms.txt` (AC4)** — see
     [Deterministic index + llms.txt regen](#deterministic-index--llmstxt-regen-ac4);
   - **only if `git status --porcelain` on the written paths is non-empty** — commits via
     `conventional-commit`, pushes, and opens or updates the release PR (AC4). If the founder
     rejected all three drafts and the deterministic regen produced byte-identical output → no
     commit, no PR, clean exit.

### Narrative artifacts (changelog / release-notes / migration-guide)

Each artifact maps onto an existing `writing-docs` quadrant template and its registry `target-path`;
`<version>` is used verbatim as the identifier:

| Artifact          | Registry quadrant | `writing-docs` template | Default `target-path`     | File convention                                                                                             |
| ----------------- | ----------------- | ----------------------- | ------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `changelog`       | reference         | Reference               | `docs/changelog/`         | Single **cumulative** file (e.g. `docs/changelog/index.md`, Keep-a-Changelog style) — each release **prepends** a new `## <version>` section grouped by change type; the file self-indexes releases newest-first |
| `release-notes`   | explanation       | Explanation             | `docs/release-notes/`     | **One page per version** (e.g. `docs/release-notes/<version>.md`); each note links its motivating ADR(s) |
| `migration-guide` | how-to            | How-to                  | `docs/migration-guides/`  | **One stub page per version** (e.g. `docs/migration-guides/<version>.md`) — headings/scaffold only (AC2)  |

- **Changelog** entry: deterministically **aggregated** from the merged-story key set — grouped by
  conventional-commit change type (Added / Changed / Fixed / …), one line per merged story
  (`<KEY> — <summary>`). Presented at the founder-confirm gate as a draft (its registry mode is
  `draft-for-review`), but its content is a mechanical aggregation, not free prose.
- **Release notes**: explanation-quadrant prose per `writing-docs`, one entry per merged story,
  each linking its motivating ADR when one exists (AC2). The prose is narrative — fully gated.
- **Migration-guide stub**: headings/scaffold **only** (AC2) — e.g. a `## Breaking changes`,
  `## Upgrade steps`, `## Rollback` skeleton with no body content. Deep migration content is
  explicitly out of scope (Epic + story out-of-scope; deferred open question).

### ADR-link resolution (AC2/AC5)

For each merged-story key, resolve its **motivating ADR** by scanning `docs/adr/**` frontmatter for
any ADR whose `source-stories:` list contains that key:

- **Match found** → the release note for that story links the ADR (relative link to the
  `docs/adr/NNNN-*.md` page, using the ADR title).
- **No match (AC5)** → the release note for that story **omits** the ADR link entirely — never
  fabricate, guess, or invent an ADR reference. A story with no ADR simply has an ADR-less note.
- Multiple ADRs may cite the same story; link each. The scan is read-only against `docs/adr/` at the
  release branch's base (`origin/<BASE-BRANCH>` head); if the repo has **no** `docs/adr/` directory
  at all, every note is ADR-less (not an error).

### Founder-confirm gate scope (decision)

AC3 names "release notes, migration stub" as the narrative content held at the gate. The
`refs/doc-types.md` registry marks **all three** release artifacts (`changelog`, `release-notes`,
`migration-guide`) as `generation-mode = draft-for-review`. This spec resolves the two by gating
**all three**: the changelog draft is presented at the same founder-confirm gate as the release
notes and migration stub. This honours the registry's `draft-for-review` mode for the changelog and
satisfies AC3 (which names the prose-heavy artifacts as a minimum, not an exhaustive list). The
changelog's content is a mechanical aggregation, so in practice the founder confirms it with little
or no editing — but it is never written un-confirmed.

### Deterministic index + llms.txt regen (AC4)

After the confirmed narrative writes, phase 2 deterministically regenerates the doc index and
`llms.txt` — the deterministic, un-gated half of the run (the same "auto rows are un-gated"
discipline `sync` applies):

- **`llms.txt`** — regenerated by reusing `sync`'s `llms-txt` algorithm
  (`refs/docs-pipeline.md` §8): an index-only file, grouping the generated pages of every enabled
  `public: yes` manifest row by Diátaxis quadrant, each entry a `title — one-line description —
  relative link` derived from page frontmatter. The newly written release pages now appear in it.
  Idempotent; committed only if changed.
- **The doc index** — for the `changelog` target-path, the cumulative changelog file **is** the
  index of releases (newest-first), maintained in place by the prepend in step (Narrative artifacts).
  For `release-notes` / `migration-guides` target-paths, if the consumer's docs tree carries a
  section index page at that path, regenerate it deterministically (append the new `<version>`
  entry); if none exists, `llms.txt` is the sole index and no separate section index is created.
  See [Open Questions](#open-questions) for the resolved default on "the doc index" identity.

Both regenerations are idempotent — re-running `release <version>` with no new merged stories and no
founder edits yields byte-identical output and therefore a no-op (AC6).

## Pipeline Contract additions — `refs/docs-pipeline.md`

This story adds a **release** section to `docs-pipeline.md`, reusing its existing skeleton (do not
fork a second ref). Required additions:

1. **Release two-phase dispatch** — reuse §2's phase-1/gate/phase-2 split; state the
   release-specific inputs (version, commit range, merged-story key set, ADR-link map) and that
   phase 1 drafts all three narrative artifacts and phase 2 writes confirmed + regenerates
   index/`llms.txt`.
2. **Merged-story enumeration algorithm** — the `git describe --tags` / range / story-key-regex
   procedure (a prose algorithm executed inline by the agent, not a committed script — matches the
   plugin's "instructions not code" style and the ADR index-regen algorithm in `adr-pipeline.md`
   §10). Include the no-tags-yet (root..head) and no-stories-merged (clean no-op) branches.
3. **ADR-link resolution** — the `source-stories` frontmatter join, match → link, no-match → omit
   (AC5), no-`docs/adr/`-dir → all ADR-less.
4. **Release artifact set + quadrant/target-path mapping** — the changelog (cumulative, prepend) /
   release-notes (per-version) / migration-guide-stub (per-version, headings-only) mapping.
5. **Release branch / PR naming + control flow** — see [Branch / PR / Control Flow](#branch--pr--control-flow);
   the cut point is the **base branch head** (unlike `sync`, which cuts from the story branch head).
6. **No-op / change-gate semantics for release** — manifest-absent silent no-op (AC6);
   no-release-row-enabled clean no-op; no-stories-merged clean no-op; commit/PR only on actual
   content change (AC4/AC6); `llms.txt` regenerated every run, committed only if changed.

The voice/format resolution chain (§4) and the deterministic-regen idempotence principle (§3/§6)
are reused verbatim — release narrative drafting resolves voice/format exactly as `sync` does
(docs-manifest → project-context → stated default).

## Branch / PR / Control Flow

| Item              | Value                                                                                                                                                                                                                       |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Branch            | `docs/release-<version>`, cut from the **base branch head** (`origin/<BASE-BRANCH>`) — **not** a story branch. Release aggregates work already merged to base, so the base tree is the correct source.                    |
| Commit            | `docs(docs): release <version>` (via `conventional-commit`)                                                                                                                                                               |
| PR title          | `docs(docs): release <version>`                                                                                                                                                                                           |
| PR base           | `<BASE-BRANCH>` from project-context (never assume `main`)                                                                                                                                                                |
| First-run PR      | Create the branch, commit, push, open the PR (`gh pr create`) against `<BASE-BRANCH>`.                                                                                                                                    |
| Re-run behaviour  | **Open or update** (AC4/AC6): if `docs/release-<version>` already exists on `origin`, check it out, `git reset --hard` onto the freshly regenerated state, then `git push --force-with-lease` to update the existing open PR — never open a duplicate. (The branch content is fully derived from confirmed drafts + deterministic regen, so a hard reset is safe.) |
| Control-flow tail | Mirror `sync` / `commands/adr.md`: after the PR is raised, drive the review loop to convergence via `/loop /sdlc:loop <PR_URL>` (falling back to `ScheduleWakeup` if the harness cannot nest `/loop`). If the run ended **before** any PR (manifest-absent silent no-op, no-row-enabled no-op, no-stories-merged no-op, or a usage STOP), release directly via `scripts/session-complete.sh` — only the manifest-absent path is silent. |

## Permissions Detail

Not a multi-role RBAC surface. The single actor is the **founder** running `/sdlc:docs release`. The
one authority boundary is the **founder-confirm gate**: the deterministic doc index + `llms.txt`
regen is written without confirmation; the three narrative artifacts (changelog, release notes,
migration stub) are written only after explicit founder confirmation at the command layer. A
dispatched subagent never runs the gate itself — the command layer owns it, exactly as
`commands/docs.md` (`sync`) and `commands/adr.md` do.

## Error Handling

| Scenario                                                                                | Behaviour                                                                                                                          |
| --------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| `.claude/project/docs-manifest.md` absent                                               | **Silent** no-op — no branch, no dispatch, no PR, no error, **no stdout** (AC6). Distinct from a usage STOP, which prints.        |
| Empty `$ARGUMENTS` / unrecognised first token                                           | Usage STOP (prints the extended usage message).                                                                                  |
| `release` with missing/empty `<version>`                                                | Usage STOP (prints the usage message).                                                                                           |
| Manifest present but no `release`-triggered row (`changelog`/`release-notes`/`migration-guide`) enabled | Clean no-op — print `no release-triggered doc types enabled…`, no PR.                                            |
| No stories merged since the last tag (or since root when no tag exists) (AC6)           | Clean no-op — print `no stories merged since <LAST_TAG> — nothing to release`, no PR.                                            |
| Repo has no tags yet                                                                    | Not an error — range is root..`origin/<BASE-BRANCH>` (first release); proceed if any stories merged.                             |
| A merged story has no motivating ADR (AC5)                                              | Its release note omits the ADR link — never fabricated.                                                                          |
| Repo has no `docs/adr/` directory                                                       | Every release note is ADR-less; not an error.                                                                                    |
| `refs/doc-types.md` unreadable/malformed                                                | Surface the failure and STOP — never generate from a partial registry.                                                          |
| Founder rejects all three drafts and deterministic regen is byte-identical              | No commit, no PR (AC4/AC6) — write phase detected an empty `git status --porcelain`.                                            |
| Founder rejects the narrative drafts but the deterministic `llms.txt`/index changed     | Still commit + PR the deterministic regen (content changed).                                                                     |
| `git describe --tags` / `git log` failure (not "no tags")                               | Surface the git failure and STOP — do not silently release an empty or partial range.                                           |
| `gh` / `raise-pr.sh` failure                                                            | STOP and surface — the write is on a branch and reviewable; never leave a raised-but-broken state silently.                     |

## Out of Scope

- **seed / audit / distill modes** — own stories (NA-54 / NA-55 / NA-57). `docs.md` keeps them as
  clean "not yet implemented" stubs; only `release` graduates from stub in this story.
- **Deep migration-guide content generation** — the migration guide is a headings/scaffold **stub
  only** (AC2). Generating actual upgrade prose/steps is a deferred open question (Epic + story).
- **Auto-triggering `release` on tag push / CI events** — later enhancement (Epic out-of-scope).
- **Creating the git tag itself** — `release <version>` documents a release; it does not cut or push
  the version tag. The tag is assumed to be created by the existing release/versioning process.
- **`llms-full.txt`** (full-content variant) — v1 ships index-only `llms.txt` (inherited from NA-52).
- **Editing `refs/doc-types.md` registry rows** — the `changelog` / `release-notes` /
  `migration-guide` rows already exist (NA-51); `release` reads them, never edits them.
- **Building or hosting a docs site** — Mintlify is this repo's choice; the plugin stays
  stack-agnostic via manifest `target-path`s (Epic out-of-scope).
- **Marketing/SEO release copy** — `gtm:docs` owns that; `sdlc release` is technical accuracy only.

## Open Questions

- [ ] **"The doc index" identity (AC4)** — AC4 says regenerate "the doc index and `llms.txt`."
      _Suggested default (adopt):_ `llms.txt` is the deterministic meta index that is always
      regenerated; the cumulative changelog file self-indexes releases (newest-first); a separate
      per-section index page is regenerated deterministically only if the consumer's docs tree
      already carries one at the `release-notes`/`migration-guides` target-path. No new index file
      type is invented.
- [ ] **`<version>` validation strictness** — _Suggested default (adopt):_ accept any non-empty
      single token (leading `v` allowed); do not hard-validate semver. A scripted caller passing a
      non-semver label (e.g. a date-based version) must not be rejected. Revisit if a consumer needs
      enforcement.
- [ ] **Changelog file shape** — cumulative single file (Keep-a-Changelog, newest-first prepend) vs
      one file per version. _Suggested default (adopt):_ cumulative single file at the `changelog`
      target-path — it is self-indexing and matches the reference-quadrant "one consultable
      surface" shape; release notes and migration guides remain one page per version.
- [ ] **Merged-story enumeration source** — git-log story-key regex over the tag range vs `gh`
      merged-PR listing. _Suggested default (adopt):_ git-log key extraction is authoritative
