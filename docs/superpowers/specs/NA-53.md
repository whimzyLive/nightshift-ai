# /sdlc:docs release mode — Technical Spec

**Story:** NA-53 — Add /sdlc:docs release mode
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-17

## Overview

Add the second `/sdlc:docs` mode — `release <version>` — which aggregates every story merged since
the last tag into whichever release artifacts the repo's manifest enables (a changelog entry,
ADR-linked release notes, and/or a headings-only migration-guide stub — see
[`ENABLED_ROWS`](#release-version--behavioural-contract)), holds the narrative content at the
founder-confirm gate, then regenerates the doc index + `llms.txt` deterministically and opens a PR.
It reuses the command shell, two-phase dispatch, and `refs/docs-pipeline.md` contract shipped by
NA-52 (the `sync` core), and the `changelog` / `release-notes` / `migration-guide` registry rows
already present in `refs/doc-types.md` (NA-51).

## Command Contract — `/sdlc:docs release <version>`

### Frontmatter `description` update

`docs.md`'s frontmatter `description` currently ends: "`release`/`seed`/`audit`/`distill` are
not-yet-implemented stubs (see Epic NA-50)". This story **must** update it — it is not cosmetic.
`refs/doc-types.md`'s `command-reference` row has `source-of-truth = command frontmatter + body`
and `generation-mode = auto`, so the very next `/sdlc:docs sync` regenerates
`docs/reference/commands/` **from this string** and publishes it into the public docs surface and
`llms.txt`. Leaving it stale would ship "release is not yet implemented" as public reference copy
for a shipped mode, and `/help` would advertise the same falsehood.

Required change: drop `release` from the stub list and state the mode it now ships — the
`description` must name **both** live modes (`sync` and `release`) and list only
`seed`/`audit`/`distill` as not-yet-implemented stubs. Mirrors the equivalent `description` update
this story makes to the agent (see [Agent Extension](#agent-extension--knowledge-engineermd)).

### Argument validation

Parse `$ARGUMENTS` into `<mode>` (first token) and the mode's remaining args. Modify the existing
validation ladder in `docs.md` so that:

1. **Empty `$ARGUMENTS`** → STOP with the usage message (extended to advertise `release`):
   `usage: /sdlc:docs sync <STORY-KEY> | release <version>  (seed|audit|distill land in later stories)`.
2. **Unrecognised first token** → STOP with the same usage message.
3. **Recognised future-mode token** (`seed`/`audit`/`distill`) → print
   `mode "<mode>" is not yet implemented (see Epic NA-50)` and exit cleanly. `release` is **removed**
   from this stub set — it now routes to the release contract below.
4. **`release` with a missing or empty `<version>`** → STOP with the usage message. This is a
   **usage STOP** (a caller error), distinct from the manifest-absent silent no-op and the
   no-stories-merged clean no-op below.
5. **`release` `<version>` — hard-validate, then normalise.** `<version>` is **not** free text: it
   becomes a git branch name (`docs/release-<VERSION>`), page ids
   (`docs/release-notes/<VERSION>.md`, `docs/migration-guides/<VERSION>.md`), and the changelog
   `## <VERSION>` heading that the [upsert rule](#changelog-upsert-rule-cumulative-file) keys on.
   It is validated exactly as `sync`'s story key is (`docs.md` validation step 4 hard-validates
   `^[A-Z][A-Z0-9]*-[0-9]+$` **because** it forms a branch/ref) — same discipline, different regex:

   ```text
   raw token must match:  ^v?[0-9A-Za-z][0-9A-Za-z.]*(-[0-9A-Za-z.]+)*$
   normalise:             strip a single leading "v"  →  VERSION
   VERSION must additionally satisfy, or the run STOPs:
     - non-empty
     - contains no ".."
     - does not end in "." or "-"
     - does not end in ".lock"
   ```

   - **Fails the regex or any rule above** → **usage STOP** with
     `invalid version "<raw>" — expected a token like 1.4.0 or v1.4.0 (letters, digits, ".", "-"; no path separators)`.
     This runs **before** any branch, dispatch, draft, or founder gate — a caller passing
     `release ../../oops` is rejected at the ladder, never after the founder has confirmed drafts.
     The regex admits no `/`, no whitespace, and no `..`, so `<VERSION>` can never escape the docs
     tree.
   - **Why the `.lock` rule is `endswith`, not per-component.** Git's `.lock` prohibition applies to
     **slash**-separated ref components. The branch is `docs/release-<VERSION>`, whose last
     slash-component is `release-<VERSION>` — that component ends in `.lock` **iff `VERSION` does**.
     So the correct and sufficient test is a plain `VERSION` `endswith(".lock")`. A per-**dot**
     component test would be vacuous: `1.4.lock` splits on `.` into `1`, `4`, `lock`, none of which
     *ends in* `.lock` (they contain no dots), so such a rule rejects nothing and `1.4.lock` would
     reach `git checkout` and fail there — after the founder had confirmed drafts. Verified:
     `git check-ref-format --branch docs/release-1.4.lock` → `fatal: not a valid branch name`.
   - **Normalisation is identity-forming, not cosmetic.** `1.4.0` and `v1.4.0` normalise to the
     **same** `VERSION` (`1.4.0`) and therefore the same branch, the same PR, the same changelog
     section, and the same release-notes / migration-guide pages. Without it the two spellings
     produce a duplicate branch, a duplicate PR (the re-run dedupe in
     [Branch / PR / Control Flow](#branch--pr--control-flow) keys on branch name), a duplicate
     changelog section, and duplicate pages. This is what makes the
     [re-run / idempotence contract](#re-run--idempotence-contract) well-defined.
   - **`VERSION` (normalised), not the raw token, is used everywhere downstream** — branch name, PR
     title, commit string, page ids, changelog heading, and the no-op notices. The raw token is
     echoed only in the invalid-version STOP message above.
   - **Case is significant and deliberately not folded.** `1.4.0-RC1` and `1.4.0-rc1` are distinct
     `VERSION`s → distinct branches and distinct page ids, which **collide** on a case-insensitive
     filesystem (macOS default). Folding case would instead break legitimately case-distinct labels,
     so v1 accepts the collision and does not fold. See [Open Questions](#open-questions).
   - Beyond the token-safety rules above, `<VERSION>` is **not** hard-validated against semver: a
     date-based or otherwise non-semver label (e.g. `2026.07.1`) passes. See
     [Open Questions](#open-questions) for the (declined) strict-semver option.

### `release <version>` — behavioural contract

The release run mirrors `sync`'s two-phase dispatch and command-layer founder-confirm gate — the
shared skeleton lives in `refs/docs-pipeline.md` §2 and is not re-derived. What is release-specific:

1. **Manifest gate (AC6).** Resolve `.claude/project/docs-manifest.md` **checkout-independently**
   (via `git show origin/<BASE-BRANCH>:.claude/project/docs-manifest.md`, the same way `sync`
   resolves it — never the working tree). If **absent** → **silent no-op**: no branch, no dispatch,
   no PR, no error, **no stdout**, exit 0. This is the zero-setup-cost guarantee for repos that
   declined the `/init` docs opt-in, identical in discipline to `sync`'s AC5.

2. **Enabled-row gate — resolve `ENABLED_ROWS` once, then thread it everywhere.** Resolve, from the
   manifest, which of the three `release`-triggered registry rows (`changelog`, `release-notes`,
   `migration-guide`) are **enabled**; call that resolved subset `ENABLED_ROWS`.

   - **What "enabled" means (absent is never default-on).** A row **present** in the manifest's
     row-table is enabled per its `enabled` column — `refs/docs-manifest-template.md`'s Fill rules
     default that column to `true` for every *written* row. A row **absent** from the table is
     **not** enabled. Absence has two causes and the distinction does not matter here: the type was
     offered and declined (recorded as a `<!-- declined: <type> -->` comment, which "does not write
     (and never overwrites) a disabled row"), or the row was **deleted**, which that template's
     header notes is *not* a permanent deactivation — "a future re-init will offer to re-append it
     unless the offer was previously declined and recorded". Re-offering is `/init`'s business, not
     `release`'s: for `release`, an absent row is simply not active this run. Never infer a missing
     row as enabled.
   - If `ENABLED_ROWS` is **empty** → **clean no-op**: print a one-line informational notice
     (`no release-triggered doc types enabled in docs-manifest.md — nothing to generate`) and exit
     without a PR. (Informational, not silent — the manifest exists, so the founder opted in; a
     fully-disabled release surface is worth surfacing, mirroring `sync`'s "found nothing to do" vs
     "opted out" distinction.)
   - **`ENABLED_ROWS` is a per-row filter, not a one-time collective gate.** It is an explicit input
     to phase 1, to the founder-confirm gate, and to phase 2, and each of those three **must**
     re-consult it per row: phase 1 drafts **only** the enabled rows' artifacts, the gate presents
     **only** those drafts, and phase 2 writes **only** those pages. This is the release-mode
     instance of the invariant `refs/docs-pipeline.md` §2 step 4 / §3 already state for `sync` —
     regeneration "never touches an unaffected row's pages". A founder who disables `release-notes`
     or `migration-guide` in their manifest must never have those pages drafted, shown, or created.
     A run whose `ENABLED_ROWS` is only `{changelog}` produces exactly one artifact, one gate item,
     and one written page.

3. **Resolve the last tag and the merged-story set (AC1).** Diff source is the **base branch**, not
   a story branch — release aggregates work already merged to base. Every git failure here is a
   **STOP**, never a silent fallthrough into the "no tags yet" branch. Discriminating the two needs
   **positive pre-checks**, not stderr matching alone: git exits `128` for both "no tags exist" and
   unrelated fatals, and — critically — a **shallow clone with no reachable tags emits the very same
   `fatal: No names found` message** as a genuine first release. Stderr therefore *cannot* separate
   "no tags exist" from "tags exist but are unreachable at this clone depth"; the shallow check below
   is what makes the fallthrough safe:

   ```bash
   git fetch origin --tags --quiet || STOP "git fetch failed"

   # The base ref must resolve BEFORE any describe — an unresolvable base is a STOP,
   # never a "no tags yet" first-release run.
   git rev-parse --verify --quiet "origin/<BASE-BRANCH>^{commit}" >/dev/null \
     || STOP "cannot resolve origin/<BASE-BRANCH>"

   # POSITIVE shallow pre-check. A shallow clone can hide every tag and still fail describe with
   # the identical "No names found" text, so it MUST be excluded before that text is trusted.
   [ "$(git rev-parse --is-shallow-repository)" = "false" ] \
     || STOP "shallow clone — cannot enumerate the release range (run: git fetch --unshallow)"

   # Session-scoped temp dir (./.tmp/<key>), removed by scripts/cleanup-tmp.sh /
   # session-complete.sh on every exit path incl. STOP — never a bare, un-trapped mktemp.
   TMP="$(bash "${CLAUDE_PLUGIN_ROOT}/scripts/tmp-dir.sh")"
   DESCRIBE_ERR="$TMP/describe.err"

   if LAST_TAG=$(git describe --tags --abbrev=0 "origin/<BASE-BRANCH>" 2>"$DESCRIBE_ERR"); then
     RANGE="$LAST_TAG..origin/<BASE-BRANCH>"        # tagged: exclusive of the tag, inclusive of head
   elif grep -q 'No names found' "$DESCRIBE_ERR"; then
     LAST_TAG=""                                    # genuine "no tags yet" — first-ever release
     RANGE="origin/<BASE-BRANCH>"                   # SINGLE-ENDED: full history to head, root inclusive
   else
     STOP "git describe failed: $(cat "$DESCRIBE_ERR")"
   fi
   ```

   - **Only `No names found` falls through**, and only *after* the shallow pre-check has passed. Do
     **not** also match `No tags can describe` — that message belongs to `--contains` / no-`--tags`
     invocations and cannot be emitted by `git describe --tags --abbrev=0`; matching it would be a
     dead alternate implying a discrimination this code does not perform.
   - **When `LAST_TAG` is set** → the commit range is `"$LAST_TAG..origin/<BASE-BRANCH>"`.
   - **When the repo has NO tags** (first-ever release) → the range is the **single-ended**
     `origin/<BASE-BRANCH>` — every commit reachable from base head, the root commit included.
     Deliberately **not** an empty range start: `"..origin/<BASE-BRANCH>"` would let git default the
     omitted left side to `HEAD`, which on an up-to-date base checkout yields an **empty** range →
     zero keys → a "no stories merged" no-op → the first-ever release silently shipping nothing
     while reporting success. Deliberately **not** `<root-sha>..origin/<BASE-BRANCH>` either — that
     form **excludes** the root commit itself. The single-ended form is the only one that is both
     non-empty and root-inclusive.
   - **Enumerate merged stories** from `RANGE` with a **delimited, machine-parseable** format —
     one record per commit, three fields:

     ```bash
     # US (0x1f) separates fields; RS (0x1e) terminates each commit record.
     # Neither byte can occur in a commit message, so the parse is unambiguous.
     git log "$RANGE" --format='%H%x1f%s%x1f%b%x1e'
     ```

     Split the output on RS into records, then each record on US into
     `(hash, subject, body)`. For each record, extract Jira story keys (regex
     `[A-Z][A-Z0-9]*-[0-9]+`) from **subject and body**; the de-duplicated union across all records
     is the merged-story key set, and each key retains the `(hash, subject, body)` of the record(s)
     that carried it.
   - **Why not `--format='%s%n%b'`.** That format emits **no record separator and no field
     separator**, and bodies are multi-line — so there is no way to tell where one commit's body
     ends and the next commit's subject begins. Every downstream rule that says "the commit that
     carried the key", "the subject", "the body", or "the most recent commit" is then underivable.
     The delimited format above is what makes those rules implementable; do not simplify it back.
   - **"Most recent" is well-defined:** `git log` emits newest-first by default, so for a key
     appearing in several records, the **first** record encountered is its most recent commit. Do
     not add `--reverse` without also inverting that rule.
   - `gh` is **not** consulted for any content that lands in a file — see the
     [`gh` boundary](#changelog-aggregation-inputs-repo-derived).
   - **No stories merged in `RANGE` (AC6)** → **clean no-op**: print a one-line notice and exit
     without a PR. The notice interpolates `LAST_TAG` **only when it is set** — never emit an empty
     interpolation:
     - `LAST_TAG` set → `no stories merged since <LAST_TAG> — nothing to release`
     - `LAST_TAG` empty (no tags yet) → `no stories merged since the start of history — nothing to release`

4. **Phase 1 dispatch — `knowledge-engineer` computes & drafts, writes nothing.** Pass it
   `VERSION`, `ENABLED_ROWS`, the resolved commit range, the merged-story key set (each key with its
   `(hash, subject, body)` records), and `origin/<BASE-BRANCH>`. Phase 1 drafts **only the artifacts
   whose row is in `ENABLED_ROWS`** — each bullet below is conditional on its row:
   - if `changelog ∈ ENABLED_ROWS` → draft the **changelog** entry for `VERSION` (see
     [Narrative artifacts](#narrative-artifacts-changelog--release-notes--migration-guide));
   - if `release-notes ∈ ENABLED_ROWS` → draft the **release notes** for `VERSION`, resolving each
     story's motivating ADR link (see [ADR-link resolution](#adr-link-resolution-ac2ac5));
   - if `migration-guide ∈ ENABLED_ROWS` → draft the **migration-guide stub** for `VERSION`
     (headings/scaffold only — no body prose);
   - returns the drafts **it actually produced** (plus the resolved merged-story set and ADR-link
     map) to the command layer, each labelled with its row so the gate and phase 2 can filter on the
     same subset. **Nothing is written to disk in phase 1.**

5. **Founder-confirm gate (command layer, AC3).** Present **every narrative draft phase 1 returned —
   and only those** (i.e. the `ENABLED_ROWS` subset, never a fixed set of three) — and wait for
   explicit founder confirmation. The founder may accept, edit, or reject each. AC3 names the
   release notes and migration stub explicitly; an enabled changelog draft rides the **same** gate
   because its registry `generation-mode` is `draft-for-review` (see
   [Founder-confirm gate scope](#founder-confirm-gate-scope-decision)). If `ENABLED_ROWS` resolved to
   a single row, the gate presents a single draft. A dispatched subagent cannot pause for
   interactive input — the gate lives at the command layer between the two dispatches, exactly as
   `sync` and `/sdlc:adr` do. **This gate is the only place release content is edited** — see
   [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them).

6. **Phase 2 dispatch — `knowledge-engineer` writes confirmed, fresh dispatch.** The command hands
   phase 2 `VERSION`, `ENABLED_ROWS`, and the founder-confirmed drafts **verbatim** (inline or via
   `scripts/tmp-dir.sh` temp files by path — never re-derived). Phase 2:
   - checks out the release branch per the [re-run / idempotence contract](#re-run--idempotence-contract)
     (a fresh branch cut from `origin/<BASE-BRANCH>` head on a first run; the **existing** branch at
     its remote head on a re-run, after that contract's guards have passed);
   - writes the confirmed content **for the rows in `ENABLED_ROWS` only** under their
     manifest-resolved `target-path`s — a row absent from `ENABLED_ROWS` gets no page, and an
     existing page belonging to a now-disabled row is left untouched (never created, never deleted);
   - applies the [changelog upsert rule](#changelog-upsert-rule-cumulative-file) when writing the
     cumulative changelog;
   - **deterministically regenerates the doc index + `llms.txt` (AC4)** — see
     [Deterministic index + llms.txt regen](#deterministic-index--llmstxt-regen-ac4);
   - **only if `git status --porcelain` on the written paths is non-empty** — commits via
     `conventional-commit` (with the `Release-Generated:` trailer — see
     [Branch / PR / Control Flow](#branch--pr--control-flow)), pushes, and opens or updates the
     release PR (AC4). If the founder rejected every draft and the deterministic regen produced
     byte-identical output → no commit, no PR, clean exit.

### Narrative artifacts (changelog / release-notes / migration-guide)

Each artifact maps onto an existing `writing-docs` quadrant template and its registry `target-path`;
the **normalised** `VERSION` is used as the identifier. A row is only in play when it is in
`ENABLED_ROWS`:

| Artifact          | Registry quadrant | `writing-docs` template | Default `target-path`    | File convention                                                                                                                                                                                                                                   |
| ----------------- | ----------------- | ----------------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `changelog`       | reference         | Reference               | `docs/changelog/`        | Single **cumulative** file (e.g. `docs/changelog/index.md`, Keep-a-Changelog style) — each release **upserts** its `## <VERSION>` section (see [upsert rule](#changelog-upsert-rule-cumulative-file)); the file self-indexes releases newest-first |
| `release-notes`   | explanation       | Explanation             | `docs/release-notes/`    | **One page per version** (e.g. `docs/release-notes/<VERSION>.md`); each note links its motivating ADR(s)                                                                                                                                          |
| `migration-guide` | how-to            | How-to                  | `docs/migration-guides/` | **One stub page per version** (e.g. `docs/migration-guides/<VERSION>.md`) — headings/scaffold only (AC2)                                                                                                                                          |

- **Changelog** entry: deterministically **aggregated** from the merged-story key set — grouped by
  conventional-commit change type (Added / Changed / Fixed / …), one line per merged story
  (`<KEY> — <summary>`). Its inputs are specified in
  [Changelog aggregation inputs](#changelog-aggregation-inputs-repo-derived) — every one of them is
  repo-derived, with **no** `gh` dependency. Presented at the founder-confirm gate as a draft (its
  registry mode is `draft-for-review`), but its content is a mechanical aggregation, not free prose.
- **Release notes**: explanation-quadrant prose per `writing-docs`, one entry per merged story,
  each linking its motivating ADR when one exists (AC2). The prose is narrative — fully gated.
- **Migration-guide stub**: headings/scaffold **only** (AC2) — e.g. a `## Breaking changes`,
  `## Upgrade steps`, `## Rollback` skeleton with no body content. Deep migration content is
  explicitly out of scope (Epic + story out-of-scope; deferred open question).

### Changelog aggregation inputs (repo-derived)

The changelog needs a **change type** and a **summary** per merged story. Both come from the
`(hash, subject, body)` records the [enumeration](#release-version--behavioural-contract) already
parsed — the **subject** for type and summary, the **body** for the breaking-change footer. No
additional git call and **no network call** is needed:

| Need        | Repo-derived source (always available)                                                                                                                                                                                                                                                                                                                                                                                     |
| ----------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Change type | The **subject's** conventional-commit `type` (this repo commits via the `conventional-commit` skill, so it is present): `feat` → **Added**; `fix` → **Fixed**; any other recognised type (`refactor`/`perf`/`docs`/`chore`/…) → **Changed**. Promoted to **Breaking** by either a `!` marker in the **subject** (`feat!:`) or a `BREAKING CHANGE:` **footer** in the body (see the footer rule below). A subject with **no** parseable prefix → **Changed** (the neutral bucket — never dropped, never guessed at). |
| Summary     | The **subject's** description text — the remainder after `type(scope):` — used verbatim, trimmed.                                                                                                                                                                                                                                                                                                                           |

- **The `BREAKING CHANGE:` test is line-anchored, never a substring search.** Per Conventional
  Commits, it is a **footer**: the token must begin a line (`^BREAKING[ -]CHANGE:`) in the body.
  A body that merely *mentions* the string in prose is **not** a breaking change. This is a live
  hazard, not a hypothetical — this spec's own commit history contains commit messages discussing
  `BREAKING CHANGE:` in prose, and a naive `"BREAKING CHANGE:" in body` test flags them as breaking.
  Match on a line anchor.
- **The `gh` boundary: `gh` never contributes to written content.** An earlier draft of this spec let
  a successful `gh` lookup replace the summary with the merged PR title, falling back to the commit
  subject when `gh` was unavailable. That makes file content depend on **ambient network state**: run
  1 with `gh` up writes PR titles, run 2 rate-limited rewrites the same release with commit subjects
  → a non-empty `git status --porcelain` → a spurious commit that *degrades* a changelog already
  under review, and a direct contradiction of the
  [re-run / idempotence contract](#re-run--idempotence-contract). So: `gh` MAY be used to enrich what
  is **displayed** to the founder at the confirm gate (e.g. showing PR titles alongside keys as
  context), and MUST NOT feed any byte that is written to a file. Written content is a pure function
  of the commit range.
- **A story whose key appears in multiple records** takes the highest-precedence change type among
  them (Breaking > Added > Fixed > Changed) and the subject of its **most recent** record as the
  summary — i.e. the first record encountered, since `git log` is newest-first. Deterministic, so a
  re-run over the same range yields byte-identical output.
- **Never fabricate.** The changelog emits exactly what the commits support. No inferred prose, no
  invented summaries — the same no-fabrication discipline the ADR-link rule applies at
  [AC5](#adr-link-resolution-ac2ac5). A bare-key line is never emitted, because the fallbacks above
  always resolve both fields.

### Changelog upsert rule (cumulative file)

The cumulative changelog is maintained by **upsert**, not by unconditional prepend — a prepend-only
rule contradicts the [idempotence contract](#re-run--idempotence-contract) below, because `release`
creates no tags (see [Out of Scope](#out-of-scope)), so a re-run of a version whose PR already
merged would find its `## <VERSION>` section already present and prepend a **second** one:

1. **Scan** the cumulative file for an existing `## <VERSION>` heading (exact match on the
   normalised `VERSION` — which is precisely what
   [validation step 5](#argument-validation)'s normalisation guarantees is a single stable
   spelling; without it `## 1.4.0` and `## v1.4.0` would be distinct sections).
2. **Section exists** → **replace it in place**: swap the existing section's body for the freshly
   aggregated content, leaving its position in the file and every other section byte-identical. A
   re-run with no new merged stories therefore rewrites the section with identical bytes → empty
   `git status --porcelain` → no commit, no PR. (The replacement is unconditional, which is why
   edits to that section are governed by
   [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them).)
3. **Section absent** → **prepend** a new `## <VERSION>` section above the newest existing section,
   preserving the file's newest-first ordering.

The upsert boundary is the `## <VERSION>` heading and everything up to (not including) the next
`## ` heading or EOF. The file's preamble (title, Keep-a-Changelog blurb) is never touched.

### ADR-link resolution (AC2/AC5)

For each merged-story key, resolve its **motivating ADR** by scanning `docs/adr/**` frontmatter for
any ADR whose `source-stories:` list contains that key:

- **Match found** → the release note for that story links the ADR (relative link to the
  `docs/adr/NNNN-*.md` page, using the ADR title).
- **No match (AC5)** → the release note for that story **omits** the ADR link entirely — never
  fabricate, guess, or invent an ADR reference. A story with no ADR simply has an ADR-less note.
- Multiple ADRs may cite the same story; link each. The scan is read-only against `docs/adr/` at the
  release branch's base (`origin/<BASE-BRANCH>` head); if the repo has **no** `docs/adr/` directory
  at all, every note is ADR-less (not an error).

### Founder-confirm gate scope (decision)

AC3 names "release notes, migration stub" as the narrative content held at the gate. The
`refs/doc-types.md` registry marks **all three** release artifacts (`changelog`, `release-notes`,
`migration-guide`) as `generation-mode = draft-for-review`. This spec resolves the two by gating
**every enabled release artifact**: an enabled changelog draft is presented at the same
founder-confirm gate as the release notes and migration stub. This honours the registry's
`draft-for-review` mode for the changelog and satisfies AC3 (which names the prose-heavy artifacts
as a minimum, not an exhaustive list). The changelog's content is a mechanical aggregation, so in
practice the founder confirms it with little or no editing — but it is never written un-confirmed.
The gate's scope is always `ENABLED_ROWS`, never a hardcoded three — a disabled row has no draft to
gate.

### Deterministic index + llms.txt regen (AC4)

After the confirmed narrative writes, phase 2 deterministically regenerates the doc index and
`llms.txt` — the deterministic, un-gated half of the run (the same "auto rows are un-gated"
discipline `sync` applies):

- **`llms.txt`** — regenerated by reusing `sync`'s `llms-txt` algorithm
  (`refs/docs-pipeline.md` §8): an index-only file, grouping the generated pages of every enabled
  `public: yes` manifest row by Diátaxis quadrant, each entry a `title — one-line description —
  relative link` derived from page frontmatter. The newly written release pages now appear in it.
  Idempotent; committed only if changed.
- **The doc index** — for the `changelog` target-path, the cumulative changelog file **is** the
  index of releases (newest-first), maintained in place by the
  [upsert rule](#changelog-upsert-rule-cumulative-file). For `release-notes` / `migration-guides`
  target-paths, if the row is enabled **and** the consumer's docs tree carries a section index page
  at that path, regenerate it deterministically (**upsert** the `<VERSION>` entry — add it if
  absent, rewrite it in place if present; never append a duplicate); if none exists, `llms.txt` is
  the sole index and no separate section index is created. See
  [Open Questions](#open-questions) for the resolved default on "the doc index" identity.

### Where edits live, and what a re-run does to them

Release content can be edited in exactly two places, and a re-run treats them **differently**. This
section is normative; the guards in
[Branch / PR / Control Flow](#branch--pr--control-flow) implement it.

| Edit made…                                                                              | Committed by                                                | A re-run…                                                                                                                                                                                                                                                                                                                          |
| --------------------------------------------------------------------------------------- | ----------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **At the founder-confirm gate** (the founder edits a draft before confirming it)        | phase 2, carrying the `Release-Generated: <VERSION>` trailer | **Re-derives it.** Phase 1 re-drafts from the commit range every run and the gate re-presents the machine draft — a prior run's gate edit is **not** carried forward. To keep it, re-apply it at this run's gate. Not silent (the founder sees and confirms what will be written), but **not preserved either**. See the OQ on seeding. |
| **On the branch, outside the pipeline** (a `/sdlc:loop` review-fix commit, a direct push) | anything **without** the trailer                            | **STOPs.** The [re-run content guard](#branch--pr--control-flow) refuses to overwrite it and hands the decision back to the founder.                                                                                                                                                                                                |

The distinction is the **trailer, not the commit subject**. Keying on the subject would be wrong:
phase 2 commits founder gate edits under exactly the normal `docs(docs): release <VERSION>` subject,
so a subject test classifies them as pipeline-authored and the guard never fires — which is correct
for gate edits (they are re-derived by design) but must not be mistaken for *preserving* them.

### Re-run / idempotence contract

Re-running `release <VERSION>` over the **same commit range**, with the founder confirming the
**same content** at the gate, recomputes **byte-identical** content for every enabled row, leaves
`git status --porcelain` empty, and therefore commits nothing and opens no PR (AC6).

**The claim is scoped, and the scope is load-bearing.** It is a statement about the *generator*:
same range + same confirmed content → same bytes. It does **not** claim that edits survive a re-run
untouched — see [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them) for what
actually happens to each kind (gate edits are re-derived and re-gated; out-of-pipeline edits STOP
the run). Five things are load-bearing, and none may be dropped:

1. **`VERSION` normalisation** ([validation step 5](#argument-validation)) — one spelling per
   release, so a re-run targets the same branch, the same PR, the same pages, and the same changelog
   section rather than creating a parallel set.
2. **Changelog upsert** ([upsert rule](#changelog-upsert-rule-cumulative-file)) — a re-run rewrites
   its section in place instead of prepending a duplicate.
3. **A parseable enumeration** ([step 3](#release-version--behavioural-contract)) — the RS/US
   delimited `git log` format; without it "the commit that carried the key" is underivable and the
   aggregation is not a function of anything well-defined.
4. **Content is a pure function of the commit range** ([aggregation inputs](#changelog-aggregation-inputs-repo-derived)) —
   no `gh`, no network, no ambient state may reach a written byte, or "byte-identical" would depend
   on whether `gh` happened to answer.
5. **Non-destructive branch re-use** ([Branch / PR / Control Flow](#branch--pr--control-flow)) — a
   re-run preserves every commit on the branch (never resets onto regenerated state, never
   force-pushes), and STOPs rather than overwrite out-of-pipeline edits to generated pages.

## Branch / PR / Control Flow

| Item                     | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Branch                   | `docs/release-<VERSION>` (the **normalised** version — see [validation step 5](#argument-validation); `1.4.0` and `v1.4.0` resolve to the same branch, never two), cut from the **base branch head** (`origin/<BASE-BRANCH>`) — **not** a story branch. Release aggregates work already merged to base, so the base tree is the correct source.                                                                                                                                                                                                                                                                                                                                                                                                     |
| Commit                   | `docs(docs): release <VERSION>` (via `conventional-commit`), carrying the trailer `Release-Generated: <VERSION>`. **The trailer is load-bearing**, not decoration: it is the only reliable marker of "this pipeline wrote this commit", and both guards below key on it. Subject-matching is not a substitute (see [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them)).                                                                                                                                                                                                                                                                                                                                                            |
| PR title                 | `docs(docs): release <VERSION>`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| PR base                  | `<BASE-BRANCH>` from project-context (never assume `main`)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| Local-branch precondition | Applies on **every** path, first-run and re-run alike, and is checked **before** either. If a **local** `docs/release-<VERSION>` exists holding commits that are not reachable from `origin/docs/release-<VERSION>` (or, when no such remote exists, any commits at all beyond `origin/<BASE-BRANCH>`) → **STOP**: `local branch docs/release-<VERSION> has unpushed commits; push, drop, or rename it, then re-run.` Never `checkout -B` over it and never create a branch on top of it. Hoisted out of the re-run rows deliberately: the first-run path is defined by the branch being absent **on `origin`**, which says nothing about a local branch of the same name — so a local-only branch with unpushed work would otherwise be clobbered. |
| First-run PR             | No remote branch (`git rev-parse --verify origin/docs/release-<VERSION>` fails) → after the local-branch precondition passes, create the branch from `origin/<BASE-BRANCH>` head, write, commit, push, open the PR (`gh pr create`) against `<BASE-BRANCH>`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| Re-run behaviour         | **Open or update; the branch's commits are never rewritten** (AC4/AC6). If `docs/release-<VERSION>` already exists on `origin` (`git rev-parse --verify origin/docs/release-<VERSION>` / `gh pr list --head docs/release-<VERSION>`) → `git fetch origin`, run the precondition + **both guards**, then check the branch out **at its remote head** (`git checkout -B docs/release-<VERSION> origin/docs/release-<VERSION>`) — a fresh checkout of the remote state, which is the single normative flow. Then write the regenerated content **on top** as a new commit and `git push` (a plain fast-forward). Convergence comes from idempotence, not from rewriting: a re-run with nothing to change produces **no commit, no push, no duplicate PR**. |
| Re-run history guard     | The **one forbidden path**: no re-run may reset the branch onto regenerated state, force-push it (`--force`/`--force-with-lease`), or discard any commit reachable from `origin/docs/release-<VERSION>`. The control-flow tail below drives `/loop /sdlc:loop` against this same branch, so it carries **review-fix commits not derived from the drafts**; a reset-to-regenerated + `--force-with-lease` would *succeed* (the local ref was just fetched) and silently revert the PR to unreviewed content. Checking the branch out at its remote head (above) does none of these — it adopts the remote's commits rather than replacing them.                                                                                                          |
| Re-run content guard     | Preserving commits protects **history**; this guard protects **content**. Before writing, find commits on the branch (relative to `origin/<BASE-BRANCH>`) that touch the `ENABLED_ROWS` target-paths and **lack the `Release-Generated: <VERSION>` trailer** — i.e. out-of-pipeline edits to generated pages. If any → **STOP**: `branch docs/release-<VERSION> carries edits to generated pages (<paths>) that this pipeline did not write; re-running would overwrite them — merge or close PR #<n>, or drop those edits, then re-run.` Necessary because phase 2 writes confirmed drafts **verbatim** and the [upsert](#changelog-upsert-rule-cumulative-file) replaces the section body unconditionally. Trailer-bearing commits and commits touching **other** paths do not trip it — proceed. |
| Control-flow tail        | Mirror `sync` / `commands/adr.md`: after the PR is raised, drive the review loop to convergence via `/loop /sdlc:loop <PR_URL>` (falling back to `ScheduleWakeup` if the harness cannot nest `/loop`). If the run ended **before** any PR (manifest-absent silent no-op, no-row-enabled no-op, no-stories-merged no-op, an invalid-version STOP, a precondition/guard STOP, or another usage STOP), release directly via `scripts/session-complete.sh` — only the manifest-absent path is silent.                                                                                                                                                                                                                                                     |

## Permissions Detail

Not a multi-role RBAC surface. The single actor is the **founder** running `/sdlc:docs release`. The
one authority boundary is the **founder-confirm gate**: the deterministic doc index + `llms.txt`
regen is written without confirmation; the enabled narrative artifacts (changelog, release notes,
migration stub) are written only after explicit founder confirmation at the command layer. A
dispatched subagent never runs the gate itself — the command layer owns it, exactly as
`commands/docs.md` (`sync`) and `commands/adr.md` do.

A second, quieter authority boundary: the founder's manifest is what enables a row. `release` never
creates a page for a row the founder disabled (`ENABLED_ROWS`, step 2), never rewrites the commits
on a branch under review (re-run history guard), never discards unpushed local work (local-branch
precondition), and never overwrites an out-of-pipeline edit to a generated page — it STOPs and hands
the decision back to the founder (re-run content guard). Gate edits are the one thing a re-run
re-derives rather than preserves, and the founder sees that at the gate before it is written — see
[Where edits live](#where-edits-live-and-what-a-re-run-does-to-them).

| Repo has no `docs/adr/` directory                                                                               | Every release note is ADR-less; not an error.                                                                                                                                                                                                                                                        |
| `refs/doc-types.md` unreadable/malformed                                                                        | Surface the failure and STOP — never generate from a partial registry.                                                                                                                                                                                                                               |
| A **local** `docs/release-<VERSION>` holds commits not reachable from its remote (or, with no remote, any at all) | **STOP** (local-branch precondition) — never discard unpushed work, and never clobber it on the first-run path either.                                                                                                                                                                                |
| Re-run: branch exists on `origin`, all page-touching commits carry the trailer                                   | Not an error — check out at the remote head, write on top, commit + fast-forward push only if content changed. Never reset onto regenerated state, never force-push.                                                                                                                                  |
| Re-run: branch carries commits touching **non**-generated paths                                                 | Not an error — proceed with the write-on-top; those commits are preserved.                                                                                                                                                                                                                            |
| Re-run: branch carries **out-of-pipeline** edits to generated pages (no `Release-Generated:` trailer)           | **STOP** (re-run content guard) — re-running would overwrite them. Surface the paths and the PR number; the founder merges/closes the PR or drops the edits, then re-runs.                                                                                                                             |
| Re-run: a prior run's **founder gate edit** is present on a generated page                                       | **Re-derived**, not preserved and not STOPped — phase 1 re-drafts and the gate re-presents; the founder re-applies the edit (or accepts the draft) before anything is written. See [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them).                                               |
| Re-run: `## <VERSION>` section already present in the cumulative changelog                                      | Not an error — the section is **replaced in place** (upsert), never prepended a second time.                                                                                                                                                                                                          |
| Founder rejects every enabled draft and deterministic regen is byte-identical                                   | No commit, no PR (AC4/AC6) — write phase detected an empty `git status --porcelain`.                                                                                                                                                                                                                  |
| Founder rejects the narrative drafts but the deterministic `llms.txt`/index changed                             | Still commit + PR the deterministic regen (content changed).                                                                                                                                                                                                                                          |
| `gh` / `raise-pr.sh` failure                                                                                    | STOP and surface — the write is on a branch and reviewable; never leave a raised-but-broken state silently.                                                                                                                                                                                           |

## Out of Scope

- **seed / audit / distill modes** — own stories (NA-54 / NA-55 / NA-57). `docs.md` keeps them as
  clean "not yet implemented" stubs; only `release` graduates from stub in this story.
- **Deep migration-guide content generation** — the migration guide is a headings/scaffold **stub
  only** (AC2). Generating actual upgrade prose/steps is a deferred open question (Epic + story).
- **Auto-triggering `release` on tag push / CI events** — later enhancement (Epic out-of-scope).
- **Creating the git tag itself** — `release <version>` documents a release; it does not cut or push
  the version tag. The tag is assumed to be created by the existing release/versioning process.
  (This is exactly why the changelog needs the
  [upsert rule](#changelog-upsert-rule-cumulative-file): a version can legitimately be re-run after
  its PR merged but before its tag is cut.)
- **Merging edits into a re-run** — when the release branch carries out-of-pipeline edits to
  generated pages, `release` STOPs rather than attempting a three-way merge of founder prose with
  freshly aggregated content. Reconciling the two automatically is out of scope for v1.
- **`llms-full.txt`** (full-content variant) — v1 ships index-only `llms.txt` (inherited from NA-52).
- **Editing `refs/doc-types.md` registry rows** — the `changelog` / `release-notes` /
  `migration-guide` rows already exist (NA-51); `release` reads them, never edits them.
- **Deleting or migrating pages for a newly-disabled row** — if a founder disables `release-notes`
  after earlier releases created pages, `release` simply stops writing them; it never deletes the
  existing pages.
- **Building or hosting a docs site** — Mintlify is this repo's choice; the plugin stays
  stack-agnostic via manifest `target-path`s (Epic out-of-scope).
- **Marketing/SEO release copy** — `gtm:docs` owns that; `sdlc release` is technical accuracy only.

## Open Questions

- [ ] **"The doc index" identity (AC4)** — AC4 says regenerate "the doc index and `llms.txt`."
      _Suggested default (adopt):_ `llms.txt` is the deterministic meta index that is always
      regenerated; the cumulative changelog file self-indexes releases (newest-first); a separate
      per-section index page is regenerated deterministically only if the consumer's docs tree
      already carries one at the `release-notes`/`migration-guides` target-path. No new index file
      type is invented.
- [ ] **Seeding a re-run's drafts from existing pages** — today phase 1 re-drafts from the commit
      range every run, so a prior run's founder gate edit is re-derived and the founder must
      re-apply it (see [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them)).
      _Suggested default (adopt for v1):_ keep re-deriving — it keeps written content a pure
      function of the commit range, which items 3-4 of the
      [idempotence contract](#re-run--idempotence-contract) depend on. _Alternative (revisit if the
      re-apply burden bites):_ when the release branch already exists, seed each draft from the
      current page content so the gate presents what is actually on the branch. This is strictly
      better UX but makes drafts a function of branch state as well as range — do not adopt it
      without re-deriving the idempotence claim.
- [ ] **`<version>` validation strictness** — _Suggested default (adopt):_ hard-validate a
      **token-safety** regex (plus the `endswith(".lock")` ref rule) and normalise away a leading
      `v` (both required: `<VERSION>` forms a branch name and page ids, and the two spellings must
      not produce duplicate branches/PRs/pages), but do **not** hard-validate semver beyond that. A
      scripted caller passing a non-semver label (e.g. a date-based version) must not be rejected.
      Revisit if a consumer needs enforcement.
- [ ] **`<version>` case folding** — `1.4.0-RC1` and `1.4.0-rc1` are distinct `VERSION`s that collide
      on case-insensitive filesystems (macOS). _Suggested default (adopt):_ do not fold case; accept
      the collision. Folding would break legitimately case-distinct labels, and the collision needs
      two spellings of one release in one repo to bite. Revisit if it bites in practice.
- [ ] **Changelog file shape** — cumulative single file (Keep-a-Changelog, newest-first, upsert per
      version) vs one file per version. _Suggested default (adopt):_ cumulative single file at the
      `changelog` target-path — it is self-indexing and matches the reference-quadrant "one
      consultable surface" shape; release notes and migration guides remain one page per version.
- [ ] **Merged-story enumeration source** — git-log story-key regex over the tag range vs `gh`
      merged-PR listing. _Suggested default (adopt):_ git-log extraction is the sole source of
      written content — keys from subjects+bodies of the RS/US-delimited records, change type +
      summary from the record's conventional-commit subject (and the body's line-anchored
      `BREAKING CHANGE:` footer). `gh` is display-only enrichment at the confirm gate and may never
      reach a written byte, so output stays byte-identical whether or not `gh` answers.
- [ ] **Migration-guide stub depth** — headings/scaffold only for v1 (per AC2). _Suggested default
      (adopt):_ ship stub-only; deep migration-content generation is deferred (Epic open question).
