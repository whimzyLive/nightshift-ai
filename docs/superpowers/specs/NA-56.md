# Wire post-QA docs sync into impl/auto — Technical Spec

**Story:** NA-56 — Wire post-QA docs sync into impl/auto
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-18

## Overview

Wire the existing `/sdlc:docs sync` core into the two implementation playbooks so a docs sync fires **automatically on the story branch right after a clean QA verdict** — landing the regenerated docs as commits in the **same PR** as the code that made them stale, with no manual `/sdlc:docs sync` step. To make `sync` usable for the recovery case this creates, this story also gives `sync` its **second diff source** — a merged-commit range — so a post-merge invocation regenerates from the landed commit instead of the WARNING-and-exit stub the four `deferred to NA-56` markers currently emit.

## Scope of this story (authoring surface)

All edits sit inside `plugins/sdlc/`, executed by the **ai-enablement-engineer** (owner of `plugins/`, `skills/`, `.agents/` per the project-context workspace→agent table).

## Trigger point + mechanism (the core decision)

### Trigger point — post-QA-clean, pre-PR-create, on the story branch

The wiring lives as **Step 6.5 of `refs/principal-engineer-playbook.md`**, inserted **after Step 6 returns `Status: clean`** and **before Step 7 creates the PR**. Because both `/impl` (its Step 4) and `/auto` (Workflow A2 step 3 and Workflow B1) execute this playbook **inline**, a single insertion satisfies AC1 for both commands — no divergent copy in `impl.md`/`auto.md`.

This is a **post-QA / pre-merge** trigger, not post-merge:

- **AC1** — "on the story branch right after a clean QA verdict": at Step 6.5 the branch `<BRANCH_PREFIX>/<STORY-KEY>` is pushed and **unmerged**, and its `$WORKTREE` is still provisioned (teardown is Step 7).
- **AC2** — "docs land in the same PR as the code": running before Step 7 means the docs commit is on `<BRANCH_PREFIX>/<STORY-KEY>` when the PR is opened, so it is part of the impl PR — no second branch, no second PR.
- **AC5** — "when the QA verdict is not clean, the phase does not run": Step 6 STOPs the whole playbook on `Status: blocked` (never reaching Step 6.5). Step 6.5 is reached **only** on the `clean` verdict.

### Mechanism — the inline post-QA dispatch variant (how `sync` composes with a run that already owns a branch+PR)

Standalone `sync` (docs-pipeline §7) cuts its own `docs/sync-<KEY>` branch, runs a two-phase founder-confirm gate, and raises its own PR. That control flow is **wrong** for a run that already owns `<BRANCH_PREFIX>/<STORY-KEY>` and will open one PR. Step 6.5 therefore invokes a new **inline post-QA variant** of the docs-sync dispatch, defined once in `refs/docs-pipeline.md` and honoured by `knowledge-engineer.md`. Its four divergences from standalone `sync`, each with the reason it exists:

1. **No `docs/sync-<KEY>` branch — writes on the story branch in `$WORKTREE`.** The orchestrator hands the agent the live `$WORKTREE` (checked out at `<BRANCH_PREFIX>/<STORY-KEY>`) and `$NX_CACHE_DIRECTORY`, exactly as Step 4/QA-Step-3 domain dispatches do. The agent does **not** check out or cut any branch. _Reason:_ docs must land on the impl branch (AC2); a separate branch would need a separate PR.
2. **Single dispatch, no founder-confirm gate.** The two-phase compute→gate→write split exists only to host the founder-confirm gate across a dispatch boundary (docs-pipeline §2). AC2 removes that gate — "PR review is the sign-off" — so the variant is a **single dispatch** that computes **and** writes the deterministic regen + `llms.txt` **and** the narrative how-to refreshes in one pass. The how-to drafts are written un-gated and reviewed in the impl PR. _Reason:_ a dispatched subagent cannot pause for interactive input; with the gate removed there is nothing to split across, and the human review moves to the PR.
3. **Commit-only; the orchestrator pushes.** The agent commits the docs changes via `conventional-commit` onto `<BRANCH_PREFIX>/<STORY-KEY>` and returns; the orchestrator pushes and runs the same push/primary-checkout guard as playbook Step 5. _Reason:_ matches the `domain-agent-handoff.md` "agent commits, orchestrator pushes" contract every other in-playbook dispatch obeys — the agent's self-raise-PR behaviour applies only to **standalone** `/sdlc:docs`/`/sdlc:adr`, never to an in-playbook dispatch.
4. **Raises no PR.** Step 7 folds the docs commit into the impl PR. _Reason:_ AC2.

The **deterministic regen algorithm, the `source:` how-to convention, the affected-row resolution, the manifest gate, and the change-gate (commit only if `git status --porcelain` non-empty)** are **unchanged** and remain owned by `refs/docs-pipeline.md` §§1–6 — the variant reuses them verbatim and the playbook points at them; it never restates them.

## Diff source — dual source + selection rule (AC3)

`sync` gains a **second diff source** so it works both before and after the story branch merges. The two sources and the rule that selects between them:

| Diff source                                | `CHANGED_FILES` / `CHANGED_DIFF` derivation                                                                                              | Used when                                                                                                                                                                            |
| ------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **story-branch-vs-base** (existing, NA-52) | `git diff [--name-only] "origin/<BASE-BRANCH>...$STORY_BRANCH"` — three-dot range, remote-tracking base, after `git fetch origin --quiet` | The story branch **exists on origin** (`origin/feat/<STORY-KEY>` or `origin/fix/<STORY-KEY>`). This is what the **post-QA phase always selects** (the branch is present and unmerged). |
| **merged-commit** (NEW, this story)        | Locate the commit(s) on `origin/<BASE-BRANCH>` carrying `<STORY-KEY>`, then `git diff [--name-only] "<sha>^..<sha>"` (see below)          | The story branch is **absent on origin** — a post-merge (squash-merged, branch deleted) or never-branched invocation of standalone `sync`.                                           |

### Selection rule (deterministic, no ambient input)

1. Resolve `STORY_BRANCH` (docs-pipeline §2 step 2 / `commands/docs.md` step 2): `origin/feat/<STORY-KEY>` preferred, `origin/fix/<STORY-KEY>` fallback.
2. **`STORY_BRANCH` resolved** → **story-branch-vs-base** source (unchanged v1 behaviour).
3. **Neither branch exists on origin** → **merged-commit** source (this replaces the current `deferred to NA-56` WARNING-and-exit).

The **post-QA phase** passes the phase's diff source explicitly (story-branch-vs-base against `<BRANCH_PREFIX>/<STORY-KEY>`) — it never runs the merged-commit branch, because at Step 6.5 the branch always exists. "The playbook selects the correct one for its phase" (AC3) means the playbook hands the agent the story-branch source; the merged-commit branch is reached only by a standalone post-merge `sync`.

### Merged-commit source — precise definition

The story branch is gone, so the diff comes from the landed commit(s) on base:

1. **Locate** the commit(s) on `origin/<BASE-BRANCH>` whose subject **or** body carries `<STORY-KEY>`, using the **`PROJECT_KEYS`-scoped alternation regex** from `refs/docs-pipeline.md` §10 (`\b(?:KEY1|KEY2|…)-[0-9]+\b`) — **never** the loose `[A-Z][A-Z0-9]*-[0-9]+` matcher (it false-positives on `UTF-8`, `SHA-1`, etc., per §10). Scan `git log origin/<BASE-BRANCH>` after `git fetch origin --quiet`.
2. **Diff derivation** — this repo squash-merges (`gh pr merge --squash`), so a story lands as a **single** commit with **one** parent: `CHANGED_FILES=$(git diff --name-only "<sha>^..<sha>")`, `CHANGED_DIFF=$(git diff "<sha>^..<sha>")`. For a true merge commit (two parents) use the first-parent form `"<sha>^1..<sha>"`.
3. **Zero commits match** `<STORY-KEY>` on base → **STOP with an explicit error** (`cannot locate a merged commit for <STORY-KEY> on origin/<BASE-BRANCH> — nothing to diff`), never a silent no-op. _Reason (Standing lesson 1):_ "the branch is gone AND no landed commit references the key" is a real failure — the caller asked to sync a story that isn't there — and must be visible, not collapsed into the benign "docs already current" path.
4. **Multiple commits match** (the story landed across several merges — e.g. a follow-up fix) → diff the **union** of each matching commit's `<sha>^..<sha>` file/hunk set, mirroring §10's "de-duplicated union across records". See [Open Questions](#open-questions) — this is the one genuinely ambiguous sub-point.

`git fetch` failure, or an unresolvable `origin/<BASE-BRANCH>`, is a **STOP** on this path exactly as the shared manifest gate (docs-pipeline §1) already requires — never a fallthrough.

## Failure-mode contract (highest-stakes decision)

The contract **splits by failure class**, because two structurally different failures can occur at Step 6.5 and they must not share one downgrade:

- **Docs-content class** (regen error, docs-commit push failure, `knowledge-engineer` returns `Status: blocked`, empty/no-change result) — a missing docs commit. **Recoverable by construction**: the code branch is untouched and `/sdlc:docs sync <STORY-KEY>` (backed by the merged-commit source this story adds) regenerates the docs after merge.
- **Workspace-integrity class** (the dispatch wrote to or committed in the **primary checkout** instead of `$WORKTREE`). **Not** a docs failure and **not** recoverable-by-construction: the corruption persists past the run — a mutated/dirty primary checkout or a stray commit on the wrong branch breaks the next branch operation and the next epic child. Contamination genuinely can occur — an agent may ignore the cwd instruction, which is exactly why playbook Step 5 verifies it with a machine check rather than trusting the handed `$WORKTREE`.

**A post-QA docs-content failure is NON-BLOCKING for the code PR, but LOUD — never silent — and points at a first-class recovery.** The impl/auto run that reached Step 6.5 is already QA-clean and about to ship; the design refuses to lose delivery of validated code over a docs-regeneration hiccup, but equally refuses to hide a broken docs pipeline. **A workspace-integrity failure is a hard STOP**, identical to Step 5 — the run does not proceed to open the impl PR over a corrupted primary checkout. Step 6.5 classifies its outcome into these **four distinct** buckets (Standing lessons 1 + 6 — distinct failures never collapse into one benign path):

| Outcome at Step 6.5                                                                                                                                    | Class                       | Playbook behaviour                                                                                                                                                                                                                                                                                                                    |
| ------------------------------------------------------------------------------------------------------------------------------------------------------ | --------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `.claude/project/docs-manifest.md` **absent** (shared manifest gate, docs-pipeline §1)                                                                 | **no-op (AC4)**             | Docs phase does not dispatch; **the impl/auto flow proceeds to Step 7 unchanged.** No warning, no report line — the repo opted out of docs. This is a no-op, **not** a failure.                                                                                                                                                        |
| Manifest present; deterministic regen byte-identical **and** no `how-to` affected (change-gate, docs-pipeline §6)                                       | **no-op**                   | Agent makes no commit (`git status --porcelain` empty); phase adds nothing to the PR. Report notes "docs: no changes". Not a failure.                                                                                                                                                                                                 |
| Manifest present; **docs-content failure** — regen error, docs-commit push failure, or `knowledge-engineer` returns `Status: blocked`                   | **failure (loud, non-block)** | **WARN, do not block.** Emit a distinct WARNING; **still proceed to Step 7 and open the impl PR** (without the docs commit). The Step 8 report carries a `Docs sync:` line naming the failure and telling the founder to run `/sdlc:docs sync <STORY-KEY>` after merge (now backed by the merged-commit source this story adds).       |
| **Workspace-integrity failure** — the post-QA dispatch changed the **primary checkout** (its HEAD moved, or its `git status --porcelain` no longer matches the pre-dispatch snapshot) | **failure (hard STOP)**     | **STOP the run**, same as playbook Step 5's guard. Do **not** open the impl PR while the primary checkout is corrupted. Surface the exact violation. This is a persistent, non-recoverable workspace corruption — the non-blocking downgrade **does not apply to it**.                                                                 |

**Why non-blocking for the docs-content class, stated explicitly (the story poses this as the decision):** blocking a QA-clean implementation PR because docs regeneration hit a transient error would invert the story's value — "docs land automatically, no extra step" is a convenience layered on top of shipping correct code, not a precondition for it. And the failure is **recoverable by construction**: the merged-commit diff source this same story adds makes the manual `/sdlc:docs sync <STORY-KEY>` recovery work **after** the PR merges. Loud-but-non-blocking + a working recovery path is the balance.

**Why the integrity class stays a hard STOP (the split, stated so it is not blanket-downgraded):** a docs miss is recoverable; a mutated primary checkout is not. Proceeding to Step 7 over a corrupted primary leaks stray commits onto the wrong branch or breaks the next branch operation — damage that outlives this run and breaks the next epic child. The recovery-by-construction argument that justifies the docs-content WARN **does not exist** here, so the same argument must not be stretched to cover it.

**Deliberate divergence — scoped to the docs-content class only.** Playbook Step 5 treats a domain agent's zero-commit return, and a primary-checkout violation, as hard **STOP**s, because the feature code and the workspace integrity are the deliverables. Step 6.5 diverges **only** for the docs-content class (a docs miss → WARNING, because docs are a secondary, regenerable artifact and the code is already validated). It does **not** diverge for the primary-checkout guard — that stays a STOP exactly as Step 5. A future edit must not read this divergence note as covering the integrity guard, and must not "consistency-fix" the docs-content WARNING into a STOP.

**A `Status: blocked` return from `knowledge-engineer` is a docs-content WARNING, not a run STOP** — this is the one place an in-playbook agent `blocked` does not halt the run. State it inline in Step 6.5 so it is not mistaken for the Step 5 / Step 6 `blocked` = STOP rule. (A primary-checkout violation is detected by the machine guard, independently of the agent's returned `Status`, and still STOPs.)

## Determinism / purity (Standing lesson 2)

The post-QA phase inherits the regen's purity guarantees from `refs/docs-pipeline.md` §3 unchanged: the deterministic `auto`-row + `llms.txt` output is a **pure function of the branch tree** — no `gh`, no network, no clock, no ambient state reaches a written byte. The change-gate (commit only on a non-empty `git status --porcelain`) still holds, so a no-source-change re-run of the same story commits nothing. The narrative how-to drafts are non-deterministic, but that is acceptable here precisely because they are reviewed in the impl PR rather than asserted byte-identical. Nothing in the wiring introduces an ambient input to the deterministic set.

## Executing-layer propagation (Standing lesson 3)

The rules the trigger imposes must reach the layer that executes them, not only the playbook:

- **`knowledge-engineer.md`** gains the inline post-QA variant in its "docs-sync dispatch — in summary" and "Branch, memory, commit, return" sections: on this variant it (a) does **not** check out or cut a branch — it works in the handed `$WORKTREE` on `<BRANCH_PREFIX>/<STORY-KEY>`; (b) runs a **single** compute+write pass with **no** founder-confirm gate; (c) **commits only, never pushes, never raises a PR**; (d) uses the diff source the playbook hands it (story-branch-vs-base). Its required-skills stay the docs-sync set (`writing-docs` loaded conditionally only when a `how-to` row is affected, plus the three always-on skills), and the `Skills loaded:` return rule for docs-sync applies to the variant.
- **`refs/docs-pipeline.md`** owns the normative definition of the variant and the merged-commit source; the playbook and `commands/docs.md` reference it. The diff-source selection rule is stated once there.
- **`refs/principal-engineer-playbook.md` Step 6.5** owns the primary-checkout guard (reused from Step 5) — the WARN-vs-STOP split is a playbook-layer decision, made where the dispatch and the guard live, not inside the agent.

## Command-reference auto-regen (Standing lesson 5)

**Do not hand-list the affected reference pages — derive them from the key.** The `command-reference` row is `commands/**`-path-keyed (docs-pipeline §3), so the affected set is **every `commands/**` file this story touches**, read off the artifact table, never a memorised subset. This story edits **three** `commands/**` files — `commands/impl.md`, `commands/auto.md`, **and `commands/docs.md`** — so a subsequent `sync` regenerates **all three** command reference pages. Enumerating only impl/auto (the accuracy edits) under-counts by one and makes an implementer treat the regenerated `docs.md` page as unexpected drift — exactly the hand-list drift class the epic keeps hitting (NA-55's `changelog` tier, the mode-count sentence). The rule is therefore: compute the affected `command-reference` set from the `commands/**` key against the artifact table, not from a literal file list. Keep every edited clause factually exact (they state the playbook now runs a post-QA docs sync before opening the PR) so the regenerated references read true. Editing `refs/**` (playbook, pipeline) does **not** mark `command-reference` affected (it is `commands/**`-keyed), but the `error-reference` content-keyed row **is** affected because the `commands/docs.md` edit touches its error-table section — expected and correct.

## Permissions Detail

Not a multi-role RBAC surface. The actors are the **automated pipeline** (`/impl`, `/auto` running the playbook inline) and the **founder** (who reviews the impl PR). The one authority change vs standalone `sync`: the **founder-confirm gate is removed** on the post-QA path — deterministic **and** narrative docs are written without an interactive gate, and **PR review is the sole sign-off** (AC2). Standalone `/sdlc:docs sync` keeps its founder-confirm gate unchanged.

## Out of Scope

- **Any new `/sdlc:docs` mode or mode-contract change.** This story adds no mode; the mode set (`sync`/`release`/`seed`/`audit`/`distill`) is untouched. It only wires `sync` and adds `sync`'s second diff source.
- **release / seed / audit / distill modes** — their own stories (NA-53/54/55 shipped; distill later). Untouched here.
- **Standalone `/sdlc:docs sync` invocations** — still gated by NA-52's contract (manifest gate, story-branch WARNING replaced by the merged-commit path, founder-confirm gate retained). This story does not change how a human-run `sync` behaves beyond adding the merged-commit source.
- **Absorbing `/sdlc:adr` into `/sdlc:docs` (NA-57).** NA-56 does not remove `/sdlc:adr`, does not delete the `seed adr` stub, and does not touch `refs/adr-pipeline.md`. The knowledge-engineer's ADR dispatch branch is untouched. **Scope fence:** if an edit reaches `adr` behaviour, it belongs to NA-57, not here.
- **Blocking the impl PR on a docs-content failure** — explicitly rejected; see the failure-mode contract. (A workspace-integrity failure **does** STOP — that is not "blocking on docs", it is refusing to ship over a corrupted checkout.)
- **Re-running the full code quality gate on the docs commit** — the docs commit writes only docs artifacts (manifest `target-path`s); in a consumer repo that is not a gated path, so no re-gate. In this SDLC repo a plugin-authoring docs write is gated per the agent's existing completion checklist — unchanged, not re-specified here.
- **Auto-triggering docs sync on CI / tag / merge webhook events** — later enhancement (epic out-of-scope). The trigger here is the in-session playbook step, not an external event.

## Open Questions

- [ ] **Multiple merged commits for one key (merged-commit source).** When several commits on base carry `<STORY-KEY>` (a story plus later follow-up fixes), which range does `sync` diff? _Suggested default (adopt):_ the **union** of each matching commit's `"<sha>^..<sha>"` file/hunk set, mirroring `refs/docs-pipeline.md` §10's "de-duplicated union across records". Deterministic and complete; no blocker. An alternative (most-recent commit only) risks missing files an earlier commit changed.
- [ ] **Post-QA phase failure — additionally open a follow-up docs issue?** On a genuine docs-content Step 6.5 failure, beyond the WARNING + recovery pointer, should the run open a tracking issue/Jira comment? _Suggested default (adopt):_ **no** — the Step 8 report line + the `/sdlc:docs sync <STORY-KEY>` recovery pointer are sufficient for a solo-dev flow; adding issue creation is scope creep and a new failure surface. Revisit if unattended-automation misses become common.
- [ ] **NA-61 template-frontmatter gap on the auto path.** The post-QA phase's `llms.txt` regen reads page frontmatter (`title`/`description`), but the `writing-docs` narrative templates emit no `description:` (three of four emit no frontmatter block) — the same gap `audit` carries (`refs/docs-pipeline.md` §"OQ carry-over"). _Suggested default (adopt):_ the post-QA path inherits the existing **skip-and-surface** behaviour (docs-pipeline §19) — it never fabricates frontmatter and never STOPs on a frontmatter-less page; **NA-61** closes the gap at the template source. No new handling in this story. Flagged because the auto-sync path does read frontmatter.
- [ ] **Docs commit ordering vs QA learnings commit.** Step 6.5 runs after QA Step 5 (learnings) and Step 6 (gate), so the docs commit is the last commit before the PR and its diff source (`origin/<BASE>...<BRANCH_PREFIX>/<STORY-KEY>`) already includes the learnings/fix commits. _Suggested default (adopt):_ keep docs last — it reflects the final tree; no reordering needed.
