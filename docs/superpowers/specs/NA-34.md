# You decide how it gets built (control section) — Technical Spec

**Story:** NA-34 — Build home you-decide-how-its-built control section
**Feature:** PRD `docs/features/` (Epic NA-29, marketing site) · Design source of truth: `docs/design/marketing-site-handoff/designs/nightshift Landing.dc.html` (section 8, `data-screen-label="Control bridge"`, component `class Component` L620–1340) + `docs/design/marketing-site-handoff/README.md` (§ Home section 8, § State Management)
**Date:** 2026-07-12

## Overview

Home section 8 ("You decide how it gets built") — three connected interactives driven by ONE client-side state machine: a triage router, a route-aware gate strip, and side-by-side comparison terminals. It lets a visitor mutate the triage config (ticket type, points, threshold, approval mode) and watch nightshift's gated route re-route and re-narrate in sync, contrasted against a scripted one-shot-AI failure run.

## Data Model

**Not applicable.** No persistence, no database, no Payload collection, no server data. Per the Jira Out-of-Scope ("Persisting any user interaction beyond the current page session") and the handoff § State Management ("No server data — everything is static content + client state"), all state is React component state in the browser, discarded on unload. No entity is created, modified, or read.

## API Surface

**Not applicable.** This section introduces **no endpoints**. It is a static Next.js App Router server component tree with one client-component island; it fetches nothing at request time and calls no Payload REST/GraphQL, no route handler, and no external API. The one outbound link (`Why an SDLC beats one-shot AI →`) is a plain in-app `<a href>` to the Why-SDLC page. Because there are zero endpoints, the Permissions table below is intentionally empty (all-public marketing page).

### State (client state only — the shared state machine)

All state lives in the `ControlSection` client island (React `useState`/`useReducer` — implementer's choice). This is the single source of truth the Jira AC4 no-desync requirement depends on: the triage router, gate strip, and right-hand terminal all derive from it. Field names and semantics are fixed by the design (`class Component`, L621 + `renderVals` L1083–1340) and MUST be preserved:

```typescript
type TicketType = 'story' | 'bug';
type ApprovalMode = 'assisted' | 'auto' | 'full-auto';
type GateStatus = 'working' | 'awaiting';

interface ControlState {
  ticketType: TicketType; // default 'story'
  storyPts: number; // 1–13, default 8 (only meaningful when ticketType === 'story')
  thresh: number; // lightweight_threshold, clamped 1–8, default 3
  approvalMode: ApprovalMode; // default 'assisted'
  gI: number; // index of the current gate within the active route
  gS: GateStatus; // current gate phase: 'working' then 'awaiting'
  gDone: boolean; // whole route completed
  raceStep: number; // 0–12, one-shot terminal scripted progression
}
```

**Route derivation (`_route()`) — pure function of `ticketType`, `storyPts`, `thresh`:**

- `ticketType === 'bug'` → `['implement', 'review']` (defect route)
- else `storyPts <= thresh` → `['refine', 'implement', 'review']` (lightweight route)
- else → `['refine', 'spec', 'plan', 'implement', 'review']` (full ceremony)

Gate count per route: full ceremony **5**, lightweight **3**, defect **2** — this satisfies Jira AC2's `5 / 3 / 2`.

**Gate lifecycle (`_startGate` / `_advanceGate`):** on mount and on **every** config change (`setStoryPts`, `incThreshold`/`decThreshold` clamped 1–8, `setTypeStory`/`setTypeBug`, `cycleMode`) the machine calls `_startGate(0)` — restarting the walk from the first gate so all three interactives re-sync (AC4). Per gate: set `gS: 'working'`, after ~1700ms (or ~400ms under reduced motion) set `gS: 'awaiting'`; if the gate auto-approves for the current mode, schedule `_advanceGate()` ~700ms later. `_advanceGate` moves to the next gate, or sets `gDone: true` at the end; under `full-auto` it re-loops via `_startGate(0)` after ~3200ms.

**Auto-approval rule (`_gateAuto(i)`):**

- `assisted` → never auto (every gate waits for `approveGate`).
- `auto` → auto-approve every gate **except** `spec` and `review` (those still wait for the user).
- `full-auto` → auto-approve every gate and loop.

**`approveGate`** advances only when `gS === 'awaiting'` and not `gDone`. **`resetGates`** = `_startGate(0)`.

**Derived view values (computed each render, names per `renderVals`):** `triageLanes` (3 lanes with lit/unlit border+bg+title colors + the ticket chip in the active lane), `triageMsg` (decision console string per route), `ctrlGates` (per-gate glyph/border/bg/color/glow/anim/connector), `raceRoute`/`routeName` (`full ceremony` | `lightweight route` | `defect route`), `raceLeft` (one-shot lines gated by `raceStep`), `raceRight` (nightshift lines gated by the gate machine + mode), `ctrlAwaiting`/`ctrlWorking`/`ctrlDone` (which terminal control shows), `approvalModeHint`, `storyBtnO`/`bugBtnO` (toggle opacities), `isStory`/`isBug`. All derived values MUST be recomputed from state — never stored — so the three interactives can never desync (AC4).

**One-shot terminal script (`raceStep` 0→12):** independent of the gate machine — a timer increments `raceStep` once/second up to 12, playing the scripted failure sequence **once** (does not loop): `> build PROJ-142` → thinking lines → diff-blob slam (`4,213 lines · one commit · no paper trail`) → `✗ reinvented your auth…` → `✗ acceptance criteria met: 2 of 7` → `❓ built the wrong thing. read all 4,213 lines…`. Under reduced motion, jump straight to `raceStep: 12`. This satisfies Jira AC3 ("the one-shot AI terminal plays its scripted failure sequence once").

All literal copy (lane titles/notes/paths, `triageMsg` variants, `approvalModeHint` variants, `CTRL_MAP` working/ready/approved strings, the one-shot failure lines, the right-terminal triage/artifact/PR lines) is **approved copy — reproduce verbatim** from the design `renderVals` and `CTRL_MAP`/`LINES`; do not rewrite (handoff Fidelity rule).

## Error Handling

No network, no user-submittable data, no validation surface — so no HTTP status codes apply. The relevant robustness cases:

| Scenario                                           | Behaviour                                                                                                                                                                                                                                                                                                                                  |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `prefers-reduced-motion: reduce`                   | All timer-driven motion is shortened/removed and end-states are shown immediately (`raceStep` jumps to 12; gate `working`→`awaiting` uses ~400ms not ~1700ms; no pulse/twinkle animation dependence). The section stays fully operable and legible — every state change is conveyed by color/glyph/text, never by motion alone (Jira AC5). |
| Points slider dragged while `ticketType === 'bug'` | Slider is hidden for bugs (`isBug` shows the `DEFECT` line instead); switching back to `story` restores the slider at the retained `storyPts`.                                                                                                                                                                                             |
| `threshold` stepper at bounds                      | Clamp to 1..8 (`Math.max(1, …)` / `Math.min(8, …)`); buttons are no-ops at the bounds, no error.                                                                                                                                                                                                                                           |
| `approve ✓` clicked when not awaiting              | No-op guard (`approveGate` returns early unless `gS === 'awaiting' && !gDone`).                                                                                                                                                                                                                                                            |
| Rapid config changes mid-walk                      | Each change calls `_startGate(0)` which `clearTimeout`s the pending gate timer before restarting — no orphaned timers, no double-advance.                                                                                                                                                                                                  |
| Component unmount (route change)                   | All `setTimeout`/`setInterval` handles cleared in the effect cleanup — no state-update-after-unmount warning.                                                                                                                                                                                                                              |

**Offline behaviour:** Not applicable — no writes, no network. The section is fully functional with no connectivity (static page + client state).

## Open Questions

- [ ] **Single file vs split components.** — Suggested default: one `control-section.tsx` holding the state machine, with the three interactives as local sub-components in the same file (keeps the shared state co-located and avoids prop-drilling the machine across files). Split into separate files only if it exceeds a comfortable size; keep the state owner single.
- [ ] **Reuse `Terminal` from `@nightshift-ai/ui` for the comparison panels?** — Suggested default: do NOT reuse if `Terminal` is a self-driving line-player; render the two panels inline with matching `--surface-terminal` token styling driven by `raceLeft`/`raceRight`. Reuse only if `Terminal` already accepts an externally-controlled line array without its own timer.
- [ ] **`approve ✓` / `run it again ↺` via `CtaButton` sizing.** — Suggested default: use `CtaButton` with the design's `size sm` (34px height) primary/secondary variants; if those variants don't exist, add them to `packages/ui` `button` (web-engineer owns `packages/ui`) rather than a one-off button in the marketing app.
- [ ] **Functional color tokens (`--cyan-400`, `--green-400`, `--amber-400`, `--red-400`) present in the marketing token layer?** — Suggested default: verify against the NA-30 global stylesheet; if absent, add them to the marketing `@theme`/token layer from the handoff `tokens/` values — never hardcode hexes in the component.
- [ ] **State container: `useState` vs `useReducer`.** — Suggested default: `useReducer` for the gate machine (discrete transitions `working`→`awaiting`→advance, plus config-change resets) with `useState` acceptable; either is fine as long as all derived values are recomputed (not stored) so the three interactives cannot desync (AC4).
