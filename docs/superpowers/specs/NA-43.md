## Overview

Add a real-role `knowledge-engineer` agent to the SDLC plugin plus a two-mode `/sdlc:adr` command that turns founder-known patterns (seed mode) and the accumulated learnings corpus (distill mode) into curated, indexed ADRs through one shared draft → propose-tags → founder-confirm → write → regenerate-index → commit/PR pipeline. Domain agents then read a deterministic per-agent index section instead of re-deriving conventions or re-appending the same learning to memory every time.

> **Nature of this story.** This is a plugin-authoring story — the deliverables are markdown agent/command/ref definitions plus a `plugin.json` edit. There is no database, HTTP API surface, web/mobile UI, or offline-sync layer, so those template sections are intentionally omitted. The sections below replace them with the equivalents for a plugin change: an artifact inventory, the agent definition, the command + shared pipeline, the distill evidence protocol, promotion criteria, index regeneration, and the read/write-path integration points.

## knowledge-engineer Agent Definition

File: `plugins/sdlc/agents/knowledge-engineer.md`.

### Frontmatter

```yaml
---
name: knowledge-engineer
description: >-
  Knowledge curator — SDLC agent that owns ADR curation: turns founder-known
  patterns and the accumulated learnings corpus into curated, indexed
  Architecture Decision Records under docs/adr/. Runs the shared ADR pipeline
  behind /sdlc:adr (seed + distill modes) and regenerates docs/adr/index.md
  deterministically from ADR frontmatter. Triggered manually via /sdlc:adr.
model: sonnet
tools: Read, Write, Edit, Bash, Skill, mcp__claude-mem__observation_search,
  mcp__claude-mem__get_observations
---
```

- **`model` — suggested `sonnet`.** Matches its sibling curation agent `ai-enablement-engineer` (also `sonnet`); domain executors are `sonnet`, reasoning-lead roles (`principal-engineer`, `qa-engineer`, `solutions-architect`, `tech-lead`, `product-manager`) are `opus`. See Open Questions for the distill-clustering trade-off.
- **`tools` — MUST whitelist the claude-mem MCP tools distill depends on.** Agent frontmatter `tools:` is an allowlist: a tool the agent does not name is not callable, so `Read, Write, Edit, Bash, Skill` alone would leave distill's `observation_search` / `get_observations` calls unavailable and the halt gate would fire **even when claude-mem is installed** (a false halt). Two mechanisms are viable with frontmatter whitelisting; **spec the explicit-name form**:
  - **Explicit MCP tool names (chosen).** List the claude-mem MCP tools by their fully-qualified harness names — `mcp__claude-mem__observation_search` and `mcp__claude-mem__get_observations` (the `mcp__<server>__<tool>` form the harness exposes for the `claude-mem` MCP server; the plan agent MUST verify the exact server slug and tool names against the installed `claude-mem` plugin's MCP manifest and use whatever it actually exposes). This makes the two tools callable while keeping the allowlist tight.
  - **Wildcard `mcp__claude-mem` (alternative).** If the harness supports a server-level wildcard, `mcp__claude-mem` (or `mcp__claude-mem__*`) admits every tool that server exposes — simpler but broader. Use only if the exact tool names cannot be pinned.
  - Because the tools are whitelisted at the frontmatter allowlist level, no runtime `ToolSearch`/opt-in is needed; the halt gate then genuinely reflects claude-mem being **absent from the environment**, not merely un-whitelisted. `Read`/`Write`/`Edit` author ADRs + regenerate index + delete promoted learnings; `Bash` runs git/`gh`/`docs/adr/` listing; `Skill` loads the pinned skills.

### Body structure (follow the existing agent template, e.g. `web-engineer.md` / `ai-enablement-engineer.md`)

1. **`${CLAUDE_PLUGIN_ROOT}` resolution note** — verbatim from the other agents (read `.claude/.sdlc-plugin-root`).
2. **Required skills (load FIRST, via the Skill tool — NOT frontmatter `skills:`, per the NA-25 resume-reinjection workaround, harness bug anthropics/claude-code#76337)**, in order:
   1. `writing-adrs`
   2. `superpowers:verification-before-completion`
   3. `gh-cli` (namespaced `sdlc:gh-cli` if the bare name does not resolve)
   4. `conventional-commit` (namespaced `sdlc:conventional-commit` if the bare name does not resolve)
3. **First steps** — read `.claude/project/project-context.md` for identity, the workspace→agent table, and quality-gate commands. Read its own memory file `.claude/memories/agents/knowledge-engineer.md` and `shared.md` if present. Read the task instructions.
   - **Active-guard scope (consumer-repo safe).** The `ai-enablement-engineer`-must-be-Active STOP applies **only to plugin-authoring writes** — i.e. a dispatch that writes under `plugins/**` (this SDLC repo). A plain ADR run in a consumer repo writes only `docs/adr/**` (+ founder-gated `.claude/memories/**` deletions) and needs **no** `ai-enablement-engineer` ownership: do not STOP on its absence there. Resolve the write-scope from project-context and gate the STOP on whether the resolved target is `plugins/**`, not on the agent roster unconditionally.
4. **Role & scope** — owns ADR curation under `docs/adr/` and the founder-gated deletion of promoted raw learning entries from `.claude/memories/**` during a distill PR (sanctioned per the analyze-protocol amendment, below). Explicitly does NOT own or edit any app/product source, and (in the SDLC repo) writes only within the `ai-enablement-engineer`-owned surface when authoring plugin changes.
5. **Pipeline** — delegate the full procedure to `${CLAUDE_PLUGIN_ROOT}/refs/adr-pipeline.md` (single source of truth); the agent body summarizes the two dispatch phases (draft-and-return, then write-after-confirmation) and links out, it does not re-inline the pipeline. **The founder-confirmation gate is NOT the agent's to run — it lives at the command layer between the two dispatches (see Command control flow).**
6. **Branch, memory, commit, return** — this command is **standalone** (not dispatched by `principal-engineer`), so in its phase-2 write dispatch the agent self-raises its own PR (like `ai-enablement-engineer` in standalone `/sdlc:analyze` mode) rather than committing-only. State this explicitly.
7. **Completion checklist** — run the consumer repo's quality-gate commands from project-context if the write touched gated paths; otherwise the "gate" is the deterministic index/frontmatter consistency check (see Index Regeneration).
8. **`Skills loaded:` return line** — required on every return, per the handoff Return format.

## /sdlc:adr Command

File: `plugins/sdlc/commands/adr.md`. Frontmatter `description:` + a `$ARGUMENTS` tail, matching `spec.md` / `analyze.md`.

### Modes (dispatched from `$ARGUMENTS`)

| Invocation              | Mode        | Corpus read?                                                    |
| ----------------------- | ----------- | --------------------------------------------------------------- |
| `/sdlc:adr "<pattern>"` | **seed**    | No — drafts inline from the founder-supplied pattern text only. |
| `/sdlc:adr --distill`   | **distill** | Yes — mines the learnings corpus for promotable candidates.     |

Both modes run the same shared pipeline; they differ only in how the draft candidate(s) are sourced and (distill only) whether learnings are deleted. If `$ARGUMENTS` is empty (no pattern, no `--distill`) → STOP with a usage message: `usage: /sdlc:adr "<pattern>"  |  /sdlc:adr --distill`.

### Shared pipeline (defined once in `refs/adr-pipeline.md`) — split across the dispatch boundary

The founder-confirmation gate needs to pause for human input, which a subagent dispatch **cannot** do (a dispatched agent runs to completion and returns; it cannot block for interactive confirmation). So the pipeline is split across the dispatch boundary exactly like `/sdlc:analyze`'s scan-then-apply flow:

**Phase 1 — dispatch `knowledge-engineer` (draft & return, writes nothing):**

1. **Draft** — produce one or more candidate ADRs from the `writing-adrs` template: full body (Title, Status `proposed`, Decision, Context, Alternatives Considered, Consequences) + YAML frontmatter (`status: proposed`, `agents: [...]`, `source-stories: [...]` — see the seed-mode `source-stories` rule below).
2. **Propose agent tags** — for each candidate, propose the `agents:` routing list (the sdlc agent identifier(s) whose future work the decision constrains).
3. **(distill only) Build the deletion list** — for each candidate, the exact memory-file entries proposed for deletion (file path + entry heading/anchor + verbatim text), subject to the shared.md audience rule below.
4. **Return** — the agent returns the drafted ADR(s), proposed tags, and (distill) the per-candidate deletion list to the command layer. **Nothing is written to disk in phase 1.**

**Founder-confirmation gate — executes at the command layer, in-session, between the two dispatches:**

- The `/sdlc:adr` command (not the subagent) presents each drafted ADR, its proposed `agents` tags, **and (distill) the exact memory entries slated for deletion per candidate**, then waits for explicit founder confirmation — identical discipline to the `/sdlc:analyze` apply gate (no inline auto-apply path). The founder may edit tags, reject individual candidates, or adjust/veto specific deletions. **Confirmation covers the drafts AND the deletions together** — no learning is deleted that the founder did not see and approve.
- If the founder confirms nothing → the command writes nothing and exits cleanly (no branch/PR).

**Phase 2 — dispatch `knowledge-engineer` again (write, only for confirmed items):**

5. **Assign number + write** — for each confirmed ADR: list `docs/adr/`, take the highest existing `NNNN` + 1 (four-digit, zero-padded, never reused), write `docs/adr/NNNN-<decision-slug>.md`.
6. **Regenerate index** — deterministically rebuild `docs/adr/index.md` from the frontmatter of every file in `docs/adr/` (see Index Regeneration). Never hand-edit the index.
7. **(distill only) Delete confirmed learnings** — remove the founder-approved raw learning entries from their source memory files **in the same PR** (git history preserves them), subject to the shared.md audience rule. Seed mode skips this step.
8. **Commit + PR** — commit via the `conventional-commit` skill, push the branch, self-raise the PR via `gh` / `raise-pr.sh`.

### Seed-mode `source-stories`

Seed mode formalises a founder-known pattern inline — there is often **no** originating Jira story. So a seed-mode ADR MAY carry an **empty or omitted `source-stories`** list, or a single optional founder-supplied key when one applies. This is consistent with `writing-adrs`, whose "pipeline-generated from stories" checklist item (`source-stories` lists the motivating Jira key(s)) applies to **distill** mode — where evidence is mined from story-keyed occurrences — not to seed. The reference to the writing-adrs checklist in `adr-pipeline.md` must state this scoping explicitly so a seed ADR with no `source-stories` is not flagged as incomplete.

### Command control flow

- Standalone command (top-level, human-run) — after the phase-2 PR is raised, drive the review loop to convergence exactly as `/spec` does (`/loop /sdlc:loop <PR_URL>`), then release the session via the loop's tail. If the harness cannot nest `/loop`, fall back to `ScheduleWakeup` (same pattern documented in `spec.md`).
- `$ARGUMENTS` tail carries the pattern string or `--distill` flag.

## Distill Evidence Protocol (`refs/adr-pipeline.md`, distill section)

### Evidence sources (cluster across all of these)

| Source                                 | Access                                                   |
| -------------------------------------- | -------------------------------------------------------- |
| `.claude/memories/agents/*.md`         | Read (per-agent learning archives).                      |
| `.claude/memories/reviews/patterns.md` | Read (QA review-pattern audit log).                      |
| PR review threads                      | `gh` (e.g. `gh pr list` / `gh api` for review comments). |
| Commit history                         | `git log`.                                               |
| claude-mem observations                | `observation_search` / `get_observations` MCP tools.     |

### Evidence contract & citations

Every distilled candidate MUST cite evidence — but the citation **kind depends on the occurrence's source**, so an empty observation DB never blocks a well-evidenced candidate:

- **claude-mem-sourced occurrences** — cite the claude-mem `observation` ID(s). Observation IDs are REQUIRED **only** for occurrences that came from claude-mem.
- **Repo-native occurrences** — cite the Jira story key(s), PR review thread(s), and/or commit SHA(s) that evidence the occurrence. No observation ID is required for these.
- A candidate satisfies the evidence gate when **each of its cited occurrences carries at least one valid citation of the appropriate kind**. A candidate may mix claude-mem and repo-native occurrences.
- **Empty observation DB → gate still satisfiable.** When claude-mem returns zero observations, a candidate evidenced entirely by repo-native citations (story keys / PR threads / commit SHAs) **fully satisfies** the evidence contract — repo-native evidence is the primary source. The `source-stories` frontmatter is populated from the cited story keys regardless of whether any observation IDs were available.

### claude-mem gates (environment, not data)

- **Tools absent → halt.** If the claude-mem MCP tools (`observation_search` / `get_observations`) are not available in the session (the plugin is not installed / not whitelisted), distill mode **halts with a clear message** (e.g. `claude-mem tools unavailable — /sdlc:adr --distill requires the claude-mem plugin; install it or use seed mode`). It does not silently proceed without them. (With the frontmatter allowlist fixed per the agent definition above, this gate fires only on genuine environment absence, never on a whitelist omission.)
- **Empty observation DB → non-fatal.** Tools present but the observation database returns zero results is **not** an error — see the evidence contract: repo-native citations carry the candidate.

> These two conditions are distinct: _tools missing_ is a halt (the environment can't support the protocol at all); _tools present but DB empty_ is a warning-and-continue (the protocol runs on repo-native evidence).

## Promotion Criteria (distill mode)

A clustered learning is promoted to an ADR candidate iff it meets **at least one** of:

| Criterion          | Definition                                                                                                                                                                                                          |
| ------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Recurrence         | Recurs across **≥ 2 cited occurrences** (each occurrence backed by a citation of the appropriate kind — observation ID for claude-mem occurrences, story key / PR thread / commit SHA for repo-native occurrences). |
| Cross-agent        | The pattern spans **more than one** sdlc agent's domain.                                                                                                                                                            |
| Durable convention | A stable convention future work must follow, even if seen once, because it is expensive to reverse or re-derive.                                                                                                    |

- Candidates failing all three are **not** promoted (reported as "below threshold", left in memory).
- On write of a confirmed promoted candidate, the corresponding **founder-approved raw learning entries are deleted from their source memory files in the same PR** — the ADR becomes the canonical record; git history preserves the deleted text. The deletion is subject to the founder gate (it appears in the phase-1 deletion list) and to the shared.md audience rule below.

### `shared.md` audience-preservation rule

`.claude/memories/agents/shared.md` is readable by **every** agent; a per-agent ADR index section is read only by the agent(s) named in its `agents:` tag. Deleting a learning from `shared.md` and replacing it with an ADR that is routed to only a subset of agents would **narrow** its visibility from all-agents to tag-list-only. Therefore:

- A learning promoted **from `shared.md`** may be deleted **only when the replacing ADR preserves the audience** — i.e. the ADR is either tagged with **every** agent the shared learning was relevant to, **or** routed to the **`General`** section (which every agent reads via the read-path integration).
- Otherwise the `shared.md` entry **stays** (the ADR is still written as the canonical record, but the raw shared learning is not deleted) — no audience is silently starved of a learning it previously saw. The phase-1 deletion list must reflect this: a shared.md deletion is only offered when the audience-preservation condition holds.

## Index Regeneration (`docs/adr/index.md`)

Deterministic, fully derived from ADR frontmatter — never hand-authored (per `writing-adrs`).

- **Grouping** — one section per distinct agent named across all ADRs' `agents:` lists, plus one **`General`** section for every ADR whose `agents` list is empty or omitted (so no ADR is ever dropped). An ADR routed to N agents appears under each of those N sections.
- **Line content** — each listing carries the ADR **number, title, and status** (e.g. `- [NNNN. Title](NNNN-slug.md) — accepted`).
- **Deterministic ordering** — sections ordered alphabetically by agent name (with `General` last); ADRs within a section ordered ascending by `NNNN`. Stated so regeneration is byte-stable and produces no spurious diffs.
- **Regeneration is idempotent** — running it twice with no ADR change yields an identical file.
- The regeneration logic lives in the pipeline ref as an algorithm the agent executes (Read frontmatter → group → sort → write); it is prose/algorithm, not a committed script, unless Open Questions resolves otherwise.

## Read-path Integration (domain agents)

Update each domain agent definition's "First steps" so it reads its own ADR index section before working:

- Add to the memory-archive read step: also read the agent's section of `docs/adr/index.md` (the section named for this agent, plus `General`), and **open the full `docs/adr/NNNN-*.md` only on demand** when a listing is relevant to the current task. The index line is cheap; the full ADR is read only when needed.
- Applies to: `database-administrator`, `platform-engineer`, `sync-engineer`, `web-engineer`, `mobile-engineer`, `ai-enablement-engineer`, **and `qa-engineer`**. (`knowledge-engineer` reads the full `docs/adr/` set as its working surface, so it needs no separate read-path line.)
  - `qa-engineer` is included because distill can promote — and delete — entries from `.claude/memories/reviews/patterns.md`, which the QA playbook consults (Step 5 and the quality gate). Without a read-path, a promoted-and-deleted review pattern would vanish from QA's view. So the QA `patterns.md` consult step must **also** read `docs/adr/index.md` (its `qa-engineer` section + `General`) and open the relevant ADR on demand.
- Guard: the read is best-effort — a missing `docs/adr/index.md` (repo has no ADRs yet) is a no-op, not an error.

## Write-path Guard (prevent re-appending covered learnings)

Two touchpoints must consult the index before appending a learning, and skip a learning an existing ADR already covers — **but only when that ADR is `accepted`**:

- **Status filter (required).** The guard matches **only `status: accepted` ADRs**. A learning whose subject matches a `superseded`, `rejected`, or `proposed` ADR is **still appended** — a superseded/rejected decision is no longer (or never was) the operative convention, and a merely `proposed` ADR is not yet binding, so suppressing the memory append against any of those would lose an active learning behind a non-binding record. The index line carries status (see Index Regeneration), so the guard can filter on it without opening the ADR.

Touchpoints:

1. **`refs/qa-engineer-playbook.md` Step 5 (Write learnings to memory)** — before the agent-memory append and the `patterns.md` append, check the relevant `docs/adr/index.md` section(s); if an **accepted** ADR already captures the learning, **soft-skip the append** and note it (the ADR is canonical). A non-accepted match does not skip.
2. **`refs/domain-agent-handoff.md` "Memory write"** — same accepted-only index-first check before a domain agent appends to `.claude/memories/agents/<name>.md` / `shared.md`.

Both are additive guards; neither changes the existing append format. The skip is soft (see Open Questions) — an append is skipped, never failed.

## Memory-ownership exception (`analyze-protocol.md`)

`refs/analyze-protocol.md` currently sanctions exactly **one** case where an agent may write another agent's memory file: the human-arbitrated memory-conflict reset (referenced from `ai-enablement-engineer.md` as the "Cross-agent memory exception"). The distill pipeline's deletion of promoted learnings — which can touch any agent's `.claude/memories/agents/*.md` (and `shared.md`, subject to the audience rule) — is a **second** cross-agent memory write and would otherwise contradict those ownership rules.

Amend `analyze-protocol.md`'s memory-ownership section to name this as a distinct, sanctioned exception:

- **Exception 2 — founder-gated distill deletion.** `knowledge-engineer`, running `/sdlc:adr --distill`, may delete promoted raw learning entries from any agent's memory file **only** as a reviewable diff/PR, **only** after the founder-confirmation gate approved the specific deletions (the phase-1 deletion list), and **only** subject to the `shared.md` audience-preservation rule. Like Exception 1, it is never a silent write.

State it once here (canonical) and have `knowledge-engineer.md`'s Role & scope point back to it, mirroring how `ai-enablement-engineer.md` points at the conflict-reset exception.

## /sdlc:analyze Recommendation Hook

- In `commands/analyze.md` (and `refs/analyze-protocol.md` if the scan/output tables live there), add a **recommendation-only** output: when the scan detects memory bloat (e.g. oversized agent memory files or repeated/duplicated learnings), include a finding that **recommends running `/sdlc:adr --distill`** to curate them into ADRs.
- This is a suggestion in the analyze report only. `/sdlc:analyze` does **not** run distill, does not write ADRs, and gains no new apply path (see Out of Scope).

## Error Handling

| Scenario                                                             | Behaviour                                                                                                        |
| -------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `/sdlc:adr` with empty `$ARGUMENTS` (no pattern, no `--distill`)     | STOP with the usage message.                                                                                     |
| Founder does not confirm at the gate                                 | Command writes nothing; report the drafted candidates and exit cleanly (no branch/PR, no phase-2 dispatch).      |
| claude-mem tools absent in distill mode                              | Halt distill with a clear "claude-mem tools unavailable" message (seed mode unaffected).                         |
| claude-mem observation DB empty in distill mode                      | Non-fatal — continue on repo-native citations, note the empty DB.                                                |
| No promotable candidates found in distill mode                       | Report "no candidates met the promotion criteria" and exit cleanly — no ADR, no PR.                              |
| Distill promotes a `shared.md` learning but the ADR narrows audience | ADR is written; the `shared.md` entry is **not** deleted (audience-preservation rule); note it in the PR.        |
| `docs/adr/` does not exist yet (first ADR)                           | Create it; next number is `0001`.                                                                                |
| ADR number collision / gap while listing `docs/adr/`                 | Always take `max(existing) + 1`; never reuse a number (including superseded/rejected).                           |
| Attempt to edit an `accepted` ADR's substance                        | Refuse — write a new superseding ADR instead (per `writing-adrs` immutability rule).                             |
| Read-path: `docs/adr/index.md` missing for a domain agent            | No-op, not an error.                                                                                             |
| `gh` / `raise-pr.sh` failure                                         | STOP and surface the error (do not leave a half-written ADR without a PR); the write is on a branch, reviewable. |

## Out of Scope

- Any change to `/sdlc:analyze` beyond it **recommending** `--distill` when memory bloat is detected (no apply path, no auto-distill).
- Retroactive backfill / migration of existing memory-file learnings into ADRs outside the normal distill promotion flow.
- Any browsing UI for ADRs beyond the generated `docs/adr/index.md`.
- Type-2 docs-freshness ownership by `knowledge-engineer` — named as a future responsibility in the story context, not built here.
- Authoring the `writing-adrs` skill itself (landed in NA-44) or any change to the ADR body/lifecycle contract it defines.
- Wiring `/sdlc:adr` into `/auto` or the automated SDLC pipeline — it is a manually-run, human-gated command.
