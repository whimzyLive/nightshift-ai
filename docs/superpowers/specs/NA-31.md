# CMS FAQ and Editorial Why-Copy Content Layer — Technical Spec

**Story:** [NA-31](https://whimzylive.atlassian.net/browse/NA-31) — Add CMS FAQ and editorial why-copy content layer
**Epic:** [NA-29](https://whimzylive.atlassian.net/browse/NA-29) — Nightshift marketing site — design-handoff build
**Feature:** docs/features/2026-07-12-nightshift-marketing-site-design-handoff.md
**Date:** 2026-07-12

## Overview

Introduce the Payload CMS content layer that serves the marketing site's editorial copy — the FAQ
entries and the Why-SDLC "why" argument paragraphs — so a maintainer can correct or refresh that
copy through the Payload admin with no code deploy. This story delivers the data model, the seed of
the approved handoff copy, and a documented query shape; the page stories (NA-35/36/38/39) do their
own rendering against it.

## Scope boundary

This is a **web-only** story owned by **web-engineer** (`apps/marketing/`). There is no separate
database, backend, mobile, or offline-sync workspace in this project — Payload (Next.js + Payload 3
on Neon Postgres) is the entire data + API surface, and the Data Model / API sections below describe
Payload collections/globals and Payload's auto-generated REST + Local API, not a separate ORM or
service layer.

## Data Model

The stack is Payload 3 with `@payloadcms/db-postgres` (Neon). New content is expressed as a Payload
**collection** (`faq`) plus a Payload **global** (`whySdlc`). Payload owns table generation and type
generation — never hand-write SQL DDL or edit `apps/marketing/src/payload-types.ts` (it is generated
by `payload generate:types`).

### New entity — `faq` collection

One document per canonical FAQ question (12 total). Home and Why-SDLC surfaces are selected with
boolean flags and per-surface ordering; where a surface intentionally shows a condensed answer, an
optional per-surface rich-text override is used, otherwise the surface renders the canonical `answer`.

File: `apps/marketing/src/collections/Faq.ts`

| --------------- | ----------------------------- | -------- | ------- | ---------------------------------------------------------------------------------------------------------------------- |
| `seedKey`       | text (unique, admin.readOnly) | Yes      | —       | Stable idempotency anchor for the seed (e.g. `positioning-wrapper`). Never edited by hand. `unique: true`.             |
| `question`      | text                          | Yes      | —       | Plain-text question, verbatim from handoff.                                                                            |
| `group`         | select                        | Yes      | —       | Eyebrow group. Options below. Stores the machine value; label carries the `//` eyebrow text.                           |

- `apps/marketing/src/payload.config.ts` — register `Faq` in `collections` and `WhySdlc` in a new

### Relationships

No cross-entity relationships. `faq` documents are flat and self-contained; `whySdlc` is a singleton
global. Home preview, Why-SDLC FAQ, and the full FAQ page are all projections of the single `faq`
collection filtered by the boolean/order fields — no join.

### Seed script

Owner decision (Epic NA-29 open-question #5): the seed is **part of the Next.js app**.

- Data module: `apps/marketing/src/seed/data.ts` — exports the typed approved copy (the 12 FAQ records
  keyed by `seedKey`, and the `whySdlc` global payload). Rich-text values are lexical JSON.
- Runner: `apps/marketing/src/seed/seed.ts` — obtains the Local API via `getPayload({ config })` and:
  - For each FAQ record: `payload.find({ collection:'faq', where:{ seedKey:{ equals } }, limit:1 })`;
    if found → `payload.update` that id, else `payload.create`. Idempotent by `seedKey`.
  - For the global: `payload.updateGlobal({ slug:'whySdlc', data })` (globals are single-instance —
    re-running only overwrites, never duplicates).
- Invocation: add a `seed` script to `apps/marketing/package.json` running the file through Payload's
  runner (`payload run src/seed/seed.ts`), so it loads env + the compiled config. Runnable via
  `pnpm --filter @nightshift-ai/marketing run seed`.

**Idempotency contract (AC ti-3):** first run against a fresh DB fully populates all 12 FAQ docs +
the global; any re-run updates in place and never duplicates or corrupts (verified by asserting
`payload.count({ collection:'faq' }) === 12` after a second run).

## API Surface

Payload auto-generates REST + a server-side Local API from the collection/global. No hand-written
endpoints, handlers, or route files are added.

### Read endpoints consumed by the page stories

| Method | Path (REST)                                                      | Auth/Route Type | Description                                                                            |
| ------ | ---------------------------------------------------------------- | --------------- | -------------------------------------------------------------------------------------- |
| GET    | `/api/faq?where[showOnHome][equals]=true&sort=homeOrder&limit=5` | public, no auth | Home FAQ preview (top 5)                                                               |
| GET    | `/api/faq?sort=faqOrder&limit=0`                                 | public, no auth | Full FAQ page (all 12 — `limit=0` = no page cap; Payload's REST default is `limit=10`) |
| GET    | `/api/faq?where[showOnWhySdlc][equals]=true&sort=whySdlcOrder`   | public, no auth | Why-SDLC page FAQ (2)                                                                  |
| GET    | `/api/globals/whySdlc`                                           | public, no auth | Why-SDLC editorial argument copy                                                       |

Admin CRUD (`POST/PATCH/DELETE /api/faq`, `POST /api/globals/whySdlc`) is authenticated — covered in
Permissions. The public site never mutates.

### Preferred read path (Local API, documented for the page stories — AC ti-5)

Page stories are React Server Components; they read via the Local API (no HTTP hop), which mirrors the
REST query shape:

```typescript
import config from '@payload-config';
import { getPayload } from 'payload';

const payload = await getPayload({ config });

// Home preview (top 5)
const homeFaqs = await payload.find({
  collection: 'faq',
  where: { showOnHome: { equals: true } },
  sort: 'homeOrder',
  limit: 5,
});

// Full FAQ page (all 12) — limit:0 disables Payload's default limit:10 page cap
const allFaqs = await payload.find({ collection: 'faq', sort: 'faqOrder', limit: 0 });

// Why-SDLC page FAQ (2)
const whyFaqs = await payload.find({
  collection: 'faq',
  where: { showOnWhySdlc: { equals: true } },
  sort: 'whySdlcOrder',
});

// Why-SDLC editorial copy
const whySdlc = await payload.findGlobal({ slug: 'whySdlc' });
```

Answer resolution rule for consumers: Home renders `homeAnswer ?? answer`; Why-SDLC renders
`whySdlcAnswer ?? answer`; the full FAQ page always renders `answer`.

### Response shapes (generated Payload types — shown for the consumer contract)

```typescript
type FaqGroup = 'positioning' | 'workflow-control' | 'setup-stack' | 'trust-quality' | 'cost-license';
```

The exact interface names/shapes are produced by `payload generate:types` into
`apps/marketing/src/payload-types.ts`; consumers import from there — the block above is the contract,
not a file to hand-write.

### Permissions

Roles derive from Payload's access model on this app (`apps/marketing/src/collections/Users.ts`):
Payload distinguishes an **authenticated user** (admin login **or** the API-key user that
`Users.useAPIKey` enables for agents) from an **anonymous** public visitor. There is no finer-grained
role system in this app.

| Role                              | List `faq` | Get `faq` | Create | Update | Delete | Read `whySdlc` | Update `whySdlc` |
| --------------------------------- | ---------- | --------- | ------ | ------ | ------ | -------------- | ---------------- |
| Anonymous (public site visitor)   | ✓          | ✓         | ✗      | ✗      | ✗      | ✓              | ✗                |
| Authenticated user (admin login)  | ✓          | ✓         | ✓      | ✓      | ✓      | ✓              | ✓                |
| API-key user (agent, `useAPIKey`) | ✓          | ✓         | ✓      | ✓      | ✓      | ✓              | ✓                |

Access functions:

- `read: () => true` on both `faq` and `whySdlc` (public site is read-only, unauthenticated).
- `create`/`update`/`delete` (collection) and `update` (global): `({ req: { user } }) => Boolean(user)`
  — any authenticated principal (admin or API key) may write; anonymous cannot.

The seed script runs through the Local API (`getPayload`), which bypasses HTTP access control by
design (server-side), so seeding is unaffected by these gates.

## Error Handling

| Scenario                                               | Behaviour                                                                                                         | HTTP Status |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------- | ----------- |
| Public GET of `faq` / `whySdlc`                        | Always allowed (read is public)                                                                                   | 200         |
| Unauthenticated mutation attempt (`POST/PATCH/DELETE`) | Rejected by access control                                                                                        | 403         |
| Empty result (e.g. no `showOnHome` docs)               | Payload returns `{ docs: [], totalDocs: 0 }` — consumer renders nothing, does not error                           | 200         |
| Global never seeded                                    | Payload returns the global with empty/undefined fields — seed guarantees this never happens in a handed-over site | 200         |
| `whySdlcOrder`/`homeOrder` missing while flag is true  | Field-level `validate` blocks the admin save with a field error                                                   | 400 (admin) |
| Duplicate `seedKey` on save                            | `unique: true` constraint rejects the write                                                                       | 400         |
| Seed re-run                                            | Upsert by `seedKey` + `updateGlobal` — no duplicates, no corruption (idempotent)                                  | n/a         |

There is no offline write path in this app — all writes are online admin/API-key operations.

## Out of Scope

  (NA-38) each do their own query/render wiring against this collection/global.
- **SEO / FAQPage JSON-LD wiring** (NA-39) — it consumes `faq` answers (flattening lexical → text so
  the schema stays text-identical to the rendered answer per the Epic constraint), but the wiring is
  its own story.
- **Home positioning / problem / trust / control paragraphs** (NA-32/33/34) — those stories are **not**
  blocked by NA-31 and keep their copy static in code for now; migrating them to CMS is a later,
  separate decision.
- **Tooling facts, agent rosters, slash commands, install snippets** — stay static in code per the PRD.
- **Analytics / adoption instrumentation** — out of scope for v1 (Epic NA-29 open-question #3).
- **A rich-text renderer component** for lexical output — provided/needed by the consuming page
  stories, not here.

## Open Questions

- [x] **FAQ content model: per-surface override vs surface-scoped rows?** — Resolved: single canonical
      `faq` document per question (12 total) with boolean surface flags + optional per-surface answer
      overrides. Rationale: DRY questions, one edit updates every surface by default, and only the 3
      intentionally-divergent answers carry an override — the alternative (19 surface-scoped rows) risks
      editor confusion and silent divergence.
- [x] **Answer field type: rich text vs plain text?** — Resolved: `richText` (lexical). The design uses
      inline `code` marks (e.g. terracotta `/auto`, `project-context.md`) that plain text cannot carry;
      NA-39 flattens lexical → plain text for the FAQPage schema.
- [x] **"Why" copy as a global or a collection?** — Resolved: a `whySdlc` **global** — single-instance
      page copy, inherently idempotent, no duplicate risk on re-seed.
- [x] **Seed ownership / re-seed policy?** — Resolved (Epic NA-29 OQ #5): seed lives in the Next.js app
      (`apps/marketing/src/seed/`), run via `payload run`, idempotent by `seedKey` + `updateGlobal`.
- [ ] **Scope of "why" paragraphs beyond the Why-SDLC page** (Epic NA-29 OQ #2) — Suggested default:
      limit this story's editorial "why" copy to the Why-SDLC page argument sections (the only positioning
      narrative gated on NA-31 via the NA-36 block). Home positioning paragraphs (NA-32/33/34) stay static
      for now and can migrate to CMS in a follow-up without changing this data model.
