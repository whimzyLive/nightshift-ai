# AI Workflow Manager Agent — Technical Spec

**Story:** NA-12 — AI workflow manager agent
**Feature:** No feature doc — product context is the NA-12 Jira story (no parent Epic)
**Date:** 2026-07-07

## Overview

Add an SDLC **domain agent** (`ai-enablement-engineer`) to the existing `sdlc` plugin that owns the repository's AI-configuration surface (`CLAUDE.md`, `AGENT.md`/`AGENTS.md`, `.agents/**`, `.claude/**`, and other popular AI-config conventions) and keeps it synchronized, plus a manual `/sdlc:analyze` command that runs a repo-wide drift/gap scan on demand. The agent's ownership resolves from per-repo config (never hard-coded) so a later handoff of `plugins/` and `skills/` from `platform-engineer` is a config edit, not a code change.

## Applicability of standard spec sections

This is a Markdown/Shell plugin repo (see `.claude/project/project-context.md` — primary language Markdown/Shell, no runtime, no database, no HTTP surface). The following standard sections are therefore **Not Applicable** and are intentionally omitted:

- **Data Model** — N/A. No database, ORM, or entities in this repo.
- **API Surface** — N/A. No HTTP endpoints; the only "surface" is a slash command, specified under [Command Surface](#command-surface).
- **Web UI / Mobile UI / Offline Sync** — N/A. No web app, mobile app, or sync layer (project-context lists only `plugins/`, `tools/`, `brand/`).

The role-based permission matrix is replaced by an [Ownership & Write-Scope Matrix](#ownership--write-scope-matrix), which is the load-bearing access-control artifact for this story (AC-5).

## Agent Definition (AC-1)

`plugins/sdlc/agents/ai-enablement-engineer.md` — front-matter and body follow the shipped `sdlc` domain-agent shape (`plugins/sdlc/agents/platform-engineer.md`).

**Front-matter:**

```yaml
---
name: ai-enablement-engineer
description: >-
  AI workflow manager — SDLC domain agent that owns and keeps the repository's
  AI configuration (CLAUDE.md, AGENT(S).md, .agents/**, .claude/**, and other
  popular AI-config conventions) synchronized, scans the repo's AI artifacts for
  drift/gaps, and applies fixes as reviewable diffs/PRs. Ownership is resolved
  per-repo, never hard-coded. Triggered manually via /sdlc:analyze and routed by
  principal-engineer when a story touches AI-config paths.
model: sonnet
tools: Read, Write, Edit, Bash, Skill
skills:
  - skill-creator
  - find-skills
  - conventional-commit
---
```

**Body (WHAT it must contain — HOW is the implementer's):**

1. The standard sdlc plugin-path-resolution preamble, reading `.claude/.sdlc-plugin-root` and substituting for `${CLAUDE_PLUGIN_ROOT}` (verbatim from the shipped `sdlc` agents' "Resolving plugin paths" block).
2. **First steps** (matching other sdlc domain agents): when dispatched by `principal-engineer`, run the pre-work branch checkout from `${CLAUDE_PLUGIN_ROOT}/refs/domain-agent-handoff.md`. Read `.claude/project/project-context.md` (identity, workspace→agent table, active-agent status); read the override `.claude/project/agents/ai-enablement-engineer.md` if present; read own memory `.claude/memories/agents/ai-enablement-engineer.md` if present; read the task instructions.
3. **Owned surface resolution** — see [Ownership Model](#ownership-model). The agent MUST compute its write-scope at runtime from (a) its declared, config-driven AI-config surface list and (b) the project-context workspace→agent table, applying the [precedence rule](#precedence-rule-metadata-inside-another-agents-area); it MUST NOT hard-code `plugins/` or `skills/`.
4. **Scan capability** — see [Drift Scan](#drift-scan--report-ac-1). Read-only enumeration of AI artifacts producing a structured drift/gap report.
5. **Apply capability** — see [Apply Flow](#apply-flow-reviewable-diff--pr--ac-2). Every write goes through a reviewable diff (commit on the impl branch when dispatched, or a self-raised PR when standalone); no silent writes; refuse any path outside resolved write-scope.
6. **Skill usage** — during scan, use `find-skills` to surface candidate skills for detected gaps and `skill-creator` to scaffold a new skill when a gap warrants one (AC-6).
7. Branch/commit/memory/return protocol — reuse `${CLAUDE_PLUGIN_ROOT}/refs/domain-agent-handoff.md` (same-plugin), like every other sdlc domain agent.

## Ownership Model

The agent resolves its write-scope at runtime from two inputs, exactly matching the mechanism documented in `EXTENDING.md` (table-driven ownership + override). It never hard-codes area paths.

### Core AI-config surface (always owned when the agent is active)

The agent owns the popular AI-config conventions below wherever they appear — **subject to the [precedence rule](#precedence-rule-metadata-inside-another-agents-area)**. This list is illustrative-but-comprehensive, **not** exhaustive: it is **config-driven and extensible per-repo** (the baseline globs ship in the agent definition; a per-repo override `.claude/project/agents/ai-enablement-engineer.md` can add more), so a newly-popular convention is added by a config edit, **not** a spec change.

| Convention / path glob | Notes |
| ---------------------- | ----- |
| `CLAUDE.md` (root) + nested `**/CLAUDE.md` | Claude Code instruction files / directory guides |
| `AGENT.md`, `AGENTS.md` (+ nested `**/AGENTS.md`) | The agents.md standard |
| `.agents/**` (incl. `.agents/skills/**`) | Agent config + agent-owned skills convention |
| `.claude/**` | Claude Code project config — `.claude/settings.json`, `.claude/skills/**`, `.claude/project/agents/*.md`, `.claude/project/skills.json` (the **skills manifest / "skills-lock"**), etc. — **minus** the read-only-authority carve-outs below |
| `.github/copilot-instructions.md` | GitHub Copilot instructions |
| `.cursor/rules/**`, `.cursorrules` | Cursor rules (new + legacy) |
| `.windsurfrules` | Windsurf rules |
| `GEMINI.md` | Gemini instruction file |
| Plugin/skill **metadata files**: `**/SKILL.md`, `**/.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json` | The "related skill/plugin refs" from the story. Whole-file scope (see below). |

> **Whole-file scope, not front-matter.** The write-scope guard is path-based — it owns a whole file or not at all. "Only touch the front-matter of a `SKILL.md`" is *behavioural guidance* for the agent, not an enforceable scope granularity; a path-based guard cannot police sub-file regions.

### Precedence rule: metadata inside another agent's area

An AI-config or metadata file can physically live **inside** an area that the workspace→agent table assigns to another agent (e.g. `plugins/**` is `platform-engineer`'s today, and a `plugins/**/CLAUDE.md` or `plugins/**/SKILL.md` sits inside it). Precedence:

- **If a matched file falls within a table-assigned area owned by another agent, it is READ/scan-only** for `ai-enablement-engineer` — drift is *reported*, not written — **until** that area is reassigned to `ai-enablement-engineer` via the workspace→agent table (the handoff).
- The core-surface **write** grant applies only to matched files **outside** any other agent's table-assigned area (e.g. root `CLAUDE.md`, root `.claude-plugin/marketplace.json`, `.claude/**`, `.agents/**`).

This keeps the broadened surface and the `plugins/` matrix rows consistent: pre-handoff, `plugins/**/SKILL.md` / `plugins/**/plugin.json` / `plugins/**/CLAUDE.md` are R for `ai-enablement-engineer` (owned+writable by `platform-engineer`); post-handoff they become W.

### Read-only authority (never written by the agent)

Carved out of `.claude/**` above:

| Path | Why read-only |
| ---- | ------------- |
| `.claude/project/project-context.md` | This file's workspace→agent table is the **source of ownership truth**. Per this project's design principle ("init owns setup — no setup agents"), it is authored by `/sdlc:init` and humans. The agent **reads** it to learn scope; letting it rewrite its own ownership would be circular. |
| `.claude/.sdlc-plugin-root`, other `.claude/.*-plugin-root` pointers | Written by init; environment wiring. |
| `.claude/memories/agents/<other-agent>.md` | Each agent owns its own memory file only. The agent may write only `.claude/memories/agents/ai-enablement-engineer.md`. |

### Extended surface (per-repo, table-driven)

Any additional area assigned to `ai-enablement-engineer` in the project-context workspace→agent table becomes owned — e.g. `plugins/`, `skills/`, `tools/`. This is the mechanism the story's follow-up uses.

### Documented scope (AC-5)

The agent definition documents the surface above, and states the runtime rule: *effective write-scope = (config-driven AI-config surface ∪ table rows assigned to `ai-enablement-engineer`) − read-only-authority paths − matched files inside another agent's table-assigned area*. The agent must (a) print its resolved write-scope at the start of any run so humans/agents can see it, and (b) **refuse and abort** before writing any path outside that scope (guard in [Apply Flow](#apply-flow-reviewable-diff--pr--ac-2)).

### Handoff-boundary design (AC-3, Dependencies)

The planned follow-up reassigns `plugins/` and `skills/` from `platform-engineer` to this agent. Because scope is table-driven, that handoff is a **config edit only** — no change to any agent's code:

1. In `.claude/project/project-context.md` workspace→agent table: change the `plugins/` owner to `ai-enablement-engineer` and add a `skills/` row owned by `ai-enablement-engineer`.
2. In `.claude/project/agents/platform-engineer.md`: remove `plugins/` from `owns:`, add it (and `skills/`) to `never:`.
3. In `.claude/project/agents/ai-enablement-engineer.md`: add `plugins/`, `skills/` to `owns:`.

This spec's contract that makes the handoff "clean rather than retrofitted": the agent reads ownership from the table at runtime and treats project-context as read-only authority — so the follow-up story only edits config, and this spec does **not** perform the reassignment (see [Out of Scope](#out-of-scope)).

## Command Surface

### `/sdlc:analyze` (AC-4)

`plugins/sdlc/commands/analyze.md` — command shape mirrors `plugins/sdlc/commands/spec.md` (front-matter `description:`, then dispatch instructions). Bootstrap is already covered by the existing `sdlc` plugin-root marker (`.claude/.sdlc-plugin-root`) and `/sdlc:init` — **no separate init command**.

| Aspect | Behavior |
| ------ | -------- |
| Trigger | Manual only — user runs `/sdlc:analyze`. No scheduling, no background trigger (see Out of Scope). |
| Arguments | `$ARGUMENTS` optional — an area/glob to narrow the scan (default: full resolved scope). |
| Default mode | **Read-only scan + report.** Dispatches `ai-enablement-engineer` in scan mode; prints the [drift/gap report](#drift-scan--report-ac-1) to the user. Writes nothing. |
| Apply | Opt-in only. After the report, the command offers to apply a selected fix; if the user confirms, the agent runs the [Apply Flow](#apply-flow-reviewable-diff--pr--ac-2) (branch + PR). Never applies silently. |
| Output | Resolved write-scope (printed first), the drift/gap list with proposed fixes, and any skill suggestions from `find-skills`. |

## Drift Scan / Report (AC-1)

Defined once in the shared ref (`plugins/sdlc/refs/analyze-protocol.md`) consumed by both the agent and the command. Read-only. Enumerate AI artifacts (repo-wide read is permitted for scanning; only *writes* are scope-limited) and detect drift/gaps:

| Check | Drift/gap detected |
| ----- | ------------------ |
| Skills on disk vs `.claude/project/skills.json` | Skill dir present but not in manifest, or manifest entry with no dir |
| Agent overrides vs skills | Override lists a skill (in `.claude/project/agents/*.md`) that resolves to no installed/bundled/project skill |
| Override directory guides vs disk | Override references a `CLAUDE.md` guide that does not exist |
| Dangling refs/scripts | A `refs/*.md` or `scripts/*` path referenced by an agent or command resolves to no file on disk |
| Workspace→agent table vs disk | Table lists a path that no longer exists, or a top-level area with no owner |
| AI-config-file freshness | Documented structure in any owned AI-config convention (`CLAUDE.md`, `AGENT(S).md`, `.cursor/rules`, `GEMINI.md`, etc.) diverges from what's on disk (plugins, commands, workspace paths) |
| Plugin/skill metadata | `SKILL.md` missing required front-matter (`name`, `description`); `plugin.json` / `marketplace.json` referencing a missing plugin path |
| Vendored-skill staleness (optional) | Compare each vendored skill's recorded pinned ref (from README) against upstream if reachable; report if a newer upstream ref exists (advisory only — refresh is manual, see [Bundled Default Skills](#bundled-default-skills-ac-6)) |
| **Skill gaps (AC-6)** | For detected gaps, run `find-skills` to surface candidate existing skills; if no fit exists, note that `skill-creator` can scaffold a new one |

Output shape — a structured report (per finding): `area` · `kind` (drift \| gap) · `detail` · `proposed fix` · `target path(s)` (each flagged W or R against the resolved write-scope). No writes.

## Apply Flow (reviewable diff / PR) — AC-2

Applying any fix is explicit and always produces a reviewable diff — never a silent write. The agent follows standard sdlc domain-agent semantics:

1. **Scope guard first** — resolve effective write-scope; if the fix's target paths are not all within scope (including the precedence rule — a file inside another agent's area is R until handoff), **refuse and abort** with the offending path listed (satisfies AC-5: "never modifies a file outside its declared scope").
2. Apply the scoped edits (Edit/Write) only within resolved write-scope; when the fix is "create a missing skill", drive it through the bundled `skill-creator` skill.
3. Commit via the `conventional-commit` skill.
4. **Reviewable diff:**
   - **Dispatched by `principal-engineer`** (as part of a story's impl): commit on the impl branch per `${CLAUDE_PLUGIN_ROOT}/refs/domain-agent-handoff.md`; `principal-engineer` opens the PR after all phases — do **not** self-raise. The commit diff is the reviewable artifact.
   - **Standalone via `/sdlc:analyze`**: create a branch `chore/ai-config-<slug>` off the base branch (`main` per project-context) and raise a PR via `${CLAUDE_PLUGIN_ROOT}/scripts/raise-pr.sh <head> main "<title>" <body-file>` (same-plugin helper). The PR **is** the reviewable diff (AC-2).
5. Report the diff/PR back to the user. Do not merge.

## Bundled Default Skills (AC-6)

Both skills are **vendored** into `plugins/sdlc/skills/<name>/SKILL.md` (the same layout as the other `plugins/sdlc/skills/*`) and listed in the agent's `skills:` front-matter so they are available by default. Vendoring — not a cross-marketplace dependency — is the chosen mechanism because both originate from `skills.sh` (a skills registry), not a Claude plugin marketplace like `postiz-agent` / `marketingskills`; a marketplace dependency is only available if/when a skill is published to a Claude marketplace.

| Skill | Source | Vet level |
| ----- | ------ | --------- |
| `skill-creator` | `https://www.skills.sh/anthropics/skills/skill-creator` (Anthropic official) | Low risk — Anthropic-authored. Confirm provenance and pin the vendored copy's version/commit. |
| `find-skills` | `https://www.skills.sh/vercel-labs/skills/find-skills` (Vercel Labs) | **Standard third-party check** (per project design principles). |

**Upstream-refresh strategy:** vendored skills are refreshed **manually** — the pinned ref is recorded in `plugins/sdlc/README.md`, and the optional "vendored-skill staleness" drift check (see [Drift Scan](#drift-scan--report-ac-1)) surfaces when a newer upstream ref exists, but never auto-updates. A refresh is a normal reviewed PR that re-vets and re-pins.

### Third-party skill vetting note

Per this project's design principle "vet third-party skills before adoption" (memory: deepline rejected; postiz-agent + marketingskills adopted), the implementation phase MUST vet before vendoring, and the spec fixes the checklist:

- Read the full `SKILL.md` (and any bundled scripts) before vendoring — reject if unreviewable.
- No network calls that exfiltrate repo content; no execution of remote code; no credential/secret access. `find-skills` must be confined to reading local files and querying the public skills registry for suggestions.
- **Licensing/attribution:** confirm the upstream license permits vendoring/redistribution; carry the upstream `LICENSE` and attribution/notice text into `plugins/sdlc/skills/<name>/` alongside the vendored `SKILL.md`; record the license, upstream URL, and pinned ref together in `plugins/sdlc/README.md`.
- Pin the vendored copy to a specific version/commit (recorded in `plugins/sdlc/README.md`).
- `skill-creator` (Anthropic-official) passes with a provenance + version + license check; `find-skills` gets the full checklist above.
- If `find-skills` fails vetting (including a license that forbids vendoring), ship without it and open a follow-up — `skill-creator` alone still satisfies "suggest/create skills", but AC-6 names both, so a fail is a flagged deviation, not a silent drop.

## Error Handling

| Scenario | Behavior |
| -------- | -------- |
| Scan finds no drift/gaps | Report "no drift detected" and exit cleanly (success). |
| No override / no table rows for the agent | Fall back to the **config-driven AI-config surface** only; warn that extended areas are unconfigured. Never error out. |
| Apply target outside resolved write-scope (incl. a file inside another agent's area) | **Refuse and abort** the apply; print the offending path(s); make no writes (AC-5). |
| `project-context.md` missing or malformed table | Scan proceeds on the core surface; report the table problem as a drift finding; do not attempt to fix project-context (read-only authority). |
| `find-skills` / `skill-creator` unavailable or offline | Degrade gracefully — skip the skill-suggestion step, still emit structural drift findings; note the skipped step in the report. |
| `raise-pr.sh` fails during standalone apply | Surface the error and PR-creation failure; leave the branch + local commit intact for manual recovery; do not retry silently. |
| Run inside another agent's memory/scope | Never write `.claude/memories/agents/<other>.md`; only the agent's own memory file. |

## Out of Scope

- **Automatic background / continuous execution** — scheduling, event triggers, watch mode. Explicitly excluded by the story; candidate future story.
- **Repository-init-time auto-discovery** of ownership/config — per the design principle "init owns setup (no setup agents)", this belongs as a future extension of `/sdlc:init`, not agent-owned logic. (The only init touch in scope here is registering the agent's override scaffold — an edit to the existing init, not a scan/auto-configure flow.)
- **Performing the `plugins/` + `skills/` ownership reassignment** in `.claude/project/project-context.md` and the two overrides. This spec **designs** the boundary so the handoff is a clean config edit; the reassignment itself is the story's named follow-up, not this story.
- **Editing generated or environment files** — `.claude/.*-plugin-root` pointers and `.claude/project/project-context.md` are read-only authority for this agent.
- Web/mobile/data/API work — none exist in this repo.
