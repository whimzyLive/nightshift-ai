# AI Workflow Manager Agent — Technical Spec

**Story:** NA-12 — AI workflow manager agent
**Feature:** No feature doc — product context is the NA-12 Jira story (no parent Epic)
**Date:** 2026-07-07

## Overview

Add an SDLC **domain agent** (`ai-enablement-engineer`) to the existing `sdlc` plugin that — once a repo opts in at init time — owns and maintains the repository's AI-configuration surface (`CLAUDE.md`, `AGENT.md`/`AGENTS.md`, `.agents/**`, `.claude/**`, other AI-config conventions) **plus `plugins/**` and `skills/**`**, and exposes a manual `/sdlc:analyze` command that scans the repo for drift, gaps, and config-vs-memory semantic conflicts. Ownership is granted per-repo via project-context (init opt-in), so `platform-engineer` no longer owns AI-related files by default.

## Agent Definition (AC-1)

`plugins/sdlc/agents/ai-enablement-engineer.md` — front-matter and body follow the shipped `sdlc` domain-agent shape (`plugins/sdlc/agents/platform-engineer.md`).

**Front-matter:**

```yaml
---
name: ai-enablement-engineer
description: >-
  AI workflow manager — SDLC domain agent that owns and keeps the repository's
  AI configuration (CLAUDE.md, AGENT(S).md, .agents/**, .claude/**), plus
  plugins/** and skills/**, synchronized once the repo opts in at init. Scans
  for drift, gaps, and config-vs-memory conflicts and applies fixes as reviewable
  diffs/PRs after human confirmation. Triggered manually via /sdlc:analyze and
  routed by principal-engineer when a story touches AI-config paths.
model: sonnet
tools: Read, Write, Edit, Bash, Skill
skills:
  - skill-creator
  - find-skills
  - conventional-commit
---
```

**Body (WHAT it must contain — HOW is the implementer's):**

1. The standard sdlc plugin-path-resolution preamble, reading `.claude/.sdlc-plugin-root` and substituting for `${CLAUDE_PLUGIN_ROOT}` (verbatim from the shipped `sdlc` agents).
2. **First steps** (matching other sdlc domain agents): when dispatched by `principal-engineer`, run the pre-work branch checkout from `${CLAUDE_PLUGIN_ROOT}/refs/domain-agent-handoff.md`. Read `.claude/project/project-context.md` (identity, workspace→agent table, active-agent status); if the agent is **not** Active there, STOP (repo not opted in — see [Error Handling](#error-handling)). Read the override `.claude/project/agents/ai-enablement-engineer.md`; read own memory; read the task instructions.
3. **Owned surface resolution** — see [Ownership Model](#ownership-model). Effective write-scope = the config-driven AI-config surface ∪ the areas the workspace→agent table assigns to `ai-enablement-engineer` (in this repo: `plugins/`, `skills/`), minus the read-only carve-outs. It MUST NOT hard-code area paths; it reads them from the table.
4. **Scan capability** — see [Drift Scan](#drift-scan--report-ac-1), including [config-vs-memory conflict analysis](#memory-conflict-analysis--resolution). Read-only; produces a structured report.
5. **Apply capability** — see [Apply Flow](#apply-flow-reviewable-diff--pr--ac-2). Applies **only after explicit human confirmation**, never auto; every write is a reviewable diff (commit on the impl branch when dispatched, or a self-raised PR when standalone); refuse any path outside resolved write-scope.
6. **Skill usage** — during scan, use `find-skills` to surface candidate skills for gaps and `skill-creator` to scaffold a new skill when a gap warrants one (AC-6).
7. Branch/commit/memory/return protocol — reuse `${CLAUDE_PLUGIN_ROOT}/refs/domain-agent-handoff.md`, like every other sdlc domain agent.

## Ownership Model

Ownership is **granted per-repo at init time** — the whole gate. There is no "future handoff": once a repo opts in, `ai-enablement-engineer` owns the AI surface (and `plugins/`, `skills/`) **now**.

### Init opt-in gate

- **Opted in** (`/sdlc:init` marks the agent Active and writes its workspace→agent rows): the agent owns and may write the surface below. `platform-engineer` no longer owns AI-related files/dirs by default.
- **Not opted in**: the agent is inactive; it does nothing and owns nothing. No effect on the repo.

The agent reads this decision from `.claude/project/project-context.md` (Active status + workspace→agent rows). That single fact — plus the read-only carve-outs — is the entire ownership model; there is no other precedence logic.

### Owned AI-config surface (config-driven, extensible)

When active, the agent owns these popular AI-config conventions wherever they appear. The list is illustrative-but-comprehensive, **not** exhaustive: it is **config-driven and extensible per-repo** (baseline globs ship in the agent definition; the per-repo override can add more), so a newly-popular convention is a config edit, not a spec change.

| Convention / path glob | Notes |
| ---------------------- | ----- |
| `CLAUDE.md` (root) + nested `**/CLAUDE.md` | Claude Code instruction files / directory guides |
| `AGENT.md`, `AGENTS.md` (+ nested `**/AGENTS.md`) | The agents.md standard |
| `.agents/**` (incl. `.agents/skills/**`) | Agent config + agent-owned skills convention |
| `.claude/**` | Claude Code project config — `.claude/settings.json`, `.claude/skills/**`, `.claude/project/agents/*.md`, `.claude/project/skills.json` (a **skills-lock**; see below) — **minus** the read-only carve-outs |
| `.github/copilot-instructions.md` | GitHub Copilot instructions |
| `.cursor/rules/**`, `.cursorrules` | Cursor rules (new + legacy) |
| `.windsurfrules` | Windsurf rules |
| `GEMINI.md` | Gemini instruction file |
| `plugins/**`, `skills/**` | Plugins and skills — owned via the workspace→agent table (this repo opts in) |
| Plugin/skill **metadata**: `**/SKILL.md`, `**/.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json` | The "related skill/plugin refs" from the story. Whole-file scope (path-based guard; front-matter-only editing is behavioural guidance, not an enforceable boundary). |

**skills-lock** = **both** `.claude/project/skills.json` **and** any lockfile convention the repo uses (e.g. a future `skills.lock`); the config-driven surface covers both, and the agent adjusts to whichever the repository actually uses.

### Read-only authority (carve-outs)

Reduced to the minimum. Even when active, the agent never writes:

| Path | Why read-only |
| ---- | ------------- |
| `.claude/project/project-context.md` | The workspace→agent table is the **source of ownership truth**, authored by `/sdlc:init` and humans ("init owns setup"). The agent reads it to learn scope; rewriting its own ownership would be circular. |
| `.claude/.*-plugin-root` pointers | Written by init; environment wiring. |
| `.claude/memories/agents/<other-agent>.md` | Each agent owns its own memory file. **Exception:** the human-arbitrated memory-conflict reset (see [Memory-Conflict Analysis](#memory-conflict-analysis--resolution)) may reset another agent's memory, but only via a reviewable diff/PR after the human picks the source of truth — never silently. |

### Documented scope (AC-5)

The agent definition documents the surface above and the runtime rule: *effective write-scope = (config-driven AI-config surface ∪ table-assigned areas) − read-only carve-outs*. At the start of any run the agent prints its resolved write-scope; before any write it **refuses and aborts** on a path outside that scope.

### Init opt-in & boundary design (AC-3)

AC-3 requires ownership be configurable per-repo (not hard-coded) with the `plugins/`+`skills/` boundary designed cleanly. Satisfied by the init opt-in:

1. `/sdlc:init` writes the workspace→agent rows (`plugins/`, `skills/` → `ai-enablement-engineer`), marks the agent Active, and scaffolds its override.
2. `platform-engineer`'s defaults drop AI-related paths — it keeps only non-AI areas (`tools/`, non-AI code).
3. A repo that never opts in leaves the agent inactive.

Because the agent resolves ownership from the table at runtime (never hard-coded) and treats project-context as read-only authority, per-repo configurability is intrinsic: flipping ownership is a project-context edit, done at init, applied immediately.

## Command Surface

### `/sdlc:analyze` (AC-4)

`plugins/sdlc/commands/analyze.md` — shape mirrors `plugins/sdlc/commands/spec.md`. Bootstrap is already covered by `.claude/.sdlc-plugin-root` + `/sdlc:init` — no separate init command.

| Aspect | Behavior |
| ------ | -------- |
| Trigger | Manual only — user runs `/sdlc:analyze`. No scheduling/background trigger (see Out of Scope). |
| Arguments | `$ARGUMENTS` optional — an area/glob to narrow the scan (default: full resolved scope). |
| Default mode | **Read-only scan + report** — drift, gaps, and config-vs-memory conflicts. Writes nothing. |
| Apply | **Only after explicit human confirmation — never auto-apply.** After the report, the command presents each fix; on confirmation the agent runs the [Apply Flow](#apply-flow-reviewable-diff--pr--ac-2). |
| Output | Resolved write-scope (printed first), the drift/gap list + [conflict report](#memory-conflict-analysis--resolution) with proposed fixes, and `find-skills` suggestions. |

## Drift Scan / Report (AC-1)

Defined once in `plugins/sdlc/refs/analyze-protocol.md`, consumed by both the agent and the command. Read-only; repo-wide read permitted, writes scope-limited.

| Check | Drift/gap detected |
| ----- | ------------------ |
| Skills on disk vs skills-lock (`.claude/project/skills.json` and/or lockfile) | Skill dir present but not in manifest, or manifest entry with no dir |
| Agent overrides vs skills | Override lists a skill that resolves to no installed/bundled/project skill |
| Override directory guides vs disk | Override references a `CLAUDE.md` guide that does not exist |
| Dangling refs/scripts | A `refs/*.md` or `scripts/*` path referenced by an agent/command resolves to no file |
| Workspace→agent table vs disk | Table lists a path that no longer exists, or an area with no owner |
| AI-config-file freshness | Documented structure in any owned AI-config convention diverges from disk |
| Plugin/skill metadata | `SKILL.md` missing required front-matter; `plugin.json`/`marketplace.json` referencing a missing plugin path |
| **Config-vs-memory semantic conflict** | An owned AI-config file (`CLAUDE.md`/`AGENTS.md`/override/etc.) asserts one semantic while agent-learning memory (`.claude/memories/agents/*.md`, project shared memory) records a contradictory one — see [Memory-Conflict Analysis](#memory-conflict-analysis--resolution) |
| Vendored-skill staleness (optional) | Recorded pinned ref (README) older than reachable upstream — advisory; refresh is manual |
| **Skill gaps (AC-6)** | For gaps, run `find-skills` to surface candidates; if none fit, note `skill-creator` can scaffold one |

Output shape — per finding: `area` · `kind` (drift \| gap \| conflict) · `detail` · `proposed fix` · `target path(s)` (each flagged W or R against resolved write-scope). No writes.

## Memory-Conflict Analysis & Resolution

`/sdlc:analyze` must also detect and help fix **conflicting semantics** — where an AI-config file says one thing and agent-learning memory records another (e.g. `CLAUDE.md` says "never push from a domain agent" while `platform-engineer`'s memory records "push after commit"). Neither side is presumed correct.

**Report:** conflicts appear in their own section of the analyze report — one entry per conflict listing the two (or more) sources, the exact contradictory assertions, and each source's file path.

**Interactive resolution flow (human-arbitrated, never silent):**

1. Present the conflict and the candidate sources of truth; the human picks which side is correct (config vs memory, or a specific file), or defers.
2. The agent **resets the wrong side** to match the chosen source of truth:
   - Wrong side is an owned AI-config file (`CLAUDE.md`/`AGENTS.md`/override/…) → edit it via the normal [Apply Flow](#apply-flow-reviewable-diff--pr--ac-2) (reviewable diff/PR).
   - Wrong side is the agent's **own** memory → edit `.claude/memories/agents/ai-enablement-engineer.md` (or project shared memory it owns) via the same reviewable flow.
   - Wrong side is **another agent's** memory → this is the documented **exception** to the "each agent owns its memory" rule: the human-confirmed reset is applied as a **reviewable diff/PR** (or under explicit human instruction), never a silent write. State this exception in the agent definition and matrix.
3. No human decision (deferred) → report only; reset nothing.

## Apply Flow (reviewable diff / PR) — AC-2

Applying any fix is explicit and always produces a reviewable diff — never a silent write, and **never without human confirmation**.

0. **Human confirmation gate** — apply only a fix the human has explicitly confirmed from the report. No inline auto-apply path.
1. **Scope guard** — resolve effective write-scope; if the fix's target paths are not all within scope (respecting the read-only carve-outs, and the memory-conflict exception only when human-arbitrated), **refuse and abort** with the offending path listed (satisfies AC-5).
2. Apply the scoped edits (Edit/Write) only within resolved write-scope; for "create a missing skill", drive it through `skill-creator`.
3. Commit via `conventional-commit`.
4. **Reviewable diff:**
   - **Dispatched by `principal-engineer`**: commit on the impl branch per `${CLAUDE_PLUGIN_ROOT}/refs/domain-agent-handoff.md`; `principal-engineer` opens the PR — do **not** self-raise.
   - **Standalone via `/sdlc:analyze`**: branch `chore/ai-config-<slug>` off `main`; raise a PR via `${CLAUDE_PLUGIN_ROOT}/scripts/raise-pr.sh <head> main "<title>" <body-file>`. The PR **is** the reviewable diff.
5. Report the diff/PR back to the user. Do not merge.

## Bundled Default Skills (AC-6)

Both skills are **vendored** into `plugins/sdlc/skills/<name>/SKILL.md` (same layout as the other `plugins/sdlc/skills/*`) and listed in the agent's `skills:` front-matter. Vendoring — not a cross-marketplace dependency — is chosen because both originate from `skills.sh` (a skills registry), not a Claude plugin marketplace.

| Skill | Source | Vet level |
| ----- | ------ | --------- |
| `skill-creator` | `https://www.skills.sh/anthropics/skills/skill-creator` (Anthropic official) | Low risk — confirm provenance and pin the vendored version/commit. |
| `find-skills` | `https://www.skills.sh/vercel-labs/skills/find-skills` (Vercel Labs) | **Standard third-party check.** |

**Upstream-refresh strategy:** refreshed **manually** — pinned ref recorded in `plugins/sdlc/README.md`; the optional staleness drift check surfaces a newer upstream ref but never auto-updates. A refresh is a normal reviewed PR that re-vets and re-pins.

### Third-party skill vetting note

Per this project's design principle "vet third-party skills before adoption" (memory: deepline rejected; postiz-agent + marketingskills adopted), the implementation phase MUST vet before vendoring:

- Read the full `SKILL.md` (and any bundled scripts) before vendoring — reject if unreviewable.
- No network calls that exfiltrate repo content; no remote code execution; no credential/secret access. `find-skills` must be confined to reading local files and querying the public skills registry.
- **Licensing/attribution:** confirm the upstream license permits vendoring/redistribution; carry the upstream `LICENSE` and attribution/notice into `plugins/sdlc/skills/<name>/` alongside the vendored `SKILL.md`; record license + upstream URL + pinned ref together in `plugins/sdlc/README.md`.
- Pin the vendored copy to a specific version/commit (recorded in README).
- `skill-creator` (Anthropic-official) passes with provenance + version + license check; `find-skills` gets the full checklist.
- If `find-skills` fails vetting (incl. a license that forbids vendoring), ship without it and open a follow-up — `skill-creator` alone still satisfies "suggest/create skills", but AC-6 names both, so a fail is a flagged deviation, not a silent drop.

## Error Handling

| Scenario | Behavior |
| -------- | -------- |
| Repo not opted in (agent not Active in project-context) | No-op — report "AI-config management not enabled; run `/sdlc:init` to opt in." Write nothing. |
| Scan finds no drift/gaps/conflicts | Report "no drift detected" and exit cleanly. |
| Apply attempted without human confirmation | Refuse — confirmation is mandatory (never auto-apply). |
| Apply target outside resolved write-scope | **Refuse and abort**; print the offending path(s); make no writes (AC-5). |
| Memory conflict with no human decision (deferred) | Report only; reset nothing. |
| Reset targets another agent's memory but not human-arbitrated | Refuse — the cross-agent memory exception applies only to a human-confirmed, reviewable reset. |
| `project-context.md` missing or malformed table | Scan proceeds on the AI-config surface; report the table problem as drift; do not edit project-context (read-only authority). |
| `find-skills` / `skill-creator` unavailable or offline | Degrade gracefully — skip the skill-suggestion step, still emit structural drift; note the skip. |
| `raise-pr.sh` fails during standalone apply | Surface the failure; leave branch + local commit for manual recovery; do not retry silently. |

## Out of Scope

(The init touch in scope here is the explicit opt-in: write ownership rows + scaffold the override — an edit to existing init, not a scan/auto-configure flow.)
- **Editing generated or environment files** — `.claude/.*-plugin-root` pointers and `.claude/project/project-context.md` are read-only authority for this agent.
- Web/mobile/data/API work — none exist in this repo.
