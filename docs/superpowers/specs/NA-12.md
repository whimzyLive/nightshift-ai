## Overview

Add a dedicated, repo-agnostic agent that owns the repository's AI-configuration surface (`CLAUDE.md`, `AGENT.md`, the skills manifest, agent overrides, plugin/skill metadata) and keeps it synchronized, plus a manual `/analyze` command that runs a repo-wide drift/gap scan on demand. ## Ownership Model

The agent resolves its write-scope at runtime from two inputs, exactly matching the mechanism documented in `EXTENDING.md` (table-driven ownership + override). It never hard-codes area paths.

### Core AI-config surface (always owned when the agent is active)

Owned regardless of the workspaceâ†’agent table, in **any** repo the agent is set up in:

| Path / pattern | Notes |
| -------------- | ----- |
| `CLAUDE.md` (root) and directory `**/CLAUDE.md` guides | AI instruction files |
| `AGENT.md` | AI instruction file |
| `.claude/project/agents/*.md` | Agent overrides (skill bindings, directory guides, ownership lines) |
| `.claude/project/skills.json` | The **skills manifest / "skills-lock"** (this repo's `skills-lock` per the story; today `{ "version": 1, "skills": [] }`) |
| `.claude/skills/**` | Project-local skill definitions |
| Plugin/skill **metadata**: `**/SKILL.md` front-matter, `**/.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json` | The "related skill/plugin refs" from the story |

### Read-only authority (never written by the agent)

| Path | Why read-only |
| ---- | ------------- |
| `.claude/project/project-context.md` | This file's workspaceâ†’agent table is the **source of ownership truth**. Per this project's design principle ("init owns setup â€” no setup agents"), it is authored by `/sdlc:init` and humans. The agent **reads** it to learn scope; letting it rewrite its own ownership would be circular. |
| `.claude/.aix-plugin-root` and other `.claude/.*-plugin-root` pointers | Written by init; environment wiring. |
| `.claude/memories/agents/<other-agent>.md` | Each agent owns its own memory file only. The agent may write only `.claude/memories/agents/ai-enablement-engineer.md`. |

### Extended surface (per-repo, table-driven)

Any additional area assigned to `ai-enablement-engineer` in the project-context workspaceâ†’agent table becomes owned â€” e.g. `plugins/`, `skills/`, `tools/`. This is the mechanism the story's follow-up uses.

### Documented scope (AC-5)

The agent definition documents the core surface above, and states the runtime rule: *effective scope = core surface âˆª (table rows assigned to `ai-enablement-engineer`) âˆ’ read-only-authority paths*. The agent must (a) print its resolved scope at the start of any run so humans/agents can see it, and (b) **refuse and abort** before writing any path outside that scope (guard in [Apply Flow](#apply-flow-reviewable-diff--pr)).

### Handoff-boundary design (AC-3, Dependencies)

The planned follow-up reassigns `plugins/` and `skills/` from `platform-engineer` to this agent. Because scope is table-driven, that handoff is a **config edit only** â€” no change to any agent's code:

1. In `.claude/project/project-context.md` workspaceâ†’agent table: change the `plugins/` owner to `ai-enablement-engineer` and add a `skills/` row owned by `ai-enablement-engineer`.
2. In `.claude/project/agents/platform-engineer.md`: remove `plugins/` from `owns:`, add it (and `skills/`) to `never:`.
3. In `.claude/project/agents/ai-enablement-engineer.md`: add `plugins/`, `skills/` to `owns:`.

This spec's contract that makes the handoff "clean rather than retrofitted": the agent reads ownership from the table at runtime and treats project-context as read-only authority â€” so the follow-up story only edits config, and this spec does **not** perform the reassignment (see [Out of Scope](#out-of-scope)).

## Drift Scan / Report

Defined once in a shared ref (`plugins/aix/refs/analyze-protocol.md`) consumed by both the agent and the command. Read-only. Enumerate AI artifacts within resolved scope and detect drift/gaps:

| Check | Drift/gap detected |
| ----- | ------------------ |
| Skills on disk vs `.claude/project/skills.json` | Skill dir present but not in manifest, or manifest entry with no dir |
| Agent overrides vs skills | Override lists a skill (in `.claude/project/agents/*.md`) that resolves to no installed/bundled/project skill |
| Override directory guides vs disk | Override references a `CLAUDE.md` guide that does not exist |
| Workspaceâ†’agent table vs disk | Table lists a path that no longer exists, or a top-level area with no owner |
| `CLAUDE.md` / `AGENT.md` freshness | Documented structure (plugins, commands, workspace paths) diverges from what's on disk |
| Plugin/skill metadata | `SKILL.md` missing required front-matter (`name`, `description`); `plugin.json` / `marketplace.json` referencing a missing plugin path |
| **Skill gaps (AC-6)** | For detected gaps, run `find-skills` to surface candidate existing skills; if no fit exists, note that `skill-creator` can scaffold a new one |

Output shape â€” a structured report (per finding): `area` Â· `kind` (drift \| gap) Â· `detail` Â· `proposed fix` Â· `target path(s)` (all within resolved scope). No writes.

## Apply Flow (reviewable diff / PR) â€” AC-2

Applying any fix is explicit and always produces a reviewable diff â€” never a silent write:

1. **Scope guard first** â€” resolve effective scope; if the fix's target paths are not all within scope, **refuse and abort** with the offending path listed (satisfies AC-5: "never modifies a file outside its declared scope").
2. Create a branch â€” `chore/aix-<slug>` (or `feat/aix-<slug>` for a new skill/plugin), off the base branch (`main` per project-context).
3. Apply the scoped edits (Edit/Write) only within resolved scope; when the fix is "create a missing skill", drive it through the bundled `skill-creator` skill.
4. Commit via the `conventional-commit` skill.
5. Raise a PR via `${CLAUDE_PLUGIN_ROOT}/scripts/raise-pr.sh <head> main "<title>" <body-file>` (the shipped atomic create+ready+review-request helper). The PR **is** the reviewable diff (AC-2).
6. Report the PR URL back to the user. Do not merge.

Both skills are **vendored** into `plugins/aix/skills/<name>/SKILL.md` (the same layout as `plugins/sdlc/skills/*`) and listed in the agent's `skills:` front-matter so they are available by default. ### Third-party skill vetting note

Per this project's design principle "vet third-party skills before adoption" (memory: deepline rejected; postiz-agent + marketingskills adopted), the implementation phase MUST vet before vendoring, and the spec fixes the checklist:

- Read the full `SKILL.md` (and any bundled scripts) before vendoring â€” reject if unreviewable.
- No network calls that exfiltrate repo content; no execution of remote code; no credential/secret access. `find-skills` must be confined to reading local files and querying the public skills registry for suggestions.
- Pin the vendored copy to a specific version/commit; record the upstream URL and pinned ref in `plugins/aix/README.md`.
- `skill-creator` (Anthropic-official) passes with a provenance + version check; `find-skills` gets the full checklist above.
- If `find-skills` fails vetting, ship without it and open a follow-up â€” `skill-creator` alone still satisfies "suggest/create skills", but AC-6 names both, so a fail is a flagged deviation, not a silent drop.

## Error Handling

| Scenario | Behavior |
| -------- | -------- |
| Scan finds no drift/gaps | Report "no drift detected" and exit cleanly (success). |
| No override / no table rows for the agent | Fall back to the **core AI-config surface** only; warn that extended areas are unconfigured. Never error out. |
| Apply target outside resolved scope | **Refuse and abort** the apply; print the offending path(s); make no writes (AC-5). |
| `project-context.md` missing or malformed table | Scan proceeds on the core surface; report the table problem as a drift finding; do not attempt to fix project-context (read-only authority). |
| `find-skills` / `skill-creator` unavailable or offline | Degrade gracefully â€” skip the skill-suggestion step, still emit structural drift findings; note the skipped step in the report. |
| `raise-pr.sh` fails during apply | Surface the error and PR-creation failure; leave the branch + local commit intact for manual recovery; do not retry silently. |
| Run inside another agent's memory/scope | Never write `.claude/memories/agents/<other>.md`; only the agent's own memory file. |

## Out of Scope

- **Repository-init-time auto-discovery** of ownership/config â€” per the design principle "init owns setup (no setup agents)", this belongs as a future extension of `/sdlc:init`, not agent-owned logic.

- **Editing generated or environment files** â€” `.claude/.*-plugin-root` pointers and `.claude/project/project-context.md` are read-only authority for this agent.
- Web/mobile/data/API work â€” none exist in this repo.
