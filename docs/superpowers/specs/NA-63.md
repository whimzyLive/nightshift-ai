# Automate plugin versioning with nx release + conventional commits вАФ Technical Spec

**Story:** NA-63
**Epic:** NA-24 (SDLC Improvements)
**Date:** 2026-07-19

## Overview

Drive independent, per-plugin SemVer releases for `plugins/sdlc` and `plugins/gtm` from Conventional Commits using `nx release` вАФ automatic bump, per-plugin git tag, and per-plugin `CHANGELOG.md`. The version source of truth is each plugin's `.claude-plugin/plugin.json` (not a `package.json`), so a custom Nx `VersionActions` module is required.

This is build-tooling work. No data model, API, permissions, sync, or web/mobile UI is involved вАФ those sections are omitted.

## Decision 1 вАФ Register plugins as Nx projects (required)

`release.projects` and a release group's `projects` list are **filters over the existing project graph** вАФ they do not create projects. `nx release` can only version projects that exist in the graph. Therefore each plugin **must** get a `project.json`. A `release.projects` glob alone is insufficient.

Add two minimal project manifests:

**`plugins/sdlc/project.json`**

```json
{
  "$schema": "../../node_modules/nx/schemas/project-schema.json",
  "name": "sdlc",
  "projectType": "library",
  "root": "plugins/sdlc",
  "targets": {}
}
```

**`plugins/gtm/project.json`**

```json
{
  "$schema": "../../node_modules/nx/schemas/project-schema.json",
  "name": "gtm",
  "projectType": "library",
  "root": "plugins/gtm",
  "targets": {}
}
```

- Project names `sdlc` and `gtm` вАФ these become the `{projectName}` used in the release tag pattern (`sdlc@X.Y.Z`).
- Empty `targets` вАФ no build/test/lint/typecheck/e2e targets are declared and none are inferred (no `tsconfig`/`eslint`/`jest` config in the plugin dirs). Implementer MUST verify after adding: `pnpm nx show project sdlc --json` and `pnpm nx run-many -t lint test build typecheck e2e --dry-run` show no new targets running for `sdlc`/`gtm`, so existing CI is unaffected.
- Run `bash tools/portability-lint.sh` after adding вАФ `project.json` carries no project-specific tokens, so it must stay clean.

## Decision 2 вАФ Version source mechanism (custom VersionActions)

Nx's default version actions (`@nx/js/src/release`) read/write `package.json`. The plugin version lives in `.claude-plugin/plugin.json`, so a custom `VersionActions` subclass is required, referenced from config via a workspace-relative path.

**Nx 23 API relied upon** (verified in `node_modules/nx/dist/src/command-line/release/version/version-actions.d.ts`):

- The `versionActions` config value is resolved with `require.resolve(path)` then `require.resolve(join(workspaceRoot, path))`. A `.ts` path is loaded via Nx's `loadTsFile`, and the class is taken from `loaded.default ?? loaded`. So the module is a workspace-relative `.ts` file exporting the subclass as `default`.
- The module extends the abstract `VersionActions` class from `nx/release/version-actions` (public deep-import).

**File:** `tools/nx-release/plugin-json-version-actions.ts` вАФ `export default class PluginJsonVersionActions extends VersionActions`.

The subclass MUST implement exactly these abstract members (WHAT each does вАФ not line-by-line HOW):

| Member | Responsibility for this project |
| ------ | ------------------------------- |
| `validManifestFilenames: string[]` | `["plugin.json"]` |
| `readCurrentVersionFromSourceManifest(tree)` | Read `version` from `<projectRoot>/.claude-plugin/plugin.json`; return `{ currentVersion, manifestPath }`. Only invoked when `currentVersionResolver: "disk"` (our fallback). |
| `readCurrentVersionFromRegistry(...)` | Not applicable вАФ return `null` (no registry). |
| `readCurrentVersionOfDependency(...)` | Return `{ currentVersion: null, dependencyCollection: null }` вАФ `plugin.json` deps are marketplace refs, not versioned Nx deps. |
| `updateProjectVersion(tree, newVersion)` | Write `newVersion` into `<projectRoot>/.claude-plugin/plugin.json` `version`, preserving key order/formatting; return a one-line log message. Use the `manifestsToUpdate` paths resolved from `manifestRootsToUpdate` (below). |
| `updateProjectDependencies(...)` | No-op вАФ return `[]`. Never rewrite the marketplace `dependencies` array. |

- No `afterAllProjectsVersioned` export is needed (no workspace lockfile to update for plugins).
- `versionActionsOptions`: none required вАФ `{}`.

The abstract-member contract above is the WHAT. The implementing agent writes the file body (JSON read/write, log strings) following it.

## Decision 3 вАФ Manifest path resolution

`plugin.json` sits at `{projectRoot}/.claude-plugin/plugin.json`, not `{projectRoot}/plugin.json`. Nx builds `manifestsToUpdate` by joining each `manifestRootsToUpdate` entry with each `validManifestFilenames` entry. Set:

```
manifestRootsToUpdate: ["{projectRoot}/.claude-plugin"]
```

Combined with `validManifestFilenames: ["plugin.json"]`, this resolves to `plugins/sdlc/.claude-plugin/plugin.json` and `plugins/gtm/.claude-plugin/plugin.json`.

## Decision 4 вАФ `nx.json` `release` block

Add this `release` key to `nx.json` (alongside the existing top-level keys; do not disturb `plugins`, `targetDefaults`, etc.):

```json
"release": {
  "projectsRelationship": "independent",
  "releaseTagPattern": "{projectName}@{version}",
  "groups": {
    "plugins": {
      "projects": ["sdlc", "gtm"],
      "projectsRelationship": "independent",
      "version": {
        "specifierSource": "conventional-commits",
        "currentVersionResolver": "git-tag",
        "fallbackCurrentVersionResolver": "disk",
        "manifestRootsToUpdate": ["{projectRoot}/.claude-plugin"],
        "versionActions": "./tools/nx-release/plugin-json-version-actions.ts",
        "adjustSemverBumpsForZeroMajorVersion": false
      },
      "changelog": {
        "projectChangelogs": {
          "file": "{projectRoot}/CHANGELOG.md"
        },
        "createRelease": false
      }
    }
  },
  "changelog": {
    "workspaceChangelog": false
  },
  "conventionalCommits": {
    "useCommitScope": true
  }
}
```

Field rationale (each verified against the Nx 23 schema / config defaults):

- `projectsRelationship: "independent"` вАФ sdlc and gtm bump/tag independently (AC1).
- `releaseTagPattern: "{projectName}@{version}"` вАФ yields `sdlc@X.Y.Z` / `gtm@X.Y.Z` (AC4). Set explicitly rather than relying on the mode default.
- `groups.plugins.projects: ["sdlc", "gtm"]` вАФ the release group; `apps/marketing` is not a project here and is excluded (AC1).
- `version.specifierSource: "conventional-commits"` вАФ bump derived from commits (AC2).
- `version.currentVersionResolver: "git-tag"` with `fallbackCurrentVersionResolver: "disk"` вАФ the current version and the "since" point come from the latest matching git tag; if no tag exists yet, fall back to reading `plugin.json` from disk. Pairs with the baseline-tag backfill (Decision 6).
- `version.versionActions` вАФ the custom module (Decision 2).
- `version.adjustSemverBumpsForZeroMajorVersion: false` вАФ **critical.** This flag defaults to `true`, which for 0.x versions downgrades every bump (breakingвЖТminor, featвЖТpatch). Both plugins are 0.x, so leaving the default would violate the required bump table. Setting `false` restores standard SemVer: featвЖТminor, fixвЖТpatch, breakingвЖТmajor (AC2).
- `changelog.projectChangelogs.file: "{projectRoot}/CHANGELOG.md"` вАФ per-plugin changelog files (AC3).
- `changelog.workspaceChangelog: false` (top level) вАФ no root/workspace `CHANGELOG.md` (root/website out of scope).
- `changelog.createRelease: false` вАФ no GitHub Release object (external publish out of scope).
- `conventionalCommits.useCommitScope: true` вАФ scope routes commits to the matching project (Decision 5). This is already the Nx default; stated explicitly for the documented contract.

Commit-type вЖТ bump map is Nx's default `DEFAULT_CONVENTIONAL_COMMITS_CONFIG` (featвЖТminor, fixвЖТpatch, everything elseвЖТnone), which already matches the required table вАФ no `types` override is needed.

## Decision 5 вАФ Conventional-commits contract

Scope routes a commit to a project. With `useCommitScope: true`, for a given project Nx considers commits that changed files under that project's root since its last tag, and uses the commit **scope** to resolve the specifier.

| Commit form | sdlc effect | gtm effect |
| ----------- | ----------- | ---------- |
| `feat(sdlc): вА¶` | minor | вАФ |
| `fix(sdlc): вА¶` | patch | вАФ |
| `feat(gtm): вА¶` | вАФ | minor |
| `fix(gtm): вА¶` | вАФ | patch |
| `feat(sdlc)!: вА¶` or body `BREAKING CHANGE:` | major | вАФ |
| `chore(sdlc): вА¶`, `docs(sdlc): вА¶`, `refactor`, `perf`, `build`, `test`, `ci`, `style` | no bump | вАФ |

Bump table (applies per project, after `adjustSemverBumpsForZeroMajorVersion: false`):

| Commit type / marker | Bump |
| -------------------- | ---- |
| `feat` | minor |
| `fix` | patch |
| `!` suffix or `BREAKING CHANGE:` footer (any type) | major |
| `chore`, `docs`, `refactor`, `perf`, `build`, `test`, `ci`, `style` | none |

**Authoring rule (document this):** one plugin per commit, and the scope must match the plugin (`sdlc` or `gtm`). Rationale/edge case to document: with `useCommitScope: true`, a commit that touches a project's files but carries a *different* scope is treated by Nx as a `patch` for that project. Keeping each commit scoped to a single plugin avoids accidental cross-project patch bumps. The existing repo commit convention already scopes by plugin (`feat(sdlc)`, `fix(gtm)`), so no workflow change is needed.

## Decision 6 вАФ Baseline tag backfill

Because `currentVersionResolver: "git-tag"`, the first automated release needs a correct "since" point. Create annotated tags at the current released versions on the commit that represents today's published state (current tip of `develop` after this work lands, i.e. the commit where `plugin.json` already reads 0.42.0 / 0.5.0):

```bash
git tag -a sdlc@0.42.0 -m "sdlc 0.42.0 (baseline)"
git tag -a gtm@0.5.0   -m "gtm 0.5.0 (baseline)"
git push origin sdlc@0.42.0 gtm@0.5.0
```

- Tag names use the exact `releaseTagPattern` (`{projectName}@{version}`) so `nx release` matches them (AC5).
- After backfill, the first real `feat`/`fix` release bumps from 0.42.0 / 0.5.0 and computes changelog from commits after these tags.
- These are one-time manual bootstrap tags. Which commit to tag is an implementer decision вАФ see Open Questions.

## Release invocation

`nx release` runs version вЖТ changelog вЖТ git commit вЖТ git tag in one pass. Commands (all via `pnpm nx`, package manager is pnpm):

```bash
# AC7 вАФ dry run proving per-plugin bump/tag/changelog with no state mutation:
pnpm nx release --dry-run

# Release the whole plugin group:
pnpm nx release

# Release a single plugin:
pnpm nx release --projects=sdlc
pnpm nx release --projects=gtm
```

- `--dry-run` performs no writes, tags, or commits вАФ it prints the computed bump, tag, and changelog diff (satisfies AC7).
- **First-release handling:** after the baseline tags are pushed (Decision 6), no special flag is needed вАФ `git-tag` resolves the current version from the baseline tag. If a release is ever run before the baseline tag exists for a project, use `pnpm nx release --first-release` (Nx skips the missing-current-version error and uses the disk fallback). Prefer the backfill path.
- Subcommand form is available if needed: `pnpm nx release version`, `pnpm nx release changelog`.
- Release is invoked manually by a maintainer. Wiring `nx release` into CI is out of scope (no AC requests it).

## Error handling

| Scenario | Behaviour |
| -------- | --------- |
| No bump-worthy commits since a plugin's last tag | Nx reports the project unchanged and skips it вАФ no tag, no changelog entry. Expected/benign. |
| `git-tag` resolver finds no tag for a project | `fallbackCurrentVersionResolver: "disk"` reads the version from `plugin.json`. Backfill (Decision 6) makes this the exception, not the rule. |
| `versionActions` path fails to resolve | Nx throws "not possible to resolve the VersionActions implementation". Fix: confirm the workspace-relative `./tools/nx-release/plugin-json-version-actions.ts` path and that `@nx/js` is installed (it is). |
| `plugin.json` missing at the resolved manifest path | Nx `validate()` throws that the project has no `plugin.json` in the manifest root. Fix `manifestRootsToUpdate`. |
| Commit scoped to the wrong plugin | Bumps the wrong project (or a stray patch). Mitigated by the one-plugin-per-commit authoring rule; caught by `--dry-run` before a real release. |
| Adding `project.json` introduces unexpected inferred targets | Verify with `pnpm nx run-many -t lint test build typecheck e2e --dry-run` before merge; none expected. |
