# Cut per-dispatch worktree and agent cold-start overhead — Technical Spec

**Story:** NA-27 (parent epic NA-24 "SDLC Improvements")
**Feature:** n/a (SDLC-plugin tooling story; no product feature doc)
**Date:** 2026-07-11

## Overview

The `/impl` and `/auto` implementation phase currently dispatches every domain agent with the
harness's per-dispatch `isolation: "worktree"`, so each dispatch creates its own throwaway git
worktree (cold `pnpm install` ≈ 1.2–1.4 GB, cold Nx/Next caches) that is never reclaimed. This
story replaces that with **one orchestrator-managed worktree per story**, a **shared Nx cache**, a
**worktree garbage-collector**, and flag-controlled **domain-agent reuse across fix rounds**
(shipped enabled; NA-23 later removes its re-injection cost) —
scoped to the impl phase only.

## Applicable agents

Per `.claude/project/project-context.md` the touched surface is entirely under `plugins/**` and
`.claude/**` config, owned by a single active agent:

| Agent                      | Why                                                                                                           |
| -------------------------- | ------------------------------------------------------------------------------------------------------------- |
| **ai-enablement-engineer** | Owns `plugins/`, `skills/`, `.agents/` and the AI-config surface — every file this story changes lives there. |

No `database-administrator`, `platform-engineer` (tools/), `web-engineer`, `sync-engineer`, or
`mobile-engineer` work. No data model, API surface, UI, or offline sync — those sections are
omitted as not applicable.

## Background — current mechanism (as-is)

- `refs/principal-engineer-playbook.md` Step 4 and `refs/qa-engineer-playbook.md` Step 3 dispatch
  each domain / fix agent with `Agent({ isolation: "worktree" })`. The Claude Code harness
  provisions a **fresh** `.claude/worktrees/agent-<hash>` per dispatch.
- The harness only auto-removes a worktree that is left **unchanged**. Every domain agent commits,
  so every worktree is left "changed" and **persists** — measured 8.7 GB / 7 dispatches on NA-22,
  host disk 99%.
- Each dispatch is a **fresh subagent** (never `SendMessage` reuse), re-paying frontmatter skill
  injection, plan/spec/override/memory reads and repo rediscovery (≈ 1.46 M subagent tokens / 7
  dispatches). Agent reuse is currently penalized by the frontmatter re-injection bug
  (anthropics/claude-code#76337, tracked as **NA-23**, ~5–6 k duplicate tokens per resume).
- Verified repo facts: `.gitignore` already ignores `.claude/worktrees`, `.nx/cache`,
  `.nx/workspace-data`, and `.tmp/`. Root `node_modules` exists (pnpm workspace). Nx `23.0.1`.
  `nx.json` sets **no** `cacheDirectory` today. Base branch is `main`. `/auto` delegates its impl
  phase straight to `refs/principal-engineer-playbook.md` (auto.md line ~359), so editing the
  playbook covers `/auto` too — no separate worktree logic lives in `auto.md`.

## Worktree lifecycle — the one coherent story

Every design section below is a slice of this single lifecycle. Read it first; the ACs then map
onto its stages.

```
provision (once, idempotent)  →  use (all domain phases)  →  fix rounds (QA + post-PR review-fix)
        │                                                              │
        └────────────── re-provision idempotently on any later entry ─┘
                                            │
                        teardown at impl-PR raise (Step 7, AC-1)
                                            │
                 GC safety-net at session end (merged-into-origin/main only, AC-2)
```

Invariants that keep the stages non-contradictory:

1. **The primary checkout NEVER checks out the story branch.** The branch is created _inside_ the
   worktree (`git worktree add -b`), so provisioning can never deadlock on "branch already checked
   out elsewhere" (Finding 1). All orchestrator git operations on the story branch (per-phase push,
   PR head) run against the worktree via `git -C "$WORKTREE"`.
2. **Provisioning is idempotent and every fix-round entry re-invokes it.** `/impl` Step 6 QA loop,
   standalone `/review-fix`, and `/sdlc:loop` all call `worktree-setup.sh` before dispatching a fix
   agent, so a torn-down or GC'd worktree is transparently re-created (Findings 3, 4).
3. **Teardown and GC use different, safe gates.** Step-7 teardown removes the impl worktree once the
   PR is raised (AC-1 literal). GC only ever removes a worktree whose branch is **merged into
   `origin/main`** — never into its own branch — so an in-flight or concurrent-session worktree is
   never destroyed (Findings 2, 6).

## Design (to-be) — what to build

### 1. Persistent per-story worktree (AC-1)

The **orchestrator** (the top-level session running the principal-engineer playbook) provisions and
owns exactly one worktree for the whole impl phase, instead of the harness making one per dispatch.

- **New script `plugins/sdlc/scripts/worktree-setup.sh <STORY-KEY> <BRANCH> <BASE-BRANCH>`** —
  idempotent, and it **owns branch creation** (the primary checkout must not check out the branch):
  - Computes a deterministic path `WT=.claude/worktrees/sdlc-<STORY-KEY>` (distinct from the
    harness's `agent-<hash>` dirs so GC can target SDLC-managed worktrees unambiguously). Path is
    already gitignored via `.claude/worktrees`.
  - Resolves the worktree in three idempotent cases:
    1. `WT` already a registered worktree → **reuse** (`git -C "$WT" fetch origin <BRANCH>` then
       fast-forward to the branch head; no-op if already current).
    2. Branch `<BRANCH>` already exists (local or `origin/<BRANCH>`) but no worktree — the
       re-provision-after-teardown case → `git worktree add "$WT" "<BRANCH>"` (tracking the existing
       branch).
    3. Neither exists — first run → `git worktree add -b "<BRANCH>" "$WT" "origin/<BASE-BRANCH>"`
       (creates the branch **in the worktree** from the base; the primary checkout is untouched).
  - Provisions `node_modules` for `WT` **once** per story with a pnpm-aware strategy (see Open
    Question 1) — because there is now a single persistent worktree, this install happens once per
    story, not once per dispatch, which is what actually reclaims the measured 8.7 GB.
  - Prints two lines for the orchestrator to capture and thread into every dispatch prompt:
    `WORKTREE=<abs path>` and `NX_CACHE_DIRECTORY=<abs shared-cache path>` (see AC-3).
  - Any hard failure (cannot create/reuse the worktree) → non-zero exit; the orchestrator STOPs the
    impl phase and reports (never silently falls back to the primary checkout for a writing agent).
- **`refs/principal-engineer-playbook.md` Step 3** (branch create) is **rewritten**: it no longer
  runs `git checkout -b <BRANCH> origin/<BASE>` in the primary checkout. Instead it calls
  `worktree-setup.sh <STORY-KEY> <BRANCH_PREFIX>/<STORY-KEY> <BASE-BRANCH>` once (which creates the
  branch inside the worktree, case 3 above), captures `WORKTREE` / `NX_CACHE_DIRECTORY`, and pushes
  the new branch from the worktree: `git -C "$WORKTREE" push -u origin <BRANCH>`.
- **`refs/principal-engineer-playbook.md` Step 4 (feature dispatch, ~lines 294–303) AND the
  `WORK_KIND=defect` variant dispatch (~lines 205, 215)** drop the harness `isolation: "worktree"`
  param (the orchestrator now owns isolation). Every dispatch prompt contract — feature phases,
  defect phase-3 (failing test) and phase-4 (fix) — gains a mandatory first instruction: _"Your
  working directory for ALL work is `<WORKTREE>` — `cd` into it before any read, edit, build, test,
  or commit. Do NOT operate in the primary checkout."_ The isolation guarantee (protect the primary
  checkout / a dev server running there) is preserved because `<WORKTREE>` is a separate checkout on
  the story branch.
- **`refs/principal-engineer-playbook.md` Step 5 (phase verification)** runs its HEAD-advance /
  push against the worktree (`git -C "$WORKTREE" ...`) and adds the primary-checkout guard from
  Finding 9 (see §5 below).
- **`refs/qa-engineer-playbook.md` Step 3 fix loop** likewise drops `isolation: "worktree"`, adds
  the same cwd instruction, and — because it can be entered in a fresh session — **calls
  `worktree-setup.sh` (idempotent) before dispatching any fix agent** so the worktree exists even
  after a prior teardown/GC (Findings 3, 4).
- **Teardown (AC-1):** `refs/principal-engineer-playbook.md` Step 7, immediately after the impl PR
  is raised, removes the impl worktree via `git worktree remove --force "$WORKTREE"`. This satisfies
  AC-1 literally (the impl worktree does not sit idle after the PR). It does **not** strand later
  review-fix rounds: any subsequent QA / `/review-fix` / `/sdlc:loop` entry re-provisions the
  worktree idempotently per invariant 2.

### 2. Worktree GC (AC-2)

- **New script `plugins/sdlc/scripts/worktree-gc.sh [STORY-KEY]`** — reclaims worktrees whose branch
  is **fully merged into `origin/<BASE-BRANCH>` (`origin/main`)**, the only safe gate:
  - Enumerate SDLC-managed worktrees under `.claude/worktrees/sdlc-*` (and any leftover harness
    `agent-*` from prior runs — see Open Question 2 for scope).
  - Resolve the merge base once: `BASE=origin/main` (from project-context Base branch), always —
    it is defined even when no `STORY-KEY` arg is passed (the SessionEnd case, Finding 6). Refresh
    it first with `git fetch origin <BASE-BRANCH>`.
  - For each worktree, read its checked-out branch `WB` and head `WH`. **Remove it iff BOTH hold:**
    (a) `WB` is **not** the base branch itself (`WB != <BASE-BRANCH>`), and
    (b) `git merge-base --is-ancestor "$WH" "$BASE"` succeeds (its work is fully merged into
    `origin/main`).
    This closes Finding 2 (a worktree can never be judged "merged into its own branch") and Finding
    6 (base is defined without a story key; an unmerged in-flight or concurrent-session worktree is
    never removed because its branch is not yet in `origin/main`).
  - On removal: `git worktree remove --force "$WT"` then delete the now-merged local branch.
  - Finish with `git worktree prune` to drop stale admin entries.
  - Always exit 0 (a GC failure must never fail the session) — mirrors the `cleanup-tmp.sh`
    contract.
- **`plugins/sdlc/scripts/session-complete.sh`** gains a call to `worktree-gc.sh` alongside its
  existing scoped-temp cleanup (both are session-teardown reclamation), so no merged worktree
  persists past session end even if Step 7 teardown was skipped (blocked/aborted run).
- **`plugins/sdlc/hooks/hooks.json` → `SessionEnd`** (currently only `cleanup-tmp.sh`) also runs
  `worktree-gc.sh` (no story-key arg — base defaults to `origin/main`) as the safety net for a
  session that errors before `session-complete.sh` (mirrors how `cleanup-tmp.sh` backstops the
  scoped-temp clean).

### 3. Shared Nx cache (AC-3)

- The shared cache location is the **primary checkout's** Nx cache, referenced by absolute path so
  every environment (the one story worktree, and any future isolated env) hits the same warm cache
  instead of recompiling cold:
  `NX_CACHE_DIRECTORY=<repo-root>/.nx/cache` (already gitignored).
- `worktree-setup.sh` emits `NX_CACHE_DIRECTORY=<abs path>`; the orchestrator threads it into every
  dispatch prompt contract as: _"Before running any `nx` command (build/test/quality gate), export
  `NX_CACHE_DIRECTORY=<abs path>` so tasks hit the shared warm cache."_ This applies to the domain
  agent's own runs and to the QA quality-gate run (`refs/qa-engineer-playbook.md`).
- `NX_CACHE_DIRECTORY` (Nx's documented env override) is chosen over an `nx.json` `cacheDirectory`
  key because a relative `nx.json` value would resolve to each worktree's own root (defeating
  sharing) and this story must not mutate the repo's committed `nx.json` build config. See Open
  Question 3.

### 4. Agent reuse across fix rounds (AC-4, flag-controlled — shipped enabled)

NA-23 ("Eliminate duplicate skill loads in SDLC dispatches") is currently **To Do**. The reuse path
is wired behind a flag; per the owner's Open-Question-4 decision the flag **ships `enabled` from day
one** (an early flip beats a forgotten manual flip after NA-23 — see Open Questions).

- Add a capability flag to `.claude/project/project-context.md` **Tooling**:
  `SDLC agent reuse = enabled` (shipped value). Until NA-23 lands, a reused (resumed) agent re-pays
  frontmatter skill injection (harness bug #76337) — an accepted token-cost trade-off, not a
  correctness risk; NA-23's landing removes that cost with no further flag change. Setting the token
  to `disabled` remains the documented off-switch.
- **`refs/qa-engineer-playbook.md` Step 3 fix loop** branches on the flag:
  - `disabled` → today's behaviour: each fix round dispatches a **fresh** `Agent(...)` of the owning
    domain type.
  - `enabled` → the **same** domain agent instance that ran the phase is reused across its fix-round
    dispatches via `SendMessage` (relaying the resume as authorized same-task control — consistent
    with `scrum-master.md`'s resume-trust rule), avoiding re-paid frontmatter/skill injection.
  - Fresh dispatch remains the explicit fallback whenever the reused instance is unavailable
    (session boundary, agent already returned/terminated).
- The reuse switch is **read-only w.r.t. NA-23**: this story does not fix #76337, only consumes the
  flag once it is set.

### 5. Primary-checkout protection guard (Finding 9)

With the harness `isolation` param dropped, the cwd guarantee is no longer prose-only — it is
machine-checked in `refs/principal-engineer-playbook.md` Step 5 (and mirrored in
`refs/qa-engineer-playbook.md` Step 3 after each fix dispatch):

- **Before** dispatching a phase, capture the primary checkout's state:
  `PRIMARY_HEAD=$(git -C "<primary-root>" rev-parse HEAD)` and confirm
  `git -C "<primary-root>" status --porcelain` is empty.
- **After** the agent returns, assert the primary checkout is unchanged: `PRIMARY_HEAD` identical
  **and** `git -C "<primary-root>" status --porcelain` still empty. If the agent wrote to (or
  committed in) the primary checkout instead of the worktree → **fail the phase and STOP**, same
  shape as the existing silent-failure STOP. This makes an ignored-cwd instruction a hard,
  detectable failure rather than a silent corruption of the primary checkout.
- The existing worktree HEAD-advance + push checks remain the completeness signal for the phase.

## Scope boundary — impl phase only

Per the story's scope directive, worktree isolation and this whole change apply **only to the impl
phase** (principal-engineer + qa-engineer domain/fix dispatches, including the `WORK_KIND=defect`
variant). The spec, plan, and refine phases (`solutions-architect`, `tech-lead`, `scrum-master`,
`product-manager`) continue operating on the **main checkout, unchanged** — no worktree
provisioning, no cache override, no reuse flag added there.

## Error Handling

| Scenario                                                            | Behaviour                                                                                                                                        |
| ------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------ |
| `worktree-setup.sh` cannot create/reuse the worktree                | Non-zero exit → orchestrator STOPs the impl phase and reports; never dispatch a writing agent into the primary checkout as a fallback.           |
| `node_modules` provisioning fails                                   | `worktree-setup.sh` surfaces the pnpm error; only a genuinely unusable env is a STOP (see Open Question 1).                                      |
| Domain agent writes to the primary checkout instead of the worktree | **Detected** by the Step-5 primary-checkout guard (HEAD changed or working tree dirty) → fail the phase and STOP (§5). No longer prose-only.     |
| `worktree-gc.sh` fails (locked worktree, removed dir)               | Log + continue; always exit 0 — GC never fails a session (mirrors `cleanup-tmp.sh`).                                                             |
| Worktree branch not yet merged into `origin/main` at GC time        | Left in place — GC removes only worktrees whose branch is fully merged into `origin/main` (protects in-flight and concurrent-session worktrees). |
| Fix round entered in a fresh session after teardown/GC              | QA / `/review-fix` / `/sdlc:loop` re-invokes `worktree-setup.sh` idempotently → worktree re-created before any fix dispatch.                     |
| `SDLC agent reuse=enabled` but reused instance unavailable          | Fall back to fresh `Agent(...)` dispatch for that fix round.                                                                                     |

## Out of Scope

- Adding worktree isolation to the spec / plan / refine phases (`solutions-architect`, `tech-lead`,
  `scrum-master`, `product-manager`) — they stay on the main checkout, unchanged.
- Fixing the frontmatter skill re-injection bug itself (anthropics/claude-code#76337) — tracked in
  **NA-23**; this story only wires agent reuse in behind a flag once that lands.
- Broader playbook prompt-contract changes beyond worktree / cache / agent-reuse wiring — tracked
  in **NA-26**.
- Changing the harness's built-in `isolation: "worktree"` behaviour — not plugin-controllable; this
  story stops relying on it for impl dispatches rather than modifying it.
- Parallelizing domain-agent dispatch — dispatch stays strictly sequential, one writer on the story
  branch at a time (unchanged).

## Open Questions

All four questions were decided by the story owner on 2026-07-11: **defaults accepted for 1–3;
question 4 decided against its suggested default** (flag ships `enabled` from day one — see 4).

- [x] **1. node_modules provisioning for the shared worktree.** — **Decided: default accepted**
      (include `--frozen-lockfile`; re-run install after a lockfile change).
      _Default:_ run `pnpm install --prefer-offline` (optionally `--frozen-lockfile`) once
      in the fresh worktree. pnpm's content-addressed global store is shared across all worktrees, so
      the install is cheap (hardlinks from the store, no re-download) and — crucially — it builds the
      correct **per-package** `node_modules` link farms that a workspace (`apps/marketing`,
      `packages/ui`, …) needs. A root-only symlink to the primary checkout's `node_modules` is
      **rejected**: pnpm workspaces place a `node_modules` in every package, which a single root-level
      link does not provide, so the first build fails and falls back to a full install anyway (no
      saving). The real disk/time win comes from having **one** persistent worktree per story (one
      install) instead of one per dispatch — not from symlinking.
- [x] **2. GC scope — SDLC-managed only, or also stray harness `agent-*` worktrees?** — **Decided:
      default accepted.**
      _Default:_ GC removes both `sdlc-*` (this story's managed worktrees) **and** any
      `agent-*` harness leftovers, because the accumulated `agent-*` dirs are the measured disk
      problem; both are gated on the same "fully merged into `origin/main`" check, so an in-flight
      worktree (SDLC-managed or harness) is never removed.
- [x] **3. Cache sharing mechanism — `NX_CACHE_DIRECTORY` env vs `nx.json cacheDirectory`.** —
      **Decided: default accepted.**
      _Default:_ `NX_CACHE_DIRECTORY` env pointing at the primary checkout's `.nx/cache`
      absolute path — does not mutate committed `nx.json` and cannot be defeated by relative-path
      resolution in each worktree.
- [x] **4. Reuse-flag source of truth.** project-context Tooling token vs probing NA-23 state. —
      **Decided: static project-context token, but shipped as `SDLC agent reuse = enabled` from day
      one** (owner decision, 2026-07-11 — an early flip beats a forgotten manual flip after NA-23).
      No runtime Jira probe on the impl hot path. **Accepted trade-off:** until NA-23 lands, a reused
      (resumed) agent re-pays frontmatter skill injection (harness bug #76337) — wasted tokens, not a
      correctness risk; the fresh-dispatch fallback (instance unavailable) is unchanged, and NA-23's
      landing removes the re-injection cost without any further flag change.
