# Make release-mode PROJECT_KEYS discoverable — Technical Spec

**Story:** NA-60 — Make release-mode PROJECT_KEYS discoverable — additional Jira keys field is inert by default
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer (related; NA-60 has **no** Jira parent link — filed as a follow-up during NA-53 PR #117 review, not a decomposed epic child)
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-18

## Overview

NA-53 gave `/sdlc:docs release` a `PROJECT_KEYS` set (primary key + an optional "Additional Jira project keys" manifest section) so a multi-project repo's legacy-key stories are not silently dropped from release artifacts — but the additional-keys field is **inert by default**: `/sdlc:init` never surfaces it, the manifest header never advertises it, and nothing warns when the release range carries commits under an unconfigured key prefix. This story makes the field **discoverable** (the manifest header documents it, and `/sdlc:init` offers it) and — the load-bearing part — makes the silent drop **self-announcing**: when `PROJECT_KEYS` is configured, the release run warns (non-silently, at the confirm gate **and** on the no-op path) about every plausible-Jira-key-shaped token that is outside `PROJECT_KEYS` and therefore missing from the release.

## The PROJECT_KEYS state space (verify this stays coherent — the core correctness contract)

Everything below must keep these **three states distinct** — none may collapse into another. The warning is an **overlay** on NA-53's unchanged emission; it never changes what is emitted, only what is _announced_.

| State                                                      | Emission (NA-53, unchanged)                                                                   | NA-60 warning overlay                                                                                                                                                                                                                                                                             |
| ---------------------------------------------------------- | -------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **A. `PROJECT_KEYS = ∅`**                                  | **Loose fallback** — emits _every_ shape-matched token, with NA-53's existing gate risk note  | **Skipped entirely.** Nothing is out-of-scope (everything shape-matched is emitted), so there is nothing to warn about and **no no-op override**. The run proceeds exactly as NA-53 defines.                                                                                                        |
| **B. `PROJECT_KEYS ≠ ∅`, no out-of-scope shape-matches**   | Strict alternation — emits the in-scope keys                                                  | **No warning line.** Normal run.                                                                                                                                                                                                                                                                  |
| **C. `PROJECT_KEYS ≠ ∅`, some out-of-scope shape-matches** | Strict alternation — emits the in-scope keys, drops the out-of-scope ones                     | **Warning fires (non-silent):** lists genuine unknown keys individually, summarizes standards-prefix matches (never drops them), names the **enabled** release rows, and gives section-aware remediation. Fires at the gate (in-scope stories exist) **or** on the no-op path (no in-scope stories). |

The old (rejected-in-review) design keyed the warning off `IN_SCOPE` unconditionally, which in **State A** forced a suppressing "nothing to release" no-op even though the loose fallback _does_ produce a release — collapsing State A into a broken no-op. The `PROJECT_KEYS ≠ ∅` gate below is what keeps A, B, C separate.

## The confirm-gate warning (AC3 — the load-bearing decision)

The warning is a **pure function of the commit range and static config** (`PROJECT_KEYS`, `ENABLED_ROWS`, manifest section presence) — no ambient input (no `gh`, no network, no clock).

### Gated on `PROJECT_KEYS ≠ ∅` (Cluster A root fix)

The **entire** warning computation and every warning print is guarded by `PROJECT_KEYS ≠ ∅`. When `PROJECT_KEYS = ∅`:

- Emission uses NA-53's loose fallback, which emits **every** shape-matched token — so **nothing is out-of-scope**, and there is nothing for this warning to say.
- The `∅` case already carries NA-53's own gate risk note ("no project key configured — matches are unfiltered"); NA-60 adds nothing and, critically, **never routes `∅` into a suppressing no-op**. The release proceeds/emits exactly as NA-53 defines.

So the warning matches **what emission actually does**: it only exists to surface keys that emission _drops_, and emission only drops keys when `PROJECT_KEYS ≠ ∅`.

### Set computation (only when `PROJECT_KEYS ≠ ∅`)

Over the `(subject, body)` records §10 already enumerated for `RANGE`:

| Set                 | Definition                                                                                                                                                                             |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `SHAPE_MATCHES`     | every **distinct** token matching the loose shape `\b[A-Z][A-Z0-9]*-[0-9]+\b` — the superset of what any emission mode could match                                                    |
| `IN_SCOPE`          | tokens whose **prefix** (the `[A-Z][A-Z0-9]*` head — unambiguous, the char class admits no `-`) is a member of `PROJECT_KEYS`. **This is exactly the set NA-53 emits in this state.** |
| `OUT_OF_SCOPE`      | `SHAPE_MATCHES − IN_SCOPE` — every shape-matched token emission **drops**. This is what the warning is about; the warning fires iff `OUT_OF_SCOPE ≠ ∅`.                               |
| `LIKELY_KEYS`       | `OUT_OF_SCOPE − (prefix ∈ STANDARDS_TOKEN_DENYLIST)` — out-of-scope tokens **not** matching a known-standards prefix → **listed individually** in the warning                        |
| `STANDARDS_MATCHES` | `OUT_OF_SCOPE ∩ (prefix ∈ STANDARDS_TOKEN_DENYLIST)` — out-of-scope tokens matching a known-standards prefix → **summarized in one aggregated line, never dropped** (Cluster B)      |

`IN_SCOPE` is subtracted **first**, so a configured key sharing a denylisted prefix is never mis-handled — it is removed as in-scope before the denylist is ever consulted.

### The denylist demotes; it never suppresses (Cluster B root fix / AC4 answer)

**AC4 decision: adopt a documented denylist — but strictly as a _display demotion_ (individual list → aggregated summary), never as a suppression. Emission continues to rely on `PROJECT_KEYS` scoping alone.**

The rejected-in-review design used the denylist to _remove_ standards tokens from the warning entirely. That reintroduces the silent drop: a real Jira project literally keyed `RFC` (`RFC-14`) is **indistinguishable by shape** from the standard `RFC-2119`, so any full-suppression denylist either over-warns on the standard or silently drops the real key. A denylist therefore **cannot** be a suppressor.

Instead, `STANDARDS_TOKEN_DENYLIST` (a plugin-side constant in `refs/docs-pipeline.md` §10, seeded from this repo's verified false-positives: `{ UTF, SHA, AES, RFC, ISO }`, prefix-level) only decides **how** an out-of-scope token is announced:

- `LIKELY_KEYS` → listed individually (they look like real missing stories).
- `STANDARDS_MATCHES` → folded into a single aggregated, clearly-demoted line that is **still shown**:
  > `(N token(s) matched common-standards prefixes (<distinct folded prefixes, e.g. RFC, SHA>) and were not listed individually — if any names a real Jira project, add its prefix as above.)`

  The parenthesised prefix list is **the distinct set of prefixes actually folded this run** (interpolated from `STANDARDS_MATCHES`), never fixed illustrative text. This is what makes "add its prefix" checkable against what is shown — an anonymous count would be barely more actionable than the silence the demotion replaces, defeating the whole point for the `RFC-14`-is-real case.

**Invariant (must hold on every path): no shape-matched out-of-scope token is ever dropped with zero notice.** A token either appears in `PROJECT_KEYS` (emitted), is listed individually (`LIKELY_KEYS`), or is counted in the summary line (`STANDARDS_MATCHES`). The denylist changes only the _prominence_, never the _presence_, of the notice — so `RFC-14` for a real project is folded into the summary, whose interpolated prefix list shows `RFC`, and the founder — reading "if any names a real Jira project, add its prefix" against a prefix they recognise — adds it. No silence, no anonymous count.

### Two exits, distinct outcomes — never collapsed

The warning is computed at enumeration (§10, command layer) and emitted at whichever exit the run actually reaches — because a gate-only warning would miss the pure silent-drop case (a range whose only keys are out-of-scope produces an empty in-scope emission set → the "no stories merged" no-op fires **before** the gate):

1. **Manifest gate** — manifest absent → silent no-op, **no warning** (zero-setup guarantee; bounds the warning to manifest-present runs).
2. **Enabled-row gate** — `ENABLED_ROWS = ∅` → clean no-op, **no warning** (a fully-disabled release surface makes key configuration moot). `ENABLED_ROWS` is resolved here, **before** enumeration, so it is available to the warning below.
3. **Resolve range + enumerate (§10)** — if `PROJECT_KEYS ≠ ∅`, compute `OUT_OF_SCOPE` over the full range, regardless of whether `IN_SCOPE` is empty. The "no stories merged" no-op keys off NA-53's **emitted** set (`IN_SCOPE` in this state), and now **splits**:
   - emitted set `= ∅` **and** `OUT_OF_SCOPE = ∅` → the existing clean no-op notice (`no stories merged since <LAST_TAG> — nothing to release`, or the no-tags wording), exit 0.
   - emitted set `= ∅` **but** `OUT_OF_SCOPE ≠ ∅` → **the warning no-op** (the pure silent-drop case). Name the out-of-scope keys + section-aware remediation; the message must **not** claim unqualified "nothing to release". Exit 0, still no PR (the pipeline never fabricates a release from unconfigured keys — it prompts the founder to configure and re-run).
   - emitted set `≠ ∅` → proceed to phase 1; carry `OUT_OF_SCOPE` (and its partition) to the gate.
4. **Phase 1 dispatch** — unchanged.
5. **Founder-confirm gate (step 5)** — if `OUT_OF_SCOPE ≠ ∅`, print the warning **alongside the drafts**, before the founder confirms. Advisory — it never blocks.

### Remediation content is correct-by-construction (Cluster C root fix)

Every warning message (gate path and no-op path) is built from **live state**, not hardcoded strings:

- **Name the actually-enabled release rows, not "the changelog".** The remediation says the missing stories would appear in `<human-named ENABLED_ROWS>` — mapping `changelog → "the changelog"`, `release-notes → "the release notes"`, `migration-guide → "the migration guide"`, listing only those enabled this run. A repo with the changelog disabled and release-notes enabled is told the stories would appear in the release notes — so re-running after configuring the key actually produces the artifact the founder expects. (`ENABLED_ROWS` is known on both exits, per step 2 above.)
- **Handle the section-absent manifest.** The remediation checks whether `.claude/project/docs-manifest.md` already contains an `## Additional Jira project keys` heading:
  - **present** → "add the prefix(es) to the existing `## Additional Jira project keys` section".
  - **absent** (an upgraded repo that never re-ran init) → "add an `## Additional Jira project keys` section (a comma-separated key list — see `refs/docs-manifest-template.md`)". Never tell a founder to edit a section that isn't there.

### Who computes and prints it

The **command layer (`commands/docs.md`)** computes `OUT_OF_SCOPE` (step 3) and prints the warning (step 3's no-op branch, or step 5's gate) — it already runs §10's enumeration and owns the interactive gate. **`refs/docs-pipeline.md`** owns the normative definitions (§10 sets + denylist-as-demotion; §14 no-op split); `commands/docs.md` references them. **`knowledge-engineer.md` is not involved** — it never renders the gate, so it never computes or prints the warning.

## AC1 — `/sdlc:init` offers the additional-keys field (Cluster D — reconsidered approach)

**The rejected-in-review design silently appended a comment-only _stub_ into the manifest.** That fought three invariants at once: NA-53's "`/sdlc:init` never writes or touches this section" guarantee (a founder who deliberately deleted the section got it back every re-init), header/body sync (the merge path wrote a body stub but never re-derived the header, so an upgraded manifest's header still named only "Voice & format"), and safe example text (the stub's literal example risked being parsed as real keys). The root problem is that **scaffolding a body stub is a mutation** — so the fix is to stop scaffolding a stub.

Adopted approach — **discoverability comes from AC2 (the header documents the section) + AC3 (the warning actively tells the founder when and how to configure it); `/sdlc:init` at most _offers_, and never writes autonomously:**

- **Fresh-write path (`init.md` Step 4g "If absent").** After writing the row table + header comment, and only if the founder accepted the docs opt-in, **offer** an optional prompt: "Does this repo carry commits under more than one Jira project key (e.g. after a Jira rename or a repo merge)? List additional prefixes, or leave blank to skip." If the founder supplies keys → **validate** each is key-shaped (`^[A-Z][A-Z0-9]*$` — a bare project prefix, no `-<number>`, no angle-bracket placeholder such as `<PREFIX>`) and re-prompt on any that is not; then write a real `## Additional Jira project keys` section containing **exactly those validated founder-supplied keys** (no stub, no example, no placeholder — honouring the template's no-`<...>`-placeholder Fill rule). If blank → **write nothing** for the section (it stays absent and founder-owned). The header written on this path already documents the section (AC2), so header and body are consistent by construction.
- **Merge path (`init.md` Step 4g "If present").** **Do not add, modify, or stub the section, and do not re-derive the header.** This preserves NA-53's non-mutation guarantee **fully** for every existing manifest — no silent re-append of a deleted section, no header/body desync. Discoverability for an upgraded (pre-NA-60) manifest is the **AC3 warning**, which fires at release time, names the enabled rows, and handles the section-absent case — the active, correct-moment discovery path that needs no manifest rewrite.

**Amended NA-53 guarantee (state it precisely so the two template assertions agree):** `/sdlc:init` never writes the section **autonomously** and never invents its content; it **may** write the exact additional keys a founder supplies at the fresh-write offer. A blank offer, and every merge-path run, leave the section untouched. This wording goes into **both** `refs/docs-manifest-template.md`'s section prose (lines ~70–89) **and** its "Fill rules" bullet (lines ~100–102) — Cluster E:146: the two must not disagree.

Why this satisfies AC1 ("offers or scaffolds … so opting in does not require reading plugin source"): the fresh-write **offer** is the "offers" arm for new repos; the **header comment (AC2)** documents the format in every founder's own manifest; and the **AC3 warning** is the universal, init-vintage-independent discovery guarantee. None requires reading plugin source.

## AC2 — manifest header documents the section (Cluster E — preserve the guardrail verbatim)

`refs/docs-manifest-template.md`'s **header comment** is written **verbatim** into every consumer manifest, so amending the template satisfies AC2 for all **future** manifests (AC2 is a template requirement; it does not retro-patch existing manifests — that is why the merge path deliberately does not rewrite headers, and why the AC3 warning is the discovery path for old manifests).

The edit **adds** the additional-keys mention and **preserves the existing guardrail clause verbatim** (Cluster E:144 — the rejected draft dropped it). The current final sentence is:

> This file may also carry an optional "Voice & format" free-form section below the table (see refs/docs-manifest-template.md) — the row table itself remains 3-column activation data only, never generation logic, a schedule, or a runtime directive.

The amended sentence keeps every word of the guardrail and only inserts the second section:

> This file may also carry, below the table, an optional "Voice & format" free-form section and an optional "Additional Jira project keys" free-form section (both described in refs/docs-manifest-template.md) — the row table itself remains 3-column activation data only, never generation logic, a schedule, or a runtime directive.

The `## Additional Jira project keys` section prose (lines ~70–89) and the "Fill rules" bullet (lines ~100–102) are updated to the **amended guarantee** wording from [AC1](#ac1--sdlcinit-offers-the-additional-keys-field-cluster-d--reconsidered-approach) — init writes only founder-supplied keys at the fresh-write offer, never autonomously; the merge path never touches it. Both assertions must state this identically.

### Resolver robustness (comment/whitespace tolerant)

Because init no longer writes any stub or example into consumer manifests, there is **no** example-parsing hazard (Cluster E:138 is moot — the only example text lives in the template ref, never in a consumer manifest). §10's additional-keys resolution keeps one small robustness rule anyway: **ignore HTML-comment spans and whitespace when parsing the section body** — a section whose only non-whitespace content is a comment (e.g. a founder commented out their keys) resolves to the **empty** additional-keys set. This is co-located with the resolution logic in `refs/docs-pipeline.md` §10 and noted in the template section; it is defensive, not load-bearing.

## Prettier landmine (NA-62) — implementer flag

This repo's Prettier corrupts **2-space-indented fenced code blocks containing an embedded blank line under a list item** (NA-62). The AC2 header block and any warning-message fenced block are authored under list items in `commands/docs.md` / the template ref; **author such fenced blocks dedented to column 0, or without an embedded blank line under the list item.** (This spec file is prettier-ignored; the flag is for the `plugins/**` edits.) Note the AC1 rework **removed** the previously-flagged init.md stub fenced block — init now writes founder-supplied keys inline, not a scaffolded block.

## Permissions Detail

Not a multi-role RBAC surface. The actors are the **founder** (runs `/sdlc:init` and `/sdlc:docs release`, reads the warning, decides whether to configure the field) and the **automated pipeline** (computes and prints the warning; never auto-configures anything). No authority change vs NA-53: the warning is **advisory** — it never blocks the release, never mutates the manifest, and never fabricates a release artifact from an unconfigured key. `/sdlc:init` writes the additional-keys section **only** from an explicit founder-supplied value; it never invents keys.

## Open Questions

- [ ] **Standards-summary line on standards-heavy repos.** With the denylist demoting (not suppressing), a release range mentioning only standards tokens (e.g. `UTF-8`) still prints the one-line aggregated summary every release. _Suggested default (adopt):_ **accept it** — the no-silent-drop invariant is worth one terse, clearly-demoted line; it is aggregated (one line regardless of count) and does not masquerade as a missing story. Revisit only if a real consumer reports fatigue.
- [ ] **Denylist granularity — prefix-level vs full-token.** _Suggested default (adopt):_ **prefix-level** (`{UTF, SHA, AES, RFC, ISO}`) — one entry covers a whole family (`SHA-1`/`SHA-256`). Because the denylist only demotes (never suppresses), the imprecision is harmless: a real `RFC-14` is still surfaced in the summary line.
- [ ] **Denylist extensibility — plugin constant vs consumer override.** _Suggested default (adopt):_ **fixed plugin constant** in v1; a consumer override is a new config surface for an advisory, non-suppressing line. Revisit if a token family not in the seed set causes recurring noise.
- [ ] **Merge-path discoverability for upgraded repos — AC1's literal "offers or scaffolds" holds for new repos only.** For a pre-NA-60 manifest, `/sdlc:init` neither offers nor scaffolds the section (the merge path deliberately does not touch it, to preserve non-mutation), so an upgraded repo's discoverability rests **solely on the AC2 header on its next fresh manifest and the AC3 warning at release time** (made actionable by the interpolated-prefix fix above). State this plainly: AC1's literal verb is satisfied for **new** repos via the fresh-write offer; the header + warning cover the upgraded case. _Suggested default (adopt):_ **acceptable** — the warning is the correct-moment, active prompt; a re-offer on merge would either re-append into a deliberate deletion or require a decline-record mechanic for a free-form section, both heavier than the warning. Revisit if founders miss it.
