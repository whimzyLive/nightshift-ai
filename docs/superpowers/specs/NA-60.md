# Make release-mode PROJECT_KEYS discoverable — Technical Spec

**Story:** NA-60 — Make release-mode PROJECT_KEYS discoverable — additional Jira keys field is inert by default
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer (related; NA-60 has **no** Jira parent link — filed as a follow-up during NA-53 PR #117 review, not a decomposed epic child)
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-18

## Overview

NA-53 gave `/sdlc:docs release` a `PROJECT_KEYS` set (primary key + an optional "Additional Jira project keys" manifest section) so a multi-project repo's legacy-key stories are not silently dropped from the changelog — but the additional-keys field is **inert by default**: `/sdlc:init` never scaffolds it, the manifest header never advertises it, and nothing warns when the release range carries commits under an unconfigured key prefix. This story makes the field **discoverable** (init scaffolds an inert stub, the header documents it) and, crucially, makes the silent drop **self-announcing** (the release confirm gate — and the no-op path — warns when the range contains plausible-Jira-key-shaped tokens outside `PROJECT_KEYS`).

## The confirm-gate warning (AC3 — the load-bearing decision)

AC3 is the point of the story: turn the silent drop into a visible prompt, self-announcing **even for a repo that never configures the additional-keys field**. The warning is a **pure function of the commit range** — no ambient input (no `gh`, no network, no clock).

### What "plausible-Jira-key shape outside PROJECT_KEYS" means (precise)

Over the `(subject, body)` records §10 already enumerated for `RANGE`, compute four sets:

| Set                 | Definition                                                                                                                                                                                                                                          |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `SHAPE_MATCHES`     | every **distinct** token in any record's subject or body matching the **loose plausible-key shape** `\b[A-Z][A-Z0-9]*-[0-9]+\b` — i.e. §10's rejected-for-emission matcher, reused here as the detection net                                          |
| `IN_SCOPE`          | the subset of `SHAPE_MATCHES` whose **prefix** (the `[A-Z][A-Z0-9]*` head, unambiguous because the char class admits no `-`) is a member of `PROJECT_KEYS` — these are exactly the merged-story keys §10 emits                                        |
| `DENYLISTED`        | the subset of `SHAPE_MATCHES` whose prefix is a member of `STANDARDS_TOKEN_DENYLIST` (see [AC4](#the-standards-token-denylist-ac4)) — the standards tokens that drove NA-53's original narrowing                                                       |
| `WARN_KEYS`         | `SHAPE_MATCHES − IN_SCOPE − DENYLISTED` — **subtract `IN_SCOPE` first, then the denylist** (a configured key that happens to share a denylisted prefix is never suppressed, because it is removed as in-scope before the denylist is ever consulted) |

`WARN_KEYS` is the set of tokens that **look like a Jira key, are not a configured key, and are not a known standards token** — precisely the class NA-53's strict emission regex silently drops. Worked example against this repo's own vocabulary (verified): with `PROJECT_KEYS = {NA}`, a range mentioning `UTF-8`, `SHA-1`, `AES-256`, `RFC-2119`, `ISO-8601`, `NA-60`, `ET-56`, and `BANANA-53` yields `IN_SCOPE = {NA-60}`, `DENYLISTED = {UTF-8, SHA-1, AES-256, RFC-2119, ISO-8601}`, and `WARN_KEYS = {ET-56, BANANA-53}` — the legacy story surfaces, the standards noise does not, and the fabricated test key `BANANA-53` correctly warns (it genuinely is a plausible key outside `PROJECT_KEYS`).

### Two distinct outcomes — never collapsed (Standing lesson: never collapse distinct outcomes)

`WARN_KEYS = ∅` and `WARN_KEYS ≠ ∅` are **structurally distinct** and neither degrades into the other:

- `WARN_KEYS = ∅` → **no warning line** printed; the run proceeds (or no-ops) exactly as NA-53 defined.
- `WARN_KEYS ≠ ∅` → a **visible warning** is printed. Advisory only — it never blocks; the founder still proceeds through the normal gate (or the run still exits 0 on the no-op path).

### Where the warning fires — BOTH the gate path AND the no-op path (the subtlety)

The naive design — "print the warning at the founder-confirm gate" — **fails the exact case AC3 targets**. Consider a repo whose release range carries commits whose *only* keys are legacy (`ET-*`), with `PROJECT_KEYS = {NA}`:

- §10 enumeration yields an **empty** merged-story set (no `NA-*` keys).
- §14's "no stories merged → clean no-op" fires and the run **exits before reaching the founder-confirm gate**.
- A gate-only warning therefore **never prints** — the silent drop persists, defeating AC3.

So the warning is computed at **enumeration** (§10, at the command layer, where the scan already runs) and emitted at whichever exit the run actually takes. Placement in the `release` route (`commands/docs.md`), with the position of each print:

1. **Manifest gate** — manifest absent → silent no-op, **no warning** (zero-setup guarantee for a repo that opted out of docs entirely; the warning is a docs-mode concern). Bounds the warning: it fires only when a manifest is present.
2. **Enabled-row gate** — `ENABLED_ROWS = ∅` → clean no-op, **no warning** (a fully-disabled release surface makes key configuration moot — even in-scope stories would not be emitted). Bounds the warning: it fires only when there is a release surface that *would* have included the dropped stories.
3. **Resolve range + enumerate (§10)** — compute `SHAPE_MATCHES`, `IN_SCOPE`, `WARN_KEYS` here, over the full range, **regardless of whether `IN_SCOPE` is empty**:
   - `IN_SCOPE = ∅` (no in-scope stories merged) → **no-op path**, which now **splits**:
     - `WARN_KEYS = ∅` → the existing clean no-op notice (`no stories merged since <LAST_TAG> — nothing to release`, or the no-tags wording), exit 0.
     - `WARN_KEYS ≠ ∅` → **the warning no-op** — this is the pure silent-drop case, and the message must not claim "nothing to release" unqualified (it would be a lie). Exit 0, still no PR (the pipeline never fabricates a changelog from unconfigured keys — it prompts the founder to configure and re-run):
       ```
       No stories under PROJECT_KEYS merged since <LAST_TAG | the start of history>, but the
       release range carries Jira-key-shaped tokens outside PROJECT_KEYS: <sorted WARN_KEYS>.
       If these belong to a legacy or secondary Jira project whose stories should appear in the
       changelog, add the prefix(es) to the "Additional Jira project keys" section of
       .claude/project/docs-manifest.md and re-run. Otherwise nothing to release.
       ```
   - `IN_SCOPE ≠ ∅` (stories merged) → proceed to phase 1; carry `WARN_KEYS` to the gate.
4. **Phase 1 dispatch** — unchanged (drafts the enabled artifacts from `IN_SCOPE`).
5. **Founder-confirm gate (step 5)** — if `WARN_KEYS ≠ ∅`, print the warning **alongside the drafts**, before the founder confirms:
   ```
   ⚠  <N> distinct Jira-key-shaped token(s) in the release range are outside PROJECT_KEYS
      (primary + additional) and are NOT in this changelog: <sorted WARN_KEYS>.
      If they belong to a legacy or secondary Jira project whose stories should appear, add the
      prefix(es) to the "Additional Jira project keys" section of
      .claude/project/docs-manifest.md and re-run. If they are unrelated tokens, ignore this notice.
   ```

One computation, emitted at whichever of the two exits (no-op path, gate path) the run reaches. The manifest-absent and `ENABLED_ROWS`-empty no-ops never warn (by the bounds above).

### Who computes and prints it (executing-layer agreement)

The **command layer (`commands/docs.md`)** computes `WARN_KEYS` (step 3) and prints it (step 3's no-op branch, or step 5's gate) — because the command layer already runs §10's enumeration before dispatching phase 1, and because the founder-confirm gate lives at the command layer (a dispatched subagent cannot pause for interactive input, per docs-pipeline §2). The **pipeline ref (`refs/docs-pipeline.md`)** owns the *normative definition* of `WARN_KEYS` and `STANDARDS_TOKEN_DENYLIST` (§10) and the no-op split (§14); `commands/docs.md` references it. `knowledge-engineer.md` is **not** involved — it never renders the gate, so it never computes or prints the warning. This is the only "who prints it" split that keeps the two layers in agreement (see [Executing-layer propagation](#executing-layer-propagation-standing-lesson-3)).

## The standards-token denylist (AC4)

**Decision: adopt a documented denylist — but scoped to the *warning* only. Emission continues to rely on `PROJECT_KEYS` scoping alone.**

The two are inverse tests and need different mechanisms:

- **Emission** (§10, unchanged) includes only tokens whose prefix is an *exact configured key* in `PROJECT_KEYS`. Standards tokens are simply not in `PROJECT_KEYS`, so they are never emitted. `PROJECT_KEYS` scoping alone is **sufficient** here — a denylist would be redundant (and risks excluding a real key). **No denylist on emission.**
- **The AC3 warning** flags everything *outside* `PROJECT_KEYS` — which, by construction, includes every standards token. Without a denylist, the warning would fire on `UTF-8`/`SHA-1`/`AES-256`/`RFC-2119`/`ISO-8601` on essentially every release, reintroducing the exact false-positive noise that drove NA-53's original narrowing. **Warning-fatigue is not a cosmetic problem: a warning that fires on every commit is one founders learn to ignore, which silently collapses the `WARN_KEYS ≠ ∅` outcome back into the `∅` (no-signal) outcome — the very degradation Standing lesson "never collapse distinct outcomes" forbids.** So `PROJECT_KEYS` scoping alone is **insufficient** for the warning, and a denylist is required to keep it credible.

`STANDARDS_TOKEN_DENYLIST` is a **plugin-side constant** in `refs/docs-pipeline.md` §10, seeded with the prefixes verified to false-positive in this repo's own history (NA-53's finding): `{ UTF, SHA, AES, RFC, ISO }` (prefix-level — one entry covers a whole family, e.g. `SHA` catches `SHA-1`/`SHA-256`/`SHA-512`). It is applied **only** to the `WARN_KEYS` computation, **after** `IN_SCOPE` subtraction, and **never** to emission.

Documented residual risk (stated inline in §10 so a future editor does not "fix" it): the denylist is **non-exhaustive** — a novel standards token (say `BASE-64` written uppercase) produces at most **one advisory line**, which a founder reads once and ignores. And a legacy Jira project literally keyed after a denylisted prefix (a project named `ISO`) would not warn — vanishingly unlikely, and the founder can still configure it by hand. Both residuals are strictly better than the warning-fatigue alternative. **The denylist is a plugin constant, not a consumer override** in v1 (see [Open Questions](#open-questions)).

## AC1 — `/sdlc:init` scaffolds the additional-keys field

Adopted approach: **`/sdlc:init` Step 4g writes an inert, comment-only `## Additional Jira project keys` stub into the manifest** — not a prompt, and the field does **not** move to `project-context.md` (moving it would fork the §10 resolution path and the template contract, both of which read the manifest). The stub makes the field's existence discoverable to every founder who opts into docs, without an extra interactive step and without ever writing a bogus key.

- **Fresh-write branch (`init.md` Step 4g "If absent").** After writing the row table and header comment, append the stub verbatim:
  ```markdown
  ## Additional Jira project keys

  <!--
    Optional. Comma-separated legacy or secondary Jira project keys whose commits /sdlc:docs
    release should also recognise (e.g. after a Jira project rename or a repo merge). Example: ET, CER
    Leave this section empty (this comment only) to recognise just the primary key from
    project-context.md. To activate, replace this comment with a plain line like:  ET, CER
  -->
  ```
  The stub is **inert**: it carries no key outside an HTML comment, so §10's resolver (with the new comment-stripping rule below) reads it as the empty additional-keys set. This satisfies AC1 — opting in no longer requires reading plugin source, because the field is now visible in every scaffolded manifest.
- **Merge branch (`init.md` Step 4g "If present").** When merging into an existing manifest that **lacks** the `## Additional Jira project keys` section (e.g. a repo init'd before NA-60), **append the same inert stub** (silently — it is comment-only and changes no behaviour; AC1 is about discoverability, and the per-row offer mechanic is for *row-table* rows, not free-form sections). A manifest that already has the section (founder-authored or previously scaffolded) is left **verbatim** — never rewritten, never re-stubbed.

### Comment-stripping resolution rule (load-bearing for the stub)

Because the stub's `Example: ET, CER` text lives **inside** an HTML comment, §10's additional-keys resolution **must strip `<!-- … -->` comment spans before parsing** the section body for comma-separated keys. A section whose only non-whitespace content is an HTML comment resolves to the **empty** additional-keys set (fall back to the primary key alone). Without this rule the scaffold's example (`ET, CER`) would be parsed as real configured keys — a silent misconfiguration on every freshly-init'd repo. This rule is added to `refs/docs-pipeline.md` §10's Story-key extraction and noted in `refs/docs-manifest-template.md`'s "Additional Jira project keys" section.

## AC2 — manifest header documents the section

## Executing-layer propagation (Standing lesson 3)

The warning's rules must reach the layer that executes them, and the layers must agree on ownership:

- **`refs/docs-pipeline.md`** owns the normative definitions: §10 gains `SHAPE_MATCHES`/`IN_SCOPE`/`DENYLISTED`/`WARN_KEYS`, the `STANDARDS_TOKEN_DENYLIST` constant, and the comment-stripping rule; §14 gains the no-op split. Stated **once** here; the command references it.
- **`commands/docs.md`** owns the *invocation*: it computes `WARN_KEYS` at step 3 (the enumeration it already runs) and prints it at step 3's no-op branch or step 5's gate. It does not restate the §10/§14 algorithm — it points at it.
- **`knowledge-engineer.md` is deliberately untouched.** The founder-confirm gate lives at the command layer (docs-pipeline §2 — a dispatched subagent cannot pause for input), and the range enumeration runs at the command layer before phase 1 is dispatched. The agent therefore neither renders the gate nor computes `WARN_KEYS`. Pushing the warning into the agent would either duplicate the computation (two sources of truth for `WARN_KEYS`) or require the agent to emit interactive gate output it structurally cannot. The single correct owner is the command layer.

## Prettier landmine (NA-62) — implementer flag

This repo's Prettier corrupts **2-space-indented fenced code blocks that contain an embedded blank line under a list item** (NA-62). The AC1 stub is authored inside `init.md` Step 4g, which is a **list context** ("If absent" / "If present" bullets), and the stub's fenced ```markdown block contains an embedded blank line (between the `##` heading and the comment). **The implementer must author that fenced block dedented to column 0 (outside the list-item indentation), or restructure so the fenced block carries no embedded blank line under a list item** — otherwise Prettier will mangle it on the next format sweep. The same caution applies to any warning-message fenced block placed under a numbered list item in `commands/docs.md`. (This spec file is prettier-ignored, so its own fenced blocks are safe; the flag is for the `plugins/**` edits.)

## Permissions Detail

Not a multi-role RBAC surface. The actors are the **founder** (runs `/sdlc:init` and `/sdlc:docs release`, reads the warning, decides whether to configure the additional-keys field) and the **automated pipeline** (computes `WARN_KEYS`, prints it, never auto-configures anything). ## Open Questions

- [ ] **Denylist granularity — prefix-level vs full-token.** _Suggested default (adopt):_ **prefix-level** (`{ UTF, SHA, AES, RFC, ISO }`). One entry covers a whole standards family (`SHA-1`/`SHA-256`/`SHA-512`), and the residual false-negative (a legacy project literally named `ISO`) is vanishingly unlikely and manually recoverable. Full-token (`UTF-8`, `SHA-1`, …) would be more precise but needs one entry per variant and still cannot be exhaustive.
- [ ] **Denylist extensibility — plugin constant vs consumer override.** _Suggested default (adopt):_ keep it a **fixed plugin-side constant** in v1. A consumer override is a new configuration surface and a new failure mode for a warning that is already advisory; revisit only if a real consumer hits recurring false-positive noise from a token family not in the seed set.
- [ ] **Merge-path stub — silent append vs per-row offer.** _Suggested default (adopt):_ **silent append** of the inert comment-only stub when the section is absent on the merge path. It changes no behaviour (comment only), and the per-row confirm mechanic is scoped to *row-table* rows, not free-form sections. Revisit if founders find unexpected diffs surprising.
- [ ] **Warning display cap.** When `WARN_KEYS` is large, should the gate/no-op notice cap the listed keys (e.g. first 20 + "+N more")? _Suggested default (adopt):_ **list all distinct keys** — `WARN_KEYS` is de-duplicated and bounded by real commit content, so a wall of text implies a genuinely large misconfiguration worth seeing in full. Add a cap only if it proves noisy in practice.
