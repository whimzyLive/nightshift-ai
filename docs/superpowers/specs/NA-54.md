# /sdlc:docs seed mode — Technical Spec

**Story:** NA-54 — Add /sdlc:docs seed mode
**Epic:** NA-50 — Public docs generation via /sdlc:docs command + knowledge-engineer
**Feature:** docs/features/2026-07-17-public-docs-generation-sdlc-docs.md
**Date:** 2026-07-17

## Overview

Add the third `/sdlc:docs` mode — `seed <type> [topic]` — which scaffolds one narrative doc of a
manifest-activated, `seed`-triggered registry type (`concept`, `tutorial`, `integration-guide`, and
— per the [type-set decision](#seed_types--the-type-set-is-registry-derived-never-hardcoded) —
`how-to`), lets the founder author its content inline at the founder-confirm gate, writes the
confirmed page to the row's manifest `target-path`, then deterministically regenerates the doc index
+ `llms.txt` and opens a PR. It reuses the command shell, the two-phase dispatch, and the
`refs/docs-pipeline.md` contract shipped by NA-52 (`sync`) and extended by NA-53 (`release`); it adds
no registry rows and edits no registry row.

**`seed` differs from its two siblings in one structural way, and that difference drives most of
this spec:** `sync` and `release` **derive** content from repo state (a branch diff; a commit range),
so both can promise byte-identical re-runs. `seed`'s page body is **founder-authored input** — not a
function of anything in the repo. This spec therefore makes **no byte-identity claim for a seeded
page body** and says so explicitly ([Re-run semantics](#re-run-semantics--what-is-and-is-not-deterministic)),
rather than inheriting a sibling's idempotence language that would be false here.

**The second structural consequence, which drives the gate ordering:** because the founder authors
the page *at the gate*, everything that can reject a run must be checked **before** that gate. A
`seed` STOP raised after the gate does not merely waste a machine's work, as it would in `sync` or
`release` — it destroys **the founder's entire page**. Every gate, guard, and validation in this spec
is hoisted ahead of the authoring moment for that reason, and where one cannot be, it is required to
preserve the authored content rather than discard it.

## Command Contract — `/sdlc:docs seed <type> [topic]`

### Frontmatter `description` update

`docs.md`'s frontmatter `description` currently ends: "``seed``/``audit``/``distill`` are
not-yet-implemented stubs (see Epic NA-50)". This story **must** update it — it is not cosmetic, and
it is the single easiest artifact in this story to miss (NA-53 nearly did).

`refs/doc-types.md`'s `command-reference` row has `source-of-truth = command frontmatter + body` and
`generation-mode = auto`, so the very next `/sdlc:docs sync` regenerates `docs/reference/commands/`
**from this string**, un-gated (auto rows are not held at the founder gate), and publishes it into the
public docs surface **and** `llms.txt`. Leaving it stale would ship "seed is not yet implemented" as
public reference copy for a shipped mode, and `/help` would advertise the same falsehood.

Required change: drop `seed` from the stub list and state the mode it now ships — the `description`
must name **three** live modes (`sync`, `release`, `seed`) and list only `audit`/`distill` as
not-yet-implemented stubs. Mirrors the equivalent `description` update this story makes to the agent
(see [Agent Extension](#agent-extension--knowledge-engineermd)).

### Usage message update

The usage string is authored in `docs.md` and reproduced in its error table. It currently reads:

```text
usage: /sdlc:docs sync <STORY-KEY> | release <version>  (seed|audit|distill land in later stories)
```

It must become:

```text
usage: /sdlc:docs sync <STORY-KEY> | release <version> | seed <type> [topic]  (audit|distill land in later stories)
```

Called out explicitly because it is a **second** place the "seed is not implemented" claim lives, and
it is printed to the founder on every usage STOP. `seed`'s own usage STOPs (below) additionally
enumerate the valid types — the generic string above is not sufficient for a bad-`<type>` rejection.

### `SEED_TYPES` — the type set is registry-derived, never hardcoded

`SEED_TYPES` is resolved **at read time** from `refs/doc-types.md`: every registry row whose
`trigger` cell contains the `seed` token, **minus `adr`** (deferred to NA-57 — see below). Today that
resolves to:

| `type`              | Registry quadrant | `generation-mode` | Default `target-path` | Named in AC1? |
| ------------------- | ----------------- | ----------------- | --------------------- | ------------- |
| `concept`           | explanation       | draft-for-review  | `docs/concepts/`      | Yes           |
| `tutorial`          | tutorial          | manual-only       | `docs/tutorials/`     | Yes           |
| `integration-guide` | how-to            | draft-for-review  | `docs/integrations/`  | Yes           |
| `how-to`            | how-to            | draft-for-review  | `docs/how-to/`        | **No**        |

**Why derived and not a literal list in `docs.md`.** `refs/doc-types.md` is the stated single source
of truth for `trigger` (`docs-manifest-template.md`: consumers "derive all four from
`refs/doc-types.md` at read time"). A hardcoded list in the command would be a second copy of a
normative fact and would drift the moment a row's `trigger` changes — the duplication defect NA-53
hit with its manifest gate stated in four places. `seed` reads the registry; the registry decides.

**`how-to` is the delta between the registry and AC1, and it is deliberate.** AC1 says `seed` is
available "for manual-only / draft-for-review registry rows (concept, tutorial, integration-guide)".
`how-to` is _also_ a draft-for-review row and _also_ carries `seed` in its trigger (`seed,sync`) — so
the registry says it is seed-able while AC1's parenthetical does not list it. This spec reads the
parenthetical as **illustrative of the rule stated before it**, not as an exhaustive override of the
registry, and includes `how-to`. Excluding it would make `/sdlc:docs seed how-to <topic>` a usage
error in direct contradiction of the registry cell that the same command reads — a
registry-vs-command divergence with no rule to resolve it. This is the one genuine ambiguity the
story leaves, and it is recorded as an [Open Question](#open-questions) rather than resolved
silently.

`how-to` is the only type with a `sync` trigger as well as `seed`. That interaction is real and is
specified under [`source:` frontmatter](#source-frontmatter--how-to-and-integration-guide-only).

Printed, exit cleanly. **Not an error, not a STOP** — the same clean-stub discipline `docs.md`

The pointer to the still-live `/sdlc:adr` is load-bearing: NA-57 removes that command, and NA-57 is

### Argument validation

Parse `$ARGUMENTS` into `<mode>` (first token) and the mode's remaining args. `<type>` is the second
token; `<topic>` is **the remainder of the argument string** (not just the third token) — a topic is
prose and routinely contains spaces (`seed concept offline sync buckets`). Modify the existing
validation ladder in `docs.md` so that, for `seed`:

**Do this first — `seed` must be struck from the ladder's step-3 future-mode stub set. It is the
reachability blocker: every other edit in this story is dead code until it is done.** `docs.md`'s
validation step 3 currently reads: "**Recognised future-mode token** (`seed`/`audit`/`distill`) →
print `mode "<mode>" is not yet implemented (see Epic NA-50)` and exit cleanly", and it
**short-circuits before any mode contract runs**. Ship every other change in this spec — the mode
table, the frontmatter, the usage string, the pipeline section, the agent branch — and
`/sdlc:docs seed concept offline sync` still prints "not yet implemented" and exits 0. Strike `seed`
from that set exactly as NA-53 struck `release` from it, leaving only `audit`/`distill`, and carry
the same explicit note (**"`seed` is not in this set** — it routes to the seed contract below"). This
is listed ahead of the ladder because it is not one rule among the others; it is the precondition for
all of them.

1. **`seed` with a missing `<type>`** → usage STOP with the usage message **plus** the valid type
   list. A **usage STOP** (caller error) — distinct from the manifest-absent silent no-op below.
2. **`<type>` = `adr`** → the clean stub above; exit 0.
3. **`<type>` not in `SEED_TYPES` ∪ `{adr}`** → usage STOP naming what was passed and enumerating
   the valid set:

   ```text
   unknown seed type "<type>" — valid types: concept, tutorial, integration-guide, how-to
   (derived from the seed-triggered rows of refs/doc-types.md)
   ```

   The enumerated list in this message is **rendered from the resolved `SEED_TYPES`**, not typed as a
   literal — otherwise it becomes the hardcoded copy the derivation rule above exists to avoid.
4. **`<topic>`, when supplied, is validated and normalised here — at the ladder, before any gate.**
   See [Topic validation + slug normalisation](#topic-validation--slug-normalisation) below. This
   placement is the direct lesson from NA-53, where a version token that forms a branch name reached
   the gate unvalidated.
5. **`<topic>` omitted** → **not** an error (the signature is `[topic]`). The command prompts for it
   — but **after** the manifest and type-activation gates, so a repo with nothing to do is never
   prompted first. See [step 4 of the behavioural contract](#seed-type-topic--behavioural-contract).

**The topic ladder is one rule with two entry points, and both go through it.** A supplied
`<topic>` and a prompted `<topic>` are validated and slugged by the **same** rule. A ladder applied
only to the CLI argument would be trivially bypassed by omitting the argument and typing the same
hostile string at the prompt — validation that the primary input path never reaches is not
validation. This is stated once in the pipeline ref and pointed at from the command; it is not
written twice.

### Topic validation + slug normalisation

`<topic>` is founder prose. `SLUG` is what is derived from it, and `SLUG` — not the raw topic — forms
a **git branch name** (`docs/seed-<type>-<SLUG>`) and a **file path** (`<PAGE>`, constructed per
[the behavioural contract](#seed-type-topic--behavioural-contract)). It therefore gets the same
hard-validation discipline NA-53's `<version>` gets. The rule:

```text
SLUG = slugify(topic):
  1. trim, lowercase
  2. replace every run of characters outside [a-z0-9] with a single "-"
  3. collapse repeated "-", then strip leading and trailing "-"

SLUG must additionally satisfy, or the run STOPs:
  - non-empty
  - no longer than 80 characters
  - not a reserved page id (see below)
```

**Normalisation is identity-forming, not cosmetic** — the same property NA-53's leading-`v` strip
has. `Offline Sync`, `offline-sync`, and `  Offline   Sync!  ` all slugify to the **same** `SLUG`
(`offline-sync`) and therefore the same branch, the same PR, and the same page — rather than three
parallel branches and three duplicate pages for one topic. Verified against the rule above:

| Raw `<topic>`        | `SLUG`         | Note                                     |
| -------------------- | -------------- | ---------------------------------------- |
| `My Topic`           | `my-topic`     | —                                        |
| `my-topic`           | `my-topic`     | identical to the above — one identity    |
| `  Offline   Sync  ` | `offline-sync` | whitespace runs collapse                 |
| `../../oops`         | `oops`         | traversal characters cannot survive      |
| `a/b/c`              | `a-b-c`        | path separators cannot survive           |
| `1.4.lock`           | `1-4-lock`     | see the `.lock` note below               |
| `x..y`               | `x-y`          | `..` cannot survive                      |
| `@{`                 | _(empty)_      | → STOP                                   |
| `...` / `----` / `-` | _(empty)_      | → STOP                                   |
| `日本語` / `☕`      | _(empty)_      | → STOP (see the STOP-message rule below) |

**Three failure modes are reachable, and all three are real. Do not add a fourth.** The slugify
charset is `[a-z0-9-]`, which makes several rules that look like they belong here — by analogy with
NA-53's version ladder — **structurally vacuous**. A vacuous rule is not free: NA-53 shipped a
`.lock` check that rejected nothing and an unreachable trailing-`-` STOP whose message misdescribed
its input, and both survived review because nobody tested them. So, explicitly:

- **No `..` rule.** `.` is not in the charset; `x..y` → `x-y`. Unreachable.
- **No `.lock` rule.** `1.4.lock` → `1-4-lock`. Verified: `git check-ref-format --branch
docs/seed-concept-1-4-lock` → **valid**. A `SLUG` cannot end in `.lock` because it cannot contain a
  `.`. Unreachable. (NA-53 needs this rule; `seed` does not. Do not add it "for parity".)
- **No trailing `.`/`-` rule.** Step 3 strips them. `topic-` → `topic`, `topic.` → `topic`.
  Unreachable.
- **No path-separator rule.** `/` is not in the charset. Verified: `../../oops` → `oops`. The docs
  tree cannot be escaped by construction — a stronger guarantee than a rejection regex, because there
  is no input that reaches the write path unsanitised. (This guarantee governs the **slug**; it does
  **not** extend to the `target-path` the slug is joined to — see
  [`PAGE` construction](#seed-type-topic--behavioural-contract).)

The three rules that **are** reachable:

- **Empty `SLUG`** → STOP. Reachable two ways — punctuation-only input (`...`, `@{`) and
  **non-ASCII-only input** (`日本語`, `☕`), both verified empty. **The message must cover both
  causes accurately**, which is exactly what NA-53's unreachable trailing-`-` STOP got wrong by
  misdescribing its own input:

  ```text
  invalid topic "<raw>" — must contain at least one ASCII letter or digit
  (the page id and branch name are derived from it)
  ```

  Say **ASCII letter or digit**, not "letters or digits": `日本語` _is_ letters, and a message
  claiming otherwise would misdescribe the input in front of the founder. The lossiness is
  deliberate and recorded as an [Open Question](#open-questions).

- **`SLUG` longer than 80 characters** → STOP:

  ```text
  invalid topic "<raw>" — the derived page id is <n> characters; shorten the topic to 80 or fewer
  ```

  Non-vacuous and empirically grounded: a 300-character filename fails with `File name too long`
  (`ENAMETOOLONG`) on this repo's filesystem while a 200-character one succeeds — verified — so the
  failure is real, and 80 keeps `<SLUG>.md` and the branch ref comfortably inside it. **STOP rather
  than truncate**, deliberately: truncation would map two distinct long topics onto one `SLUG` →
  one branch, one page, one silently overwriting the other — the identity-collision class that
  NA-53's `V1.4.0`/`v1.4.0` normalisation exists to close. Better to reject loudly than to collide
  silently.

- **`SLUG` is a reserved page id** → STOP:

  ```text
  invalid topic "<raw>" — "<SLUG>" is a reserved page id at <target-path>: it is the section index
  this run regenerates. Choose a different topic.
  ```

  The reserved set is **`index`** (the section-index filename convention this pipeline already uses —
  NA-53's cumulative changelog is `docs/changelog/index.md`). This rule is **not** a slug-charset
  rule, which is why it is a third mode rather than a variant of the two above: `index` is a
  perfectly well-formed slug. It is a **collision** rule, and without it `seed concept index`
  destroys its own output inside a single run:

  1. `PAGE` resolves to `docs/concepts/index.md`. The page-exists gate passes — no index page exists
     yet, so the run looks like a legitimate first seed.
  2. Phase 2 writes the founder's authored page to `docs/concepts/index.md`.
  3. The [index regen](#deterministic-index--llmstxt-regen-ac4) then fires **because phase 2 just
     created a section index page at that `target-path`** — the very condition it tests for — and
     rewrites the founder's page as a generated index, **in the same commit**. The founder's work is
     destroyed by the run that created it, and the PR shows only a generated index.

  Two independent rules close this, and both are required (each covers the other's gap):

  - **This STOP**, which keeps a reserved id out of `SLUG` at the ladder, before any authoring.
  - **The index regen's "already carries a section index page" test MUST be evaluated against the
    pre-write tree state** (the branch/base head as fetched), **never the post-write working tree.**
    Otherwise any page this run creates can satisfy the regen's own precondition — a self-triggering
    write. This is the deeper fix and it holds even if the reserved set is later found incomplete;
    it is restated where the regen is specified.

**`SLUG` (normalised), not the raw topic, is used everywhere downstream** — branch name, PR title,
commit string, page id, and the page-exists gate. The raw topic is echoed **only** in the STOP
messages above, and used verbatim **only** as the human topic handed to phase 1 for drafting.

All of these are **usage STOPs raised at the ladder** — before any branch, dispatch, draft, or
founder gate. A caller passing `seed concept ../../oops` never reaches a write path (and, per the
table above, could not escape the tree even if it did).

### `seed <type> [topic]` — behavioural contract

The seed run mirrors `sync`/`release`'s two-phase dispatch and command-layer founder-confirm gate —
the shared skeleton lives in `refs/docs-pipeline.md` §2 and is **not re-derived**. What is
seed-specific, **in this order** (the order is load-bearing and stated once):

**Everything through step 5 runs before the founder authors anything.** That is the ordering
principle for this whole list: for `seed`, a post-gate STOP discards the founder's entire page, so
every rejection is hoisted ahead of the gate wherever it can be.

1. **Argument validation** — the ladder above, including a supplied `<topic>`.

2. **Manifest gate (AC5).** Shared with `sync` and `release` — defined **once** in
   `refs/docs-pipeline.md`'s [manifest-gate section](#shared-manifest-gate-rename-a-required-mechanical-edit)
   (base-ref resolution pre-check → checkout-independent `git show` → silent no-op on genuine
   absence). This spec does **not** re-derive it; `seed` is its third consumer.

   The pre-check ordering is load-bearing here for exactly the reason NA-53 documents: a bare
   `git show origin/<BASE-BRANCH>:<path>` failure is ambiguous between "the manifest genuinely does
   not exist" (the intended silent no-op) and "the **ref** does not resolve" (a fresh/shallow CI
   clone, or a renamed/deleted base branch). Without the `git fetch` + `git rev-parse --verify
origin/<BASE-BRANCH>^{commit}` pre-check running **first**, an opted-in repo with a real manifest
   gets a **silent no-op reporting success while writing nothing**. Manifest genuinely absent →
   **silent no-op**: no prompt, no branch, no dispatch, no PR, no error, **no stdout**, exit 0 (AC5).

3. **Type-activation gate (AC2).** Resolve the requested `<type>`'s row from the manifest; call it
   `SEED_ROW`. The type is **activated** iff its row is **present** in the manifest row-table **and**
   its `enabled` column is `true`.

   - **Absent is never activated.** A row **present** is enabled per its `enabled` column
     (`docs-manifest-template.md`'s Fill rules default it to `true` for every _written_ row). A row
     **absent** from the table is **not** activated. Absence has two causes and the distinction does
     not matter here: the type was offered at `/init` and declined (recorded as a
     `<!-- declined: <type> -->` comment, which never writes a disabled row), or the row was deleted
     (which a future re-init may re-offer — `/init`'s business, not `seed`'s). **Never infer a
     missing row as enabled.** This is the `seed` instance of NA-53's `ENABLED_ROWS` absent-row rule.
   - **Not activated (absent _or_ `enabled = false`) → report and make no write** (AC2), before any
     prompt, branch, dispatch, or draft:

     ```text
     doc type "<type>" is not activated in .claude/project/docs-manifest.md — nothing seeded
     (add or enable its row to seed this type)
     ```

     **Informational, not silent** — the manifest exists, so the founder opted in; a request for a
     specific, deactivated type is worth surfacing. Deliberately distinct from step 2's silent
     no-op, and the same "found nothing to do" vs "opted out" distinction `sync` and `release` draw.
     Do not dispatch `knowledge-engineer`.
   - **`target-path` comes from `SEED_ROW`**, not from the registry default. The registry's
     `target-path` is a default token that `/init` pre-fills and the founder retargets; the manifest
     row is authoritative at read time.

4. **Topic resolution (prompt if omitted).** If `<topic>` was not supplied, prompt the founder for it
   now — at the command layer, which is the only layer that can pause for input (§2's opening
   rationale) — and run the **identical** ladder from
   [Topic validation](#topic-validation--slug-normalisation) on the answer. A prompted topic that
   fails the ladder is the same usage STOP as a supplied one. Placed **after** gates 2–3 so a repo
   with no manifest is never prompted before its silent no-op, and a deactivated type is reported
   rather than prompted for.

5. **Page-exists + branch-state gate — all of it pre-gate.** This step resolves `PAGE`, decides
   first-run vs re-run, and runs **every** guard that could reject the run. All of its inputs (base
   ref, branch state on `origin`) are available now; none requires the founder's content. Running any
   of it later would mean STOPping **after** the founder has authored the page — total loss of their
   work, for a condition that was already knowable here.

   **Construct `PAGE` by normalising `target-path` first — the naive join fails OPEN.** Every
   registry `target-path` ends in `/` (`docs/concepts/`, `docs/tutorials/`, …), and a
   founder-retargeted one may or may not. So a plain
   `PAGE = <target-path> + "/" + <SLUG> + ".md"` yields `docs/concepts//offline-sync.md`. That is
   not a cosmetic blemish — it silently defeats this gate, because git and the filesystem
   **disagree** about the path:

   - `git show "origin/<BASE-BRANCH>:docs/concepts//offline-sync.md"` **does not resolve**, even
     when `docs/concepts/offline-sync.md` is published — git matches tree paths literally and there
     is no empty-named subtree. Verified against this repo.
   - The **filesystem collapses `//`** — writing `docs/concepts//offline-sync.md` opens
     `docs/concepts/offline-sync.md`. Verified.

   The gate is on the losing side of that disagreement: it concludes "exists nowhere → first run",
   phase 1 scaffolds, and phase 2 writes through the malformed path **onto the published file**,
   destroying it — the exact outcome this gate exists to prevent, in the exact case it exists for,
   with no STOP and no diff the founder would recognise as a delete. And because every registry
   `target-path` ships with a trailing slash, this is the **default** path, not an edge case. So:

   ```text
   PAGE = rstrip(SEED_ROW.target-path, "/") + "/" + SLUG + ".md"
   ```

   Normalise once, at construction. The **same** `PAGE` string is then used everywhere downstream —
   the gate, phase 1, phase 2's write, and the guard path set — never re-joined per call site (a
   second join is a second chance to reintroduce the defect).

   **Check `PAGE` checkout-independently** at the base ref — never the working tree, for the same
   reason the manifest is read that way (a stale local checkout must not decide whether a page
   exists):

   ```bash
   git show "origin/<BASE-BRANCH>:$PAGE" >/dev/null 2>&1
   ```

   The base ref is already known to resolve (step 2's pre-check ran), so a failure here is
   unambiguous — it means the path does not exist at a known-good ref. Therefore:

   - **Exists at `origin/<BASE-BRANCH>`** → **STOP**:

     ```text
     <PAGE> already exists on <BASE-BRANCH> — seed authors new pages, it does not rewrite published
     ones. Edit the page directly, or re-seed under a different topic.
     ```

     `seed` is a **create** verb: AC3 says it "scaffolds the doc and the founder authors content
     inline". Silently overwriting a published page — one that may carry months of hand edits and, for
     a `how-to`, a founder-authored `source:` list — is not scaffolding. Refuse and hand the decision
     back.
   - **Does not exist at base, but the seed branch exists on `origin`** → this is a **re-run** of an
     unmerged seed (its PR is still open). Allowed — **but the local-branch precondition and both
     re-run guards run now, here, before phase 1 dispatches.** Their full definitions live in
     [Branch / PR / Control Flow](#branch--pr--control-flow); this step is where they are
     **evaluated**.
   - **Exists nowhere** → first run. The **local-branch precondition still runs** — it applies on
     every path (a local branch of the same name with unpushed commits says nothing about `origin`).

6. **Phase 1 dispatch — `knowledge-engineer` scaffolds & drafts, writes nothing.** Pass it `<type>`,
   `SEED_ROW` (its `target-path`), `SLUG`, the **raw** topic text, the normalised `PAGE`, and
   `origin/<BASE-BRANCH>`. Phase 1:
   - loads `writing-docs` **unconditionally** (see
     [Agent Extension](#agent-extension--knowledge-engineermd)) and selects the quadrant template for
     `<type>`'s registry `quadrant` (see [Seeded page artifacts](#seeded-page-artifacts));
   - scaffolds the page from that template, filling what the topic and (for corpus-backed types) the
     learnings corpus support — **never inventing facts to fill a section**; `writing-docs`'s "No
     TBDs in published docs" rule applies, so an unsupported section is left for the founder to
     author at the gate rather than padded with plausible prose;
   - emits the required [page frontmatter](#page-frontmatter-required--llmstxt-reads-it);
   - returns the scaffold to the command layer. **Nothing is written to disk in phase 1.**

7. **Founder-confirm gate (command layer, AC3) — this is where the founder authors.** Present the
   scaffold. The founder may **accept, edit, author over, or reject** it. This is the mode's whole
   point: AC3 is "the founder authors content inline through the founder-confirm gate", so unlike
   `sync`/`release` — where the gate mostly confirms machine-derived content — here the gate is the
   **authoring surface**, and heavy editing is the expected path, not the exception. `tutorial` is
   `manual-only` in the registry, which this spec reads as its strongest form: the scaffold is a
   skeleton and the prose is the founder's. A dispatched subagent cannot pause for interactive input
   — the gate lives at the command layer between the two dispatches, exactly as `sync`, `release`,
   and `/sdlc:adr` do.

   - **Validate the confirmed content here, at the gate, while it still exists** — specifically the
     required `title`/`description` frontmatter (see
     [Page frontmatter](#page-frontmatter-required--llmstxt-reads-it)). The gate is the **last**
     layer that holds the founder's content in an editable, interactive context; phase 2 is a fresh
     dispatch that receives it as an opaque payload. A content check deferred to phase 2 can only
     STOP — it cannot ask the founder to fix the missing line — so it converts a one-line correction
     into the loss of a whole authored page.
   - **Founder rejects / does not confirm** → write nothing, no branch, no PR, no phase-2 dispatch;
     report and exit cleanly. Adopted from `commands/adr.md`'s identical row. There is nothing else
     to write: unlike `release`, a seed run has **no** deterministic half that could still be worth
     committing — the index/`llms.txt` regen (step 8) is a _consequence_ of the page write, so with
     no page there is nothing for it to reindex.
   - **`gh` and the learnings corpus MAY enrich what is displayed here; neither may feed a written
     byte except through the founder's confirmation.** See
     [Content purity](#content-purity--the-seed-variant-of-the-gh-boundary).

8. **Phase 2 dispatch — `knowledge-engineer` writes confirmed, fresh dispatch.** The command hands
   phase 2 `<type>`, `SEED_ROW`, `SLUG`, the normalised `PAGE`, and the founder-confirmed page
   content **verbatim** (inline or via `scripts/tmp-dir.sh` temp files by path — **never re-derived**;
   phase 2 is a fresh dispatch with no visibility into phase 1 or the gate). Phase 2:
   - checks out the seed branch per [Branch / PR / Control Flow](#branch--pr--control-flow) (fresh
     from `origin/<BASE-BRANCH>` head on a first run; the **existing** branch **at its remote head**
     on a re-run). The precondition and both guards **already passed at step 5** — phase 2 does not
     re-derive them. It **may** re-verify them as a narrow TOCTOU backstop (the branch could have
     moved while the founder authored), and if one fires there it **MUST preserve the confirmed
     content** — write it to the session temp dir and surface the path in the STOP — never discard
     it. A guard that destroys the founder's page while protecting someone else's commit has traded
     one loss for another;
   - writes the confirmed content to `PAGE` — and **only** to `PAGE`. `seed` writes exactly one
     page, for exactly one row: `SEED_ROW`. It never touches another row's pages, which is the `seed`
     instance of §3's "never touches an unaffected row's pages" invariant;
   - **deterministically regenerates the doc index + `llms.txt` (AC4)** — see
     [Deterministic index + llms.txt regen](#deterministic-index--llmstxt-regen-ac4), including its
     **independent** `llms-txt` row check and its **pre-write** index-existence test;
   - **only if `git status --porcelain` on the written paths is non-empty** — commits via
     `conventional-commit` (with the `Seed-Generated:` trailer), pushes, and opens or updates the
     seed PR (AC4).

### Seeded page artifacts

Each type maps onto an existing `writing-docs` quadrant template via its **registry `quadrant`** —
the mapping is a lookup, not a new table to maintain:

| `type`              | Registry quadrant | `writing-docs` template | Default `target-path` | File convention | `source:` key                   |
| ------------------- | ----------------- | ----------------------- | --------------------- | --------------- | ------------------------------- |
| `concept`           | explanation       | Explanation             | `docs/concepts/`      | `<SLUG>.md`     | N/A (not a how-to page)         |
| `tutorial`          | tutorial          | Tutorial                | `docs/tutorials/`     | `<SLUG>.md`     | N/A (not a how-to page)         |
| `integration-guide` | how-to            | How-to                  | `docs/integrations/`  | `<SLUG>.md`     | Offered at the gate — see below |
| `how-to`            | how-to            | How-to                  | `docs/how-to/`        | `<SLUG>.md`     | Offered at the gate — see below |

One page per seed run, always a **new** page (per the
[page-exists gate](#seed-type-topic--behavioural-contract)). The `target-path` is `SEED_ROW`'s, not
the registry default, and is **trailing-slash-normalised before the join** (same section).

### Page frontmatter (required — `llms.txt` reads it)

Every seeded page **MUST** carry:

```yaml
---
title: <human title for the page>
description: <one line, used verbatim as the llms.txt description>
---
```

**This is a real gap in the templates, not a restatement of an existing rule.** Verified by
inspection of `writing-docs/SKILL.md`: the **How-to guide template** is the only one that emits
frontmatter (`title` + the `source:` glob list), the **Tutorial**, **Reference**, and **Explanation**
templates emit **no frontmatter block at all**, and **none of the four** emits `description:`. So
three of `seed`'s four types would, template-faithfully, produce a page with no frontmatter
whatsoever.

The consequence is concrete and identical to the one NA-53 documents for the changelog preamble:
§8's `llms.txt` regen derives every entry from the generated page's frontmatter (`title — one-line
description — relative link`). A seeded page with no frontmatter either **drops out of `llms.txt`
silently** — the page is published but invisible to the index that exists to expose it, the exact
failure `seed` is supposed to prevent — or makes the regen **fail on a page it expects to carry
frontmatter**.

**Where the check lives is itself load-bearing:** phase 1 emits the frontmatter as part of the
scaffold, and **the command layer validates it at the founder-confirm gate**, where the content still
exists and the founder can fix a missing line in situ. It is **not** deferred to phase 2. Phase 2 is
a fresh dispatch holding an opaque payload: a STOP there cannot prompt for a correction, so a founder
who blanked `description:` while authoring four hundred lines would lose all of it to a one-line
defect — reintroducing, one step later, exactly the "STOP after the founder has authored" failure
this rule exists to prevent. Phase 2 MAY keep a defensive assertion, but it must preserve the content
(temp dir + surfaced path) rather than discard it.

Fixing the templates themselves is a wider, cross-mode change: this story states the rule in the
pipeline ref, as NA-53 did for release pages (§12). But that is now the **second** ref section
carrying the same rule, and the [Open Question](#open-questions) below no longer treats the template
fix as merely an option — it recommends it as a follow-up, because
[the `llms.txt` regen's exposure to pre-existing frontmatter-less pages](#deterministic-index--llmstxt-regen-ac4)
is the same defect surfacing a third time.

### `source:` frontmatter — `how-to` and `integration-guide` only

Both map onto the how-to quadrant, whose template emits a `source:` glob list. `refs/docs-pipeline.md`
§5 is normative and unchanged: a how-to page **with** `source:` is auto-refresh-drafted by `sync` when
`CHANGED_FILES` intersects its globs; a page **without** it is **never** auto-refreshed — "the
deliberate opt-in boundary — authoring/opting a page in is a founder action (add `source:`), not
something `sync` infers."

`seed` **is** that founder action, and it is the natural moment to make the choice. So:

- The scaffold presents the `source:` list (with the template's explanatory comment) at the gate, and
  the founder supplies globs or removes the key.
- **Founder supplies globs** → the page is written with `source:` and future `sync` runs will draft
  refreshes of it. This is the seed → sync handoff that makes `how-to`'s `seed,sync` trigger
  coherent.
- **Founder omits it** → the key is **omitted entirely** (not written empty). Per §5, the page is
  simply never auto-refreshed. Not an error, and not something `seed` may infer a value for —
  inferring globs would opt the founder into automated refreshes of a page they hand-authored.
- `concept` and `tutorial` pages **never** carry `source:` — it is a how-to-quadrant key, and §5's
  match semantics apply only to how-to rows (§3's table). Do not emit it for them.

### Content purity — the `seed` variant of the `gh` boundary

NA-53 established the hard rule that **content written to a file must be a pure function of repo
state** — `gh` enrichment feeding written content made output depend on ambient network state and
broke idempotence. That rule's _rationale_ does not transfer to `seed`, so its _form_ must not be
copied blindly. The `seed` rule is:

> **The written page is exactly what the founder confirmed at the gate, byte for byte — never
> re-derived, re-enriched, or re-drafted at write time.**

The founder-confirm gate **is** the purity boundary.

**Scope this rule to content derivation — it is not a prohibition on phase 2 touching the network.**
Phase 2 consults no corpus, no `gh`, and no network **for any byte of the page**: it receives the
confirmed content and writes it unchanged. It self-evidently **does** perform the mechanical git/`gh`
operations the write itself requires — `git fetch origin`, `git push`, `gh pr create`, and the PR
number a guard STOP must quote — because raising the PR is AC4. An absolute reading ("phase 2
consults no network") would forbid those and silently fail the very AC the phase exists to satisfy;
that reading is available precisely because
[Required addition #10](#required-additions) lands this rule **verbatim** in the ref as the
implementer's normative source, so the wording there must carry this scope, not just this spec.

This scoping is why `seed` can safely let phase 1 consult ambient sources that `release` must not:

- `refs/doc-types.md` gives `concept` the `source-of-truth` **`learnings corpus`**, and `how-to` /
  `integration-guide` **`founder-authored draft + learnings corpus`** — so phase 1 **may** query
  claude-mem for corpus material to enrich the scaffold for those three types. (`tutorial`'s
  `source-of-truth` is **`founder-authored`** only — no corpus consultation for it.)
- `gh` may likewise enrich what is **displayed**.

Both are safe **because** `seed` makes no byte-identity claim (see
[Re-run semantics](#re-run-semantics--what-is-and-is-not-deterministic)) and because everything they
touch is a _draft a human reads and rewrites_ before any byte is written. A corpus hit that differs
between runs changes what the founder is _offered_, never what is _written_ behind their back. Were
`seed` ever to gain a "re-derive on re-run" path, this reasoning would have to be re-derived with it
— it is contingent on the gate, not on the sources being trustworthy.

### claude-mem availability — never a halt for `seed`

If the claude-mem MCP tools are unavailable, phase 1 **proceeds without corpus enrichment** and says
so at the gate. It does **not** halt.

This is a deliberate divergence from `distill`, and `commands/adr.md` already draws exactly this line
in its error table: _"claude-mem tools absent in distill mode → halt distill with a clear 'claude-mem
tools unavailable' message. **Seed mode is unaffected.**"_ The reason is structural, not stylistic:
`distill`'s entire input **is** the corpus, so without it there is nothing to distill; `seed`'s input
is the **founder**, and the corpus is enrichment. Halting would block a founder from authoring a page
they can write unaided.

### Deterministic index + llms.txt regen (AC4)

After the confirmed page write, phase 2 deterministically regenerates the doc index and `llms.txt` —
the un-gated half of the run (the same "auto rows are un-gated" discipline `sync` and `release`
apply):

- **`llms.txt`** — **only if the `llms-txt` row is present and enabled in the manifest, checked
  independently of `SEED_ROW`.** `llms-txt` is a `sync`-triggered row; it is not the row `seed` was
  invoked for, so its state is **never** inferred from `SEED_ROW`'s activation — a founder may
  perfectly well have `concept` enabled and `llms-txt` declined. Absence is never inferred as enabled
  (the same discipline as the type-activation gate): a founder who declined `llms-txt` at `/init`
  (row absent) must never have it written or overwritten by a `seed` run. If **disabled or absent**,
  phase 2 does not write or touch `llms.txt` at all this run — any existing file is left exactly
  as-is. When it **is** enabled: reuse §8's algorithm verbatim — index-only, grouping the generated
  pages of every enabled `public: yes` row by Diátaxis quadrant, each entry a `title — one-line
description — relative link` derived from page frontmatter. The newly seeded page appears in it.
  Idempotent; committed only if changed.

  This rule must reach `knowledge-engineer.md`, which performs the write — see
  [Agent Extension](#agent-extension--knowledge-engineermd). NA-53's equivalent rule landed in the ref,
  the spec, and the command but **not** in the agent, re-opening the defect at the only layer that
  executes it.

- **Frontmatter-less pages the regen encounters are SKIPPED, never fatal, and never fabricated.**
  This story mandates frontmatter on the **page it writes**, but §8's algorithm reads frontmatter
  from **every** enabled `public: yes` page in the tree — and the templates emit none, so pages
  written before this rule existed (a hand-authored tutorial, a `sync`-drafted how-to) may well lack
  it. That exposure is pre-existing and shared with `sync`/`release`; `seed` merely walks into it. The
  rule:
  - A page missing `title`/`description` is **omitted from `llms.txt`** and its path **surfaced in
    the phase-2 output and the PR body**. Loud, not silent.
  - It is **never a STOP.** A STOP mid-phase-2 would tear the write — the founder's page committed,
    the index not — and would let an unrelated page's defect block a valid seed the founder just
    authored.
  - Its `title`/`description` are **never** inferred from the body or filename. That would fabricate
    published index copy, which is the fabrication boundary this pipeline holds everywhere else.

  This is the same underlying defect as
  [Page frontmatter](#page-frontmatter-required--llmstxt-reads-it), now surfacing a **third** time —
  which is the substance of the Open Question recommending the template fix.

- **The doc index** — if the consumer's docs tree already carries a section index page at
  `SEED_ROW`'s `target-path`, regenerate it deterministically (**upsert** the page's entry — add if
  absent, rewrite in place if present; never append a duplicate). If none exists, `llms.txt` is the
  sole index and **no separate section index is created**. No new index file type is invented.
  Identical to NA-53's resolved default for the same question.

  **The "already carries a section index" test is evaluated against the PRE-write tree state** — the
  branch/base head as fetched — **never the post-write working tree.** Otherwise a page this run just
  wrote can satisfy the regen's own precondition and be rewritten as a generated index inside the
  same commit: exactly the `seed <type> index` self-destruction the
  [reserved-page-id STOP](#topic-validation--slug-normalisation) closes from the other side. Both
  rules are required; neither is redundant.

- **All four seed types are `public: yes`** in the registry, so a seeded page always belongs in
  `llms.txt` when that row is enabled. (`adr` is the registry's only `public: no` row — and it is
  out of scope here, routed to NA-57. NA-57 must not let an ADR reach `llms.txt`; that is its AC3,
  not this story's rule.)

### Re-run semantics — what is and is not deterministic

**`seed` makes no byte-identity claim for a seeded page body, and this spec will not imply one.**
NA-53's idempotence contract ("same range + same confirmed content → same bytes") is a statement
about a **generator** whose input is repo state. `seed`'s page body has no such input: it is founder
prose typed at a gate. Re-running `seed concept offline-sync` re-prompts and re-authors; asserting
byte-identity would be false, and NA-53's review caught precisely that class of overclaim twice.

What **is** true, stated precisely:

| Property                                      | Holds?  | Scope / why                                                                                                                                                                                        |
| --------------------------------------------- | ------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Page **identity** is stable across re-runs    | **Yes** | `SLUG` normalisation — one branch, one PR, one page id per topic, regardless of how the topic was typed                                                                                             |
| Page **body** is byte-identical across re-runs | **No**  | It is founder-authored input, re-authored at each run's gate. No claim is made                                                                                                                      |
| **Write-time fidelity**                       | **Yes** | Phase 2 writes the confirmed content **verbatim** — the bytes written are the bytes the founder saw. This, not byte-identity, is `seed`'s content guarantee                                         |
| **`llms.txt` / index regen** is deterministic | **Yes** | Pure function of the enabled `public: yes` rows' page frontmatter (§8). Re-running it over an unchanged page set yields byte-identical output; committed only if changed                            |
| **Commit/PR only on actual content change**   | **Yes** | Phase 2 commits only if `git status --porcelain` on the written paths is non-empty. A re-run where the founder confirms an unchanged page and the index is unchanged → no commit, no PR, clean exit |
| Branch commits are **preserved** across re-runs | **Yes** | Never reset, never force-pushed — see [Branch / PR / Control Flow](#branch--pr--control-flow)                                                                                                       |

### Where edits live, and what a re-run does to them

Seed content can be edited in exactly two places, and a re-run treats them **differently**. This
section is normative; the guards in [Branch / PR / Control Flow](#branch--pr--control-flow) implement
it.

| Edit made…                                                                                 | Committed by                                                  | A re-run…                                                                                                                                                                                                                                                                                                                                  |
| ------------------------------------------------------------------------------------------ | ------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **At the founder-confirm gate** (the founder authors/edits the scaffold before confirming) | phase 2, carrying the `Seed-Generated: <type>/<SLUG>` trailer | **Re-presents a fresh scaffold.** Phase 1 re-scaffolds from the template + topic every run — a prior run's authored prose is **not** carried forward into the new scaffold. Not silent (the founder sees exactly what will be written before confirming), but **not preserved either**. See the OQ on seeding a re-run from the branch page. |
| **On the branch, outside the pipeline** (a `/sdlc:loop` review-fix commit, a direct push)  | anything **without** the trailer                              | **STOPs — at step 5, before the founder authors.** The [re-run content guard](#branch--pr--control-flow) refuses to overwrite it and hands the decision back to the founder.                                                                                                                                                                |

The distinction is the **trailer, not the commit subject**. Keying on the subject would be wrong for
exactly the reason NA-53 documents: phase 2 commits founder gate edits under the normal
`docs(docs): seed <type> <SLUG>` subject, so a subject test classifies them as pipeline-authored and
the guard never fires — correct for gate edits (re-scaffolded by design), but it must not be mistaken
for _preserving_ them.

For `seed`, the gate-edit row bites harder than it does for `release`: a release gate edit is a tweak
to machine-aggregated content, whereas a seed gate edit **is the entire page**. Re-running a seed
whose PR is open therefore means re-authoring. That is a real UX cost, accepted for v1, and it is the
subject of the most consequential [Open Question](#open-questions) in this spec.

   (registry is the single source of truth for `trigger`; a literal list drifts).

6. **Type-activation gate** — present + `enabled = true`; absent-is-never-activated with the two

8. **Seeded page artifact set** — the type → registry-quadrant → `writing-docs` template →

9. **Page frontmatter** — `title` + `description` required on every seeded page; **the template gap

    **byte of the page**, but it does perform the mechanical `git fetch` / `git push` /

11. **claude-mem availability** — absent tools → proceed without enrichment, note it at the gate,

12. **Seed branch / PR naming + control flow** — see

13. **No-op / change-gate semantics for seed** — the manifest gate's base-ref pre-check ordered

## Agent Extension — `knowledge-engineer.md`

The agent already carries three dispatch branches (ADR, docs-sync, release). This story adds a
**fourth**: `seed`. Every rule below must land **in the agent file** — it is the layer that performs
the write, and NA-53's `llms.txt` enable-gate defect re-opened precisely because the rule reached the
ref, the spec, and the command but not the agent.

- **Frontmatter `description`** — extend to state it also runs the `/sdlc:docs seed` pipeline
  (scaffolding a manifest-activated narrative doc type for inline founder authoring, then writing the
  confirmed page and regenerating the doc index + `llms.txt`). Note: the agent's `description` is
  itself `agent-reference` source-of-truth (`auto` row) — a stale one publishes into the public docs
  surface exactly as `docs.md`'s does.
- **Role & scope** — add: in a `seed` dispatch it scaffolds one page of the requested activated type
  (phase 1) and, on founder-confirmed content, writes that page and deterministically regenerates the
  doc index + `llms.txt` (phase 2), writing under the manifest's resolved `target-path` (and, when
  authoring plugin changes in this SDLC repo, only within the `ai-enablement-engineer`-owned surface —
  the existing Active-guard scope note already covers this). It writes **exactly one page, for exactly
  one row**, and never a page for a row it was not invoked for.
- **Pipeline pointer** — point the seed dispatch at `refs/docs-pipeline.md` §§15–19 as the single
  source; do not re-inline.
- **Narrow the always-on preamble's byte-identity gate — it currently binds `seed` to a claim this
  spec refutes.** The Required-skills preamble introduces `verification-before-completion` for **all**
  dispatch types as the skill that "governs the change-gate / idempotence check: confirm a
  no-source-change re-run yields **byte-identical output** before claiming completion". That is a
  sound gate for the three existing dispatches, all of which derive their content. Applied to `seed`
  it is unsatisfiable by construction: a seeded page body is founder-authored, so a no-source-change
  re-run **does not** yield byte-identical output, and the spec says so deliberately (see
  [Re-run semantics](#re-run-semantics--what-is-and-is-not-deterministic)). Left unnarrowed, the agent
  either fails its own completion checklist on every seed run, or an implementer "reconciles" the
  contradiction by asserting the idempotence this spec spent a section refuting — the propagation
  failure NA-53 hit, in its most damaging form (the executing layer contradicting the contract).
  Required: scope the preamble's parenthetical per dispatch, or add an explicit seed carve-out —
  **for a seed dispatch the byte-identity gate applies to the deterministic index/`llms.txt` regen
  only; the page body's gate is write-time fidelity** (the bytes written are the bytes the founder
  confirmed). Keep `verification-before-completion` loaded — only the claim it verifies changes.
- **claude-mem tool-use note — the existing note is now wrong by omission.** The agent currently
  states the claude-mem MCP tools are used by ADR distill and are "**not** used by `release`". A seed
  dispatch **does** use them, for the three corpus-backed types (`concept`, `how-to`,
  `integration-guide` — per their registry `source-of-truth` cells), and **not** for `tutorial`
  (`founder-authored`). Add: their **absence never halts a seed dispatch** — proceed without
  enrichment and surface that at the gate (contrast distill, which halts). No new tool grants are
  needed: both tools are already in the `tools:` allowlist, alongside
  `Read`/`Write`/`Edit`/`Bash`/`Skill`.
- **Required-skills branch — state the `seed` rule explicitly; do NOT inherit the docs-sync gate.**
  The docs-sync branch loads `writing-docs` **conditionally** — "only once you've resolved that at
  least one `how-to` row is affected this run". That condition is **sync-scoped**: it keys on
  `sync`-triggered how-to row _affectedness_, which a seed run never computes and which resolves to
  **zero rows** on a seed dispatch. Inheriting it would leave a seed phase 1 scaffolding a tutorial or
  concept with **no Diátaxis template loaded** — the same trap NA-53 had to close for `release`.
  Required wording: a **seed** dispatch loads `writing-docs` **unconditionally in phase 1** (a seed
  phase 1 always scaffolds exactly one page, and all four seed types are quadrant-templated), and —
  like `sync` and `release` phase 2 — never needs it in phase 2, which writes only what the founder
  already confirmed. Add as a **fourth** named branch (ADR / docs-sync / release / **seed**), not as a
  clause on an existing bullet.
- **Branch, memory, commit, return** — add a seed sub-section as a **pointer** to
  `refs/docs-pipeline.md`'s seed branch/PR section (not a restatement), naming only the

  `git status --porcelain` on the written paths is non-empty**, and still append the memory learning
  if it is empty.
- **Completion checklist** — add a seed branch. Its gate is **not** an aggregation-idempotence check
  and **not** a page-body byte-identity check (see the preamble narrowing above): confirm the written
  page is **byte-identical to the founder-confirmed content** (write-time fidelity — never re-drafted
  at write time), that it carries `title` + `description` frontmatter, that it landed at the
  manifest-resolved, **trailing-slash-normalised** `target-path`, that `source:` was emitted only for

## Branch / PR / Control Flow

**All of the precondition and guard rows below are evaluated at
[step 5](#seed-type-topic--behavioural-contract) — before phase 1 dispatches and before the founder
authors anything.** They are defined here and evaluated there. Every input they need (base ref,
branch state on `origin`, the trailer on each commit) is available at step 5; none needs the
founder's content. Deferring any of them to phase 2 would mean a STOP that discards the founder's
authored page to protect a condition that was knowable before they typed a word.

| Item                      | Value                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                             |
| ------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Branch                    | `docs/seed-<type>-<SLUG>` (the **normalised** slug — `My Topic` and `my-topic` resolve to the same branch, never two), cut from the **base branch head** (`origin/<BASE-BRANCH>`) — **not** a story branch. A seed authors a new page against current base; there is no story diff to read. Mirrors `commands/adr.md`'s `docs/adr-<slug>` shape, with `<type>` included because two types may legitimately seed the same topic (`docs/seed-concept-offline-sync` and `docs/seed-tutorial-offline-sync` are distinct pages and must be distinct branches).                                                                                                                                                                                             |
| Commit                    | `docs(docs): seed <type> <SLUG>` (via `conventional-commit`), carrying the trailer `Seed-Generated: <type>/<SLUG>`. **The trailer is load-bearing**, not decoration: it is the only reliable marker of "this pipeline wrote this commit", and both guards below key on it. Subject-matching is not a substitute (see [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them)).                                                                                                                                                                                                                                                                                                                                                          |
| PR title                  | `docs(docs): seed <type> <SLUG>`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| PR base                   | `<BASE-BRANCH>` from project-context (never assume `main`)                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| Local-branch precondition | Applies on **every** path, first-run and re-run alike, evaluated at step 5. If a **local** `docs/seed-<type>-<SLUG>` exists holding commits not reachable from `origin/docs/seed-<type>-<SLUG>` (or, when no such remote exists, any commits at all beyond `origin/<BASE-BRANCH>`) → **STOP**: `local branch docs/seed-<type>-<SLUG> has unpushed commits; push, drop, or rename it, then re-run.` Never `checkout -B` over it. Hoisted out of the re-run rows deliberately: the first-run path is defined by the branch being absent **on `origin`**, which says nothing about a local branch of the same name.                                                                                                                                       |
| First-run PR              | No remote branch (`git rev-parse --verify origin/docs/seed-<type>-<SLUG>` fails) → after the precondition passes, create the branch from `origin/<BASE-BRANCH>` head, write, commit, push, open the PR (`gh pr create`) against `<BASE-BRANCH>`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| Re-run behaviour          | **Open or update; the branch's commits are never rewritten.** If the branch exists on `origin` → `git fetch origin`, run the precondition + **both guards at step 5**, then (in phase 2) check it out **at its remote head** (`git checkout -B docs/seed-<type>-<SLUG> origin/docs/seed-<type>-<SLUG>`) — the single normative flow. Write the confirmed content **on top** as a new commit and `git push` (plain fast-forward). A re-run where nothing changed produces **no commit, no push, no duplicate PR**.                                                                                                                                                                                                                                     |
| Re-run history guard      | The **one forbidden path**: no re-run may reset the branch onto regenerated state, force-push it (`--force` / `--force-with-lease`), or discard any commit reachable from `origin/docs/seed-<type>-<SLUG>`. **This is a deliberate divergence from §7 (`sync`) — do not harmonise it back.** §7 resets and force-pushes safely _because_ a sync branch's content is fully derived. A seed branch is the opposite extreme: its content is founder-authored **and** the control-flow tail drives `/loop /sdlc:loop` against it, so it carries review-fix commits. A reset-to-fresh + `--force-with-lease` would _succeed_ (the local ref was just fetched) and silently revert the PR to unreviewed content. Checking out at the remote head does none of these. |
| Re-run content guard      | Preserving commits protects **history**; this guard protects **content**. Evaluated at step 5. Find commits on the branch (relative to `origin/<BASE-BRANCH>`) that touch **any path phase 2 writes this run** and **lack the `Seed-Generated: <type>/<SLUG>` trailer** — i.e. out-of-pipeline edits to generated pages. That path set is **not** just `PAGE`: phase 2 also rewrites `llms.txt` (when its row is enabled) and any regenerated section index page — a `/sdlc:loop` review-fix or founder edit to either is exactly the out-of-pipeline content this guard protects, and scoping the scan to `PAGE` alone would silently destroy it with no STOP. If any qualifying path is found → **STOP**: `branch docs/seed-<type>-<SLUG> carries edits to generated pages (<paths>) that this pipeline did not write; re-running would overwrite them — merge or close PR #<n>, or drop those edits, then re-run.` Trailer-bearing commits, and commits touching **other** paths, do not trip it — proceed. |
| Control-flow tail         | Mirror `sync` / `release` / `commands/adr.md`: after the PR is raised, drive the review loop to convergence via `/loop /sdlc:loop <PR_URL>` (falling back to `ScheduleWakeup` if the harness cannot nest `/loop`). If the run ended **before** any PR (manifest-absent silent no-op, deactivated-type report, `adr` stub, founder-declined-at-gate, a usage STOP, a page-exists STOP, or a precondition/guard STOP), release directly via `${CLAUDE_PLUGIN_ROOT}/scripts/session-complete.sh` — only the manifest-absent path is silent.                                                                                                                                                                                                               |

## Permissions Detail

Not a multi-role RBAC surface. The single actor is the **founder** running `/sdlc:docs seed`. The
authority boundaries, in order of how much they matter for this mode:

1. **The founder-confirm gate is the authoring surface, not just an approval.** Nothing narrative is
   written un-confirmed; for `seed`, "confirmed" usually means "written by the founder at the gate".
   The deterministic doc index + `llms.txt` regen is the only content written without confirmation —
   and only as a consequence of a page the founder did confirm. A dispatched subagent never runs the
   gate itself; the command layer owns it, exactly as `commands/docs.md` (`sync`/`release`) and
   `commands/adr.md` do.
2. **The founder's manifest decides what may be seeded.** `seed` never creates a page for a type the
   founder did not activate (type-activation gate) — and never infers an absent row as activated.
3. **`seed` never rewrites published work.** A page already on the base branch is a STOP, not an
   overwrite (page-exists gate) — which is why that gate's path construction is
   [normalised rather than naively joined](#seed-type-topic--behavioural-contract): a gate that fails
   open is not a boundary at all.
4. **`seed` never rewrites the commits on a branch under review** (re-run history guard), never
   discards unpushed local work (local-branch precondition), and never overwrites an out-of-pipeline
   edit to a generated page — it STOPs and hands the decision back (re-run content guard).
5. **`seed` never discards the founder's own authored content to satisfy a guard.** Every rejection
   runs before the authoring gate; the one check that cannot (a TOCTOU re-check in phase 2) must
   preserve the content and surface where it was kept.

Gate-authored content is the one thing a re-run re-scaffolds rather than preserves, and the founder
sees exactly that at the gate before anything is written — see
[Where edits live](#where-edits-live-and-what-a-re-run-does-to-them).

| claude-mem tools unavailable                                                                       | **Never a halt for `seed`** — phase 1 proceeds without corpus enrichment and says so at the gate. (Contrast `distill`, which halts: the corpus is `seed`'s enrichment but `distill`'s entire input. Mirrors `commands/adr.md`'s existing row.)                                                                  |
| `<type>` is `tutorial` and the corpus is available                                                 | Not consulted — `tutorial`'s registry `source-of-truth` is `founder-authored` only.                                                                                                                                                                                                                          |
| `gh` unavailable / rate-limited                                                                    | Never affects **page content** — `gh` is display-only enrichment at the gate. It does gate the PR itself: a `gh pr create` failure is the `raise-pr.sh` row below. The written page is the founder-confirmed bytes either way.                                                                                 |
| Manifest present but the `llms-txt` row is disabled or absent                                      | Not an error — `llms-txt` is a `sync`-triggered row, checked **independently** of the row `seed` was invoked for. Phase 2 does not write or touch `llms.txt` this run; any existing file is left exactly as-is.                                                                                                |
| No section index page exists at `SEED_ROW`'s `target-path`                                         | Not an error — `llms.txt` is the sole index; no separate section index is created. No new index file type is invented. The existence test reads the **pre-write** tree, so a page this run wrote can never satisfy it.                                                                                         |
| Re-run: founder confirms an unchanged page and the index is unchanged                              | No commit, no PR — write phase detected an empty `git status --porcelain` (AC4).                                                                                                                                                                                                                              |
| Re-run: branch exists on `origin`, all page-touching commits carry the trailer                     | Not an error — check out at the remote head, write on top, commit + fast-forward push only if content changed. **Never reset, never force-push.**                                                                                                                                                             |
| Re-run: branch carries commits touching **non**-generated paths                                    | Not an error — proceed with the write-on-top; those commits are preserved.                                                                                                                                                                                                                                    |
| Re-run: branch carries **out-of-pipeline** edits to generated pages (no `Seed-Generated:` trailer) | **STOP at step 5**, before the founder authors — surface the paths and the PR number. Never overwritten.                                                                                                                                                                                                      |
| A guard condition first becomes true **while the founder is authoring** (TOCTOU)                   | Phase 2's optional re-check may STOP — but it **MUST preserve the confirmed content** (session temp dir, path surfaced in the STOP), never discard it. Protecting someone else's commit by destroying the founder's page trades one loss for another.                                                          |
| Re-run: a prior run's **gate-authored** page content is present on the branch                      | **Re-scaffolded**, not preserved and not STOPped — phase 1 re-scaffolds and the gate re-presents; the founder re-authors (or re-pastes) before anything is written. For `seed` this is the whole page, not a tweak. See the OQ on branch-seeded drafts.                                                        |
| A **local** `docs/seed-<type>-<SLUG>` holds commits not reachable from its remote                  | **STOP** (local-branch precondition, evaluated at step 5) — never discard unpushed work, and never clobber it on the first-run path either.                                                                                                                                                                   |
| `gh` / `raise-pr.sh` failure                                                                       | STOP and surface — the write is on a branch and reviewable; never leave a raised-but-broken state silently.                                                                                                                                                                                                   |

## Out of Scope

- **`seed adr <pattern>`** — routed to `refs/adr-pipeline.md` by **NA-57** (the `/sdlc:adr`
  absorption story, which NA-54 blocks). This story ships a clean stub that points at the still-live
  `/sdlc:adr`; NA-57 owns both the implementation **and** the deletion of that stub and its pointer.
- **`distill` mode** — NA-57. **`audit` mode** — NA-55. Both remain clean "not yet implemented" stubs
  in `docs.md`; only `seed` graduates from stub here.
- **Re-seeding / rewriting an existing published page** — the page-exists gate STOPs. `seed` creates;
  editing a published page is a hand edit, and refreshing a `source:`-bearing how-to is `sync`'s job.
- **Seeding multiple pages, or multiple types, in one run** — one invocation seeds exactly one page
  for exactly one type.
- **Editing `refs/doc-types.md` registry rows** — the five `seed`-triggered rows already exist
  (NA-51); `seed` reads them, never edits them. Adding a new doc type is a registry change, not a
  `seed` run. (This story does correct that file's **header prose**, which staleley claims only `sync`
  exists — see the artifact table. No row is touched.)
- **Adding `description:` (or any frontmatter) to `writing-docs`' quadrant templates** — this story
  mandates the frontmatter in the pipeline ref (as NA-53 did for release pages) rather than editing
  the skill, which would change `sync`'s behaviour too. **Recommended as a follow-up story** — see
  Open Question 3; the rule now has three call sites and a live regen exposure.
- **Retro-fixing frontmatter on pages that already lack it** — the regen skips and surfaces them
  (see [the regen rule](#deterministic-index--llmstxt-regen-ac4)); backfilling is not `seed`'s job.
- **Scaffolding the manifest's optional free-form sections** ("Voice & format", "Additional Jira
  project keys" — the inert-by-default field NA-60 tracks). `seed` reads the 3-column row table and
  nothing else; NA-60 owns the discoverability of the keys field, and `seed` has no interaction with
  it (it enumerates no story keys). `seed` never writes to the manifest.
- **`llms-full.txt`** (full-content variant) — v1 ships index-only `llms.txt` (inherited from NA-52).
- **Deleting or migrating pages for a newly-deactivated type** — if a founder disables `concept`
  after earlier seeds created pages, `seed` simply refuses to seed more; it never deletes existing
  pages.
- **Building or hosting a docs site** — Mintlify is this repo's choice; the plugin stays
  stack-agnostic via manifest `target-path`s (Epic out-of-scope).
- **Marketing/SEO doc copy** — `gtm:docs` owns that; `sdlc seed` is technical accuracy only.

## Open Questions

- [ ] **Is `how-to` a seedable type?** (The one genuine story-vs-registry ambiguity — see
      [`SEED_TYPES`](#seed_types--the-type-set-is-registry-derived-never-hardcoded).) AC1 names
      "concept, tutorial, integration-guide"; the registry gives `how-to` a `seed,sync` trigger and
      `draft-for-review` mode, making it seedable by the same rule AC1 states. _Suggested default
      (adopt):_ derive `SEED_TYPES` from the registry, which includes `how-to`. Excluding it would
      make `seed how-to` a usage error contradicting the registry cell the command itself reads, and
      `how-to`'s `seed,sync` trigger would then have a `seed` token no mode honours. _Founder
      call:_ if AC1's list is meant as exhaustive, the fix belongs in the **registry** (drop `seed`
      from `how-to`'s trigger), not in a command-side exception — otherwise the divergence just moves.
- [ ] **Seeding a re-run's scaffold from the branch page** — today phase 1 re-scaffolds from the
      template + topic every run, so a prior run's authored prose is not carried forward and the
      founder re-authors at the gate (see
      [Where edits live](#where-edits-live-and-what-a-re-run-does-to-them)). This bites much harder
      for `seed` than it did for `release`: a release gate edit is a tweak to aggregated content,
      whereas a seed gate edit **is the entire page**. _Suggested default (adopt for v1):_ keep
      re-scaffolding — it is the simplest correct behaviour and matches `release`'s resolved default.
      _Alternative (strongly worth revisiting, and cheaper here than for `release`):_ when the seed
      branch already exists, seed the scaffold from the current page content so the gate presents what
      is actually on the branch. Unlike `release`, this does **not** endanger an idempotence claim —
      `seed` makes none for a page body (see
      [Re-run semantics](#re-run-semantics--what-is-and-is-not-deterministic)) — so the objection that
      blocked it for `release` does not apply. The cost is that drafts become a function of branch
      state; the benefit is not re-typing a page.
- [ ] **Should `writing-docs`' quadrant templates emit `title` + `description` frontmatter? (This
      spec now recommends yes, as a follow-up story.)** Only the How-to template emits frontmatter at
      all, and none emits `description:` — yet `llms.txt` derives every entry from it. The rule has
      now been written **three** times from three directions: §12 for release pages (NA-53), §§15–19
      for seeded pages (this story), and the
      [regen's skip-and-surface rule](#deterministic-index--llmstxt-regen-ac4) for the
      **pre-existing** frontmatter-less pages the templates have already produced — which is no longer
      hypothetical, since the regen reads every enabled `public: yes` page. _Suggested default (adopt
      for this story):_ mandate it in the pipeline ref and validate at the gate; do not edit the skill
      here, because its templates are shared with `sync` and a template change is a cross-mode
      behaviour change that wants its own story and review. _Founder call (recommended):_ open that
      follow-up. Fixing the templates emits the frontmatter at authoring time for **every** mode and
      lets all three restatements collapse into one — the alternative is a fourth restatement the next
      time a mode writes a page.
- [ ] **Non-ASCII topics slug lossily or STOP** — `Ünïcodé` → `n-cod`; `日本語` → empty → STOP.
      _Suggested default (adopt):_ accept it. ASCII-only page ids and branch names are the safe,
      portable choice, the STOP message names the real constraint, and the founder can always supply
      an ASCII topic while titling the page in any script (the page `title` is free text — only the
      **id** is slugged). _Revisit if:_ a consumer repo documents in a non-Latin script, where every
      topic would STOP.
- [ ] **`SLUG` length cap of 80** — _Suggested default (adopt):_ 80, STOP rather than truncate.
      Empirically bounded (a 300-char filename fails `ENAMETOOLONG`, 200 succeeds — verified), and
      STOP avoids truncation collapsing two topics onto one page. The exact number is a judgement
      call; the STOP-not-truncate rule is not.
- [ ] **The reserved page-id set is `{index}`** — _Suggested default (adopt):_ reserve `index`, the
      section-index filename convention this pipeline already uses (NA-53's changelog is
      `docs/changelog/index.md`). _Residual risk:_ a consumer whose docs tree names its section index
      something else (`README.md`, `overview.md` — Mintlify and Docusaurus both have their own
      conventions) could seed a topic that collides with it. The **pre-write existence test** is what
      makes that survivable rather than destructive, which is why both rules are specified. _Revisit
      if:_ a consumer's index convention differs — the set may need to be manifest-derived rather than
      a literal.
- [ ] **`seed` on an existing published page** — _Suggested default (adopt):_ STOP (page-exists
      gate). `seed` is a create verb; a published page may carry months of hand edits and a
      founder-authored `source:` list, and silently overwriting them is the failure this gate exists
      to prevent. _Alternative (declined for v1):_ a `--force` / re-author flag. Declined because the
      safe path already exists — edit the page directly — and a force flag on a write path this
      destructive wants a stronger case than convenience.
- [ ] **Branch name includes `<type>`** (`docs/seed-<type>-<SLUG>`, not `docs/seed-<SLUG>`) —
      _Suggested default (adopt):_ include it. Two types may legitimately seed the same topic (a
      concept **and** a tutorial about offline sync), which are distinct pages at distinct
      `target-path`s and must not collide on one branch/PR. Diverges from `commands/adr.md`'s
      `docs/adr-<slug>` only because ADR has exactly one target path and `seed` has four.
