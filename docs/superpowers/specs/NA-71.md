# Render Marketing Site as Static Build Output — Technical Spec

**Story:** [NA-71](https://whimzylive.atlassian.net/browse/NA-71) — Render marketing site as static build output
**Epic:** [NA-29](https://whimzylive.atlassian.net/browse/NA-29) — nightshift marketing site
**Feature:** docs/features/2026-07-12-nightshift-marketing-site-design-handoff.md
**Owner agent:** web-engineer (`apps/marketing/`)
**Date:** 2026-07-24

## Overview

Convert the three DB-backed public marketing routes (Home `/`, `/faq`, `/why-sdlc`) from request-time `force-dynamic` rendering to build-time static HTML, so the served site makes zero runtime database requests. Payload/Neon stays for the admin surface and build-time reads only, never on the serving path.

> **This is a rendering-mode conversion, not a feature.** There is no new data model, no new API surface, no new UI, and no new offline sync. The Payload collections/globals and the page component trees are unchanged. Only *when* the Payload reads execute (build time vs request time) changes — plus one narrow error-handling adjustment in the `lib/` helpers so a real DB outage fails the build rather than silently shipping empty pages (see Web UI / Rendering and Error Handling).

| --- | --- | --- | --- |

| `/faq` | `apps/marketing/src/app/(frontend)/faq/page.tsx` | ƒ Dynamic (`force-dynamic`) | ● / ○ Static (SSG) |

| `/team` | `apps/marketing/src/app/(frontend)/team/page.tsx` | ○ Static (already — zero DB dependency) | ○ Static (unchanged) |
| `/admin/*` | `apps/marketing/src/app/(payload)/admin/[[...segments]]/page.tsx` | ƒ Dynamic | ƒ Dynamic (out of scope — stays) |
| `/api/*` | `apps/marketing/src/app/(payload)/api/**/route.ts` | ƒ Dynamic | ƒ Dynamic (out of scope — stays) |

> **`/team` is NOT a proof-of-pattern for this conversion.** `team/page.tsx` performs **no** Payload read at all — it is a synchronous server component that only imports static `teamMetadata` and static roster TypeScript. It confirms that a page with *zero* DB dependency prerenders statically, but it does **not** exercise the risky path this story introduces: a **build-time Payload Local API read against a live, migrated Neon DB** (Home, `/faq`, `/why-sdlc`). That build-time-Neon-reachability path (see Build / CI Constraint) has **no existing in-repo example** and is the genuinely new risk this story carries.

## Web UI / Rendering (core of this spec)

The public pages already read Payload through its **Local API** (`getPayload({ config })` — native in-process Node calls, no HTTP). Removing the `force-dynamic` opt-out lets Next.js execute those reads once at `next build` and emit static HTML. No component tree or markup change is required; the only source changes are the per-route opt-out removal and a narrowing of the `lib/` helper catch blocks (below).

### Per-route change

For each of the three routes, the **only** source change is removing the dynamic opt-out and its now-stale explanatory comment:

| File | Remove | Keep unchanged |
| --- | --- | --- |
| `app/(frontend)/page.tsx` | `export const dynamic = 'force-dynamic';` (line 35) + the preceding NA-16 rationale comment (lines 31–34) | `metadata` const, `async function HomePage`, the `await getHomeFaqs()` read, JSON-LD, full section tree |
| `app/(frontend)/faq/page.tsx` | `export const dynamic = 'force-dynamic';` (line 19) + the preceding rationale comment (lines 15–18) | `metadata` const, `async function FaqPage`, `await getFaqPageGroups()`, JSON-LD, section tree |
| `app/(frontend)/why-sdlc/page.tsx` | `export const dynamic = 'force-dynamic';` (line 24) + the preceding rationale comment (lines 20–23) | `metadata` const, `async function WhySdlcPage`, `Promise.all([getWhySdlcContent(), getWhySdlcFaqs()])`, JSON-LD, section tree |

The remaining `// Async page — the single top-level boundary…` comment block on each page describes the RTL nested-async-boundary constraint (NA-16) and stays; only the DB/force-dynamic rationale comment is removed.

> **Comment hygiene:** NA-70 review (memory 15510) flagged explanatory comments that restate policy as a lint/style violation. Do **not** replace the removed comment with a new "// static now" comment — remove it cleanly.

### Static-eligibility verification (AC2)

A route silently opts back into dynamic rendering if it calls `cookies()`, `headers()`, reads `searchParams`, or sets `dynamic`/`revalidate = 0`. Confirmed by inspection across all four public routes in `app/(frontend)/`:

- **No** `cookies()`, `headers()`, or `searchParams` usage in any public route (repo-wide grep of `app/(frontend)` returns zero matches).
- No `export const revalidate` / `generateStaticParams` on any public route.
- After the change, no public route has a `dynamic` export; `app/(payload)/*` (admin + api) has its own Payload-provided dynamic config and is untouched.

The implementer must re-run this grep after the edit to prove no public route reintroduced a dynamic trigger.

### Data helpers — narrow the catch (required change)

`apps/marketing/src/lib/faq.ts` (`getHomeFaqs`, `getFaqPageGroups`) and `apps/marketing/src/lib/why-sdlc.ts` (`getWhySdlcContent`, `getWhySdlcFaqs`) each call `getPayload({ config })` and a Payload query inside a single try/catch that returns an empty fallback (`[]` / `null`) on **any** error. At request time (today) that fallback merely degrades one section. At **build** time it is dangerous: a total Neon outage would be swallowed and `next build` would **succeed with contentless pages**, silently shipping an empty site to the CDN.

**Required change (web-engineer — `lib/faq.ts` + `lib/why-sdlc.ts`):** narrow each helper's catch so it distinguishes two failure classes:

- **Connection / init failures** — a failure to construct the Payload client or reach the DB (`getPayload({ config })` throwing, or the query rejecting with a connection/authentication/DB-unreachable error) — must **rethrow**, so `next build` fails fast on a real outage.
- **Row-level / data-shape defects** — a single malformed row or a benign query issue that does not indicate the DB is down — may still be swallowed to the empty fallback, preserving today's resilience for isolated defects.

A query that legitimately returns **zero rows** (e.g. no FAQ has `showOnHome = true`) is **not** an error — the helper returns `[]`/`null` and the page renders its empty state, exactly as today. This root-cause fix removes any need for a per-route emptiness allowlist and makes the build-fails-fast-on-outage behaviour in Error Handling true by construction, so the contradiction cannot reappear.

> Implementation detail (mechanism of the connection-vs-row discrimination — e.g. inspecting the error type/code from the postgres adapter, or attempting a cheap connection probe before the query) is left to the web-engineer; the **contract** above is the spec requirement.

## Build / CI Constraint

Because the Payload Local API reads now run at `next build` instead of per request, **the build environment must reach a migrated, reachable Neon database** to read CMS content. This inverts the NA-16-era posture (force-dynamic was chosen specifically to avoid needing a DB at build time). This is the genuinely new risk — no existing route in the repo exercises a build-time Local API read against Neon (`/team`, the only other static page, reads no DB).

Requirements the build/deploy environment must satisfy:

| Requirement | Detail |
| --- | --- |
| `DATABASE_URL` present at build | Consumed by `postgresAdapter` in `payload.config.ts`; the same var already used at runtime. Must be set in the CI/build env, not only the serve env. |
| `PAYLOAD_SECRET` present at build | `getPayload({ config })` initialises the full Payload instance; missing secret throws. |
| Neon reachable from the build runner | Network egress from CI to the Neon host during `next build`. |
| Schema migrated | The build reads the current `faq` / `whySdlc` schema; the Neon instance must be on the migrated schema (already true in production per the story). |

**Tradeoff (document in the PR description per AC5):** content changes now require a **rebuild + redeploy** to appear on the live site — there is no request-time DB read on the serving path. Instant/live content updates are explicitly out of scope and accepted by the story. In exchange the served site is a static CDN artifact with zero serving-path DB dependency.

## Verification Criteria

1. **Build-output rendering symbols (AC3).** `pnpm nx build @nightshift-ai/marketing` route table shows `/`, `/faq`, `/why-sdlc`, and `/team` marked static (`○` Static or `●` SSG) — **no `ƒ` (Dynamic)** against any public route.
2. **Admin/API unaffected (AC4).** `/admin/*` and `/api/*` may remain `ƒ` in the same output — expected, not a regression.
3. **No dynamic triggers (AC2).** Post-edit grep of `app/(frontend)` shows no `cookies()`, `headers()`, `searchParams`, `dynamic`, or `revalidate`.
4. **Visual/markup identity (AC6).** Rendered content and markup of all four pages are unchanged vs pre-change — only the rendering mode differs. Existing component tests (e.g. `faq.spec.ts`) remain green under `pnpm nx run-many -t test`.
5. **Build reaches the DB.** The build completes with populated content (FAQ entries, why-sdlc arguments render), proving the build-time Local API read succeeded against Neon.
6. **Outage fails the build (helper-catch narrowing).** With the DB deliberately unreachable, `next build` **fails** (a connection/init error rethrows) rather than producing an empty-but-successful build — proving the narrowed catch works.

## Error Handling

| Scenario | Behaviour | Rationale |
| --- | --- | --- |
| Neon **unreachable / auth fails / client init fails** at build time | The narrowed `lib/` helper catch **rethrows** the connection/init error → the page's `await` rejects → **`next build` fails fast**. No empty page is shipped. | Shipping a blank/contentless page to the CDN is worse than a failed build; a failed build blocks the bad deploy. Made true by construction via the helper-catch narrowing (Web UI / Rendering). |
| A **single malformed row** or benign data-shape defect | The helper's catch swallows it and returns the empty fallback (`[]` / `null`); the page renders its empty state for that section and the build still succeeds. | Preserves existing resilience for isolated data defects without failing the whole build. |
| **Zero matching rows** (e.g. no `showOnHome` FAQs) | Not an error — the query succeeds and returns `[]`; the section renders nothing (existing behaviour). Build succeeds. | A legitimately-empty result is distinct from a DB outage; only the outage fails the build. |

Rows 1 and 3 are reconciled by the helper-catch narrowing: a **connection/init failure** fails the build, a **legitimately-empty query result** does not. There is no longer any path where a total DB outage silently ships an empty site.

## Out of Scope

- **Deploy-hook / rebuild-on-content-change automation** (Vercel Deploy Hook / webhook). Content updates are a manual/CI-triggered rebuild for now.
- **On-demand ISR** (`revalidatePath`, `revalidate`, `dynamicParams`) — explicitly not this approach.
- **The `[slug]` route + Pages/blocks freeform-CMS feature** — abandoned ISR effort (closed PR #168, spec reverted #167). Not present on `develop` (confirmed: `payload.config.ts` collections are `[Faq, Media, Users]`, no Pages collection, no `[slug]` route). A separate future story if ever revisited, and it must be built in this static style.
- **Dynamic admin editing / DB-in-serving-path** — deferred until real-time admin editing is needed. `/admin/*` and `/api/*` stay dynamic.
- **Any redesign** of the existing pages — visual output must be unchanged.
