# Render Marketing Site as Static Build Output — Technical Spec

**Story:** [NA-71](https://whimzylive.atlassian.net/browse/NA-71) — Render marketing site as static build output
**Epic:** [NA-29](https://whimzylive.atlassian.net/browse/NA-29) — nightshift marketing site
**Feature:** docs/features/2026-07-12-nightshift-marketing-site-design-handoff.md
**Owner agent:** web-engineer (`apps/marketing/`)
**Date:** 2026-07-24

## Overview

Convert the three DB-backed public marketing routes (Home `/`, `/faq`, `/why-sdlc`) from request-time `force-dynamic` rendering to build-time static HTML, so the served site makes zero runtime database requests. Payload/Neon stays for the admin surface and build-time reads only, never on the serving path.

> **This is a rendering-mode conversion, not a feature.** There is no new data model, no new API surface, no new UI, and no new offline sync. The Payload collections/globals, the `lib/` data helpers, the page component trees, and the rendered markup are all unchanged. Only *when* the Payload reads execute (build time vs request time) changes.

## Data Model

**No changes — this is a rendering-mode conversion.** The `faq` collection, the `whySdlc` global, `Media`, and `Users` (`apps/marketing/src/payload.config.ts`) are untouched. No migration.

## API Surface

**No changes — this is a rendering-mode conversion.** No new endpoints. The existing Payload REST/GraphQL surface under `app/(payload)/api/*` is explicitly out of scope and stays dynamic (it is the admin editing surface, not the public serving path).

## Web UI / Rendering (core of this spec)

The public pages already read Payload through its **Local API** (`getPayload({ config })` — native in-process Node calls, no HTTP). Removing the `force-dynamic` opt-out lets Next.js execute those reads once at `next build` and emit static HTML. No component, hook, or markup change is required.

### Static-eligibility verification (AC2)

A route silently opts back into dynamic rendering if it calls `cookies()`, `headers()`, reads `searchParams`, or sets `dynamic`/`revalidate = 0`. Confirmed by inspection across all four public routes in `app/(frontend)/`:

- **No** `cookies()`, `headers()`, or `searchParams` usage in any public route (repo-wide grep of `app/(frontend)` returns zero matches).
- No `export const revalidate` / `generateStaticParams` on any public route.
- After the change, the only `dynamic` exports remaining anywhere under `app/` are none on public routes; `app/(payload)/*` (admin + api) has its own Payload-provided dynamic config and is untouched.

The implementer must re-run this grep after the edit to prove no public route reintroduced a dynamic trigger.

### Data helpers — no change required

`apps/marketing/src/lib/faq.ts` (`getHomeFaqs`, `getFaqPageGroups`) and `apps/marketing/src/lib/why-sdlc.ts` (`getWhySdlcContent`, `getWhySdlcFaqs`) already:

- Use the Payload **Local API** via `getPayload({ config })` — works identically at build time as at request time (in-process, no server needed).
- Wrap every read in try/catch with a graceful empty fallback (`[]` for lists, `null` for the global).

**Decision: keep the graceful-empty fallbacks as-is** — they protect the *page render* from a transient malformed row, and the pages already tolerate empty data (`content?.intro ?? null`, `faqs.map`, etc.). See Error Handling for the interaction with build-time DB failure. No edit to the `lib/` files.

## Build / CI Constraint

Because the Payload Local API reads now run at `next build` instead of per request, **the build environment must reach a migrated, reachable Neon database** to read CMS content. This inverts the NA-16-era posture (force-dynamic was chosen specifically to avoid needing a DB at build time).

Requirements the build/deploy environment must satisfy:

| Requirement | Detail |
| --- | --- |
| `DATABASE_URL` present at build | Consumed by `postgresAdapter` in `payload.config.ts`; the same var already used at runtime. Must be set in the CI/build env, not only the serve env. |
| `PAYLOAD_SECRET` present at build | `getPayload({ config })` initialises the full Payload instance; missing secret throws. |
| Neon reachable from the build runner | Network egress from CI to the Neon host during `next build`. |
| Schema migrated | The build reads the current `faq` / `whySdlc` schema; the Neon instance must be on the migrated schema (already true in production per the story). |

**Tradeoff (document in the PR description per AC5):** content changes now require a **rebuild + redeploy** to appear on the live site — there is no request-time DB read on the serving path. Instant/live content updates are explicitly out of scope and accepted by the story. In exchange the served site is a static CDN artifact with zero serving-path DB dependency.

## Verification Criteria

1. **Build-output rendering symbols (AC3).** `pnpm nx build @nightshift-ai/marketing` route table shows `/`, `/faq`, `/why-sdlc`, and `/team` marked static (`○` Static or `●` SSG) — **no `ƒ` (Dynamic)** against any public route.
2. **Admin/API unaffected (AC4).** `/admin/*` and `/api/*` may remain `ƒ` in the same output — expected, not a regression.
3. **No dynamic triggers (AC2).** Post-edit grep of `app/(frontend)` shows no `cookies()`, `headers()`, `searchParams`, `dynamic`, or `revalidate`.
4. **Visual/markup identity (AC6).** Rendered content and markup of all four pages are unchanged vs pre-change — only the rendering mode differs. Existing component tests (e.g. `faq.spec.ts`) remain green under `pnpm nx run-many -t test`.
5. **Build reaches the DB.** The build completes with populated content (FAQ entries, why-sdlc arguments render), proving the build-time Local API read succeeded against Neon.

## Error Handling

| Scenario | Behaviour | Rationale |
| --- | --- | --- |
| Neon **unreachable / auth fails** at build time | `getPayload({ config })` initialisation throws → the affected page's `await` rejects. Since there is no request-time fallback, the correct posture is **build fails fast** — do not silently ship an empty page. | Shipping a blank/contentless page to the CDN is worse than a failed build; a failed build blocks the bad deploy. See Open Question 1. |
| A **single malformed row** or partial query error | The `lib/` helper's try/catch returns the empty fallback (`[]` / `null`); the page renders its empty state for that section and the build still succeeds. | Preserves existing resilience for isolated data defects without failing the whole build. |
| **Missing rows** (e.g. no `showOnHome` FAQs) | Helper returns `[]`; section renders nothing (existing behaviour). Not an error. | Unchanged from today. |

Because the helpers swallow DB errors and return empty, a *total* DB outage at build could otherwise produce a successfully-built-but-empty site. The implementer must ensure a build-time DB outage does not silently ship empty pages — resolve via Open Question 1.

## Out of Scope

- **Deploy-hook / rebuild-on-content-change automation** (Vercel Deploy Hook / webhook). Content updates are a manual/CI-triggered rebuild for now.
- **On-demand ISR** (`revalidatePath`, `revalidate`, `dynamicParams`) — explicitly not this approach.
- **The `[slug]` route + Pages/blocks freeform-CMS feature** — abandoned ISR effort (closed PR #168, spec reverted #167). Not present on `develop` (confirmed: `payload.config.ts` collections are `[Faq, Media, Users]`, no Pages collection, no `[slug]` route). A separate future story if ever revisited, and it must be built in this static style.
- **Dynamic admin editing / DB-in-serving-path** — deferred until real-time admin editing is needed. `/admin/*` and `/api/*` stay dynamic.
- **Any redesign** of the existing pages — visual output must be unchanged.
