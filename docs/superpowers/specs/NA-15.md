# Atomic Design Skill — Technical Spec

**Story:** NA-15 — Build and publish new atomic design skill
**Feature:** —  (no parent Epic; story is self-contained)

## Overview

Author a new **workspace-authored, public** Claude Code skill, `atomic-design`, that teaches an AI/developer to decompose any frontend UI into Brad Frost's five-level atomic hierarchy (atoms → molecules → organisms → templates → pages). The skill is framework-agnostic — pure UI/UX structuring guidance with no stack-specific implementation detail — and ships under the repo's published `skills/` directory alongside the other public skills.

## Skill Placement & Naming

The repo places skills in **two** locations, defined in `plugins/sdlc/commands/init.md` — do not mix them:

1. `.claude/skills/<name>/` — **consumed skills** this repo's own agents invoke. Each is a **real, copied directory** (init.md installs content here — never a symlink): either vendored from a declared upstream `source` cloned at a pinned ref, plugin-installed, or a locally hand-authored skill (e.g. `nightshift-design`). External skills are vendored via the human-confirmed **vet-and-pin** flow (clone at a pinned ref, vet, commit) — **never** `npx skills add`, which `plugins/sdlc/refs/analyze-protocol.md` explicitly forbids during a scan. Consumed skills are tracked by the **sync trio** (see Publication & Registration).
2. `skills/<name>/` — **workspace-authored skills committed to the repo and published for external consumers** (`hono-api`, `typeorm`, `electrodb`). These are published-for-others: they are **not** invoked by this repo's own agents and are deliberately absent from the sync trio.

The `atomic-design` skill belongs in the **second location**: framework-agnostic guidance authored here and hosted for any external frontend team to consume. It is the same class as `hono-api`/`typeorm`/`electrodb` — published-for-others, not consumed by this repo's own agents.

**Decision — placement:**

| Item | Value |
| --- | --- |
| Skill directory | `skills/atomic-design/` |
| Primary skill file | `skills/atomic-design/SKILL.md` |
| Skill identifier (`name` frontmatter) | `atomic-design` |
| Real directory or symlink | **Real directory** (workspace-authored, committed content) |

Directory and identifier use the same lowercase, hyphenated token, matching every existing published skill.

## Skill File Structure

### File layout

```
skills/atomic-design/
  SKILL.md            # required — the skill overview: frontmatter + rationale + workflow + heuristics
  evals/
    evals.json        # required — sample UI descriptions with expected five-level decompositions
  references/
    hierarchy.md      # required — expanded per-level guidance (the five levels in depth)
```

**DECIDED (PR #55 review):** the skill ships with the **`references/` split from the start** — `SKILL.md` (overview + workflow + heuristics) plus `references/hierarchy.md` (the expanded per-level detail). This is the chosen layout, **not** the single-file-until-200-lines default: the maintainer selected the progressive-disclosure split up front. `SKILL.md` stays the scannable overview and links out to `references/hierarchy.md` for the deep per-level content.

Every existing published skill (`hono-api`, `typeorm`, `electrodb`) ships `evals/evals.json`, so `atomic-design` must too — for AC-5 parity. A `scripts/` directory is **waived** for this conceptual skill (nothing to execute); the published precedents carry one, but the evals here are declarative decomposition cases, not runnable scripts.

**`references/` naming note:** `references/hierarchy.md` follows the `skill-creator` upstream convention (a `references/` subdirectory of progressive-disclosure files). This deliberately **diverges** from `typeorm`'s in-repo precedent, which uses a single flat `reference.md`. Keeping `references/` is the decided choice here (see Resolved Decisions) — the divergence is named so no implementer "corrects" it to match typeorm.

### Frontmatter (required)

YAML frontmatter with exactly the two keys every skill in this repo uses — `name` and `description`. No other keys.

```yaml
---
name: atomic-design
description: <one-line trigger sentence — see below>
---
```

- `name` MUST equal the directory name: `atomic-design`.
- `description` MUST follow the repo's "Use when …" trigger convention (see `electrodb`, `hono-api`, `typeorm`) so the skill auto-surfaces at the right moment. It must name the trigger scenario and the five levels, and stay framework-agnostic.
- `description` MUST stay within the platform limit (≤ 1024 characters); target 1–3 sentences.

**Suggested `description` (implementer may refine wording, must keep meaning):**

> `Use when structuring or decomposing any frontend UI into reusable components, regardless of framework. Applies Brad Frost's atomic design methodology — atoms, molecules, organisms, templates, and pages — to guide consistent component hierarchy, naming, and composition decisions.`

### Body structure (required sections, in order)

The `SKILL.md` body MUST cover, at minimum, the following sections. Exact headings/wording are the implementer's call; the content coverage is fixed by the acceptance criteria. Per the decided `references/` split, Section 3's expanded per-level detail lives in `references/hierarchy.md` and is linked from `SKILL.md`.

| # | Section | Must contain |
| --- | --- | --- |
| 1 | Title + one-paragraph summary | What atomic design is; that this skill is framework-agnostic. |
| 2 | Why atomic design (rationale) | The problem it solves — consistency, reuse, shared vocabulary, scalable UI systems (AC-3). Synthesised from Frost ch. 1–2, not copied. |
| 3 | The five levels | One subsection per level: **Atoms → Molecules → Organisms → Templates → Pages**. Each: definition, defining trait, an abstract (non-framework) example, and the "when does X become Y" boundary rule (AC-2). Overview in `SKILL.md`; expanded per-level guidance in `references/hierarchy.md`. |
| 4 | How to apply it (decomposition workflow) | An ordered, tool-agnostic procedure an AI/developer follows to decompose an arbitrary UI into the five levels (AC-6). |
| 5 | Decision heuristics / boundaries | Rules of thumb for classifying an element and for promoting/demoting between levels (e.g. atom vs molecule threshold). |
| 6 | Common pitfalls / anti-patterns | e.g. over-nesting organisms, treating pages as templates, framework leakage. |

### The five-level hierarchy — required coverage (AC-2)

Each level MUST be defined in framework-neutral terms. Reference framing (implementer synthesises, does not quote verbatim):

| Level | Framework-neutral definition | Boundary rule to state |
| --- | --- | --- |
| Atoms | The smallest indivisible UI building blocks (a label, an input field, a single control). | An atom has no meaningful sub-parts; splitting it removes function. |
| Molecules | A small group of atoms bonded together to do one thing (a search field = label + input + button). | Becomes a molecule when ≥ 2 atoms combine for a single, reusable purpose. |
| Organisms | Relatively complex groups of molecules and/or atoms forming a distinct section (a header, a product card grid). | Becomes an organism when it forms a standalone, recognisable interface region. |
| Templates | Page-level layout skeletons that place organisms into a structure, using placeholder content. | Concerned with structure/arrangement, not real content. |
| Pages | Specific instances of templates with real representative content. | The concrete, testable end state; used to validate the design system. |

### evals (required)

`skills/atomic-design/evals/evals.json` MUST ship alongside `SKILL.md`, matching the published-skill precedent. Its content is a set of declarative cases: sample UI descriptions paired with the expected five-level decomposition (which elements are atoms vs molecules vs organisms vs templates vs pages), so the skill's guidance can be checked against known-good answers. Mirror the shape/format used by `skills/hono-api/evals/evals.json`, `skills/typeorm/evals/evals.json`, and `skills/electrodb/evals/evals.json`.

### Body budget & progressive disclosure

- Keep `SKILL.md` scannable. The expanded per-level detail lives in `skills/atomic-design/references/hierarchy.md` (the decided `references/` split — see Resolved Decisions), linked from Section 3, keeping `SKILL.md` as the overview + workflow + heuristics.
- Size `SKILL.md` in line with the concise-skill norm seen in existing published skills (roughly the size of `hono-api`/`typeorm` SKILL.md); push depth into `references/hierarchy.md` rather than growing the overview.

## Framework-Agnostic Requirements (AC-4)

The skill MUST be verifiably free of stack-specific detail. Concretely:

- **No** framework or library names as prescribed tooling: React, Vue, Angular, Svelte, Solid, React Native, Expo, SwiftUI, Flutter, Tailwind, CSS-in-JS, etc.
- **No** code blocks in a specific framework's syntax (no JSX, no `.vue` SFCs, no Swift/Kotlin). Illustrative examples MUST be described in prose or framework-neutral pseudo-structure (e.g. "a *search-field* molecule = *label* atom + *text-input* atom + *button* atom").
- **No** web-only or mobile-only assumptions (no "DOM", "route", "screen", "native module" as required concepts). Neutral terms: "component", "screen/view" only as generic UI regions.
- Vocabulary stays at the **UI/UX structuring** layer — hierarchy, composition, naming, reuse.

Note: framework names appear in the `skills-map.yml` `when:` detection triggers (React, Vue, Next.js, React Native, Expo) purely so `/init` surfaces the skill in those repos — that is registration metadata, **not** skill content, and is exempt from this content scan.

**Verification (part of the impl, not a separate gate):** a case-insensitive scan of `SKILL.md` (and any `references/*`) for the banned framework/library tokens above returns zero prescriptive hits. Brad Frost / "atomic design" attribution and generic words are fine.

## Publication & Registration (AC-5)

"Published and publicly available" is satisfied by committing the skill into the repo's public `skills/` tree — the same mechanism by which `hono-api`, `typeorm`, and `electrodb` are already public. There is **no separate publish step, package, or registry push.**

**Sync-trio decision.** `plugins/sdlc/commands/init.md` defines a **"sync trio"** — the installed `.claude/skills/<name>/` directory + its `.claude/project/skills.json` entry + its agent-override skill list — which must stay aligned **for skills this repo's own agents invoke**. The precedent for published-for-others skills (`hono-api`, `typeorm`, `electrodb`) is that they live under `skills/` only and are **absent from `.claude/project/skills.json` and from every agent override** — because this repo does not consume them. `atomic-design` is the same class (published for external frontend teams, not invoked by nightshift's SDLC/GTM agents), so it stays out of the sync trio.

**Registration parity (AC-5).** Every published skill is also registered in `plugins/sdlc/refs/skills-map.yml` with a `source` + `path` so other repos can discover and install it via `/init`. `hono-api`, `typeorm`, and `electrodb` each carry `source: https://github.com/whimzyLive/nightshift-ai.git` + `path: skills/<name>`. AC-5's "consistent with other public skills" therefore **puts skills-map registration in scope** for `atomic-design`.

**Required registration actions:**

| Action | Required? | Rationale |
| --- | --- | --- |
| Create `skills/atomic-design/SKILL.md` + `evals/evals.json` + `references/hierarchy.md` | **Yes** | The deliverable (AC-1, AC-5); ships with the decided `references/` split. |
| Register in `plugins/sdlc/refs/skills-map.yml` (`source` + `path`) | **Yes** | AC-5 parity — every published skill (`hono-api`, `typeorm`, `electrodb`) is registered there so `/init` in other repos can install it. |
| Add a `.claude/project/skills.json` entry | **No** | `skills.json` tracks skills this repo's own agents consume. Published-for-others skills are absent from it (as are `hono-api`/`typeorm`/`electrodb`); the current manifest is `{"version": 1, "skills": []}`. |
| Add to an agent override's skill list | **No** | Same rationale — a binding implies local consumption, which this skill does not have. |
| Add a `.claude/skills/atomic-design/` copy | **No** | `.claude/skills/` holds **consumed** skills (real copied dirs). A published-for-others artifact lives only under `skills/`; there is no symlink or copy step. |

Net: the **mandatory registration is (a) the committed `skills/atomic-design/` directory** (with `SKILL.md` + `evals/evals.json` + `references/hierarchy.md`) **and (b) its `plugins/sdlc/refs/skills-map.yml` entry.** Everything else (`skills.json`, agent overrides, `.claude/skills/`) is intentionally omitted to match the published-for-others precedent.

**`skills-map.yml` entry** (`when` triggers decided per PR #55 review — the five frontend frameworks below):
