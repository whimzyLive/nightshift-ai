# Atomic Design Skill — Technical Spec

**Story:** NA-15 — Build and publish new atomic design skill
**Feature:** —  (no parent Epic; story is self-contained)
**Date:** 2026-07-09

## Overview

Author a new **workspace-authored, public** Claude Code skill, `atomic-design`, that teaches an AI/developer to decompose any frontend UI into Brad Frost's five-level atomic hierarchy (atoms → molecules → organisms → templates → pages). The skill is framework-agnostic — pure UI/UX structuring guidance with no stack-specific implementation detail — and ships under the repo's published `skills/` directory alongside the other public skills.

## Skill Placement & Naming

The repo has three skill tiers (per the ai-enablement-engineer override — do not mix):

1. `.agents/skills/<name>/` — CLI-installed external skills (`npx skills add`), pinned in root `skills-lock.json`.
2. `.claude/skills/<name>` — Claude Code discovery surface: a symlink to tier 1, or a real dir for local hand-authored **consumed** skills (e.g. `nightshift-design`).
3. `skills/<name>/` — **workspace-authored skills committed to the repo and published for external consumers** (e.g. `hono-api`, `typeorm`, `electrodb`, `nx-*`).

The `atomic-design` skill is a **tier-3 workspace-authored public skill**: framework-agnostic guidance authored here and hosted for any external frontend team to consume. It is the same class as `hono-api`/`typeorm`/`electrodb`, which are published-for-others and are deliberately **not** consumed by this repo's own agents.

## Skill File Structure

### File layout

```
skills/atomic-design/
  SKILL.md            # required — the skill itself (frontmatter + body)
  references/         # optional — progressive-disclosure detail files (see below)
    hierarchy.md      # optional — expanded per-level guidance if SKILL.md would exceed the body budget
```

The MVP is a **single self-contained `SKILL.md`**. A `references/` file is only introduced if the body would otherwise become too long to stay scannable (see "Body budget"). No `scripts/` or `evals/` directory is required for a conceptual skill.

### Frontmatter (required)

YAML frontmatter with exactly the two keys every skill in this repo uses — `name` and `description`. No other keys.

```yaml
---
name: atomic-design
description: <one-line trigger sentence — see below>
---
```

- `name` MUST equal the directory name: `atomic-design`.
- `description` MUST follow the repo's "Use when …" trigger convention (see `electrodb`, `hono-api`, `typeorm`) so the skill auto-surfaces at the right moment. It must name the trigger scenario and the five levels, and stay framework-agnostic.
- `description` MUST stay within the platform limit (≤ 1024 characters); target 1–3 sentences.

**Suggested `description` (implementer may refine wording, must keep meaning):**

> `Use when structuring or decomposing any frontend UI into reusable components, regardless of framework. Applies Brad Frost's atomic design methodology — atoms, molecules, organisms, templates, and pages — to guide consistent component hierarchy, naming, and composition decisions.`

### Body structure (required sections, in order)

The `SKILL.md` body MUST cover, at minimum, the following sections. Exact headings/wording are the implementer's call; the content coverage is fixed by the acceptance criteria.

| # | Section | Must contain |
| --- | --- | --- |
| 1 | Title + one-paragraph summary | What atomic design is; that this skill is framework-agnostic. |
| 2 | Why atomic design (rationale) | The problem it solves — consistency, reuse, shared vocabulary, scalable UI systems (AC-3). Synthesised from Frost ch. 1–2, not copied. |
| 3 | The five levels | One subsection per level: **Atoms → Molecules → Organisms → Templates → Pages**. Each: definition, defining trait, an abstract (non-framework) example, and the "when does X become Y" boundary rule (AC-2). |
| 4 | How to apply it (decomposition workflow) | An ordered, tool-agnostic procedure an AI/developer follows to decompose an arbitrary UI into the five levels (AC-6). |
| 5 | Decision heuristics / boundaries | Rules of thumb for classifying an element and for promoting/demoting between levels (e.g. atom vs molecule threshold). |
| 6 | Common pitfalls / anti-patterns | e.g. over-nesting organisms, treating pages as templates, framework leakage. |

### The five-level hierarchy — required coverage (AC-2)

Each level MUST be defined in framework-neutral terms. Reference framing (implementer synthesises, does not quote verbatim):

| Level | Framework-neutral definition | Boundary rule to state |
| --- | --- | --- |
| Atoms | The smallest indivisible UI building blocks (a label, an input field, a single control). | An atom has no meaningful sub-parts; splitting it removes function. |
| Molecules | A small group of atoms bonded together to do one thing (a search field = label + input + button). | Becomes a molecule when ≥ 2 atoms combine for a single, reusable purpose. |
| Organisms | Relatively complex groups of molecules and/or atoms forming a distinct section (a header, a product card grid). | Becomes an organism when it forms a standalone, recognisable interface region. |
| Templates | Page-level layout skeletons that place organisms into a structure, using placeholder content. | Concerned with structure/arrangement, not real content. |
| Pages | Specific instances of templates with real representative content. | The concrete, testable end state; used to validate the design system. |

### Body budget & progressive disclosure

- Keep `SKILL.md` scannable. If the combined body materially exceeds the concise-skill norm seen in existing tier-3 skills (roughly the size of `hono-api`/`typeorm` SKILL.md), move the expanded per-level detail into `skills/atomic-design/references/hierarchy.md` and link to it from Section 3, keeping `SKILL.md` as the overview + workflow.
- **Suggested default:** start single-file; only split if it grows past ~200 body lines.

## Framework-Agnostic Requirements (AC-4)

The skill MUST be verifiably free of stack-specific detail. Concretely:

- **No** framework or library names as prescribed tooling: React, Vue, Angular, Svelte, Solid, React Native, SwiftUI, Flutter, Tailwind, CSS-in-JS, etc.
- **No** code blocks in a specific framework's syntax (no JSX, no `.vue` SFCs, no Swift/Kotlin). Illustrative examples MUST be described in prose or framework-neutral pseudo-structure (e.g. "a *search-field* molecule = *label* atom + *text-input* atom + *button* atom").
- **No** web-only or mobile-only assumptions (no "DOM", "route", "screen", "native module" as required concepts). Neutral terms: "component", "screen/view" only as generic UI regions.
- Vocabulary stays at the **UI/UX structuring** layer — hierarchy, composition, naming, reuse.

**Verification (part of the impl, not a separate gate):** a case-insensitive scan of `SKILL.md` (and any `references/*`) for the banned framework/library tokens above returns zero prescriptive hits. Brad Frost / "atomic design" attribution and generic words are fine.

## Publication & Registration (AC-5)

"Published and publicly available" is satisfied by committing the skill into the repo's public `skills/` tree — the same mechanism by which `hono-api`, `typeorm`, and `electrodb` are already public. There is **no separate publish step, package, or registry push.**

**Sync-trio decision.** The ai-enablement-engineer override defines a "sync trio" (install location + `.claude/project/skills.json` entry + agent-override skill list) that must stay aligned **for skills this repo's own agents invoke**. The precedent for tier-3 *published-for-others* skills (`hono-api`, `typeorm`, `electrodb`) is that they are **committed to `skills/` only** and are **absent from `skills.json` and from every agent override** — because this repo does not consume them. `atomic-design` is the same class (published for external frontend teams, not invoked by nightshift's SDLC/GTM agents).

**Required registration actions:**

| Action | Required? | Rationale |
| --- | --- | --- |
| Create `skills/atomic-design/SKILL.md` (+ optional `references/`) | **Yes** | The deliverable (AC-1, AC-5). |
| Add to root `skills-lock.json` | **No** | `skills-lock.json` pins **CLI-installed external** skills under `.agents/skills/` only. Workspace-authored skills are never locked (`hono-api`/`typeorm`/`electrodb` are absent from it). Hand-editing the lockfile is explicitly forbidden. |
| Add a `.claude/project/skills.json` entry | **No (default)** | Reserved for skills this repo consumes/binds. Tier-3 published-for-others skills are not tracked there. See Open Questions if the team decides to also *use* it locally. |
| Add to an agent override's skill list | **No (default)** | Same rationale — binding implies local consumption. |
| Create a `.claude/skills/atomic-design` symlink | **No** | Symlinks are the discovery surface for locally-consumed skills; not needed for a published-only artifact. |
| Add to `plugins/sdlc/refs/skills-map.yml` | **Optional (see Open Questions)** | That map drives `/init` stack-based suggestions. Adding an entry would let other repos discover/install `atomic-design`; not required to satisfy any AC. |

Net: the **only mandatory registration is the committed `skills/atomic-design/` directory itself.** Everything else is intentionally omitted to match the published-for-others precedent.

## Authoring Method

The ai-enablement-engineer authors this via its plugin-bundled `skill-creator` skill (frontmatter + concise body conventions) and MUST mirror the structure/tone of existing tier-3 skills (`hono-api`, `typeorm`, `electrodb`) for consistency. Source material is Brad Frost's guide (table of contents + chapters 1–5, linked in the story) — content is **synthesised, not copied verbatim**, to avoid reproducing the source text.

## Error Handling / Quality Gates

No runtime error handling (static Markdown artifact). The equivalent "failure modes" are authoring-quality gates the implementer must satisfy before the PR is ready:

| Gate | Pass condition |
| --- | --- |
| Frontmatter valid | Exactly `name` + `description`; `name` == `atomic-design` == dir name; `description` ≤ 1024 chars. |
| Hierarchy completeness | All five levels present, each with definition + boundary rule + neutral example (AC-2). |
| Rationale + usage present | Sections for "why" (AC-3) and "how to apply / decompose" (AC-6) both exist. |
| Framework-agnostic scan | Zero prescriptive framework/library tokens (see Framework-Agnostic Requirements) (AC-4). |
| Placement correct | File at `skills/atomic-design/SKILL.md`, real directory, not a symlink. |
| No source plagiarism | Content synthesised, not pasted from bradfrost.com. |

## Out of Scope (from the story, restated)

- Framework-specific implementation guides (React/Vue/mobile atomic-design patterns).
- Retrofitting or reclassifying any existing components in this repo to the new skill.
- Design tooling or Figma library changes.
- Adding the skill to `skills-lock.json`, or building any publishing pipeline beyond the git commit.
- Binding the skill to any nightshift agent as a consumed skill (it is published-for-others).
