# Full FAQ Page — Technical Spec

**Story:** NA-38 — Build the full FAQ page
**Epic:** NA-29 — Nightshift marketing site — design-handoff build
**Feature/handoff:** `docs/design/marketing-site-handoff/designs/faq.dc.html` (authoritative layout) · screenshots `docs/design/marketing-site-handoff/screenshots/0{1,2,3}-faq.png`
**Date:** 2026-07-12

## Overview

Build the standalone `/faq` page in `apps/marketing`: all twelve CMS FAQ entries rendered as five topic-grouped, solid-card accordions, numbered continuously Q.01–Q.12, with a single answer open at a time across the whole page and an install CTA at the bottom. Content is sourced entirely from the existing Payload `faq` collection — nothing is hard-coded.

## Scope & Agents

Single-agent story. **web-engineer** owns everything here (`apps/marketing/`). No database, backend-infra, or sync work. Data Model and Offline Sync sections are stated only to record that they are unaffected.

## Data Model

**No schema changes.** This story consumes the existing `faq` collection (created in NA-31, `apps/marketing/src/collections/Faq.ts`) read-only. No migration, no new fields, no collection edits.

Fields consumed by this page (all already present):

| Field      | Type                    | Used for                                                            |
| ---------- | ----------------------- | ------------------------------------------------------------------- |
| `id`       | number                  | React key / accordion open-key                                      |
| `question` | text                    | Accordion row heading                                               |
| `answer`   | richText (Lexical JSON) | Accordion answer body (the full-page answer — **not** `homeAnswer`) |
| `group`    | select (5 values)       | Grouping + eyebrow label                                            |
| `faqOrder` | number                  | Global ordering and continuous Q.NN number (1–12)                   |

`group` select values → eyebrow labels (already the option labels in the collection, text-identical to the handoff):

| `group` value      | Eyebrow label           | Handoff item count |
| ------------------ | ----------------------- | ------------------ |
| `positioning`      | `// positioning`        | 2 (Q.01–Q.02)      |
| `workflow-control` | `// workflow & control` | 4 (Q.03–Q.06)      |
| `setup-stack`      | `// setup & stack`      | 2 (Q.07–Q.08)      |
| `trust-quality`    | `// trust & quality`    | 2 (Q.09–Q.10)      |
| `cost-license`     | `// cost & license`     | 2 (Q.11–Q.12)      |

`faqOrder` is globally continuous (1–12) and each group occupies a contiguous block, so the continuous Q.NN number equals `faqOrder`, and grouping the `faqOrder`-sorted list by first-appearance of `group` yields the handoff group order without a hand-maintained list.

### Relationships

None. `faq` is a standalone (non-tenant) Payload collection; no parent/child relations are read on this page.

## API Surface

**No new HTTP endpoints.** All reads use the **Payload Local API** (`getPayload({ config }).find(...)`) inside the page's server component — the same in-process pattern as `getHomeFaqs` (`apps/marketing/src/lib/faq.ts`) and the why-sdlc reads. No REST/GraphQL route is added or called from the browser.

### Data access (Payload Local API)

| Operation          | Collection | Access | Where                  | Sort       | Limit |
| ------------------ | ---------- | ------ | ---------------------- | ---------- | ----- |
| List all page FAQs | `faq`      | public | _(none — all entries)_ | `faqOrder` | `100` |

`depth: 0` (no relationship population needed). `limit: 100` is a safe ceiling over the 12 seeded entries; adding future entries requires no code change.

### Data hook shape

Extend `apps/marketing/src/lib/faq.ts` with a grouped reader. Fully-typed, no `any`:

```typescript
import type { Faq } from '../payload-types';

/** Union of the collection's five group values (from payload-types). */
type FaqGroup = Faq['group'];

export interface FaqPageItem {
  id: Faq['id'];
  question: Faq['question'];
  answer: Faq['answer']; // Lexical richText JSON — server-rendered via <RichText/>
  /** Continuous 1-based page number for the Q.NN label (equals faqOrder). */
  number: number;
}

export interface FaqPageGroup {
  key: FaqGroup;
  eyebrow: string; // e.g. '// positioning'
  items: FaqPageItem[];
}

/**
 * All FAQs for the standalone /faq page, ordered by `faqOrder`, partitioned
 * into the five topic groups in first-appearance order. Uses the full `answer`
 * field (not `homeAnswer`). Returns [] on any Payload/DB failure — same
 * graceful-empty convention as getHomeFaqs.
 */
export async function getFaqPageGroups(): Promise<FaqPageGroup[]>;
```

The eyebrow label is derived from a single fixed `Record<FaqGroup, string>` map (source of truth: the collection select labels), never re-typed inline per group.

## Web UI

**Route:** `/faq` (already exists as a placeholder that must be replaced).

**Pages:**

| Route  | File                                             | Description                                               |
| ------ | ------------------------------------------------ | --------------------------------------------------------- |
| `/faq` | `apps/marketing/src/app/(frontend)/faq/page.tsx` | Async server component — replaces the current placeholder |

`page.tsx` becomes an `async` server component that: awaits `getFaqPageGroups()`, renders `FaqHero` → `FaqGroups` → `FaqCta`, and exports a Next `metadata` object (see below). Rendering mode matches the existing home/why-sdlc frontend pages (server component, build-time data fetch via Payload Local API against Neon).

**Components** (new, under `apps/marketing/src/components/faq/`):

| Component          | File                         | Kind                    | Responsibility                                                                                                                                            |
| ------------------ | ---------------------------- | ----------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `FaqHero`          | `faq/faq-hero.tsx`           | server                  | Hero: breadcrumb (Home / FAQ), `Eyebrow`, h1 "Everything builders ask", subhead, terra glow — per handoff hero section.                                   |
| `FaqGroups`        | `faq/faq-groups.tsx`         | server                  | Maps `FaqPageGroup[]` → per-item `answer: <RichText data={...} disableContainer/>` (server-side Lexical→JSX), passes flat groups to the client accordion. |
| `FullFaqAccordion` | `faq/full-faq-accordion.tsx` | client (`'use client'`) | Renders the five solid-card groups; manages **one** open key across the entire page; renders each row with the shared row visual/animation.               |
| `FaqCta`           | `faq/faq-cta.tsx`            | server                  | Bottom install CTA: h2 "Question answered? Put a ticket in tonight.", two `InstallSnippet`s, GitHub star `CtaButton`, back-to-overview link, terra glow.  |

**Reuse:**

- `Eyebrow`, `InstallSnippet`, `CtaButton` from `@nightshift-ai/ui` (already exported).
- `RichText` from `@payloadcms/richtext-lexical/react` — server-rendered, exactly as in `faq-preview.tsx` / `why-sdlc/page-faq.tsx`.
- The **row visual + measured-max-height animation contract** already implemented as `FaqRow` inside `home/faq-accordion.tsx` (Q.NN mono label in `--accent`, 18px question, `+`/`−` glyph, `overflow-hidden` + measured `scrollHeight` max-height transition, `motion-reduce:transition-none`). `FullFaqAccordion` must present the same row contract. Whether to extract `FaqRow` into a shared module and import it, or reproduce it, is an implementation choice — but the two accordions must not visually diverge.

**Single-open-across-page behaviour (AC2):** The open state is a single value held by `FullFaqAccordion` spanning all five groups (e.g. the item `id` or continuous `number`, `null` = all closed). Opening any row sets that value; opening another replaces it; clicking the open row clears it. This differs from `home/FaqAccordion`, whose open index is scoped to its single card — do **not** give each group card its own independent open state.

**Continuous numbering (AC1):** Q.NN label uses `item.number` (= `faqOrder`), not the row's index within its group — so numbering runs Q.01→Q.12 unbroken across group boundaries.

**Data hooks:** `getFaqPageGroups()` (new, in `lib/faq.ts`) — the only data read.

**State:** No global/client store. Local `useState` open-key inside `FullFaqAccordion` only. The server components carry no client state.

**Route permissions:** Public route, no guard (marketing site has no auth-gated frontend routes).

**Page metadata (in scope, minimal):** `page.tsx` exports a static Next `metadata` object with `title` and `description` for the FAQ page. Values are plain copy consistent with the hero ("Everything builders ask" / the grouped-FAQ subhead) — no external data dependency. Structured data (FAQPage JSON-LD) is **out of scope** (see below).

## Error Handling

| Scenario                                        | Behaviour                                                                                                                                                                                                             | Result                         |
| ----------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------ |
| Payload/DB read fails in `getFaqPageGroups`     | Catch, `console.error('[faq]', error)`, return `[]` (same convention as `getHomeFaqs`)                                                                                                                                | Groups section renders nothing |
| `getFaqPageGroups()` returns `[]`               | `FaqGroups` returns `null` (mirror `PageFaq`'s empty guard); hero + CTA still render                                                                                                                                  | Page never throws / never 404s |
| A group's `items` is empty (no entry for group) | That group card is skipped (not rendered as an empty card)                                                                                                                                                            | No empty solid card            |
| A row answer richText is empty/malformed        | `RichText` renders nothing for the body; row heading + toggle still work                                                                                                                                              | Row remains operable           |
| `prefers-reduced-motion: reduce` (AC5)          | No JS branch needed: the site-wide `global.css` reduce guard zeroes `transition-duration !important`, and rows carry `motion-reduce:transition-none`; open/close is instant, content stays fully legible and operable | Instant, accessible toggle     |

**No offline behaviour** — no offline writes exist.

- Adoption analytics / instrumentation for install-copy or star click-through (explicitly out of scope per the Epic).
- Search / filter over questions, deep-linking to a specific open question, expand-all — the handoff shows plain grouped accordions only.
