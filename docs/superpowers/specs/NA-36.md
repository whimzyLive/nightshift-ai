# Build the Why SDLC page — Technical Spec

**Story:** NA-36 — Build the Why SDLC page
**Feature:** No PRD/feature doc; product context is the NA-36 Jira story (self-contained, no parent Epic) plus its dependencies NA-30 (site chrome + tokens) and NA-31 (CMS FAQ + editorial why-copy content layer).
**Date:** 2026-07-12

## Overview

Build the standalone `/why-sdlc` editorial page: a hero, five argument sections wired to a scroll-lit gate rail beside a sticky crossfading visual pane, a 2-question CMS-sourced page FAQ, and a page CTA. All editorial copy is read from the Payload content layer already shipped by NA-31; this story is the render/interaction wiring only.

## Scope & Layers

Web-only story — sole active agent: **web-engineer** (`apps/marketing/`, `packages/ui/`). No database, backend-service, mobile, or offline-sync work. The Data Model and CMS schema already exist (NA-31); this story adds **no** schema changes.

## Data Model

**No new or modified entities.** The content this page renders already exists from NA-31 and is registered in `apps/marketing/src/payload.config.ts`:

- **`whySdlc` global** (`apps/marketing/src/globals/WhySdlc.ts`) — `intro` group (`eyebrow`, `heading`, `lead` richText, `scrollHint`) + `arguments` array (exactly 5 rows: `eyebrow`, `heading`, `body` richText). Seeded by `whySdlcSeedData` in `apps/marketing/src/seed/data.ts`.
- **`faq` collection** (`apps/marketing/src/collections/Faq.ts`) — the two page-FAQ rows are the entries with `showOnWhySdlc = true`, ordered by `whySdlcOrder` (1, 2), each carrying a `whySdlcAnswer` richText override (falls back to `answer`). Seeded in the same file.

Generated types (`apps/marketing/src/payload-types.ts`) already expose `WhySdlc` and `Faq` — consume these; never edit the generated file.

### Relationships

- `whySdlc` global (1) → (5) embedded `arguments` rows (array field; fixed `minRows: 5, maxRows: 5`).
- `faq` collection (many) → filtered to (2) rows where `showOnWhySdlc = true`.

No cross-tenant scoping applies — this is a single-tenant public marketing site.

## API Surface

**No new HTTP/REST endpoints.** Content is read server-side through the Payload **local API** (`getPayload({ config })`) at the page's top-level async boundary, mirroring the existing `getHomeFaqs()` reader in `apps/marketing/src/lib/faq.ts`. Two new reader functions are added in a new module `apps/marketing/src/lib/why-sdlc.ts`:

| Reader                | Payload call                                                                                                                | Returns                  |
| --------------------- | --------------------------------------------------------------------------------------------------------------------------- | ------------------------ |
| `getWhySdlcContent()` | `payload.findGlobal({ slug: 'whySdlc', depth: 0 })`                                                                         | `WhySdlcContent \| null` |
| `getWhySdlcFaqs()`    | `payload.find({ collection: 'faq', where: { showOnWhySdlc: { equals: true } }, sort: 'whySdlcOrder', limit: 2, depth: 0 })` | `WhySdlcFaqItem[]`       |

Both follow the NA-16 / `faq.ts` resilience convention: wrap in `try/catch`, log with a namespaced tag, and return the empty sentinel (`null` / `[]`) on any Payload/DB failure so a content-layer outage degrades the section rather than throwing the route.

### Request / Response shapes

```typescript
import type { Faq, WhySdlc } from '../payload-types';

// One argument section (verbatim shape of a whySdlc.arguments row).
export interface WhySdlcArgument {
  eyebrow: WhySdlc['arguments'][number]['eyebrow'];
  heading: WhySdlc['arguments'][number]['heading'];
  body: WhySdlc['arguments'][number]['body']; // Lexical SerializedEditorState — pass to <RichText>
}

// The hero intro block.
export interface WhySdlcIntro {
  eyebrow: WhySdlc['intro']['eyebrow'];
  heading: WhySdlc['intro']['heading'];
  lead: WhySdlc['intro']['lead']; // richText
  scrollHint: WhySdlc['intro']['scrollHint'];
}

export interface WhySdlcContent {
  intro: WhySdlcIntro;
  arguments: WhySdlcArgument[]; // always length 5 when content is seeded
}

// One page-FAQ entry (whySdlcAnswer wins over answer, mirroring getHomeFaqs).
export interface WhySdlcFaqItem {
  id: Faq['id'];
  question: Faq['question'];
  answer: Faq['answer']; // resolved: whySdlcAnswer ?? answer
}
```

All fields are typed from the generated `payload-types`. No `any`.

**Reused UI (from `@nightshift-ai/ui`, no changes):** `Eyebrow`, `CtaButton`, `InstallSnippet`. RichText via `@payloadcms/richtext-lexical/react` (server-side, `disableContainer`) exactly as `faq-preview.tsx` uses it.

**Static illustration content:** the five right-pane mono illustrations, the gate-rail glyph logic, the five captions (`what one-shot abstraction hands you`, `how software teams already ship`, `five gates — control returns at each`, `one opaque blob vs five reviewable units`, `the same principles, one verb at a time`), and both CTA-kicker strings are static in code (they are UI, not editable copy) — transcribed verbatim from the design handoff `docs/design/marketing-site-handoff/designs/why-sdlc.dc.html`. Only the hero intro, the five argument card bodies/headings/eyebrows, and the two FAQ Q&As come from the CMS.

**Scroll-progress rules (from the handoff `renderVals`/`_setup`, `data-props` script):**

- A section counts as **passed** when its bounding-rect `top` crosses `0.7 × window.innerHeight`; the last section (index 4) sets `reached = 5`. `reached` is monotonic (only increases) so fast/jump scrolls never un-light a passed node.
- The **active** section (drives the crossfade + caption) is the highest-indexed section whose `top` crosses `0.5 × window.innerHeight`.
- Gate node states: `passed` (`✓`, terra, faint glow), `current` (`⊘`, terra, pulsing `ns-gatepulse`), `idle` (`⊘`, dim). Visual layers fade with `opacity` + a small `translateY` offset keyed to `active`.

**State:** local React state inside the `scroll-progress` client island only (`{ reached, active }` via `useState` + one effect wiring `scroll`/`resize`, cleaned up on unmount). No global/store changes; the rest of the page is server-rendered.

**Route permissions:** `/why-sdlc` is a public route, consistent with the other marketing pages — no gating.

## Error Handling

| Scenario                                                  | Behaviour                                                                                                     | Result                                                                                                                                                                    |
| --------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `getWhySdlcContent()` throws (DB/Payload down)            | Catch, log `['why-sdlc', error]`, return `null`                                                               | Hero + argument sections render nothing (each component guards on nullish content and returns `null`); page shell (nav/footer/CTA) still renders. No route throw, no 500. |
| `whySdlc` global has content but `arguments.length !== 5` | Render whatever rows exist; gate rail renders one node per present section                                    | Defensive — seed guarantees 5, but the rail must not assume exactly 5.                                                                                                    |
| `getWhySdlcFaqs()` returns `[]`                           | `page-faq.tsx` returns `null`                                                                                 | FAQ section omitted; rest of page unaffected (same convention as `FaqPreview`).                                                                                           |
| `prefers-reduced-motion: reduce`                          | Gate-rail pulse, visual crossfade, and CTA-kicker transition are disabled/instant; `reached` initialises to 5 | All five argument cards, both visuals' text, the FAQ, and the CTA remain fully reachable and legible (AC6).                                                               |

No form submissions or client writes on this page → no offline/queue behaviour.
