**Story:** NA-52 — Add /sdlc:docs command + sync mode

## Overview

Ship the `/sdlc:docs` command shell, `refs/docs-pipeline.md` (mirroring `refs/adr-pipeline.md`'s shape), and the first mode — `sync <STORY-KEY>` — which diff-drives deterministic regeneration of frontmatter-driven reference docs plus `llms.txt`, and drafts gated how-to refreshes for the narrative rows a diff touches. This is the deterministic foundation the other four `/sdlc:docs` modes (release / seed / audit / distill) build on.

## Command Contract — `/sdlc:docs`

### Mode table (v1 ships only `sync`)

| Invocation                     | Mode    | v1 status                                                                                  |
| ------------------------------ | ------- | ------------------------------------------------------------------------------------------ |
| `/sdlc:docs sync <STORY-KEY>`  | **sync** | **In scope — this story**                                                                  |
| `/sdlc:docs release <version>` | release | Out of scope (NA-53) — the command surfaces a "not yet implemented" stub, not an error     |
| `/sdlc:docs seed <type> [topic]` | seed  | Out of scope (NA-54/55) — stub                                                             |
| `/sdlc:docs audit [--dry-run]` | audit   | Out of scope (NA-54/55) — stub                                                             |
| `/sdlc:docs distill`           | distill | Out of scope (NA-56) — stub                                                                |

### Argument validation

Parse `$ARGUMENTS` into `<mode>` (first token) and the mode's remaining args. Apply, in order:

1. **Empty `$ARGUMENTS`** → STOP with the usage message:
   `usage: /sdlc:docs sync <STORY-KEY>  (release|seed|audit|distill land in later stories)`.
2. **Unrecognised first token** (not one of `sync`/`release`/`seed`/`audit`/`distill`) → STOP with the same usage message.
3. **Recognised future-mode token** (`release`/`seed`/`audit`/`distill`) → print `mode "<mode>" is not yet implemented (see Epic NA-50)` and exit cleanly. Not an error, not a STOP — a deliberate stub so the surface is stable before the modes land.
4. **`sync` with a missing or malformed story key** — the story key must match `^[A-Z][A-Z0-9]*-[0-9]+$` (e.g. `NA-52`). If the key is absent or fails the regex → STOP with the usage message. This is a **usage STOP** (a caller error), distinct from the manifest-absent silent no-op below.

### `sync <STORY-KEY>` — behavioural contract

1. **Manifest gate (AC5).** Resolve `.claude/project/docs-manifest.md`. If **absent** → **silent no-op**: no branch, no dispatch, no PR, no error, **no stdout** — exit cleanly (exit 0). This is the zero-setup-cost guarantee for repos that declined the `/init` docs opt-in, and is deliberately distinct from a usage STOP (which does print a message).

2. **Resolve the story branch (v1 diff source).** `sync` never depends on the currently-checked-out branch. It resolves the story's branch from the story key, `git fetch`ing first so remote refs are current:

   ```bash
   git fetch origin --quiet
   # Resolve the story branch: feat/<STORY-KEY> preferred, fix/<STORY-KEY> fallback.
   STORY_BRANCH=""
   for cand in "feat/<STORY-KEY>" "fix/<STORY-KEY>"; do
     git rev-parse --verify --quiet "origin/$cand" >/dev/null && { STORY_BRANCH="origin/$cand"; break; }
   done
   ```

   - **Neither `origin/feat/<STORY-KEY>` nor `origin/fix/<STORY-KEY>` exists** → this is a **post-merge (or never-branched) invocation**. Emit an explicit **WARNING** (never a silent clean exit that reads as success):
     `WARNING: no story branch (feat|fix)/<STORY-KEY> found on origin. v1 sync is branch-diff-only; post-merge sync (diffing the merged commit range) is deferred to NA-55. Nothing regenerated.` — then exit. The distinction matters: a genuine "docs already current" run is a clean no-op, whereas "I could not find the branch to diff" must be visible so the founder knows nothing ran.

3. **Compute the changed-file set AND the unified diff (v1, single source).** Diff the resolved story branch against `<BASE-BRANCH>` (from project-context), producing **both** artifacts the affected-row resolution needs — the name-only file list **and** the unified diff content for those files:

   ```bash
   CHANGED_FILES=$(git diff --name-only "<BASE-BRANCH>...$STORY_BRANCH")     # path-keyed rows
   CHANGED_DIFF=$(git diff "<BASE-BRANCH>...$STORY_BRANCH")                    # content-keyed rows
   ```

   Both are derived from the same `<BASE-BRANCH>...$STORY_BRANCH` three-dot range (changes on the story branch since it diverged from base). Dual diff-source selection (story-branch-vs-develop **vs** merged-commit) is **out of scope** here — see Out of Scope.

4. **Affected-row resolution.** For each **enabled** manifest row, look up its registry row in `refs/doc-types.md` and keep only rows whose `trigger` contains `sync` (v1: the seven `auto` reference/meta rows + the `how-to` row). A row is **affected** per its keying kind (see the source-of-truth map): a **path-keyed** row is affected when `CHANGED_FILES` intersects its source glob; a **content-keyed** row (`hooks-contract`, `error-reference`) is affected when a name-matched file's **hunks in `CHANGED_DIFF`** touch the relevant section. `llms-txt` is affected on **every** run (AC4).

5. **Deterministic regen (auto rows, no gate — AC3).** For each affected `auto` row, regenerate its reference-doc set into the row's `target-path` deterministically from its `source-of-truth` frontmatter — idempotent (re-running with no source change yields a byte-identical tree). `llms-txt` regenerates every run.

6. **Narrative drafts (draft-for-review rows, gated — AC3).** For each affected `how-to` row, draft a **refresh** of the how-to page(s) whose `source:` frontmatter intersects `CHANGED_FILES` (see the `source:` convention below), using the `writing-docs` skill. These drafts are **held at the founder-confirm gate** at the command layer — never written un-confirmed.

7. **Founder-confirm gate (command layer).** Present the deterministic regen summary (informational — auto rows are not gated) and each narrative draft (gated). The founder may accept/edit/reject each narrative draft. The gate is **skipped** when there are zero narrative drafts — the run proceeds straight to the write phase.

8. **Write + PR only on change (AC6).** On a **branch cut from the story branch head** (see Branch / PR / Control Flow — the tree must contain the changed source so regen reads it), write the deterministic regen + `llms.txt` + founder-confirmed narrative drafts. If, after writing, `git status --porcelain` on the target paths is **empty** (deterministic output was byte-identical and no narrative draft confirmed) → no commit, no PR, exit cleanly. Otherwise commit and **open or update** the sync PR.

### Source-of-truth map (diff → affected doc-type)

`sync` derives "affected" from this map (the registry's `source-of-truth` column made concrete against the SDLC plugin layout; a consumer repo's equivalent globs resolve from its own project structure). The **keying** column states whether a row is matched against the name-only `CHANGED_FILES` list or against hunks in the unified `CHANGED_DIFF`:

| Doc-type            | `generation-mode` | Keying       | Affected when…                                                                                         |
| ------------------- | ----------------- | ------------ | ------------------------------------------------------------------------------------------------------ |
| `command-reference` | auto              | path         | `CHANGED_FILES` contains any `commands/**` file                                                        |
| `agent-reference`   | auto              | path         | `CHANGED_FILES` contains any `agents/**` file                                                          |
| `skill-reference`   | auto              | path         | `CHANGED_FILES` contains any `**/SKILL.md`                                                              |
| `config-reference`  | auto              | path         | `CHANGED_FILES` contains `.claude/project/project-context.md` or a config template under `refs/*-template.md` |
| `hooks-contract`    | auto              | content      | a name-matched `.claude/settings*.json` or referenced hook script has hunks in `CHANGED_DIFF` touching a hooks block |
| `error-reference`   | auto              | content      | a name-matched command/agent/ref file has hunks in `CHANGED_DIFF` touching an "Error Handling" / error-table section |
| `llms-txt`          | auto              | always       | **every run** (AC4)                                                                                    |
| `how-to`            | draft-for-review  | path         | `CHANGED_FILES` intersects an existing how-to page's `source:` frontmatter glob-list (see below)       |

### `source:` frontmatter convention (normative)

The how-to refresh opt-in is governed by a `source:` key in each how-to page's frontmatter. This spec defines it normatively; the implementation task extends `writing-docs`'s how-to template to emit it.

- **Key name:** `source` (top-level YAML frontmatter key on a how-to page).
- **Value:** a YAML list of repo-root-relative **glob** strings identifying the source files whose change should trigger a refresh draft of this page. Example:

  ```yaml
  ---
  title: How to configure the sync review loop
  source:
    - plugins/sdlc/commands/loop.md
    - plugins/sdlc/scripts/pr-loop-status.sh
  ---
  ```

- **Match semantics:** a how-to page is **affected** iff at least one entry in its `source:` list matches at least one path in `CHANGED_FILES` under standard glob semantics (`*` within a path segment, `**` across segments). Matching is against the name-only changed-file set — it is a path-keyed row.
- **Absent key:** a how-to page with **no** `source:` frontmatter is **never** auto-refreshed by `sync`. This is the deliberate opt-in boundary — authoring/opting a page in is a founder action (add `source:`), not something `sync` infers.
- **Implementation task:** extend the how-to structure template in `plugins/sdlc/skills/writing-docs/SKILL.md` so newly authored how-to pages emit a `source:` frontmatter block (with a one-line comment explaining it is the `/sdlc:docs sync` refresh opt-in). This keeps the convention discoverable at authoring time rather than buried in the pipeline ref.

### llms.txt format (v1 decision)

`sync` regenerates `llms.txt` at the `llms-txt` row's `target-path` (registry default `llms.txt`, repo-root) deterministically on every run: an **index-only** file (no full-content `llms-full.txt` in v1), grouping the **generated pages of every enabled, `public: yes` manifest row** by Diátaxis quadrant, each entry a `title — one-line description — relative link` line derived from the generated page's frontmatter. Idempotent, no narrative synthesis. (Resolves the Epic's llms.txt open question for v1 in the narrowest deterministic form; `llms-full.txt` remains a later enhancement.)

#### llms-txt registry alignment (required)

`refs/doc-types.md`'s `llms-txt` row currently reads `source-of-truth = aggregated command/agent/skill frontmatter`, which describes only a subset of what this spec generates from (all enabled `public: yes` rows, not just command/agent/skill). The registry self-check in `doc-types.md` forbids any section — or the manifest template — from diverging from the registry. **Implementation task:** update the `llms-txt` row's `source-of-truth` cell to `generated pages of all enabled public:yes manifest rows` so the registry and this spec agree. This is a one-cell edit within the existing registry table; no schema change, no other row touched.

## Pipeline Contract — `refs/docs-pipeline.md`

`docs-pipeline.md` is authored to **mirror `refs/adr-pipeline.md`'s shape** (copy the skeleton; do not abstract a shared ref — same "copy the shape, do not generalize" rule `doc-types.md` and the Epic both state). It is referenced by both `commands/docs.md` and `agents/knowledge-engineer.md` so the contract lives in exactly one place. Required sections:

1. **Purpose + ownership note** — single-source pipeline for `/sdlc:docs`; all writes land under paths resolved from the consumer repo's manifest/project-context; a plugin-authoring write (touching `plugins/**` in this SDLC repo) stays within the `ai-enablement-engineer` write-scope.
2. **Two-phase dispatch across the confirmation boundary** — a dispatched subagent cannot pause for interactive input, so the founder-confirm gate lives at the command layer between two `knowledge-engineer` dispatches, identical in spirit to `adr-pipeline.md` §2 and `/sdlc:analyze`'s scan-then-apply split:
   - **Phase 1 (compute & draft, writes nothing):** resolve manifest → resolve story branch → compute `CHANGED_FILES` + `CHANGED_DIFF` → affected rows; produce the deterministic regen content for auto rows + `llms.txt`; draft narrative (how-to) refreshes via `writing-docs`; return the regen summary, the deterministic content, and the narrative drafts to the command layer.
   - **Founder-confirm gate (command layer):** present the informational auto-regen summary and each gated narrative draft; founder accepts/edits/rejects narrative drafts. Skipped when zero narrative drafts.
   - **Phase 2 (write confirmed, fresh dispatch):** the command hands phase 2 the deterministic content **and** the founder-confirmed narrative drafts **verbatim** (inline or via session temp-dir files by path, per `scripts/tmp-dir.sh` — the same payload-contract discipline as `adr-pipeline.md` §2). Phase 2 checks out a branch cut from the story branch head (so the tree contains the changed source), writes everything, then commit/push/open-or-update the PR — but only if content changed (AC6).
3. **Deterministic regen algorithm** — the reference-doc and `llms.txt` regeneration is a **prose algorithm executed inline, not a committed script** (matches the ADR index-regen algorithm in `adr-pipeline.md` §10 and the plugin's "instructions not code" style). Idempotent: re-running with no source change → byte-identical output.
4. **Voice/format resolution** — narrative drafting resolves voice + output format via `writing-docs`'s chain: `.claude/project/docs-manifest.md` "Voice & format" section → `.claude/project/project-context.md` → stated default. Never hardcoded.
5. **`source:` refresh convention** — restate the normative `source:` frontmatter rule (key, glob-list value, match semantics, absent-key = never-refreshed) so pipeline readers see it alongside the drafting flow.
6. **No-op / change-gate semantics** — manifest-absent silent no-op (AC5); story-branch-missing WARNING (not a silent success); commit/PR only on actual content change (AC6); `llms.txt` regenerated every run but committed only if changed (AC4 + AC6).
7. **Branch / PR naming + control flow** — see below.

## Agent Extension — `knowledge-engineer.md`

The existing `knowledge-engineer` agent (owner of the knowledge lifecycle) gains the docs-sync dispatch role. Modifications:

- **Frontmatter `description`** — extend to state it also runs the `/sdlc:docs` sync pipeline (currently reads "Triggered manually via /sdlc:adr").
- **Role & scope** — add: in a docs-sync dispatch it regenerates frontmatter-driven reference docs + `llms.txt` deterministically and drafts gated how-to refreshes, writing under the manifest's resolved `target-path`s (and, when authoring plugin changes in this SDLC repo, only within the `ai-enablement-engineer`-owned surface — the existing Active-guard scope note already covers this).
- **Pipeline pointer** — point at `refs/docs-pipeline.md` as the single source for the docs-sync procedure (as it already points at `refs/adr-pipeline.md` for ADRs). Do not re-inline.
- **Required-skills conditional on dispatch type.** The agent's required-skills list is **branched by which pipeline the dispatch runs**, so an ADR run never loads doc-authoring skills and a docs-sync run never loads ADR-authoring skills:
  - **ADR dispatch** (seed/distill via `/sdlc:adr`) → `writing-adrs`, plus the always-on `verification-before-completion`, `gh-cli`, `conventional-commit`.
  - **docs-sync dispatch** (via `/sdlc:docs sync`) → `writing-docs`, plus `conventional-commit` and `gh-cli` (`verification-before-completion` remains applicable for the change-gate / idempotence check).
  No new tool grants are required — `sync` needs `Read`/`Write`/`Edit`/`Bash`/`Skill`, all already in the agent's `tools:` allowlist. The claude-mem MCP tools are not used by `sync`.
- **`Skills loaded:` return line — aligned with the branch above.** An ADR dispatch lists the ADR set; a docs-sync dispatch lists `writing-docs` (when a narrative draft was produced), `conventional-commit`, `gh-cli`, `verification-before-completion` — or the literal `none`. A docs-sync return must **not** list `writing-adrs`, and an ADR return must not list `writing-docs`.

## Branch / PR / Control Flow

| Item              | Value                                                                                                                                                                                                                                                                                          |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Branch            | `docs/sync-<STORY-KEY>`, cut from the **story branch head** (`origin/feat/<STORY-KEY>`, fallback `origin/fix/<STORY-KEY>`) — **not** `<BASE-BRANCH>`. The story branch carries the changed source the deterministic regen must read; branching off base would regenerate from a tree missing those changes. |
| Commit            | `docs(docs): sync <STORY-KEY> reference docs` (via `conventional-commit`)                                                                                                                                                                                                                        |
| PR title          | `docs(docs): sync <STORY-KEY>`                                                                                                                                                                                                                                                                   |
| PR base           | `<BASE-BRANCH>` from project-context (never assume `main`)                                                                                                                                                                                                                                       |
| First-run PR      | Create the branch, commit, push, open the PR (`gh pr create`) against `<BASE-BRANCH>`.                                                                                                                                                                                                           |
| Re-run behaviour  | **Open or update** (AC6): before creating, check whether `docs/sync-<STORY-KEY>` already exists on `origin` (`git rev-parse --verify origin/docs/sync-<STORY-KEY>` / `gh pr list --head docs/sync-<STORY-KEY>`). If it does → check it out, **`git reset --hard` onto the freshly regenerated state** (the branch content is fully derived, so a hard reset is safe and keeps history clean), then **`git push --force-with-lease`** to update the existing open PR — never open a duplicate PR. |
| Control-flow tail | Mirror `commands/adr.md`: after the PR is raised, drive the review loop to convergence via `/loop /sdlc:loop <PR_URL>` (falling back to `ScheduleWakeup` if the harness cannot nest `/loop`). If the run ended **before** any PR, release directly via `scripts/session-complete.sh` — this covers the manifest-absent silent no-op, the story-branch-missing WARNING, a usage STOP, and a clean "nothing changed" no-op alike (all four release without a PR; only the manifest-absent path is silent). |

## Permissions Detail

Not a multi-role RBAC surface. The single actor is the **founder** running `/sdlc:docs`. The one authority boundary is the **founder-confirm gate**: auto (deterministic) doc types are written without confirmation; narrative (how-to) drafts are written only after explicit founder confirmation at the command layer. A dispatched subagent never runs the gate itself (it cannot pause for input) — the command layer owns it, exactly as `commands/adr.md` does.

## Error Handling

| Scenario                                                                                        | Behaviour                                                                                                                             |
| ----------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------- |
| `.claude/project/docs-manifest.md` absent                                                       | **Silent** no-op — no branch, no dispatch, no PR, no error, **no stdout** (AC5). Distinct from a usage STOP, which prints.           |
| Empty `$ARGUMENTS` / unrecognised first token                                                   | Usage STOP (prints the usage message).                                                                                              |
| `sync` with missing/malformed story key (fails `^[A-Z][A-Z0-9]*-[0-9]+$`)                        | Usage STOP (prints the usage message).                                                                                              |
| Recognised future-mode token (`release`/`seed`/`audit`/`distill`)                               | Print "mode not yet implemented (see NA-50)" and exit cleanly — not an error.                                                       |
| `sync` but no `origin/feat/<STORY-KEY>` or `origin/fix/<STORY-KEY>` (post-merge / never-branched) | Emit the explicit WARNING (v1 is branch-diff-only; post-merge deferred to NA-55); exit **without** a silent success.                |
| `refs/doc-types.md` unreadable/malformed                                                         | Surface the failure and STOP — never regenerate from a partial registry.                                                            |
| Manifest present but no enabled `sync`-triggered row affected, and `llms.txt` unchanged          | Clean no-op — no commit, no PR (AC6).                                                                                               |
| Deterministic regen produced byte-identical output and no narrative draft confirmed              | No commit, no PR (AC6) — write phase detected an empty `git status --porcelain`.                                                    |
| Founder rejects all narrative drafts but deterministic content changed                           | Still commit + PR the deterministic regen (AC6 — content changed).                                                                  |
| Story branch diff yields no changed files                                                        | Only `llms.txt` is (re)generated; commit/PR only if it changed.                                                                     |
| `gh` / `raise-pr.sh` failure                                                                      | STOP and surface — the write is on a branch and reviewable; never leave a raised-but-broken state silently.                        |

## Out of Scope

- **release / seed / audit / distill modes** — own stories (NA-53 … NA-56). `docs.md` ships them as clean "not yet implemented" stubs, not errors.
- **Dual diff-source selection** (story-branch-vs-develop **vs** merged-commit) — added by the playbook-integration story (NA-55). v1 uses the single story-branch-vs-base source, and post-merge invocation emits a WARNING rather than diffing a merged commit range.
- **Authoring brand-new narrative pages** (new how-to / tutorial / concept) — that is `seed` mode. `sync` only refreshes **existing** how-to pages whose `source:` intersects the diff.
- **`llms-full.txt`** (full-content variant) — v1 ships index-only `llms.txt`.
- **Building or hosting a docs site** — Mintlify is this repo's choice; the plugin stays stack-agnostic via manifest `target-path`s (Epic out-of-scope).
- **Absorbing `/sdlc:adr` into `/sdlc:docs`** — NA-56. The standalone `/sdlc:adr` command and `refs/adr-pipeline.md` are untouched by this story (beyond `knowledge-engineer` gaining a second, branched dispatch path).
- **Auto-triggering on CI / tag events** — later enhancement (Epic out-of-scope).

Documented in both `docs-pipeline.md` and the `writing-docs` how-to template.

- [ ] **Future-mode stub vs hard error** — unimplemented modes print a "not yet implemented" notice and exit cleanly. _Suggested default (adopt):_ stub, not error — keeps the command surface stable as later stories fill it in, and avoids a scripted caller treating an expected gap as a failure.
