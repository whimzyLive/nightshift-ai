# Generic activation-gated reference-doc generation for /sdlc:docs — Implementation Plan

> **For agentic workers:** all tasks in this plan are owned by a single domain agent —
> `ai-enablement-engineer` (DEPENDENCY-FREE in the phase ladder). Phases are the spec's three
> strictly-ordered delivery phases (P1 → P2 → P3); do not reorder or parallelise. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-65.md](../specs/NA-65.md)
**Story:** [NA-65](https://whimzylive.atlassian.net/browse/NA-65) — Epic NA-50
**Date:** 2026-07-21
**Agents required:** `ai-enablement-engineer` (sole domain — all files under `plugins/sdlc/**` and `.claude/project/**`)

## Goal

Make the `/sdlc:docs` reference-doc pipeline repo-agnostic and activation-gated: strip every
Nightshift-specific string out of the shipped `sdlc` plugin refs, gate artifact-reference doc-types on
declared `reference-roots`, add a product-reference family (api/config/schema/cli/error) resolved by a
shared source resolver, and re-express Nightshift's own docs scope via its manifest so `sync` output is
unchanged for Nightshift.

## Architecture

The "generator" is prose executed inline by the `knowledge-engineer` agent — there is no runtime code,
data model, API, or UI. Deliverables are edits to plugin-owned prose refs (`plugins/sdlc/refs/*.md`),
Nightshift's `.claude/project/docs-manifest.md`, and dispatch-and-snapshot test fixtures. Work is
single-domain and single-branch: all commits land on the already-checked-out `feat/NA-65` branch.

## Global Constraints (copied verbatim from spec — apply to every task)

- **Zero Nightshift-specific strings** may remain in `plugins/sdlc/refs/doc-types.md` or
  `plugins/sdlc/refs/docs-pipeline.md` after Phase 1 — no `plugins/{sdlc,gtm}`, no `gtm`
  `marketing-context-template`, no nx-mirror-exclusion strings (`agents/`, `.opencode/**`, `.codex/**`,
  `.gemini/**`, `.github/**`). Grep gate in each phase's verify catches leaks.
- **Only `llms-txt` stays `applies-when: always`** among reference-family rows. Narrative/release rows
  are out of scope and remain `always`.
- **8-column registry schema is preserved** — `family` is descriptive prose, NOT a 9th column. The
  manifest's core 3-column identity is preserved; `source` / `contract` are two OPTIONAL trailing
  columns (legacy 3-column manifests stay valid).
- **Source resolver order is fixed:** contract → source → convention-scan → skip. Convention-scan
  honors the NA-64 transform-not-mirror rule (reference it, do not restate the body).
- **Fail-loud vs. skip:** an explicitly configured `contract:`/`source:`/root that is missing or
  malformed → FAIL LOUD at the confirm gate. An absent *conventional* path (nothing configured) →
  fall through to the next rung. Never emit an empty page or empty `llms.txt` entry for a skipped row.
- **Prettier fixed-point:** `plugins/sdlc/scripts/check-plugin-docs-format.sh` must pass over every
  edited plugin ref (NA-62 gate). Keep the large refs Prettier-stable.
- **Idempotence (AC3):** re-running the resolver against an unchanged contract/source/root set yields
  byte-identical pages.

## Execution order

All phases are sequential — one agent (`ai-enablement-engineer`), one at a time, on the shared
`feat/NA-65` branch. P2 depends on P1's activation model; P3 depends on P2's resolver. There is no DB,
backend, sync, web, or mobile phase — those agents are not involved in this story.

- **Phase 1** — de-leak + activation foundation (maps to AC1, AC2)
- **Phase 2** — product-reference framework + `api-reference` (maps to AC3)
- **Phase 3** — remaining product types + family-resolved `config-reference` (maps to AC4)
- **Phase 4** — cross-cutting fixture + idempotence gate (maps to AC5)
