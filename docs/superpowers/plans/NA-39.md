# Wire SEO meta, JSON-LD, and llms.txt — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-39.md](../specs/NA-39.md)
**Story:** [NA-39](https://whimzylive.atlassian.net/browse/NA-39) — Epic [NA-29](https://whimzylive.atlassian.net/browse/NA-29)
**Date:** 2026-07-12
**Agents required:** web-engineer (only)
**Estimated complexity:** Low (1 phase)

**Goal:** Wire the search/AI-discovery layer into the existing `apps/marketing` Next.js app — per-page meta/OG/Twitter tags, `@graph` JSON-LD on Home / Why SDLC / FAQ, and static `llms.txt` + `robots.txt` — every value sourced verbatim from `docs/gtm/site-brief.md`.

**Architecture:** A new `apps/marketing/src/lib/seo/` module holds frozen config constants, fully-typed schema.org node interfaces + static node constants, a Lexical→plaintext helper, per-page `Metadata` builders, and a tiny `<JsonLd>` server component. Three existing async Server routes export `metadata` and render `<JsonLd>` from their already-fetched FAQ data (no extra Payload fetch, single async boundary preserved). Two static text files land in `public/`.

**Tech Stack:** Next.js App Router (React Server Components), TypeScript (no `any`), Payload 3 (`@payloadcms/richtext-lexical/plaintext` `convertLexicalToPlaintext`, v3.85.2), Nx, pnpm, Tailwind v4 (no UI in this story).

## Global Constraints

- **Zero new SEO copy.** Every title / description / OG / JSON-LD / llms.txt / robots.txt value is verbatim from `docs/gtm/site-brief.md` (§3 JSON-LD, §4 meta/OG, §5 llms.txt) — copy exactly, author nothing.
- **No `any`.** All JSON-LD node shapes use the narrow local interfaces defined in the spec; no `schema-dts` dependency added.
- **Text-identity (AC4).** Each FAQPage `answer.text` MUST be produced by running `convertLexicalToPlaintext` on the **same** resolved Lexical `answer` value the page renders — never re-source answer text from the brief.
- **Absolute URLs (AC5).** Every `url` / `@id` / `item` / canonical / OG URL in every emitted node MUST be fully qualified. Canonical host = `https://github.com/whimzyLive/nightshift-ai`; OG images = `https://raw.githubusercontent.com/whimzyLive/nightshift-ai/main/og-image.png` and `…/og-why-sdlc.png` (brief §4).
- **No empty FAQPage node.** When a route's `faqs` array is empty, omit the FAQPage node entirely — never emit a node with an empty `mainEntity`.
- **No new data model / API / migration.** `faq` collection + `whySdlc` global consumed read-only via existing `lib/faq.ts` / `lib/why-sdlc.ts`.
- **Out of scope:** og-image asset generation; `/team` meta; `/faq` meta/OG copy changes (FAQ gains JSON-LD only, keeps NA-38 `metadata`); `sitemap.xml`; `WebSite` node; host-swap.
- **Verification project name:** nx project = `@nightshift-ai/marketing` (targets: `lint`, `test`, `build`).

## Execution order

Single sequential phase on the shared `feat/NA-39` branch — one agent (web-engineer). Within the phase, tasks are strictly ordered: SEO module foundations first (routes consume them), then static assets + layout base, then the three route wirings, then verification.

## Notes / assumptions (spec-gap fills)

- **FAQPage `@id` anchors:** each `buildFaqPageNode` call passes the absolute `…#faq` anchor from site-brief §3 for that surface; the implementer copies the exact `@id` strings from the brief per route (Home §3, Why SDLC §3 "New page" block, FAQ page §3).
- **llms.txt / robots.txt delivery = static `public/` files** (spec Open Question default) — content is fully static and `public/` serves at site root ahead of the `(payload)` admin catch-all.
- **`metadata` via static `const`** (not `generateMetadata`) — all Home/Why-SDLC meta values are static brief copy with no request/CMS dependency.
- Resolved-answer field names (`homeAnswer ?? answer`, `whySdlcAnswer ?? answer`) are taken from the spec's text-identity section; the implementer uses whatever the page already binds before `<RichText>` so schema text cannot drift from visible copy.
