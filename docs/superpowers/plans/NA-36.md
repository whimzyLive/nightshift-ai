# Build the Why SDLC page — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-36.md](../specs/NA-36.md)
**Date:** 2026-07-12
**Agents required:** web-engineer (only active agent for this story)
**Goal:** Render the standalone `/why-sdlc` editorial page — hero, five scroll-lit argument sections wired to a gate rail beside a sticky crossfading visual pane, a 2-question CMS FAQ, and a page CTA whose kicker turns green once all five gates pass.

**Architecture:** Two new CMS readers (`apps/marketing/src/lib/why-sdlc.ts`) fetch the pre-existing `whySdlc` global and filtered `faq` rows via the Payload local API at the page's single top-level async boundary. The page (`(frontend)/why-sdlc/page.tsx`) is an async server component that fans resolved data into server sections; scroll state lives in one `scroll-progress.tsx` client context island wrapping the argument rail, the (server-rendered) page FAQ passed as `children`, and the CTA kicker so the rail and CTA share one `{ reached, active }` source of truth.

**Tech Stack:** TypeScript, Next.js (App Router), Payload 3 (local API), React (server + client components), Tailwind v4 (CSS-first tokens), Nx, pnpm, Jest + RTL. UI is `.tsx` only.

## Global Constraints

- No schema, `payload.config.ts`, collection, global, migration, or seed changes — all content pre-exists from NA-31. This story is render/interaction wiring only.
- Consume generated types from `apps/marketing/src/payload-types.ts` (`WhySdlc`, `Faq`) — never edit the generated file. No `any`.
- Readers follow the NA-16 / `faq.ts` resilience convention: `try/catch`, log with a namespaced tag (`['why-sdlc', error]`), return the empty sentinel (`null` / `[]`) on any Payload/DB failure.
- Keep the Payload fetch at the single top-level async boundary in `page.tsx`; every child stays synchronous and RTL-testable (per NA-16 memory: a second nested async Server Component breaks RTL).
- Reuse `Eyebrow`, `CtaButton`, `InstallSnippet` from `@nightshift-ai/ui` unchanged. RichText via `@payloadcms/richtext-lexical/react` server-side with `disableContainer` (as `faq-preview.tsx` does).
- Static UI copy (five right-pane mono illustrations, gate-rail glyphs, five captions, both CTA-kicker strings) is transcribed verbatim from the design handoff `docs/design/marketing-site-handoff/designs/why-sdlc.dc.html` and co-located in code — it is UI, not editable CMS copy.
- Site chrome (starfield, nav with `active: 'Why SDLC'`, footer, cursor glow, tokens) comes from the NA-30 layout shell — do NOT render chrome in this page.
- `prefers-reduced-motion: reduce`: gate-rail pulse, visual crossfade, and CTA-kicker transition disabled/instant; `reached` initialises to 5 (AC6).
- Payload-mock test convention (from `faq.spec.ts`): `jest.mock('@payload-config', () => ({ default: {} }), { virtual: true })` and `jest.mock('payload', () => ({ getPayload: jest.fn() }))`.

### Task 1: CMS reader module + interfaces

**Files:**

- Create: `apps/marketing/src/lib/why-sdlc.ts`

**Interfaces:**

- Consumes: generated `WhySdlc`, `Faq` from `../payload-types`; `getPayload` from `payload`; `config` from `@payload-config`.
- Produces:
  - `interface WhySdlcArgument { eyebrow; heading; body }` (each field typed from `WhySdlc['arguments'][number]`)
  - `interface WhySdlcIntro { eyebrow; heading; lead; scrollHint }` (each from `WhySdlc['intro']`)
  - `interface WhySdlcContent { intro: WhySdlcIntro; arguments: WhySdlcArgument[] }`
  - `interface WhySdlcFaqItem { id: Faq['id']; question: Faq['question']; answer: Faq['answer'] }`
  - `async function getWhySdlcContent(): Promise<WhySdlcContent | null>`
  - `async function getWhySdlcFaqs(): Promise<WhySdlcFaqItem[]>`

- [ ] **Step 1: Write failing tests** — mirror `faq.spec.ts`. Mock `@payload-config` (virtual) and `payload`. Add a `makeWhySdlcGlobal(overrides)` helper (intro group + 5-row `arguments` array) and reuse a `makeFaqDoc` helper (`showOnWhySdlc: true`, `whySdlcOrder`, optional `whySdlcAnswer`). Assert:
  - `getWhySdlcContent()` calls `payload.findGlobal({ slug: 'whySdlc', depth: 0 })` and returns `{ intro, arguments }` with `arguments.length === 5`.
  - `getWhySdlcContent()` returns `null` (and logs) when `findGlobal` rejects.
  - `getWhySdlcFaqs()` calls `payload.find({ collection: 'faq', where: { showOnWhySdlc: { equals: true } }, sort: 'whySdlcOrder', limit: 2, depth: 0 })`.
  - `getWhySdlcFaqs()` prefers `whySdlcAnswer` over `answer` when present; falls back to `answer` when absent.
  - `getWhySdlcFaqs()` returns `[]` (and logs) when `find` rejects.

- [ ] **Step 2: Run tests, verify they fail** — `pnpm nx test marketing -- --testPathPattern=why-sdlc.spec` → FAIL (module not found).

- [ ] **Step 3: Implement `why-sdlc.ts`** — the two interfaces blocks above; `getWhySdlcContent()` wraps `findGlobal({ slug: 'whySdlc', depth: 0 })` in `try/catch`, maps the global to `{ intro, arguments }`, logs `['why-sdlc', error]` and returns `null` on throw; `getWhySdlcFaqs()` wraps `find(...)` with the exact query above, maps `docs` to `{ id, question, answer: doc.whySdlcAnswer ?? doc.answer }`, logs and returns `[]` on throw.

- [ ] **Step 4: Run tests, verify pass** — `pnpm nx test marketing -- --testPathPattern=why-sdlc.spec` → PASS.

**Interfaces:**

- [ ] **Step 1: Write failing tests** — provider mounts children; `useScrollProgress()` default is `{ reached: 0, active: 0 }`; under mocked `matchMedia('(prefers-reduced-motion: reduce)') === true`, `reached` initialises to `5` and no `scroll`/`resize` listeners are attached; listeners are removed on unmount.

- [ ] **Step 3: Implement** — context holding `{ reached, active }`. One passive `scroll`/`resize` effect scans the five `[data-why-sec]` nodes: a section is **passed** when its rect `top` crosses `0.7 × window.innerHeight` (last index sets `reached = 5`); `reached` is monotonic (only increases). **active** = highest-indexed section whose `top` crosses `0.5 × window.innerHeight`. Under `prefers-reduced-motion: reduce`, set `reached: 5` and skip listeners. Clean up listeners on unmount.
