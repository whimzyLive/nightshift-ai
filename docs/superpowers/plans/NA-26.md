# Enforce project-skill loading in domain-agent dispatches — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-26.md](../specs/NA-26.md)
**Story:** NA-26 — Enforce project-skill loading in domain-agent dispatches
**Epic:** NA-24 — SDLC Improvements
**Date:** 2026-07-11
**Agents required:** `ai-enablement-engineer` (only)

**Goal:** Add a contract-level enforcement loop so the dispatcher names applicable override skills in the prompt, the domain agent declares the skills it loaded in its return, and the orchestrator verifies the returned set against the named set — closing the NA-22 non-deterministic-skill-loading gap without reverting the NA-23 frontmatter workaround.

**Architecture:** Pure plugin-prose / contract change under `plugins/sdlc/**`. One shared return-format artifact (the `Skills loaded:` line) is defined once in `refs/domain-agent-handoff.md` and flows into all 6 domain-agent profiles that delegate their return to it. Two orchestration playbooks (principal Step 4/5, QA Step 3) gain the naming item and the mechanical verification. No code, data model, API, UI, or sync surface is touched.

**Tech Stack:** Markdown playbooks / agent profiles / refs; one JSON version field. No compiler or typecheck — verification is grep-based internal-consistency + cross-file contract-consistency self-checks.

## Global Constraints

- **Ownership:** every path below lives under `plugins/sdlc/**` — owned solely by `ai-enablement-engineer` per the project-context workspace→agent table. `plugins/**` is a published hosted artifact; edit in place, never delete.
- **Version bump:** `plugins/sdlc/.claude-plugin/plugin.json` `version` `0.29.0` → `0.30.0` (minor — new enforced behavior contract). This is the ONLY version-bearing occurrence in the plugin (verified: no other `0.29.0` string).
- **US spelling throughout** — "behavior", not "behaviour".
- **Line counts are status-dependent:** complete return = **exactly 3 lines** (Status / Summary / Skills loaded); blocked return = **exactly 4 lines** (Status / Note / Summary / Skills loaded). No prose may say "three lines" unconditionally.
- **`Skills loaded:` reports only runtime override (project) skills** — never the frontmatter-preloaded generic skills (`executing-plans`, `conventional-commit`, `test-driven-development`, `verification-before-completion`).
- **`none` semantics are mechanical, keyed off the dispatch prompt** — see spec § "`none` semantics". `Skills loaded: none` passes iff the prompt declared no applicable skills; if the prompt named skills, `none`/missing/empty/partial is a failure.
- **Two verification paths differ (do NOT claim they are consistent):** principal Step 5 = redispatch-once-then-STOP; QA Step 3 = blocked-immediately (no redispatch).
- **Line numbers below are content anchors captured at plan time** — re-locate by the quoted text, not by line number.
