## Goal

Land the Payload CMS content layer that serves the marketing site's editorial copy — a `faq`
collection (12 canonical questions) and a `whySdlc` global (page argument copy) — plus an idempotent
seed of the approved handoff copy and generated types. **No page rendering** — that is NA-35/36/38/39.

## Architecture

`apps/marketing/` is Next.js + Payload 3 on Neon Postgres — the entire data + API surface. New content
is a Payload **collection** (`faq`) and a Payload **global** (`whySdlc`); Payload owns table + type
generation. The seed lives inside the Next.js app and runs through the Local API (`getPayload`),
upserting by `seedKey` and overwriting the global — fully idempotent. Page stories read via the Local
API in RSCs; this story ships no renderers.

## Global Constraints

- **Web-only story, single owner: web-engineer** — all files under `apps/marketing/`. No DB, backend,
  mobile, or offline-sync workspace exists.
- **Never hand-write SQL DDL** — Payload manages the Postgres schema (table add for the new
  collection/global).
- **Never hand-edit `apps/marketing/src/payload-types.ts`** — it is generated by `payload generate:types`.
- **Copy is verbatim from the committed handoff** — do not re-author. Sources: full FAQ
  `docs/design/marketing-site-handoff/designs/faq.dc.html` (`GROUPS` array); Home overrides
  `docs/design/marketing-site-handoff/designs/nightshift Landing.dc.html` (FAQ section); Why-SDLC
  overrides + argument copy `docs/design/marketing-site-handoff/designs/why-sdlc.dc.html`.
- **Rich-text fields are lexical (Payload `richText`)**; seed rich-text values are lexical JSON.
- **UI/styling rules do not apply** — this story adds no `.tsx`/UI; it is config + data only.
- **Package/project name:** `@nightshift-ai/marketing` (nx project `marketing`).
- **Idempotency contract (AC ti-3):** a second seed run leaves exactly 12 `faq` docs
  (`payload.count({ collection: 'faq' }) === 12`) and never duplicates/corrupts the global.

## Execution order

Single sequential phase on the shared `feat/NA-31` branch — web-engineer only.

Ordering within the phase respects dependencies: schema (collection + global) → config registration →
type generation → migration → seed data → seed runner → package script → full verification.

## Phase 1 — Web (CMS content layer) [web-engineer]

- [ ] [web-engineer] Create `apps/marketing/src/collections/Faq.ts` — the `faq` collection config.
  - `slug: 'faq'`, `admin.useAsTitle: 'question'`, `admin.group: 'Content'`,
    `admin.defaultColumns: ['question','group','faqOrder','showOnHome','showOnWhySdlc']`.
  - `access.read: () => true`; `access.create`/`update`/`delete`: `({ req: { user } }) => Boolean(user)`.
  - Fields (exact names, types, required, defaults per spec Data Model table):
    - `seedKey` — text, required, `unique: true`, `admin.readOnly: true`.
    - `question` — text, required.
    - `group` — select, required. Options (value → label): `positioning` → `// positioning`,
      `workflow-control` → `// workflow & control`, `setup-stack` → `// setup & stack`,
      `trust-quality` → `// trust & quality`, `cost-license` → `// cost & license`.
    - `answer` — richText (lexical), required.
    - `faqOrder` — number, required.
    - `showOnHome` — checkbox, required, `defaultValue: false`.
    - `homeOrder` — number, optional; field-level `validate`: error if empty while `showOnHome` is true.
    - `homeAnswer` — richText (lexical), optional.
    - `showOnWhySdlc` — checkbox, required, `defaultValue: false`.
    - `whySdlcOrder` — number, optional; field-level `validate`: error if empty while `showOnWhySdlc` is true.
    - `whySdlcAnswer` — richText (lexical), optional.
- [ ] [web-engineer] Create `apps/marketing/src/globals/WhySdlc.ts` — the `whySdlc` global config.
  - `slug: 'whySdlc'`, `admin.group: 'Content'`.
  - `access.read: () => true`; `access.update: ({ req: { user } }) => Boolean(user)`.
  - Fields:
    - `intro` — group, required, with subfields: `eyebrow` (text, required), `heading` (text, required),
      `lead` (richText lexical, required), `scrollHint` (text, required).
    - `arguments` — array, required, `minRows: 5`, `maxRows: 5`, with subfields per row:
      `eyebrow` (text, required), `heading` (text, required), `body` (richText lexical, required).
- [ ] [web-engineer] Modify `apps/marketing/src/payload.config.ts` — import `Faq` and `WhySdlc`; add
      `Faq` to the `collections` array (leave `Users`, `Media` untouched); add `globals: [WhySdlc]`.
- [ ] [web-engineer] Regenerate types: run `payload generate:types` (via the app's payload runner) so
      `apps/marketing/src/payload-types.ts` gains `Faq`, `WhySdlc`, and the `FaqGroup` union — do NOT hand-edit.
- [ ] [web-engineer] Create/apply the Payload DB migration for the new `faq` + `whySdlc` tables
      (`payload migrate:create` then `payload migrate`, or dev push against the local Neon/`docker-compose`
      Postgres). No hand-written SQL DDL.
- [ ] [web-engineer] Create `apps/marketing/src/seed/data.ts` — typed approved copy, verbatim from handoff.
  - Export the 12 FAQ records keyed by `seedKey` (values, groups, questions, faqOrder, Home flag+order,
    Why-SDLC flag+order per the spec's Seed content mapping table), with lexical-JSON `answer` for all 12
    and lexical-JSON overrides only for the 3 divergent rows: `homeAnswer` on `workflow-full-ceremony`
    and `setup-repo-stack`; `whySdlcAnswer` on `workflow-slow-me-down`.
    - `showOnHome: true` rows (homeOrder): `positioning-wrapper` (1), `workflow-full-ceremony` (2),
      `setup-repo-stack` (3), `trust-agent-output` (4), `cost-what-does-it-cost` (5).
    - `showOnWhySdlc: true` rows (whySdlcOrder): `positioning-guardrail` (1), `workflow-slow-me-down` (2).
    - `faqOrder` 1–12 in the table's row order.
  - Export the `whySdlc` global payload: `intro` (eyebrow/heading/lead/scrollHint) + exactly 5
    `arguments` (`// 01…05` blocks), lexical JSON for `lead` and each `arguments[].body`, verbatim from
    `why-sdlc.dc.html`.
- [ ] [web-engineer] Create `apps/marketing/src/seed/seed.ts` — idempotent Local-API runner.
  - `const payload = await getPayload({ config })`.
  - For each FAQ record: `payload.find({ collection:'faq', where:{ seedKey:{ equals } }, limit:1 })`;
    if a doc exists → `payload.update({ collection:'faq', id, data })`, else
    `payload.create({ collection:'faq', data })`.
  - For the global: `payload.updateGlobal({ slug:'whySdlc', data })`.
- [ ] [web-engineer] Modify `apps/marketing/package.json` — add script
      `"seed": "payload run src/seed/seed.ts"` (runnable via `pnpm --filter @nightshift-ai/marketing run seed`).
- [ ] [web-engineer] Verify seed idempotency (AC ti-3): run the seed twice against a fresh dev DB; after
      the second run assert `payload.count({ collection:'faq' }) === 12` and the global has 5 `arguments` —
      no duplicates, no corruption.
- [ ] [web-engineer] Verify: `pnpm nx build marketing` (Next.js/Payload typecheck), `pnpm nx lint marketing`,
      and `pnpm nx test marketing` all pass. (`marketing` has no separate `typecheck` target — `build`
      type-checks; project-context lists no standalone typecheck command.)
