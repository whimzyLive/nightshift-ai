# AI Workflow Manager Agent — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-12.md](../specs/NA-12.md) (merged to `main`, 3 owner-review rounds — source of truth)
**Story:** NA-12 — AI workflow manager agent (no parent Epic)
**Date:** 2026-07-07
**Agents required:** `platform-engineer` (this repo's workspace→agent table assigns `plugins/` to `platform-engineer`; the reassignment of `plugins/` + `skills/` to the new `ai-enablement-engineer` is itself a deliverable of this story, applied at the very end — so every implementation task is tagged `[platform-engineer]`).

**Goal:** Add an SDLC domain agent `ai-enablement-engineer` to the `sdlc` plugin that (once a repo opts in at init) owns the repo's AI-config surface plus `plugins/**`/`skills/**`, exposes `/sdlc:analyze` for drift/gap/memory-conflict scanning, and applies fixes only as reviewable diffs after human confirmation; vendor `skill-creator` + `find-skills`; and apply the ownership reassignment to this repo's config.

**Architecture:** All deliverables are Markdown/plugin assets under `plugins/sdlc/**` plus this repo's `.claude/project/**` config. The agent mirrors the shape of `plugins/sdlc/agents/platform-engineer.md`; the command mirrors `plugins/sdlc/commands/spec.md`. A shared ref `plugins/sdlc/refs/analyze-protocol.md` holds the scan protocol consumed by both. Ownership is config-driven (read from the project-context workspace→agent table at runtime, never hard-coded). Two skills are vendored from `skills.sh` with pinned refs + license/attribution after vetting.

**Tech Stack:** Markdown (agents/commands/refs/skills), Shell (tooling). No database, web, mobile, or sync layers exist in this repo — those SDLC phases are intentionally skipped. Base branch: `main`. Work happens on the already-checked-out `feat/NA-12` branch.

## Global Constraints

- Agent definition shape mirrors `plugins/sdlc/agents/platform-engineer.md` verbatim structure (front-matter → path-resolution preamble → First steps → Role → Skills → Constraints → handoff/return).
- The `${CLAUDE_PLUGIN_ROOT}` path-resolution preamble is copied **verbatim** from the shipped sdlc agents (the blockquote in `platform-engineer.md` lines 13-16). Shared assets resolve same-plugin via `${CLAUDE_PLUGIN_ROOT}` — no cross-plugin dependency.
- Command shape mirrors `plugins/sdlc/commands/spec.md`.
- Apply flow: **only after explicit human confirmation, never auto-apply**; every write is a reviewable diff; scope guard **refuses and aborts** on any target path outside resolved write-scope (AC-5).
- Effective write-scope = (config-driven AI-config surface ∪ table-assigned areas) − read-only carve-outs. Never hard-code area paths — read them from the project-context workspace→agent table.
- Read-only carve-outs (never written by the agent): `.claude/project/project-context.md`, `.claude/.*-plugin-root` pointers, other agents' memory files (except the human-arbitrated memory-conflict reset via reviewable diff).
- Vendored skills: read full `SKILL.md` + any scripts before vendoring; no exfiltrating network calls / no RCE / no secret access; confirm upstream license permits vendoring; carry `LICENSE`/attribution alongside the vendored `SKILL.md`; pin version/commit; record license + upstream URL + pinned ref in `plugins/sdlc/README.md`.
- File naming: kebab-case. Any shell script touched must pass `bash tools/portability-lint.sh` (org-neutral portability gate).
- This repo has **no** typecheck/test command configured (`.claude/project/agents/platform-engineer.md` "Local dev"). Verification for Markdown deliverables is structural: front-matter validity, anchor/link resolution, and the AC-5 end-to-end scope check.
