# Atomic Design Skill — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-15.md](../specs/NA-15.md)
**Date:** 2026-07-10
**Agents required:** `ai-enablement-engineer` (sole owner of `skills/` and `plugins/`)

**Goal:** Author and publish a framework-agnostic `atomic-design` skill under the repo's public `skills/` tree, and register it in `plugins/sdlc/refs/skills-map.yml` so other repos can discover it.

**Architecture:** A workspace-authored, published-for-others skill (same class as `hono-api`/`typeorm`/`electrodb`). Ships with the decided progressive-disclosure `references/` split: `SKILL.md` (overview + workflow + heuristics) + `references/hierarchy.md` (expanded per-level detail) + `evals/evals.json` (declarative decomposition cases). Registration is a `skills-map.yml` entry with five frontend `when:` triggers. No consumption wiring (`.claude/project/skills.json`, agent overrides, `.claude/skills/` copy) — matches the published-for-others precedent.

**Tech Stack:** Markdown + YAML + JSON. No code, no runtime, no build step. Publication = git commit into the public repo.

## Execution order

This story is **entirely `ai-enablement-engineer` work** — it touches only `skills/` and `plugins/`, both owned by that agent. It is marked **DEPENDENCY-FREE** in the SDLC ladder: it consumes no artifact from, and produces no artifact for, any domain agent.

**No database-administrator, platform-engineer, sync-engineer, web-engineer, or mobile-engineer phases apply.** The spec's Artifact Classification marks Data Model, API Surface, Backend, Web UI, Mobile UI, Offline Sync, and Permissions all **N/A** (docs/skill artifact, no code). Those phases are intentionally omitted.

All tasks below run sequentially, one agent, on the shared `feat/NA-15` branch.

## Global Constraints

Copied verbatim from the spec — every task's requirements implicitly include these:

- **Placement:** skill directory is `skills/atomic-design/`; primary file `skills/atomic-design/SKILL.md`. Real committed directory — **never a symlink**.
- **Frontmatter:** exactly two keys, `name` and `description`. No other keys. `name` MUST equal the directory name `atomic-design`. `description` ≤ 1024 characters, follows the "Use when …" trigger convention.
- **Framework-agnostic (AC-4):** skill **content** (`SKILL.md` + `references/*`) must contain **zero prescriptive framework/library tokens** — no React, Vue, Angular, Svelte, Solid, React Native, Expo, SwiftUI, Flutter, Tailwind, CSS-in-JS; no JSX / `.vue` SFC / Swift / Kotlin code blocks; no "DOM", "route", "native module" as required concepts. "component" and "screen/view" allowed only as generic UI regions. Brad Frost / "atomic design" attribution is fine. Framework names in `skills-map.yml` `when:` triggers are registration metadata and **exempt** from this scan.
- **No source plagiarism:** content synthesised from Brad Frost's guide (ToC + ch. 1–5), **not** copied verbatim.
- **Published-for-others:** the skill is **absent** from `.claude/project/skills.json`, from every agent override, and from `.claude/skills/`. Do NOT add any of those.
- **Sizing:** keep `SKILL.md` scannable — roughly the size of `hono-api`/`typeorm` SKILL.md (~4 KB); push per-level depth into `references/hierarchy.md`.

- [ ] **Step 1: Write the frontmatter block**

Exactly two keys, at the top of the file:

```yaml
---
name: atomic-design
description: Use when structuring or decomposing any frontend UI into reusable components, regardless of framework. Applies Brad Frost's atomic design methodology — atoms, molecules, organisms, templates, and pages — to guide consistent component hierarchy, naming, and composition decisions.
---
```

Wording of `description` may be refined but MUST keep the meaning, name the trigger scenario + the five levels, stay framework-agnostic, and stay ≤ 1024 chars. `name` MUST be exactly `atomic-design`.

- [ ] **Step 2: Write the body sections, in this order**

Mirror the tone/structure of `skills/hono-api/SKILL.md` and `skills/typeorm/SKILL.md` (concise, scannable, `## Core principle`-style headings). Required coverage (exact headings are the author's call; content is fixed):

1. **Title + one-paragraph summary** — what atomic design is; that this skill is framework-agnostic. (AC-1)
2. **Why atomic design (rationale)** — the problem it solves: consistency, reuse, shared vocabulary, scalable UI systems. Synthesised from Frost ch. 1–2, not copied. (AC-3)
3. **The five levels (overview)** — one short paragraph per level (Atoms → Molecules → Organisms → Templates → Pages): definition + defining trait + a framework-neutral example. Keep this an overview and **link out** to `references/hierarchy.md` for the deep per-level detail (the decided `references/` split). (AC-2)
4. **How to apply it (decomposition workflow)** — an ordered, tool-agnostic procedure an AI/developer follows to decompose an arbitrary UI into the five levels. (AC-6)
5. **Decision heuristics / boundaries** — rules of thumb for classifying an element and promoting/demoting between levels (e.g. atom-vs-molecule threshold: ≥ 2 atoms combining for a single reusable purpose).
6. **Common pitfalls / anti-patterns** — e.g. over-nesting organisms, treating pages as templates, framework leakage.

Examples MUST be prose or framework-neutral pseudo-structure, e.g. "a _search-field_ molecule = _label_ atom + _text-input_ atom + _button_ atom". No JSX/SFC/Swift code blocks.

### Task 2: Author `references/hierarchy.md` (expanded per-level detail)

**Files:**

- Create: `skills/atomic-design/references/hierarchy.md`

**Interfaces:**

- Consumes: linked from `SKILL.md` Section 3 (relative link `references/hierarchy.md` added in Task 1 Step 3).

- [ ] **Step 1: Write one section per level, in order**

Five sections — **Atoms → Molecules → Organisms → Templates → Pages** — each containing, per AC-2: a framework-neutral **definition**, the **defining trait**, an **abstract (non-framework) example**, and the **"when does X become Y" boundary rule**. Use exactly this framing (synthesised, not quoted):

| Level     | Framework-neutral definition                                                                                    | Boundary rule to state                                                         |
| --------- | --------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------ |
| Atoms     | The smallest indivisible UI building blocks (a label, an input field, a single control).                        | An atom has no meaningful sub-parts; splitting it removes function.            |
| Molecules | A small group of atoms bonded to do one thing (a search field = label + input + button).                        | Becomes a molecule when ≥ 2 atoms combine for a single, reusable purpose.      |
| Organisms | Relatively complex groups of molecules and/or atoms forming a distinct section (a header, a product-card grid). | Becomes an organism when it forms a standalone, recognisable interface region. |
| Templates | Page-level layout skeletons placing organisms into a structure with placeholder content.                        | Concerned with structure/arrangement, not real content.                        |
| Pages     | Specific instances of templates with real representative content.                                               | The concrete, testable end state; used to validate the design system.          |
