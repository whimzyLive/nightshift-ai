# ET-68 — One-command project-context scaffolder (`/sdlc:init`)

**Triage:** lightweight (3 pts) · direct implementation, no spec gate.

## Goal

Add a `/sdlc:init` slash command to the `sdlc` plugin that interactively scaffolds everything a new
consuming repo needs to run nightshift SDLC: `.claude/project/project-context.md`, the per-agent
override files for the selected active agents, the `.claude/.sdlc-plugin-root` marker, and a
post-init checklist for the Jira custom fields that must be configured by hand. It also gates on the
`gh`/`acli` prerequisites and walks the user through `acli` authentication before writing anything.

## Acceptance criteria → design

1. **Interactive, one-at-a-time prompts → real `project-context.md`.** Collect project name, Jira
   project key, Jira site, base branch, package manager, typecheck/test commands, and the active
   domain-agent set, then write `.claude/project/project-context.md` populated with the supplied
   values. No placeholder tokens (`<...>`, `acme`, `TODO`) remain in the output.
2. **Scaffold override files for the active agent set only.** For each active domain agent the user
   selected, write `.claude/project/agents/<agent>.md` from the EXTENDING.md override contract.
   Pipeline-only agents (product-manager, scrum-master, …) get no override — they read
   `project-context.md` directly.
3. **Prerequisite gate.** Detect `gh` and `acli` via `command -v`. If either is missing, print the
   correct install command for the user's OS (macOS/Homebrew, Debian/apt, generic) and **exit
   without creating any files**.
4. **`acli` auth flow.** Probe `acli jira auth status`; if not authenticated, guide the user through
   `acli jira auth login` (site, email, API token) and re-verify before any file generation. An auth
   failure surfaces an actionable error and stops.
5. **Post-init checklist.** After writing files, print the Jira custom fields that must be configured
   (`Story point estimate`, `AI Workflow` single-select with `Full Auto`/`Auto`/`Assisted`) and the
   `acli` commands to verify them — so the project is usable end-to-end.

## Out of scope (per story)

- Creating/modifying Jira custom fields via API (instruct only).
- Idempotent re-init / migration of an already-initialised project.
- Auto-installing `gh`/`acli` (detect + instruct only).

## Files

- `plugins/sdlc/commands/init.md` — the command (model-executed prose: gate → auth → collect →
  write → checklist, in that order).
- `README.md` — add `/init` to the commands table; tick the roadmap item.

## Step order (enforced in the command)

`prereq gate (AC-3)` → `acli auth (AC-4)` → `collect values (AC-1)` → `write project-context +
overlays + plugin-root marker (AC-1, AC-2)` → `post-init checklist (AC-5)`. The gate runs first so a
missing CLI never leaves partial files; auth is verified before any write.
