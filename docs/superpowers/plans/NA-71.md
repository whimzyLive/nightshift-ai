# Render Marketing Site as Static Build Output — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Spec:** [docs/superpowers/specs/NA-71.md](../specs/NA-71.md)
**Story:** [NA-71](https://whimzylive.atlassian.net/browse/NA-71) — Render marketing site as static build output
**Date:** 2026-07-24
**Agents required:** web-engineer (only)

**Goal:** Convert the three DB-backed public marketing routes (Home `/`, `/faq`, `/why-sdlc`) from request-time `force-dynamic` rendering to build-time static HTML so the served site makes zero runtime database requests.

**Architecture:** This is a pure rendering-mode conversion — no new data model, API, UI, or offline sync. Removing each route's `force-dynamic` opt-out lets Next.js execute the existing Payload Local API reads once at `next build` and emit static HTML. A narrowing of the `lib/` helper catch blocks makes a real DB outage fail the build (rethrow connection/init errors) instead of silently shipping empty pages, while still swallowing isolated row-level defects.

**Tech Stack:** Next.js (App Router), Payload 3 Local API (`getPayload`), Neon Postgres (`postgresAdapter`), TypeScript, Nx, pnpm.

**Estimated complexity:** Low (1 phase, web-engineer only).

## Global Constraints

- **Verification bar (AC3):** `pnpm nx build @nightshift-ai/marketing` route table shows `/`, `/faq`, `/why-sdlc`, and `/team` marked static — `○` (Static) or `●` (SSG) — with **no `ƒ` (Dynamic)** against any public route.
- **Admin/API stay dynamic (AC4):** `/admin/*` and `/api/*` may remain `ƒ` in the same output — expected, not a regression. Do not touch `app/(payload)/*`.
- **No dynamic triggers (AC2):** post-edit grep of `app/(frontend)` must show zero `cookies()`, `headers()`, `searchParams`, `dynamic`, or `revalidate` usages.
- **Visual/markup identity (AC6):** rendered content and markup of all four pages must be unchanged vs pre-change — only the rendering mode differs. Existing component tests (e.g. `faq.spec.ts`) stay green.
- **Build reaches the DB:** the build completes with populated content, proving the build-time Local API read succeeded against a migrated, reachable Neon (`DATABASE_URL` + `PAYLOAD_SECRET` present at build time).
- **Outage fails the build:** with the DB deliberately unreachable, `next build` **fails fast** (a connection/init error rethrows) rather than producing an empty-but-successful build.
- **Comment hygiene (memory 15510):** remove the stale `force-dynamic` rationale comment cleanly — do NOT replace it with a new "// static now" comment.
- All work lands on the existing `feat/NA-71` branch (already checked out off `develop`). This plan doc ships in the same PR as the implementation — no separate plan branch or PR.

No database-administrator, platform-engineer, sync-engineer, or mobile-engineer work: the spec's Data Model, API Surface, Backend, Mobile, and Offline Sync sections are all "No changes". **Interfaces:**
- Consumes: nothing from other tasks. `getPayload({ config })` from `payload`, `config` from `@payload-config`.
- Produces: the four helpers keep their exact existing signatures/return types — `getHomeFaqs(): Promise<HomeFaqItem[]>`, `getFaqPageGroups(): Promise<FaqPageGroup[]>`, `getWhySdlcContent(): Promise<WhySdlcContent | null>`, `getWhySdlcFaqs(): Promise<WhySdlcFaqItem[]>`. New behaviour: a **connection/init failure rethrows**; a **row-level/data-shape defect** still returns the empty fallback (`[]` / `null`); a **zero-row query result** is not an error and returns the empty fallback.

**Contract (from spec — Web UI / Rendering + Error Handling):**
- **Connection / init failure** — `getPayload({ config })` throwing, or the query rejecting with a connection/authentication/DB-unreachable error — must **rethrow** so `next build` fails fast.
- **Row-level / data-shape defect** — a single malformed row or benign query issue that does not indicate the DB is down — may still be swallowed to the empty fallback.
- **Zero matching rows** (e.g. no `showOnHome` FAQs) — not an error; the query succeeds and the helper returns `[]`/`null`; the page renders its empty state.

**Mechanism (implementer's choice, spec leaves it open):** introduce one shared classifier in a small helper (e.g. `apps/marketing/src/lib/is-connection-error.ts`) used by all four catch blocks so the logic is DRY. Discriminate on the postgres adapter / `pg` error surface — connection/init errors carry a Postgres connection `code` (e.g. `ECONNREFUSED`, `ENOTFOUND`, `ETIMEDOUT`, or SQLSTATE class `08*` / `28*` for connection/authentication), and `getPayload` init failures (missing `PAYLOAD_SECRET`, adapter construction) surface before any query. Anything not classified as connection/init is treated as row-level and swallowed. If uncertain whether an error is connection-class, **prefer rethrow** (fail closed) — a false rethrow fails a build loudly, a false swallow silently ships an empty site.

- [ ] **Step 3: Add the shared classifier**

Create `apps/marketing/src/lib/is-connection-error.ts`:

```ts
/**
 * True when an error indicates the Payload client could not be constructed or
 * the database was unreachable/unauthenticated — i.e. a build-breaking outage,
 * not an isolated row defect. Connection-class errors carry a Node/pg network
 * `code` or a Postgres SQLSTATE in the connection (08*) / auth (28*) classes.
 */
export function isConnectionOrInitError(error: unknown): boolean {
  if (!(error instanceof Error)) return false;
  const code = (error as { code?: unknown }).code;
  if (typeof code === 'string') {
    if (['ECONNREFUSED', 'ENOTFOUND', 'ETIMEDOUT', 'ECONNRESET'].includes(code)) {
      return true;
    }
    // Postgres SQLSTATE: 08xxx = connection exception, 28xxx = invalid auth
    if (/^(08|28)/.test(code)) return true;
  }
  return false;
}
```
