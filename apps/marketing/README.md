# @nightshift-ai/marketing

Next.js 16 (App Router) marketing site with an embedded Payload CMS 3 admin,
backed by Postgres (Neon in the cloud). Site routes live in
`src/app/(frontend)/` (`/`, `/why-sdlc`, `/team`, `/faq`); the Payload
admin/API lives in `src/app/(payload)/` (generated — don't hand-edit).

Content: the `faq` collection + `whySdlc` global drive the FAQ and Why-SDLC
copy; `users` (admin auth, API keys enabled) and `media` (uploads) are the
built-ins.

## Requirements

Two env vars in `apps/marketing/.env` (copy `.env.example`):

| Var              | Value                                                         |
| ---------------- | ------------------------------------------------------------- |
| `DATABASE_URL`   | Postgres connection string (local docker DB or a Neon branch) |
| `PAYLOAD_SECRET` | Strong secret — generate with `openssl rand -hex 32`          |

```bash
cp .env.example .env      # then fill both values
```

## Local development

Run the nx targets below from the repo root (project `@nightshift-ai/marketing`,
alias `marketing`).

```bash
# 1. Start a local Postgres (docker compose)
pnpm nx local-start marketing

# 2. Apply the schema, then seed content (idempotent — safe to re-run)
pnpm nx run marketing:migrate
pnpm nx seed marketing

# 3. Dev server → http://localhost:3000
pnpm nx dev marketing

# When done
pnpm nx local-stop marketing
```

- Site pages: `/`, `/why-sdlc`, `/team`, `/faq`
- Payload admin: `/admin` · REST API: `/api` · GraphQL: `/api/graphql`

The seed loads the 12 canonical FAQs and the `whySdlc` global from the approved
handoff copy; re-running it leaves exactly 12 FAQ docs (no duplicates).

Prod-parity check locally:

```bash
pnpm nx build marketing && pnpm nx start marketing
```

## Deployment (Vercel)

No `vercel.json` — Vercel auto-detects Next.js.

1. **Import the repo** into Vercel. For this monorepo, set the project **Root
   Directory** to `apps/marketing`, or keep the repo root and set the build
   command to `pnpm nx build marketing`.
2. **Environment variables** (all environments):
   - `DATABASE_URL` → Neon **prod** branch, **pooled** connection string
     (serverless-safe).
   - `PAYLOAD_SECRET` → a strong secret, kept stable across deploys.
3. **Run migrations on deploy** — Payload does not auto-create tables in
   production. Apply migrations in the build step so the schema is ready before
   the Next build:
   ```
   Build command:  pnpm nx run marketing:migrate && pnpm nx build marketing
   ```
   (or run the migrate target via a Vercel deploy hook / one-off job).
4. **Seed prod once** — run `pnpm nx seed marketing` against the prod
   `DATABASE_URL` a single time (idempotent, so re-deploys are safe).

## Notes

- Schema is managed by **Payload migrations** (`src/migrations/`) — apply them
  with `pnpm nx run marketing:migrate`. To author a new one, run
  `payload migrate:create` from `apps/marketing/`. Do **not** hand-write SQL DDL
  and do **not** hand-edit `src/payload-types.ts` (generated by
  `payload generate:types`).
- Pages that read CMS content are `force-dynamic` and **fail-soft** (render
  empty on DB error), so a build never breaks if the DB is unreachable at build
  time — but pages stay empty until the DB is migrated + seeded.
- SEO metadata / JSON-LD use `metadataBase`; confirm it points at the real
  deploy domain before launch.
